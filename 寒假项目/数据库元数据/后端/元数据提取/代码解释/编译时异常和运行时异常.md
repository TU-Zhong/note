这是一个非常经典的 Java 面试题，也是写代码时每天都会遇到的概念。

为了让你彻底搞懂，我们先看一张 Java 异常体系的家族图谱：

在 Java 中，所有的异常都继承自 `Throwable` 类。我们今天只关注 `Exception` 分支下的两大家族：**编译时异常**和**运行时异常**。

我用一个**“坐飞机”**的例子来比喻，保证你秒懂。

---

### 1. 编译时异常 (Compile-time / Checked Exception)

别名：受检异常 (Checked Exception)

特点：“安检必须过”

这就好比你去坐飞机，**安检（编译器）** 这一关是必须要过的。

- 安检员（编译器）会强制检查你有没有带身份证（处理异常）。
    
- 如果你没带，安检员直接把你拦在登机口，**不让你上飞机（代码编译不通过，红线报错，无法运行）**。
    

#### 为什么要有它？

因为有些事情是程序员无法控制的。

比如：你写代码去读取一个文件，但是那个文件可能被用户删了；或者你去连数据库，网线可能被拔了。

Java 觉得这些风险太高，所以强制你在写代码的时候就必须想好：“万一出事了怎么办？”

#### 常见的例子（正好是你毕设里用到的）：

- **`ClassNotFoundException`**: 你用 `Class.forName("com.xxx.Driver")` 加载驱动时。Java 会问：“万一用户没传这个 Jar 包咋办？你必须处理！”
    
- **`SQLException`**: 你调用 `driver.connect()` 时。Java 会问：“万一数据库密码改了连不上咋办？你必须处理！”
    
- **`IOException`**: 文件读写操作。
    

#### 怎么处理？

必须二选一，否则代码无法运行：

1. **自己扛 (`try-catch`)**：我在现场就解决这个问题。
    
2. **甩锅 (`throws`)**：我解决不了，抛给调用我的人去解决（你之前的 Controller 代码里把异常抛给 Spring 框架处理就是这种）。
    

---

### 2. 运行时异常 (Runtime / Unchecked Exception)

别名：非受检异常 (Unchecked Exception)

特点：“走路摔跟头”

这就好比你在大马路上走。

- 交警（编译器）**不会**强制检查你的鞋带散没散。
    
- 你可以正常上路走（代码编译通过，没有红线）。
    
- 但是！如果你走路不看路，或者鞋带散了，走着走着就**摔倒了（程序运行中突然崩溃报错）**。
    

#### 为什么要有它？

这类异常通常是因为程序员逻辑写得烂或者不严谨造成的。

比如：你访问了空对象，或者数组越界了。

Java 觉得：“这种低级错误你应该自己避免，我不每行代码都盯着你了，否则代码里全是 try-catch 没法看了。”

#### 常见的例子：

- **`NullPointerException` (NPE)**: 空指针异常。你毕设里如果 `conn` 没连上是 null，你还去调 `conn.getMetaData()`，就会报这个。
    
- **`ArithmeticException`**: 算术异常。比如 `int a = 10 / 0;`。
    
- **`ArrayIndexOutOfBoundsException`**: 数组越界。数组长度是3，你非要取第4个。
    
- **`ClassCastException`**: 类型转换错误。把“猫”强行转成“狗”。
    

#### 怎么处理？

- 一般**不**用 `try-catch` 来处理（虽然可以）。
    
- **正确的做法是改代码逻辑**：比如在调用前加一个 `if (obj != null)` 的判断。
    

---

### 3. 一张表总结对比

|**特性**|**编译时异常 (Checked)**|**运行时异常 (Runtime)**|
|---|---|---|
|**继承关系**|继承自 `Exception` 但不是 `RuntimeException`|继承自 `RuntimeException`|
|**发生时间**|**写代码时** (编译阶段)|**程序跑起来后** (运行阶段)|
|**编译器反应**|**爆红！** 必须处理 (try/throws)，否则不让运行|**沉默**。编译能通过|
|**产生原因**|外部环境因素 (文件不存在、网络断了)|**程序员逻辑错误** (空指针、除零)|
|**处理策略**|必须捕获或抛出|通常修改代码逻辑避免发生|

---

### 4. 结合你的毕设项目

来看看你写的代码，你就知道区别了：

**场景 A：连接数据库 (编译时异常)**

Java

```
// 这是必须处理的！因为 Java 怕你连不上数据库
public Connection getConn() throws Exception { // 👈 必须 throws 或者 try-catch
    // Class.forName 可能会抛出 ClassNotFoundException
    Class.forName("com.microsoft.sqlserver.jdbc.SQLServerDriver");
    
    // driver.connect 可能会抛出 SQLException
    return driver.connect(url, props); 
}
```

**场景 B：从 Map 取连接 (运行时异常)**

Java

```
// 前端传了个错误的 token
String token = "wrong-token";
Connection conn = connectionCache.getConnection(token); // 此时 conn 是 null

// 下面这行代码编译时不会报错，看起来很完美
// 但是运行的时候，如果 conn 是 null，直接炸了 -> NullPointerException
conn.getMetaData(); 
```

### 💡 总结

- **编译时异常**是 Java 逼你写的“应急预案”。
    
- **运行时异常**是 Java 信任你，但你如果不小心写出的“逻辑Bug”。
    

如果你在写代码时看到 IntelliJ IDEA 画了**红色波浪线**，那就是**编译时异常**。如果代码跑着跑着突然控制台**喷出一大堆红字**，那通常是**运行时异常**。
