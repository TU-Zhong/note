### Day 7: 基础信息提取 (Table & View)
[[Result更改统一返回]]
你的逻辑是 `metaData.getTables(...)`，这非常正确。但是这个方法有 4 个参数，如果你传不对，可能会查出来几千张系统表（System Tables），把你的前端卡死。

#### 🛠️ 避坑指南

1. 参数过滤（过滤系统表）：
    
    SQL Server 里有很多系统表（如 sys.objects）。你只想给用户看用户表。
    
    Java
    [[改写，添加 连接缓存，防止连接丢失]]
    ```
    // 这里的 types 数组是关键，只查 表 和 视图
    String[] types = {"TABLE", "VIEW"};
    
    // 参数说明：(Catalog/数据库名, Schema/模式名, TableNamePattern/表名匹配, Types/类型)
    // 这里的 "%" 代表匹配所有表名
    ResultSet rs = metaData.getTables(null, "dbo", "%", types); 
    ```
    
    - **注意**：SQL Server 的默认 Schema 通常是 `dbo`。如果你传 `null`，可能会把其他用户的表也查出来（取决于权限）。建议允许前端传 schema，如果前端没传，默认用 `null` 或 `dbo`。
        
2. 数据封装 (DTO)：
    
    不要直接返回 ResultSet，要在后端封装成一个 List 对象返回给前端。
    
    Java
    
    ```
    // 建议定义的简单类
    public class TableVO {
        private String name;    // 表名
        private String type;    // TABLE 或 VIEW
        private String remarks; // 注释 (SQL Server 这个字段有时需要特殊配置才能读到)
    }
    ```

### Day 8: 字段与细节提取 (Columns)

你的难点识别很准：**数据类型映射**确实是个麻烦事。

#### 🛠️ 避坑指南

1. 关于数据类型映射：
    
    你提到要处理 datetime2 -> Java 类型的映射。
    
    - **如果是为了“展示”**：其实你不需要手动映射。`metaData.getColumns()` 返回的结果里，有一个字段叫 **`TYPE_NAME`**，它会直接告诉你数据库里的原生类型名称（例如 "nvarchar", "datetime2"）。直接把这个字符串展示给前端即可，用户就是想看这个。
        
    - **如果是为了“生成代码”**：那你才需要去匹配 `DATA_TYPE` (int) 到 Java 类型的映射。做“浏览器”的话，直接用 `TYPE_NAME` 最省事。

### ⚠️ 一个必须要解决的架构问题：连接管理

这可能是你 Day 7/8 面临的**最大技术挑战**。

- **问题**：HTTP 是无状态的。
    
    1. 用户调用 `/connect` 接口，你创建了一个 `Connection`。接口返回，这个 `Connection` 如果没保存，就被垃圾回收或关闭了。
        
    2. 用户紧接着调用 `/getTables`，服务器会问：“你是谁？刚才那个连接去哪了？”
        
- 解决方案（Session 缓存大法）：
    
    你需要在一个全局的地方（比如 Map<String, Connection>）暂存这个连接。
    
    - **步骤 1**：`/connect` 成功后，生成一个 UUID (比如 `conn-123`) 返回给前端，并把 `Connection` 对象存入 Map：`connectionCache.put("conn-123", connection)`。
        
    - **步骤 2**：前端调用 `/getTables` 时，必须在 Header 或参数里带上这个 `conn-123`。
        
    - **步骤 3**：后端用 `conn-123` 从 Map 里取出活着的 Connection，然后调用 `getMetaData()`。
