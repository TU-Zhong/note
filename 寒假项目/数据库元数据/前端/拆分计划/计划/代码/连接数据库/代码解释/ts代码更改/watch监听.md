这两段代码共同构成了 Vue 3 中**组件双向绑定**的“传声筒”。

因为在 Vue 里，数据流动是**单向**的（父传子），但弹窗的显示状态需要**双向同步**。这两段 `watch` 就是为了解决“父组件以为开着，子组件其实关了”这种不同步的情况。

---

### 第一段：接听指令（外部 $\rightarrow$ 内部）

JavaScript

```
watch(() => props.modelValue, (newVal) => {
  visible.value = newVal
})
```

- **谁在盯着谁？** 子组件在盯着父组件传进来的 `modelValue`。
    
- **人话解释**：当父组件在外面把变量（比如 `showDialog`）改成 `true` 时，子组件通过 `props` 感知到了。
    
- **动作**：子组件立刻把自己的内部变量 `visible` 也改成 `true`。
    
- **为什么要这么写？** * 因为 `visible` 是控制 `el-dialog` 显示的直接开关。
    
    - 如果父组件想让弹窗跳出来，它只能改自己的变量。子组件必须“听话”，发现父组件变了，自己也得跟着变。
        

---

### 第二段：汇报工作（内部 $\rightarrow$ 外部）

JavaScript

```
watch(visible, (newVal) => {
  emit('update:modelValue', newVal)
})
```

- **谁在盯着谁？** 子组件在盯着自己内部的 `visible` 变量。
    
- **人话解释**：当用户点击了弹窗右上角的 **“X”** 或者点击了 **“取消”** 按钮时，Element Plus 会自动把子组件内部的 `visible` 改成 `false`。
    
- **动作**：一旦 `visible` 变了，子组件立刻发射一个信号（`emit`）给父组件：“报告老板，我这边已经关门了，请把你的变量也改成 `false`！”
    
- **为什么要这么写？** * 如果你不写这段，用户点“X”关闭了弹窗，子组件是关了，但父组件里的变量还是 `true`。
    
    - 当你下次再点打开按钮时，父组件一看：“变量本来就是 `true` 啊，没变化”，就不会再触发第一段 `watch`，**导致弹窗再也打不开了**。
        

---

### 为什么写法不一样？（一个有 `() =>`，一个没有）

这是一个非常细微但专业的软件工程细节：

1. **`() => props.modelValue`**：
    
    `props` 是一个**响应式对象**，而不是一个具体的 `ref`。在 Vue 3 的 `watch` 里，如果你想监听对象里的某一个属性，必须写成 **getter 函数**（即那个箭头函数）的形式，否则 `watch` 抓不到它的变化。
    
2. **`visible`**：
    
    因为它本身就是一个 `ref`（`const visible = ref(false)`）。`watch` 可以直接监听 `ref` 对象，不需要加箭头函数。
    

---

### 总结：双向奔赴

这两段代码合起来，就像是在父子组件之间建了一座**实时同步的桥梁**：

- **桥 A（第一段）**：保证了父组件点按钮时，子组件能跳出来。
    
- **桥 B（第二段）**：保证了子组件关掉时，父组件能“收到信儿”，把状态重置。
