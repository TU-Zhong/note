这个命名绝对是有说法的！使用 `__row_id`（带有**双下划线前缀**）是前端开发和软件工程中一个非常经典且专业的**防御性编程**习惯。

你可以把它理解为给数据打上的一个“隐形水印”。这么命名主要有三大核心原因：

### 1. 极度防冲突（防止污染真实数据）

试想一下，如果你直接把这个虚拟 ID 命名为 `id` 或者 `row_id`，会发生什么？

万一用户数据库里的表（比如一张订单表）刚好也有一个列叫做 `row_id`，那你前端强行塞进去的这个虚拟 ID，就会**瞬间覆盖、破坏掉用户原本的真实数据**！

加上双下划线 `__` 后，就极大地降低了与真实数据库列名撞车的概率。

### 2. 业界通用的“元数据 / 私有属性”标识

在编程界（不仅仅是 JS，还包括 Python 的 `__init__`，Vue 源码底层的 `__v_isRef` 等），**双下划线（Dunder）** 是一种强烈的视觉暗示。

它在无声地告诉接手这段代码的其他程序员：

> “嘿！这是一个**系统内部使用**的元数据（Meta Data）。它是前端为了实现某种交互（比如追踪脏检查）自己悄悄加进去的，**它不是真实的业务数据，不要把它渲染到页面上，也不要随便修改它！**”

### 3. 方便在提交给后端前“一键剔除”

这是最实用的一个原因！

当用户在页面上修改完数据，点击“提交”按钮时，你需要把数据发给 Spring Boot 后端。但后端数据库里根本没有 `__row_id` 这个字段，发过去肯定会报 SQL 语法错误。

因为有了统一的 `__` 前缀，你在发送请求前，只需写一行极其优雅的代码，就能把所有内部标识摘得干干净净：

JavaScript

```
// 提交前清洗数据：过滤掉所有以 __ 开头的属性
const cleanData = {};
Object.keys(dirtyRow).forEach(key => {
  if (!key.startsWith('__')) {
    cleanData[key] = dirtyRow[key];
  }
});
```

`__row_id` 就像是咱们在前端搭建的一座“脚手架”，它帮我们稳稳地追踪增删改状态。等大楼建好（准备提交给后端）时，因为它的名字很特殊，我们可以一眼认出并拆除它，不留痕迹。
