JVM 的内存区域（也称为**运行时数据区**，Runtime Data Areas）是 Java 虚拟机在执行 Java 程序的过程中，将其所管理的内存划分成的若干个不同数据区域。

理解这些区域对于排查内存溢出（OOM）、性能调优以及理解多线程至关重要。

根据《Java 虚拟机规范》，JVM 内存区域主要分为两大类：**线程私有区域**和**线程共享区域**。

---

### 一、 线程私有区域 (Thread-Private)

这些区域的生命周期与线程相同（随线程创建而创建，随线程结束而销毁），因此**不需要进行垃圾回收（GC）**，也不存在线程安全问题。

#### 1. 程序计数器 (Program Counter Register)

- **作用：** 是一块较小的内存空间，可以看作是当前线程所执行的字节码的**行号指示器**。分支、循环、跳转、异常处理等基础功能都依赖它。
    
- **特点：**
    
    - 如果线程正在执行的是 Java 方法，计数器记录的是正在执行的虚拟机字节码指令的地址。
        
    - 如果执行的是 Native 方法，计数器值为空（Undefined）。
        
    - **它是唯一一个在 JVM 规范中没有规定任何 `OutOfMemoryError` 情况的区域。**
        

#### 2. Java 虚拟机栈 (Java Virtual Machine Stack)

- **作用：** 描述 Java 方法执行的内存模型。每个方法在执行时都会创建一个**栈帧 (Stack Frame)**。
    
- **栈帧包含的内容：**
    
    - **局部变量表：** 存放基本数据类型（int, boolean 等）和对象引用。
        
    - **操作数栈：** 也就是执行引擎计算数据的“草稿纸”。
        
    - **动态链接：** 指向常量池的方法引用。
        
    - **方法出口：** 方法返回的地址。
        
- **异常：**
    
    - `StackOverflowError`：线程请求的栈深度大于虚拟机所允许的深度（例如无限递归）。
        
    - `OutOfMemoryError`：如果栈可以动态扩展，但无法申请到足够的内存。
        

#### 3. 本地方法栈 (Native Method Stack)

- **作用：** 与虚拟机栈发挥的作用非常相似，区别在于虚拟机栈为 Java 方法服务，而本地方法栈则为虚拟机使用到的 **Native 方法**（通常由 C/C++ 编写）服务。
    
- **注：** 在 HotSpot 虚拟机中，本地方法栈和虚拟机栈是合二为一的。
    

---

### 二、 线程共享区域 (Thread-Shared)

这些区域在虚拟机启动时创建，被所有线程共享。这里是**垃圾回收（GC）的重点区域**。

#### 1. Java 堆 (Java Heap)

- **作用：** JVM 中内存最大的一块。**几乎所有的对象实例和数组**都在这里分配内存。
    
- **结构（分代模型）：** 虽然现代 GC（如 ZGC）可能不再分代，但经典模型通常将堆分为：
    
    - **新生代 (Young Generation)：** 包含 Eden 区和两个 Survivor 区（S0, S1）。新对象通常在此诞生。
        
    - **老年代 (Old Generation)：** 存放长期存活的对象。
        
- **异常：** 如果堆中没有内存完成实例分配，且堆也无法再扩展时，抛出 `OutOfMemoryError` (Java heap space)。
    

#### 2. 方法区 (Method Area)

- **作用：** 用于存储已被虚拟机加载的**类信息（Class）、常量、静态变量 (static)、即时编译器编译后的代码缓存**等数据。
    
- **演变（面试重点）：** “方法区”是规范中的逻辑概念，不同 JDK 版本的**实现**不同：
    
    - **JDK 7 及之前：** 实现为**永久代 (PermGen)**。它使用 JVM 内存，容易出现 OOM。
        
    - **JDK 8 及之后：** 永久代被移除，替换为**元空间 (Metaspace)**。
        
- **元空间的特点：** 它主要使用**本地内存 (Native Memory)**，而不属于虚拟机内存。这意味着它受限于物理内存大小，极大地降低了 OOM 的风险。
    

#### 3. 运行时常量池 (Runtime Constant Pool)

- 是方法区的一部分。Class 文件中除了有类的版本、字段、方法等描述信息外，还有一项是常量池表，用于存放编译期生成的==各种字面量和符号引用==。这部分内容在类加载后进入运行时常量池。
    

---

### 三、 特殊区域：直接内存 (Direct Memory)

- **说明：** 它**不是**虚拟机运行时数据区的一部分，也不是 JVM 规范中定义的内存区域。
    
- **作用：** 在 JDK 1.4 中新加入了 NIO (New Input/Output) 类，引入了一种基于通道 (Channel) 与缓冲区 (Buffer) 的 I/O 方式。它可以使用 Native 函数库直接分配**堆外内存**，然后通过一个存储在 Java 堆中的 `DirectByteBuffer` 对象作为这块内存的引用进行操作。
    
- **优势：** 避免了在 Java 堆和 Native 堆之间来回复制数据，在某些场景（如网络传输、大文件读写）下性能显著提高。
    
- **异常：** 虽然不受 Java 堆大小限制，但受本机总内存限制，也会导致 OOM。
    

---

### 总结与对比

|**内存区域**|**所属**|**是否垃圾回收**|**常见异常**|**核心存储内容**|
|---|---|---|---|---|
|**程序计数器**|线程私有|否|无|当前指令地址|
|**虚拟机栈**|线程私有|否|StackOverflow / OOM|局部变量、操作数栈、方法出口|
|**本地方法栈**|线程私有|否|StackOverflow / OOM|Native 方法相关数据|
|**Java 堆**|**线程共享**|**是 (主要)**|**OOM**|**对象实例、数组**|
|**方法区 (元空间)**|**线程共享**|是 (较少)|OOM (Metaspace)|类信息、常量、静态变量|
