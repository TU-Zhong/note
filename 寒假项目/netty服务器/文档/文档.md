# 🛡️ Project Aegis (神盾) 架构实施手册 v1.0

项目目标：构建一个高吞吐、低延迟、可水平扩展的 TCP 长连接网关。

技术底座：Java 17/21 + Spring Boot 3.x + Netty 4.1 + Redis 7 + Docker。

核心原则：

1. **IO 与业务分离**：绝不阻塞 Netty IO 线程。
    
2. **防御性编程**：假设客户端是恶意的（防爆、防攻击）。
    
3. **无状态网关**：网关本身只持连接，状态下沉至 Redis。

## 🏗️ Phase 0: 顶层设计规范 (Design Specs)

在写一行代码前，必须锁死以下规范，严禁随意更改。

### 0.1 通信协议规范 (Binary Protocol)

放弃文本协议，采用紧凑的二进制协议。所有多字节数据必须遵循 **Big Endian (网络字节序)**。

| **字段名**     | **长度 (Byte)** | **类型**  | **说明**                               |
| ----------- | ------------- | ------- | ------------------------------------ |
| **Magic**   | 2             | `short` | 固定值 `0xCAFE`，用于快速丢弃非法包               |
| **Version** | 1             | `byte`  | 协议版本，当前为 `0x01`                      |
| **Type**    | 1             | `byte`  | 指令类型 (1=Req, 2=Resp, 3=Ping, 4=Pong) |
| **Cmd**     | 2             | `short` | 业务指令码 (如 1001=Login)                 |
| **Length**  | 4             | `int`   | **Body 的长度** (不包含头部)                 |
| **Body**    | N             | `bytes` | 具体的业务数据 (Protobuf/JSON bytes)        |

- **防御规范**：单包最大限制 (`MAX_FRAME_LENGTH`) 设为 **64KB**。超过此长度的包直接断开连接，防止内存炸弹。
    

### 0.2 线程模型规范 (Threading Model)

这是系统不崩溃的关键。

- **Boss Group (1)**: 仅负责 TCP 三次握手。
    
- **Worker Group (CPU*2)**: 仅负责 IO 读写、编解码、快速的心跳检测。**严禁查库、严禁复杂计算**。
    
- **Business Group (CPU_4 ~ CPU_16)**: 独立的 `ThreadPoolExecutor`，负责具体的业务逻辑（登录、转发）。
    

### 0.3 工程目录结构规范

Plaintext

```
com.yourname.aegis
├── core
│   ├── netty           // ServerBootstrap, LifeCycle
│   ├── codec           // Encoder, Decoder, FrameCheck
│   └── session         // SessionManager (Local & Remote)
├── protocol            // Packet definitions
├── logic               // Business Logic (Service)
│   └── executor        // **Business Thread Pool**
├── infra               // Infrastructure
│   ├── config          // Spring Config
│   └── utils           // HexUtil, ThreadUtil
└── AegisApplication.java
```

## ⚡ Phase 2: 核心通信层实现 (The "Reactor")

### 2.1 启动类与优雅停机

在 `NettyServer` 类中：

- **配置**：开启 `SO_KEEPALIVE`，开启 `TCP_NODELAY` (禁用 Nagle 算法，降低延迟)。
    
- **Backlog**：`SO_BACKLOG` 设置为 1024 (处理瞬时大量连接)。
    
- **Shutdown Hook**：利用 Spring 的 `@PreDestroy`，必须调用 `bossGroup.shutdownGracefully()`，确保数据发完再死。
    

### 2.2 编解码器链 (The Pipeline) - **高危环节**

Pipeline 的顺序决定了生死。请严格按照以下顺序添加到 `ChannelPipeline`：

1. **`IdleStateHandler`**: (Read=60s, Write=0, All=0)。60秒没读到数据，触发空闲事件。
    
2. **`GlobalTrafficShapingHandler`** (可选): 限制全局带宽，防止被把网卡打满。
    
3. **`LengthFieldBasedFrameDecoder`**: **核心防粘包组件**。
    
    - `maxFrameLength` = 65536
        
    - `lengthFieldOffset` = 4 (Magic+Ver+Type+Cmd 之后?) **NO!** 仔细看 0.1 的协议，Magic(2)+Ver(1)+Type(1)+Cmd(2) = 6 bytes。如果 Length 在第 4 个位置，那表格里的顺序要调整，或者 offset 要算对。
        
    - _修正建议_：把 `Length` 放在最前面或者紧跟 Magic 之后最容易处理。这里我们严格按 0.1 表格：`Offset` 应为 6 (2+1+1+2)。
        
4. **`AegisProtocolDecoder`**: 自定义解码器。
    
    - **检查 Magic**：如果是 `0xCAFE` -> 继续；否则 -> `ctx.close()` (直接踢掉，不解释)。
        
    - **读取 Body**：根据 Length 读取字节数组。
        
5. **`AegisProtocolEncoder`**: 编码器，将 Java 对象转为二进制流。

## 🧠 Phase 3: 业务与线程隔离 (The "Brain")

### 3.1 线程隔离实现

这是区分“玩具”与“产品”的分水岭。

Java

```
@Component
public class BusinessThreadPool {
    private final ExecutorService executor;

    public BusinessThreadPool() {
        // 使用有界队列，防止 OOM
        this.executor = new ThreadPoolExecutor(
            8, 16, 60L, TimeUnit.SECONDS,
            new ArrayBlockingQueue<>(5000),
            new ThreadFactoryBuilder().setNameFormat("logic-pool-%d").build(),
            new ThreadPoolExecutor.CallerRunsPolicy() // 满载后的拒绝策略
        );
    }

    public void submit(Runnable task) {
        executor.submit(task);
    }
}
```

### 3.2 改造 Handler (`GameHandler`)

Java

```
@Component
@ChannelHandler.Sharable
public class GameHandler extends SimpleChannelInboundHandler<AegisPacket> {

    @Autowired private BusinessThreadPool businessThreadPool;
    @Autowired private LoginService loginService;

    @Override
    protected void channelRead0(ChannelHandlerContext ctx, AegisPacket msg) {
        // IO 线程只负责“收”，立刻转交给业务线程去“办”
        businessThreadPool.submit(() -> {
            try {
                // 1. 设置 MDC (日志追踪)
                MDC.put("sessionId", ctx.channel().id().asShortText());
                
                // 2. 路由分发
                if (msg.getCmd() == 1001) {
                    loginService.handle(ctx, msg);
                }
                // ... 其他逻辑
            } finally {
                MDC.clear();
            }
        });
    }
}
```

## 💾 Phase 4: 分布式状态管理 (Redis)

### 4.1 会话管理 (Session Manager)

- **LocalSession**: `ConcurrentHashMap<String, Channel>`。用于：收到 MQ 消息后，判断用户是否在当前机器，如果在，直接 `channel.writeAndFlush`。
    
- **RemoteSession**: Redis Key: `user:location:{uid}` -> Value: `192.168.1.5:8888`。
    
- **登录逻辑**:
    
    1. 校验 Token。
        
    2. 保存 `Channel` 到本地 Map。
        
    3. 写入 Redis，设置 TTL = 5分钟。
        
    4. 开启 Netty 侧的心跳响应。
        

### 4.2 心跳保活 (Keepalive)

- **客户端**：每 30 秒发一个 Ping 包。
    
- **服务端**：
    
    - 收到 Ping -> 回复 Pong -> **Redis 续期 (Expire)**。
        
    - 触发 `userEventTriggered` (IdleStateEvent) -> 说明 60s 没收到 Ping -> **关闭连接** -> 删除 Redis Key。

## 🛡️ Phase 5: 安全与可观测性 (Defense & Observe)

### 5.1 握手检查 (The Gatekeeper)

在 `channelActive` 中：

Java

```
@Override
public void channelActive(ChannelHandlerContext ctx) {
    // 启动一个定时炸弹
    ctx.executor().schedule(() -> {
        if (!SessionManager.isLogin(ctx.channel())) {
            // 3秒内未登录，直接断开
            ctx.close(); 
        }
    }, 3, TimeUnit.SECONDS);
}
```

### 5.2 日志追踪 (Traceability)

利用 MDC (Mapped Diagnostic Context)。

所有业务日志必须带上 [SessionID] 和 [UserID]。这样在 Docker logs 里，你才能通过 grep 还原某个用户的完整操作路径。

## 🧪 Phase 6: 验证与验收标准

### 6.1 编写压测客户端 (Headless Client)

不要用 Telnet 了，写一个纯 Java Client 程序：

1. 启动 500 个线程。
    
2. 每个线程建立连接，发送 Login 包。
    
3. 随机休眠 1-5秒，发送 Ping 包。
    
4. **混沌测试**：在程序运行时，手动 `docker restart aegis-redis`，观察你的网关是否报错，是否能在 Redis 回来后自动恢复服务。
    

### 6.2 验收清单 (Checklist)

1. [ ] **端口检查**：`netstat -an | grep 8888`，确认 Netty 监听。
    
2. [ ] **粘包测试**：客户端一次性发送 100 个拼在一起的小包，服务端必须触发 100 次业务逻辑，不能合并，也不能丢包。
    
3. [ ] **OOM 测试**：发送一个声明长度为 1GB 但实际只有 1KB 的包，服务端必须立刻断开，且内存无波动。
    
4. [ ] **空闲踢人**：客户端连上后不发任何包，60秒后必须被服务端强制断开。
