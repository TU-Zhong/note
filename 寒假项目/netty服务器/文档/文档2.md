# 🛡️ Project Aegis (神盾) 进阶实施手册 v1.1

**当前阶段目标**：将一个简单的“回声玩具”升级为具备**自我修复能力**（断线重连）和**业务路由能力**（群聊/私聊）的生产级 IM 系统核心。

**核心原则**：

1. **连接保活**：默认为网络是不稳定的，必须有心跳机制。
    
2. **容灾恢复**：客户端断开后必须能自动重连。
    
3. **会话管理**：服务器必须知道每个连接背后是谁（Session 映射）。

## 🏗️ Phase 0: 协议规范升级 (Protocol Upgrade)

为了支持心跳和不同类型的消息，我们需要明确定义 `Type` 字段。

### 0.1 协议常量定义

在 `AegisPacket` 或常量类中锁死以下定义：

|**字段**|**值**|**含义**|**处理逻辑**|
|---|---|---|---|
|**Magic**|`0xCAFE`|魔数|校验失败直接断开|
|**Type**|`1`|**业务请求 (Request)**|交给业务 Handler 处理 (如登录、聊天)|
|**Type**|`2`|**业务响应 (Response)**|客户端收到后展示给用户|
|**Type**|`3`|**PING (心跳请求)**|客户端发，服务端收|
|**Type**|`4`|**PONG (心跳响应)**|服务端回，客户端收|

## 💓 Phase 3: 连接健壮性建设 (Infrastructure)

这是你简历中**“熟悉 Netty 心跳机制与断线重连”**这一行的来源。

### 3.1 服务端：空闲检测 (Idle Detection)

**目标**：踢掉那些“占着茅坑不拉屎”的死连接。

- **修改 Pipeline**：在 `LengthFieldBasedFrameDecoder` 之前添加 Netty 原生的 `IdleStateHandler`。
    
- **参数设置**：ReaderIdleTime = 30秒 (30秒没读到数据就触发事件)。
    
- **事件处理**：在 `AegisProtocolDecoder` 或新的 Handler 中捕获 `userEventTriggered`。
    
    - 如果收到 `IdleStateEvent` -> 打印日志 -> `ctx.close()`。
        

### 3.2 客户端：定时心跳 (Heartbeat)

**目标**：告诉服务器“我还活着，别踢我”。

- **修改 Pipeline**：同样添加 `IdleStateHandler`。
    
- **参数设置**：WriterIdleTime = 10秒 (10秒没写数据，就手痒)。
    
- **逻辑实现**：
    
    - 捕获 `userEventTriggered`。
        
    - 如果触发 `WRITER_IDLE` -> 发送一个 `Type=3 (PING)` 的包。
        

### 3.3 客户端：断线自动重连 (Auto Reconnect)

**目标**：拔掉网线再插上，程序能自动恢复。

- **监听位置**：客户端 Handler 的 `channelInactive` (连接断开) 方法。
    
- **核心逻辑**：
    
    - 打印“连接断开，3秒后重连...”。
        
    - 使用 `ctx.channel().eventLoop().schedule(...)` 调度任务。
        
    - 在任务里重新调用 `connect()` 方法。
        
    - **注意**：要处理好重连失败的递归逻辑（指数退避策略，比如第一次等3秒，第二次等5秒...）。

## 💬 Phase 4: 业务功能实现 (The Chat Room)

这是你可视化的来源，让两个黑框框能对话。

### 4.1 会话管理器 (Session Manager)

**目标**：不再是广播，而是精确找到那个人。

- 创建一个单例类 `SessionManager`。
    
- **核心数据结构**：`ConcurrentHashMap<String, Channel> userMap`。
    
    - Key: 用户ID (userId)
        
    - Value: Netty 的连接通道 (Channel)
        
- **方法**：
    
    - `bind(userId, channel)`: 登录成功时调用。
        
    - `getChannel(userId)`: 想给某人发消息时调用。
        
    - `unbind(channel)`: 用户断线时调用。
        

### 4.2 业务指令路由 (Router)

**目标**：根据 Cmd 决定干什么。

修改服务端的 `SimpleChannelInboundHandler`：

1. **登录 (Cmd=1001)**：
    
    - 解析 Body (`username:password`)。
        
    - 调用 `SessionManager.bind(username, ctx.channel())`。
        
    - 回复：`登录成功`。
        
2. **单聊消息 (Cmd=2001)**：
    
    - 解析 Body (`targetUser:messageContent`)。
        
    - 调用 `SessionManager.getChannel(targetUser)`。
        
    - 如果找到了 Channel -> `targetChannel.writeAndFlush(msg)`。
        
    - 如果没找到 -> 回复发送者“用户不在线”。
        
3. **群聊消息 (Cmd=2002)**：
    
    - 遍历 `SessionManager` 里的所有 Channel。
        
    - 除了自己，给其他人全发一遍。
