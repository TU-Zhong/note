这是一个非常经典的数据库基础问题，也是在 Java 后端开发和面试中经常被问到的概念。

在关系型数据库中，索引（Index）就像是一本书的“目录”，用来加快查询速度。而**唯一**和**非唯一**的区别，主要在于**是否允许数据重复**。

---

### 1. 唯一索引 (Unique Index) 对应 `nonUnique = false`

**定义**：唯一索引不仅能像普通目录一样加速查询，它还多了一层“**约束**”的作用——它强制要求被索引的这一列（或组合列）中的**所有值都必须是唯一的，不能有重复**。

- **特点**：
    
    - 保证数据的完整性（Data Integrity）。如果你试图插入一条重复的数据，数据库会直接报错（例如：`Duplicate entry`）。
        
    - 大多数数据库（包括你正在用的 SQL Server）允许在唯一索引列中包含 `NULL` 值，但通常只能有一个 `NULL`。
        
- **常见应用场景**：
    
    - **主键 (Primary Key)**：主键天生就是一个唯一索引（且不能为空）。
        
    - **业务唯一键**：用户的身份证号、手机号、注册邮箱、商品的条形码等。
        

> **举个例子**：在 `User` 表中，我们将 `email` 字段设置为唯一索引。
> 
> - 如果系统里已经有了 `admin@test.com`，再有用户想用 `admin@test.com` 注册，数据库层就会直接拒绝，不需要后端代码再去写 `select count(*)` 来判断。
>     

### 2. 非唯一索引 (Non-Unique Index / 普通索引) 对应 `nonUnique = true`

**定义**：非唯一索引是最基本的索引类型。它的目的**纯粹是为了加快查询速度**，没有任何数据约束的作用。

- **特点**：
    
    - 允许被索引的列中出现重复的值，也允许多个 `NULL` 值。
        
    - 当你通过这个索引查询时，可能会返回多行数据。
        
- **常见应用场景**：
    
    - **经常作为查询条件的普通字段**：用户的年龄、性别、商品的类别、订单的创建时间等。
        

> **举个例子**：在 `User` 表中，我们将 `age` (年龄) 字段设置为非唯一索引。
> 
> - 数据库里可以有几百个 `age = 22` 的用户。当我们执行 `SELECT * FROM User WHERE age = 22` 时，数据库通过索引能极快地把这几百个人找出来，而不用全表扫描。
>     

---

### 📊 核心区别对比表

|**特性**|**唯一索引 (Unique Index)**|**非唯一索引 / 普通索引 (Normal Index)**|
|---|---|---|
|**核心目的**|加速查询 + **保证数据唯一性**|**仅加速查询**|
|**是否允许重复值**|❌ 不允许|✅ 允许|
|**性能**|查询极快（找到一个就停止匹配）|查询很快（需要匹配所有相同的值）|
|**JDBC 元数据标识**|`NON_UNIQUE = false`|`NON_UNIQUE = true`|
|**常见字段**|手机号、邮箱、身份证号|姓名、年龄、分类状态|

### 💡 在你的毕业设计中的意义

在你的可视化平台中，这个字段（`nonUnique`）对于前端展示非常重要：

- 如果 `nonUnique = false`（唯一索引），你可以在前端界面上给这个索引加一个特殊的图标（比如一把**金色的钥匙**），表示这是一个非常重要的约束条件。
    
- 如果 `nonUnique = true`（非唯一索引），你可以用一个普通的**放大镜图标**，表示它只是用来加速搜索的。
