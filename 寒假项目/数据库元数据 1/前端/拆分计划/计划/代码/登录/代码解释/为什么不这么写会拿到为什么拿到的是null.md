这是一个非常反直觉、但完全符合网络安全逻辑的设计。

简单直接的答案是：**因为这个 `OPTIONS` 请求不是你发的，而是浏览器“背着你”自动发的。浏览器为了安全，故意没给它带上 Token。**

为了让你彻底明白，我们来看一下浏览器内部发生了什么：

### 1. 形象的比喻：先遣谈判

假设你要带着**巨款（Token）**去一个陌生的**城堡（后端服务器）**做交易。

- **你的代码（Axios）**：命令浏览器“带上巨款，冲进去！”
    
- **浏览器（保安）**：拦住你说：“慢着！那个城堡跟咱们不是一个地盘的（跨域），万一是个黑店怎么办？巨款会被吞的！”
    
- **浏览器的动作**：
    
    1. 浏览器先派出一个**手里什么都没拿**的**谈判专家（OPTIONS 请求）**去城堡。
        
    2. 谈判专家只问一句话：“**嘿！如果我兄弟待会儿带着 Authorization 这种高级货来，你们收不收？**”
        
    3. **注意：** 因为是去“问话”的，为了防止意外，谈判专家**绝对不能带巨款（Token）**。万一城堡这时候把你拒了，巨款还在也没损失。
        

这就是为什么你在拦截器里拦截 `OPTIONS` 请求时，`request.getHeader("Authorization")` 拿到的全是 `null` —— **因为它本来就两手空空！**

### 2. 技术层面的解剖

当你在 Vue 里写下这段代码时：

JavaScript

```
axios.get('/api/data', { 
    headers: { 'Authorization': 'token_123' } 
})
```

浏览器实际上**按顺序**发了两个 HTTP 请求：

#### 🕵️ 第一步：OPTIONS 请求 (预检 Preflight)

这是浏览器自动构建的，**它的 Headers 里只有这些信息**：

- `Origin`: http://localhost:5173 (我从哪来)
    
- `Access-Control-Request-Method`: GET (我想用 GET 方法)
    
- `Access-Control-Request-Headers`: Authorization (我想带个 Authorization 头，行不行？)
    
- **Authorization**: **(无！根本没发！)** ❌
    

#### 🚚 第二步：GET 请求 (实际请求)

只有当服务器对第一步回答“OK (200)”并且说“允许 Authorization”之后，浏览器才会发真正的请求：

- `Origin`: http://localhost:5173
    
- **Authorization**: **token_123** (终于带上了！) ✅
    

### 3. 现在的结局

如果你的拦截器里**没有**写那句 `if ("OPTIONS"...) return true`：

1. **谈判专家 (OPTIONS)** 两手空空地来了。
    
2. **拦截器** 像个死板的门卫，直接搜身：“Token 呢？没带 Token？**滚（401）！**”
    
3. **浏览器** 看到谈判专家被打回来了，判定这个服务器“不可理喻”。
    
4. **浏览器** 直接取消第二步的行动。
    
5. **结果**：你的 Vue 控制台报错 `CORS Error`，真正的请求胎死腹中。
    

### 总结

你之所以拿到 `null`，是因为**出于安全考虑，浏览器在进行跨域询问（OPTIONS）时，严禁携带任何敏感数据（包括 Token 和 Cookie）。**

所以，那段 `return true` 的代码，本质上是告诉拦截器：**“别搜谈判专家的身，让他直接过，他是来谈条件的，不是来办业务的。”**
