

# 📖 剧情对话场景生成器 (PRD)

## 一、 产品概述

**1. 产品定位**
一款面向内容创作者（如 B 站剧情向视频 UP 主）的轻量级桌面端视觉小说与对话场景编辑器。
**2. 核心价值**
通过自动化的 UI 排版、高自由度的图层挂载以及基于“关键帧继承”的编辑逻辑，极大地缩短传统使用 PS/PR 进行图文对齐和排版的耗时，一键输出沉浸式对话截图。
**3. 技术架构预期**

* **宿主环境**：桌面端客户端（推荐 Tauri 或 Electron），以获取本地文件系统的完全读写权限。
* **前端视图**：基于 Vue 3 + Pinia + Element Plus 构建，利用响应式数据池管理复杂的画布图层状态和跨组件联动。

---

## 二、 全局设计原则：高可扩展与数据驱动

为了避免后续迭代陷入硬编码的泥潭，整个系统的底层必须遵守以下契约：

* **配置与逻辑分离**：所有的预设样式（如气泡类型、系统默认背景、字体库）必须以 JSON 或单独的配置文件形式存在，系统启动时动态加载。
* **资源引用解耦**：工程文件中不保存图片的 Base64 全量数据，而是统一将本地图片拷贝至当前工程的 `assets` 目录下，并使用相对路径进行引用（如 `./assets/char_a_smile.png`）。
* **组件化渲染**：剧本流中的每一个气泡、画布上的每一个立绘/挂件，均封装为独立的 Vue 组件，通过接收统一格式的 `props`（包含坐标、层级、内容、缩放比）进行动态渲染。

## 三、 核心功能模块

### 1. 首页与工程管理模块 (Project Management)

**需求描述**：应用的入口，负责本地数据的持久化与无损恢复。

| 功能名称     | 交互逻辑与说明                                                                   | 扩展性要求                                                       |
| -------- | ------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **新建工程** | 弹出系统原生目录选择器，选择保存路径并命名。系统自动创建 `.arkchat`（或自定义后缀）工程文件，并伴生一个 `assets` 资源文件夹。 | 工程文件本质上为一个标准的 JSON 序列化对象，必须包含版本号 `version` 字段，以兼容未来数据结构的升级。 |
| **读取工程** | 从首页列表中点击或通过拖拽 `.arkchat` 文件载入，秒级还原上次编辑的所有剧本流、画布状态和全局配置。                   | 解析引擎需具备容错机制，若部分本地素材丢失，渲染时应显示占位符而非直接崩溃。                      |
| **打包导出** | 支持将工程文件与依赖的资源文件夹打包成一个 `.zip`，方便跨设备流转。                                     | /                                                           |

### 2. 角色与资源全局中心 (Global Asset Library)

**需求描述**：集中管理所有可复用的视觉元素，避免在对话节点中重复配置。

| 功能名称         | 交互逻辑与说明                                     | 扩展性要求                                                                |
| ------------ | ------------------------------------------- | -------------------------------------------------------------------- |
| **角色档案管理**   | 创建角色并分配全局唯一 `characterId`。支持设置角色名称、默认字体颜色。  | 数据结构需支持无限拓展字段。                                                       |
| **立绘与表情包配置** | 为单一角色上传多张图片资产。支持对图片进行标签化管理（如：默认、愤怒、思考、小挂件）。 | 每张图片生成独立的 `assetId`，剧本流中仅记录 ID，由 Pinia 全局状态库统一映射。                    |
| **预设样式库面板**  | 提供多套预设气泡（常规方舟、爆炸咆哮、云朵思考）。                   | 气泡不使用硬编码的 CSS 样式。需封装一套 `BubbleTemplate` 注册表，包含边框、背景、Padding、文本对齐等参数。 |

### 3. 核心工作区：剧本流与关键帧画布 (Script Flow & Canvas)

**需求描述**：编辑器的灵魂，采用“左侧/下方剧本流控制 + 右侧/上方全局画布实时预览”的布局，或直接在画布内嵌滚动列表。
- **独立文本样式面板**：在每一个对话节点的编辑区域内，提供专属的样式控制栏。
    
- **可调参数**：用户可以针对当前这句台词，单独修改**字号（Font Size）**、**字体颜色（Color）**、**字重（加粗 Bold）**、**字形（倾斜 Italic）**。

| 功能名称 | 交互逻辑与说明 | 扩展性要求 |
| --- | --- | --- |
| **对话节点编排** | 无限滚动列表。支持下拉选择发言角色、文本输入、气泡形态切换。支持插入系统旁白（无头像、无三角箭头，居中排版）。 | 节点类型需使用枚举（`RoleChat`, `SystemNarrator`, `Action`），方便后续低成本增加新排版模式。 |
| **自适应气泡** | 气泡宽度根据文本内容自动撑开（`fit-content`），达到最大宽度限制后自动换行。三角箭头自动挂载在说话人同侧。 | / |
| **关键帧画面继承** | **核心逻辑**：点击“新增下一句”时，系统默认克隆上一节点的完整“画布快照”（Canvas Snapshot）。 | `snapshot` 对象需深度拷贝（Deep Copy），切断对象引用，确保每一句的画面可以独立修改而不污染其他节点。 |
| **高自由度画布** | 画面中的立绘、表情、小道具等元素支持鼠标拖拽改变坐标（X, Y），支持滚轮/拖柄缩放（Scale），支持调整图层顺序（Z-index）。可多角色同屏。 | 画布元素需统一抽象为 `RenderNode` 接口，包含基础变换矩阵属性，渲染引擎只需遍历 `RenderNode` 列表进行绝对定位绘制即可。 |

### 4. 沉浸式预览与输出 (Preview & Export)

**需求描述**：为最终的“人工截图”服务，提供纯净的视觉呈现。

| 功能名称 | 交互逻辑与说明 | 扩展性要求 |
| --- | --- | --- |
| **全局 UI 配置** | 支持上传自定义全局背景图，调整背景模糊度。支持一键修改剧本流的全局字体大小与间距。 | 样式变量通过 CSS Variables (`--var`) 注入，实现全局一键换肤。 |
| **纯净模式切换** | 点击“预览”按钮或快捷键（如 F11），瞬间隐藏所有控制面板、拖拽框、新增按钮，仅保留渲染好的画布与当前选中的对话气泡。 | 建立全局的 `AppMode` 状态（`EDIT` 或 `PREVIEW`），所有非展示类 UI 组件通过 `v-show` 绑定该状态。 |

## 四、 核心数据结构设计规范示例

为了保证绝不硬编码，整个系统的数据流应当围绕一个高扩展性的 JSON 树展开。以下是核心状态池（State）的结构建议：

```json
{
  "projectInfo": {
    "version": "1.0.0",
    "globalStyle": {
      "backgroundImage": "./assets/bg_room.jpg",
      "fontFamily": "System Default"
    }
  },
  "characterDict": {
    "char_01": {
      "name": "维什戴尔",
      "assets": {
        "asset_001": "./assets/w_smile.png",
        "asset_002": "./assets/w_angry.png"
      }
    }
  },
  "storyFlow": [
    {
      "nodeId": "node_101",
      "type": "RoleChat",
      "speakerId": "char_01",
      "text": "你踏马为什么会在我房间里！！！",
      "bubbleTemplateId": "preset_shout", 
      "canvasSnapshot": {
        "elements": [
          {
            "elementId": "el_01",
            "assetId": "asset_002",
            "position": { "x": 120, "y": 300 },
            "scale": 1.2,
            "zIndex": 10
          }
        ]
      }
    }
  ]
}

```

## 五、 开发实施建议

1. **第一阶段（MVP 最小可行性产品）**：跑通 Tauri/Electron 的本地文件保存闭环。实现固定的一种方舟气泡和纯文本列表。
2. **第二阶段（画布升级）**：引入 VueUse 库中的 `useDraggable` 等组合式 API，完成立绘和挂件在画布上的自由拖拽与状态记录。
3. **第三阶段（扩展与打磨）**：完成关键帧继承逻辑、接入预设 SVG 异形气泡、完善纯净预览模式的快捷键映射。
