MyBatis 支持嵌套查询，主要通过两种方式来实现：**嵌套 Select 查询 (Nested Selects)** 和 **嵌套结果映射 (Nested Results)**。这两种方式都用于处理对象之间的关联关系，比如一对一（`<association>`）和一对多（`<collection>`）。

下面详细介绍这两种方式：
嵌套select查询 [[标签解释]]
### 1. 嵌套结果映射 (Nested Results using Joins) - 推荐方式

这种方式通常性能更好，因为它通过**一次SQL查询（使用JOIN）**获取所有需要的数据，然后在 `<resultMap>` 中通过嵌套的 `<association>` 或 `<collection>` 标签来组织这些数据，将扁平的SQL结果集映射成层次化的Java对象。

**优点：**

- 通常性能较好，因为只需要执行一次数据库查询。
- 数据一次性加载，避免了N+1问题。

**缺点：**

- SQL语句可能比较复杂，因为需要编写JOIN。
- 如果关联对象非常多，可能会导致主查询返回大量重复数据（在JOIN的"一"方）。

#### 示例：查询博客及其作者（一对一）和所有文章（一对多）

**数据库表结构 (假设):**

- `Author` 表:
    - `author_id` (PK)
    - `author_name`
    - `author_bio`
- `Blog` 表:
    - `blog_id` (PK)
    - `blog_title`
    - `author_id` (FK references Author)
- `Post` 表:
    - `post_id` (PK)
    - `post_title`
    - `post_content`
    - `blog_id` (FK references Blog)

**Java 实体类:**

Java

```
// Author.java
public class Author {
    private Integer authorId;
    private String authorName;
    private String authorBio;
    // getters and setters
}

// Post.java
public class Post {
    private Integer postId;
    private String postTitle;
    private String postContent;
    // getters and setters
}

// Blog.java
public class Blog {
    private Integer blogId;
    private String blogTitle;
    private Author author; // 一对一
    private List<Post> posts; // 一对多
    // getters and setters
}
```

**Mapper XML (`BlogMapper.xml`):**

XML

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.BlogMapper">

    <resultMap id="blogResultMap" type="com.example.model.Blog">
        <id property="blogId" column="blog_id"/>
        <result property="blogTitle" column="blog_title"/>

        <association property="author" javaType="com.example.model.Author">
            <id property="authorId" column="author_id"/>
            <result property="authorName" column="author_name"/>
            <result property="authorBio" column="author_bio"/>
        </association>

        <collection property="posts" ofType="com.example.model.Post">
            <id property="postId" column="post_id"/>
            <result property="postTitle" column="post_title"/>
            <result property="postContent" column="post_content"/>
            </collection>
    </resultMap>

    <select id="selectBlogWithAuthorAndPosts" resultMap="blogResultMap">
        SELECT
            b.blog_id,
            b.blog_title,
            a.author_id,         -- Author的列，需要别名或保证唯一性
            a.author_name,
            a.author_bio,
            p.post_id,           -- Post的列
            p.post_title,
            p.post_content
        FROM
            Blog b
        LEFT JOIN
            Author a ON b.author_id = a.author_id
        LEFT JOIN
            Post p ON b.blog_id = p.blog_id
        WHERE
            b.blog_id = #{id}
    </select>
</mapper>
```

**Mapper 接口 (`BlogMapper.java`):**

Java

```java
package com.example.mapper;
import com.example.model.Blog;
public interface BlogMapper {
    Blog selectBlogWithAuthorAndPosts(int id);
}
```

**说明：**

- `blogResultMap` 定义了如何将JOIN查询的结果映射到 `Blog` 对象及其关联的 `Author` 和 `List<Post>`。
- `<association property="author" javaType="com.example.model.Author">` 用于映射一对一的 `author` 属性。内部的 `<id>` 和 `<result>` 标签映射 `Author` 类的属性。
- `<collection property="posts" ofType="com.example.model.Post">` 用于映射一对多的 `posts` 列表。`ofType` 指定了列表中元素的类型。
- SQL语句使用 `LEFT JOIN` 来一次性获取所有相关数据。MyBatis 会根据 `blog_id` (主对象的ID) 来正确地将多个 `Post` 行组合到对应的 `Blog` 对象的 `posts` 列表中。

### 2. 嵌套 Select 查询 (Nested Selects) - 可能导致 N+1 问题

这种方式会为每个主对象执行额外的SQL查询来加载其关联对象。即，先执行一个查询获取主对象列表，然后对列表中的每个主对象，再分别执行一个或多个查询来获取其关联对象。

**优点：**

- SQL语句相对简单，每个查询只关注自己的表。
- 可以实现**延迟加载 (Lazy Loading)**，即只有在实际访问关联属性时才执行嵌套查询。

**缺点：**

- **N+1 问题**：如果主查询返回N条记录，那么为了加载关联对象，可能会额外执行N次（或N乘以关联类型的数量）查询，导致数据库交互过多，性能低下。

**Mapper XML (`BlogMapper.xml`):**

XML

```xml
<?xml version="1.0" encoding="UTF-8" ?>
<!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" "http://mybatis.org/dtd/mybatis-3-mapper.dtd">
<mapper namespace="com.example.mapper.BlogMapper">

    <select id="selectAuthorById" resultType="com.example.model.Author">
        SELECT author_id, author_name, author_bio FROM Author WHERE author_id = #{id}
    </select>

    <select id="selectPostsByBlogId" resultType="com.example.model.Post">
        SELECT post_id, post_title, post_content FROM Post WHERE blog_id = #{blogId}
    </select>

    <resultMap id="blogResultMapWithNestedSelect" type="com.example.model.Blog">
        <id property="blogId" column="blog_id"/>
        <result property="blogTitle" column="blog_title"/>

        <association property="author"
                     javaType="com.example.model.Author"
                     column="author_id"
                     select="selectAuthorById"/>

        <collection property="posts"
                    ofType="com.example.model.Post"
                    column="blog_id"
                    select="selectPostsByBlogId"/>
    </resultMap>

    <select id="selectBlogById" resultMap="blogResultMapWithNestedSelect">
        SELECT
            blog_id,
            blog_title,
            author_id  -- 这个 author_id 会被传递给 selectAuthorById
        FROM Blog
        WHERE blog_id = #{id}
    </select>
</mapper>
```

**Mapper 接口 (`BlogMapper.java`):**

Java

```java
package com.example.mapper;

import com.example.model.Author; // 如果需要单独调用
import com.example.model.Post;   // 如果需要单独调用
import com.example.model.Blog;

public interface BlogMapper {
    // 主查询
    Blog selectBlogById(int id);

    // 供嵌套查询使用 (也可以是其他Mapper中的方法)
    Author selectAuthorById(int id);
    List<Post> selectPostsByBlogId(int blogId);
}
```

**说明：**

- `<association>` 和 `<collection>` 标签使用了 `select` 属性，指向另一个已定义的 `select` 语句的ID。
- `column` 属性指定了将主查询结果中的哪些列的值作为参数传递给嵌套的 `select` 语句。
    - 如果嵌套查询需要多个参数，`column` 可以这样写：`column="{prop1=col1, prop2=col2}"`，其中 `prop1`, `prop2` 是嵌套查询方法参数名（或`param1`, `param2`，或通过`@Param`注解指定），`col1`, `col2` 是主查询结果的列名。
- **延迟加载配置**： 可以在 MyBatis 的配置文件 (`mybatis-config.xml`) 中启用延迟加载：
    
    XML
    
    ```xml
    <settings>
        <setting name="lazyLoadingEnabled" value="true"/>
        <setting name="aggressiveLazyLoading" value="false"/>
        <setting name="lazyLoadTriggerMethods" value="equals,clone,hashCode,toString"/>
    </settings>
    ```
    
    也可以在 `<association>` 或 `<collection>` 标签上通过 `WorkspaceType="lazy"` (或 `eager`) 单独控制。

### 如何选择？

1. **性能优先/数据总是需要**：优先选择**嵌套结果映射 (Joins)**。它通常更快，因为它只与数据库交互一次。
2. **延迟加载/SQL简化**：如果关联数据不是每次都需要，或者希望保持各个SQL语句的简单性，可以考虑**嵌套Select查询**，并启用延迟加载。但务必注意潜在的N+1问题，并进行性能测试。
3. **数据量和复杂度**：如果JOIN导致SQL过于复杂或返回的冗余数据过多，有时嵌套Select配合缓存或仔细的延迟加载策略可能是一个折衷方案。

在大多数情况下，**嵌套结果映射（Joins）是更推荐的方案**，尤其是在关联数据几乎总是被使用的情况下。如果担心JOIN的复杂性或返回过多冗余数据，可以考虑将非常复杂的查询拆分成更小的部分，并在服务层进行组装，但这会增加Java代码的复杂性。
