好的，我们来谈谈 MyBatis 中实现分页查询的“语法”或更准确地说是**方法**。

需要明确的关键点是：**MyBatis 核心本身并没有提供一个统一的、跨数据库的“分页语法”XML标签（比如像 `<limit>` 或 `<page>` 这样的东西）**。因为 SQL 的分页语法在不同的数据库（MySQL, PostgreSQL, Oracle, SQL Server等）中是不同的。

因此，在 MyBatis 中实现分页查询主要有以下几种方式：

### 1. 在 Mapper XML 中直接编写数据库特定的分页 SQL

这是最直接、最透明的方式。你需要在你的 `<select>` 语句中包含特定数据库的分页子句，并将分页所需的参数（如偏移量 `offset` 和限制数量 `limit` 或页码 `pageNum` 和页面大小 `pageSize`）作为方法的参数传入。

**示例 (MySQL 的 `LIMIT ... OFFSET ...`)**

- **Mapper 接口 (`OrderMapper.java`)**
    
    Java
    
    ```
    import org.apache.ibatis.annotations.Param;
    // ... 其他 import ...
    
    public interface OrderMapper {
        List<Order> findOrdersPaged(
            @Param("offset") int offset,
            @Param("limit") int limit,
            @Param("status") String status // 其他查询条件
        );
    }
    ```
    
- **Mapper XML (`OrderMapper.xml`)**
    
    XML
    
    ```
    <select id="findOrdersPaged" resultType="com.example.model.Order">
        SELECT orderId, customerId, orderDate, totalAmount, status
        FROM Orders
        WHERE status = #{status} -- 示例：包含其他条件
        ORDER BY orderDate DESC  -- 分页必须有 ORDER BY
        LIMIT #{limit} OFFSET #{offset}
        -- 或者 MySQL 的简写: LIMIT #{offset}, #{limit}
    </select>
    ```
    

**优点：**

- 非常明确，直接控制生成的 SQL。
- 性能可控，因为你写的是原生分页语句。

**缺点：**

- SQL 语句与特定数据库绑定，如果需要切换数据库，需要修改 XML。
- 需要在 XML 中手动添加分页逻辑。

### 2. 使用 `RowBounds` 参数 (逻辑分页 - 需谨慎使用)

MyBatis 允许你在调用 Mapper 方法时传入一个 `org.apache.ibatis.session.RowBounds` 对象作为参数，它包含了 `offset` 和 `limit` 信息。

- **Mapper 接口 (`OrderMapper.java`)**
    
    Java
    
    ```
    // 接口方法签名不需要包含 RowBounds 参数
    List<Order> findAllOrdersFiltered(@Param("status") String status);
    ```
    
- **Mapper XML (`OrderMapper.xml`)**
    
    XML
    
    ```
    <select id="findAllOrdersFiltered" resultType="com.example.model.Order">
        SELECT orderId, customerId, orderDate, totalAmount, status
        FROM Orders
        WHERE status = #{status}
        ORDER BY orderDate DESC
    </select>
    ```
    
- **Java 调用代码**
    
    Java
    
    ```
    import org.apache.ibatis.session.RowBounds;
    // ... 其他 import ...
    
    try (SqlSession session = MyBatisSqlSessionFactoryUtil.openSession()) {
        OrderMapper mapper = session.getMapper(OrderMapper.class);
        int offset = 20; // (page 3 - 1) * 10
        int limit = 10;  // page size
        RowBounds rowBounds = new RowBounds(offset, limit);
    
        // 将 RowBounds 作为额外参数传递给 selectList 方法
        List<Order> orders = session.selectList("com.example.mapper.OrderMapper.findAllOrdersFiltered", "SHIPPED", rowBounds);
        // 或者如果 Mapper 方法只有一个参数，可以直接这样调用 (MyBatis 3.4.6+):
        // List<Order> orders = mapper.findAllOrdersFiltered("SHIPPED", rowBounds); // 需要确认具体版本和配置
    }
    ```
    

**工作方式与风险：**

- MyBatis 会识别 `RowBounds` 参数。
- **关键风险点**：如果底层的 JDBC 驱动不支持或 MyBatis 配置不允许使用游标分页 (Cursor/Scrollable ResultSet)，MyBatis 会**查询出所有符合条件的记录**到内存中，然后再通过 `subList()` 的方式在 Java 层面截取所需的页面数据。
- **性能问题**：对于大数据量的表，将所有结果加载到内存中再分页是**极其低效且可能导致内存溢出 (OutOfMemoryError)** 的。
- **结论**：`RowBounds` 方式简单，但潜在的性能风险很大，**通常不推荐用于生产环境的大数据量分页**，除非你非常确定你的环境和配置能够进行基于数据库游标的高效分页（这比较少见且配置复杂）。

### 3. 使用分页插件 (物理分页 - 最常用)

这是在实际项目中**最常用、最推荐**的方式。通过引入第三方分页插件（如 **PageHelper**），可以在不修改 Mapper XML 的情况下实现物理分页。

- **原理**: 这些插件通常使用 MyBatis 的**拦截器 (Interceptor)** 机制。它们会拦截即将执行的 SQL 语句，根据你在 Java 代码中设置的分页参数（通常通过 `ThreadLocal` 传递），自动分析 SQL，并将其改写成对应数据库的物理分页 SQL（添加 `LIMIT`/`OFFSET`, `ROWNUM`, `ROW_NUMBER()` 等）。
    
- **示例 (使用 PageHelper)**
    
    1. **添加依赖** (例如 Maven `pom.xml`):
        
        XML
        
        ```
        <dependency>
            <groupId>com.github.pagehelper</groupId>
            <artifactId>pagehelper-spring-boot-starter</artifactId> <version>最新版本</version>
        </dependency>
        ```
        
    2. **配置插件** (例如在 Spring Boot application.properties 或 mybatis-config.xml 中配置 Interceptor)。
    3. **Mapper 接口和 XML (保持原样，不需要分页逻辑)**:
        
        Java
        
        ```
        // OrderMapper.java
        List<Order> findOrders(@Param("status") String status);
        ```
        
        XML
        
        ```
        // OrderMapper.xml
        <select id="findOrders" resultType="com.example.model.Order">
            SELECT orderId, customerId, orderDate, totalAmount, status
            FROM Orders
            WHERE status = #{status}
            ORDER BY orderDate DESC
        </select>
        ```
        
    4. **Java 调用代码**:
        
        Java
        
        ```
        import com.github.pagehelper.PageHelper;
        import com.github.pagehelper.PageInfo;
        // ... 其他 import ...
        
        try (SqlSession session = MyBatisSqlSessionFactoryUtil.openSession()) {
            OrderMapper mapper = session.getMapper(OrderMapper.class);
        
            int pageNum = 3;
            int pageSize = 10;
        
            // 1. 启动分页 (仅对接下来执行的第一次MyBatis查询有效)
            PageHelper.startPage(pageNum, pageSize);
        
            // 2. 执行正常的查询
            List<Order> orders = mapper.findOrders("SHIPPED"); // XML里没有分页语句
        
            // 3. (可选) 使用 PageInfo 包装查询结果，获取更详细的分页信息
            PageInfo<Order> pageInfo = new PageInfo<>(orders);
        
            System.out.println("总记录数: " + pageInfo.getTotal());
            System.out.println("总页数: " + pageInfo.getPages());
            System.out.println("当前页码: " + pageInfo.getPageNum());
            System.out.println("每页数量: " + pageInfo.getPageSize());
            System.out.println("当前页数据条数: " + pageInfo.getSize());
            System.out.println("当前页数据: ");
            pageInfo.getList().forEach(System.out::println);
        
            // 底层 PageHelper 已经自动将 SQL 修改为类似:
            // SELECT ... FROM Orders WHERE status = ? ORDER BY orderDate DESC LIMIT ?, ?
            // (具体SQL取决于数据库方言)
        }
        ```
        

**优点：**

- **代码简洁**: Mapper XML 无需关心分页细节，保持干净。
- **数据库无关**: 插件负责处理不同数据库的分页语法，提高了代码的可移植性。
- **功能丰富**: 插件通常还提供总记录数查询、分页信息封装等便捷功能。

**缺点：**

- 增加了一个外部依赖。
- 需要理解插件的工作原理和配置。
- 对于非常复杂的、包含子查询或特殊结构的 SQL，插件自动改写有时可能出错（虽然现代插件已经很智能）。

### 总结

MyBatis 本身没有统一的分页“语法”，实现分页主要靠：

1. **手动在 XML 中写数据库特定的分页 SQL**：最直接，性能可控，但需处理数据库差异。
2. **使用 `RowBounds` 参数**：写法简单，但有**严重性能风险**（内存分页），一般不推荐。
3. **使用分页插件 (如 PageHelper)**：**最常用和推荐**的方式，通过拦截器实现物理分页，代码简洁，对开发者透明。

根据你的项目需求、团队习惯和对性能的要求来选择合适的方式。对于大多数新项目，使用成熟的分页插件是最高效的选择。
