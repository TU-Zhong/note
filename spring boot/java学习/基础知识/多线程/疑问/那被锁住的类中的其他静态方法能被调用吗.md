

用银行金库和窗口的比喻，让你彻底明白类锁的规则！🏦

---

### **直接答案**
如果其他静态方法**也加了 `synchronized`** → **不能立即调用，要等待锁释放！**  
如果其他静态方法**没加 `synchronized`** → **随便调用，无需等待！**

---

### **详细解释**
#### **1. 锁住类的情况**
当某个线程进入 `synchronized` 静态方法时，它获取的是**整个类的锁**（即 `类名.class` 对象）。此时：
- **其他 `synchronized` 静态方法** → 被阻塞（需要等锁）
- **普通静态方法** → 可以直接调用（不涉及锁）

---

#### **2. 代码验证**
```java
public class Bank {
    // 同步静态方法1
    public static synchronized void 操作金库A() {
        System.out.println("金库A开始操作...");
        sleep(3000); // 模拟耗时
        System.out.println("金库A操作完成");
    }

    // 同步静态方法2
    public static synchronized void 操作金库B() {
        System.out.println("金库B开始操作...");
    }

    // 普通静态方法
    public static void 查询汇率() {
        System.out.println("汇率查询成功");
    }

    private static void sleep(long ms) {
        try { Thread.sleep(ms); } catch (Exception e) {}
    }
}
```

**测试代码**：
```java
new Thread(() -> Bank.操作金库A()).start(); // 线程1：进入同步方法A
new Thread(() -> Bank.操作金库B()).start(); // 线程2：尝试进入同步方法B → 被阻塞！
new Thread(() -> Bank.查询汇率()).start();   // 线程3：普通方法 → 直接执行！
```

**运行结果**：
```
金库A开始操作...
汇率查询成功 （线程3立刻执行）
...3秒后...
金库A操作完成
金库B开始操作... （线程2此时才执行）
```

---

### **核心规则表**
| **方法类型**               | **是否被类锁阻塞** | **原因**                     |
|--------------------------|------------------|-----------------------------|
| `synchronized` 静态方法    | ✅ 是            | 需要同一把类锁（Bank.class） |
| 普通静态方法               | ❌ 否            | 不涉及锁                     |
| `synchronized` 实例方法    | ❌ 否            | 锁的是实例对象（this），与类锁无关 |

---

### **关键结论**
1. **类锁是全局的**：所有 `synchronized` 静态方法共享同一把锁  
2. **锁的是方法访问权**：同一时间只能有一个线程执行任意一个 `synchronized` 静态方法  
3. **非同步方法不参与锁竞争** → 随时执行

---

### **比喻总结**
把类锁想象成**银行总部的金库大门**：
- 金库A和金库B的操作（`synchronized` 静态方法）都需要总部钥匙（锁）  
- 同一时间只能有一个人拿钥匙（线程）操作金库（不论A还是B）  
- 查询汇率（普通静态方法）在大厅自助机操作，不需要钥匙！
