

用银行窗口排队的例子，1分钟搞懂锁的公平性！ 🏦

---

### **一句话回答**
锁的公平性就是解决**“插队”问题**的规则：
- **公平锁** → 先排队的线程先拿到锁（先到先得）
- **非公平锁** → 新来的线程可能插队抢锁（效率更高但可能饿死早期线程）

---

### **1. 公平锁（Fair Lock）**
#### **比喻**：
银行设有**取号机**，所有客户按取号顺序排队。窗口叫号时，严格按号码顺序处理（1号、2号、3号...）。  

**代码示例**：
```java
ReentrantLock fairLock = new ReentrantLock(true); // true表示公平锁
```

---

### **2. 非公平锁（Non-Fair Lock）**
#### **比喻**：
银行没有取号机，每次窗口空闲时，所有人一拥而上抢窗口。虽然大部分情况遵循先来后到，但新来的客户可能挤到前面抢到机会。

**代码示例**：
```java
ReentrantLock unfairLock = new ReentrantLock(); // 默认非公平锁
// 或
ReentrantLock unfairLock = new ReentrantLock(false);
```

---

### **对比表**
| **对比项**       | **公平锁**                          | **非公平锁**                          |
|------------------|------------------------------------|---------------------------------------|
| **获取顺序**      | 严格按线程等待顺序（先到先得）       | 允许插队（可能后到的线程先拿到锁）      |
| **吞吐量**        | 较低（维护队列开销大）               | 较高（减少线程切换）                   |
| **线程饥饿**      | 不会发生                           | 可能发生（早期线程长期等待）            |
| **适用场景**      | 线程等待时间差异大，要求公平性       | 高并发场景，追求吞吐量                 |
| **实现复杂度**    | 高（需维护等待队列）                 | 低                                     |

---

### **代码验证公平性**
```java
public class FairnessDemo {
    private static final ReentrantLock lock = new ReentrantLock(true); // 改为false测试非公平锁
    
    public static void main(String[] args) {
        for (int i = 1; i <= 5; i++) {
            new Thread(() -> {
                for (int j = 0; j < 2; j++) {
                    lock.lock();
                    try {
                        System.out.println(Thread.currentThread().getName() + " 获得锁");
                        Thread.sleep(100);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    } finally {
                        lock.unlock();
                    }
                }
            }, "线程-" + i).start();
        }
    }
}
```

**公平锁输出示例**（顺序基本一致）：
```
线程-1 获得锁
线程-2 获得锁
线程-3 获得锁
线程-4 获得锁
线程-5 获得锁
线程-1 获得锁
线程-2 获得锁
...
```

**非公平锁输出示例**（可能出现插队）：
```
线程-1 获得锁
线程-1 获得锁 ← 同一线程连续获取
线程-3 获得锁 ← 新线程插队
线程-5 获得锁
...
```

---

### **实际开发中的选择**
- **默认推荐非公平锁**：性能更高（`synchronized`也是非公平锁）
- **特殊场景用公平锁**：如交易系统要求严格按订单到达时间处理
- **避免过度公平**：维护队列需要额外开销，可能降低吞吐量

---

### **总结口诀**  
公平锁，不插队；非公平，效率高；  
默认不用公平锁，特殊需求再上轿！ 🚀
