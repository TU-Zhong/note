好的，我们来详细了解一下 Java 中的偏向锁、轻量级锁和重量级锁。这些都是 Java 虚拟机 (JVM) 为了提高 `synchronized` 关键字的性能而引入的锁优化机制。

在 Java 中，`synchronized` 关键字是我们常用的线程同步方式。早期的 `synchronized` 实现非常依赖操作系统提供的互斥量 (Mutex)，这种实现在并发竞争激烈时性能开销较大（需要用户态到内核态的切换）。为了在不同争用情况下都能有较好的性能，JVM 引入了锁的升级（或称膨胀）概念，从偏向锁 -> 轻量级锁 -> 重量级锁。

理解这些锁的关键在于理解 Java 对象头（Object Header）中的 **Mark Word** 部分，因为锁的状态信息就记录在这里。

---

### 1. 偏向锁 (Biased Locking)

**引入目的：** 在大多数情况下，锁不仅不存在多线程竞争，而且总是由**同一个线程**多次获得。偏向锁的核心思想是，如果一个线程获得了锁，那么锁就“偏向”于这个线程，后续这个线程进入和退出同步块时，不需要进行 CAS (Compare-And-Swap) 操作或互斥量的获取，从而降低获取锁的代价。
[[不需要进行 CAS (Compare-And-Swap) 操作或互斥量的获取这个是什么意思？]]

**工作原理：**

1. **初次获取：** 当一个线程第一次访问同步块并获取锁时，<mark style="background: #FFF3A3A6;">会在对象头（Mark Word）和栈帧中的锁记录里存储锁偏向的线程ID</mark>。同时，Mark Word 中的“偏向锁标记位”设为1，“锁标记位”设为01（表示偏向锁状态）。
2. **后续访问（同一线程）：** 当这个线程再次请求锁时，JVM 只需检查 Mark Word 中记录的线程ID是否与当前线程ID一致。
    - 如果一致，表示该线程已经获得了锁，可以直接执行同步代码块，无需任何同步操作（如CAS）。这是偏向锁性能最高的场景。
3. **锁撤销与升级：**
    - **不同线程尝试获取：** 当有**另一个线程**尝试获取这个偏向锁时，偏向模式宣告结束。
    - **偏向锁的撤销 (Revocation)：** 需要等待全局安全点 (Global Safepoint)，暂停拥有偏向锁的线程，检查该线程是否还在执行同步代码块中的代码。
        - 如果原线程不再活动或已退出同步块，则将对象头设置为无锁状态或重新偏向于新的线程。
        - 如果原线程仍然持有锁（即仍在同步块内），则需要将锁升级为轻量级锁。此时，原线程的锁记录会被填充，对象头的 Mark Word 会指向这个锁记录。
    - **调用对象的 `hashCode()` 或 `wait()`/`notify()`：** 如果一个对象已经处于偏向锁状态，当有线程调用其 `hashCode()` 方法（因为 Mark Word 可能需要存储哈希值）或者 `wait()`/`notify()`（需要更重量级的 Monitor 支持），偏向锁会被撤销并可能升级。

**优点：**

- 在只有一个线程持续访问同步代码块的场景下，加锁和解锁的开销几乎为零，性能提升非常明显。

**缺点：**

- 如果线程竞争比较激烈，或者有多个线程交替访问同一个锁，偏向锁的撤销和锁升级会带来额外的性能开销。
- 偏向锁的撤销过程可能需要暂停持有锁的线程（STW - Stop The World 的一部分），这在某些情况下是不可接受的。
- 在 JDK 15 及以后版本，偏向锁默认被禁用，因为现代应用中，完全无竞争或单线程反复获取同一锁的场景相对减少，其带来的维护成本和复杂性可能超过了收益。可以通过 JVM 参数 `-XX:+UseBiasedLocking`（在支持的版本中）显式开启。

### 2. 轻量级锁 (Lightweight Locking)

[[线程尝试通过 CAS 原子地将对象头中的 Mark Word（里面包含了锁的状态信息）更新为一个指向自己栈上锁记录的指针这个对象头是什么东西，自己栈又是什么东西？]]

**引入目的：** 当偏向锁被撤销后，或者关闭了偏向锁的场景下，如果线程对锁的竞争不激烈（即线程交替执行同步块，但同一时刻只有一个线程在尝试获取锁，或者锁的持有时间非常短），轻量级锁可以避免重量级锁的操作系统互斥量调用，从而提高性能。它基于“多数锁在整个同步周期内都不存在竞争”的假设。

**工作原理：**

1. **加锁过程：**
    - JVM 在当前线程的栈帧中创建一块名为“锁记录 (Lock Record)”的空间，用于存储锁对象目前的 Mark Word 的拷贝（Displaced Mark Word）。
    - JVM 使用 CAS 操作尝试将对象头中的 Mark Word 更新为指向锁记录的指针。
        - **成功：** 如果 CAS 操作成功，当前线程就获得了锁，对象头的锁标志位设为 "00"（表示轻量级锁状态）。
        - **失败：** 如果 CAS 操作失败，说明存在其他线程竞争锁。
2. **自旋等待 (Spinning)：**
    - 如果 CAS 失败，当前线程并不会立即挂起，而是会进行“自旋”等待。即执行一个忙循环（空转），期望在短时间内锁的持有者会释放锁。
    - 自旋的次数和策略是 JVM 动态调整的（例如，自适应自旋：如果之前自旋成功过，就允许更长时间的自旋）。
3. **锁升级：**
    - 如果自旋一定次数后仍然没有获取到锁，或者有其他线程也在自旋等待同一个锁，轻量级锁就会膨胀 (inflate) 为重量级锁。此时，对象头的 Mark Word 会被修改为指向重量级锁（Monitor）的指针，锁标志位变为 "10"。后面等待锁的线程会被阻塞。

**解锁过程：**

1. 使用 CAS 操作尝试将栈帧中拷贝的 Displaced Mark Word 替换回对象头。
    - **成功：** 表示没有竞争发生，解锁完成。
    - **失败：** 表示在持有锁期间，有其他线程尝试获取锁并可能导致锁升级（例如，锁已经膨胀为重量级锁）。这时需要进入重量级锁的解锁路径，唤醒等待队列中的线程。


**优点：**

- 在短时间、低竞争的情况下，通过 CAS 和自旋避免了操作系统互斥量的开销，性能较高。

**缺点：**

- 如果锁竞争激烈，或者锁被长时间持有，自旋会白白消耗 CPU 资源。
- 最终如果还是要升级到重量级锁，那么 CAS 和自旋的开销就成了额外的负担。

### 3. 重量级锁 (Heavyweight Locking)

**引入目的：** 当锁竞争非常激烈（多个线程同时请求锁），或者一个线程持有锁的时间过长，轻量级锁的自旋就显得不划算了。此时，需要一种更传统的锁机制，它能够让获取不到锁的线程进入阻塞状态，释放 CPU，等待被唤醒。这就是重量级锁，它依赖于操作系统层面的互斥量 (Mutex Lock) 来实现，在 Java 中通常与对象的监视器 (Monitor) 相关联。

**工作原理：**

1. 当锁膨胀为重量级锁后，对象头的 Mark Word 会存储指向一个 Monitor 对象的指针。
2. Monitor 内部维护了一个等待队列 (Wait Set) 和一个入口队列 (Entry List / Contention List)。
3. **加锁过程：**
    - 线程尝试获取 Monitor 的所有权。
    - 如果 Monitor 未被占用，则该线程成为 Monitor 的持有者。
    - 如果 Monitor 已被其他线程占用，则当前线程会被放入入口队列，并进入阻塞 (Blocked) 状态，放弃 CPU 执行权，等待操作系统调度。
4. **解锁过程：**
    - 持有 Monitor 的线程执行完同步代码块后，会释放 Monitor。
    - 释放 Monitor 时，会唤醒入口队列中等待的一个或多个线程，这些被唤醒的线程会重新尝试获取锁。选择哪个线程被唤醒取决于具体的调度策略（可能是公平或非公平的）。

**优点：**

- 在锁竞争激烈、锁持有时间长的场景下，能够有效地管理线程，避免 CPU 空转。线程阻塞后不会消耗 CPU。

**缺点：**

- 涉及操作系统内核态和用户态之间的切换，上下文切换成本较高，因此性能开销最大。

### 锁升级（膨胀）过程总结

Java 中 `synchronized` 锁的升级路径通常是单向的，目的是在不同场景下尽可能地优化性能：

**无锁状态 -> 偏向锁 -> 轻量级锁 -> 重量级锁**

1. **对象创建时**：默认是无锁状态（但也可能根据 JVM 参数直接启用偏向锁）。
2. **第一个线程访问**：升级为偏向锁（如果开启了偏向锁）。
3. **出现其他线程竞争（撤销偏向锁后）**：升级为轻量级锁。
4. **轻量级锁自旋失败或竞争加剧**：升级为重量级锁。

**注意：** 这个升级路径不是绝对的，例如：

- 如果偏向锁被禁用 (`-XX:-UseBiasedLocking`)，对象会直接尝试进入轻量级锁状态。
- 如果一开始就有多个线程竞争，可能会直接跳过偏向锁和轻量级锁，直接膨胀为重量级锁（虽然 JVM 会尽量尝试轻量级锁）。
- 锁的降级（例如从重量级锁降回轻量级锁）理论上是可能的，但在 HotSpot JVM 中实现得非常有限或不常见，因为成本和复杂性很高。通常认为锁升级是单向的。

这些锁优化机制对开发者是透明的，我们只需要正确使用 `synchronized` 关键字，JVM 会在底层根据实际的竞争情况自动选择和升级锁的状态，以期达到最佳的性能。
