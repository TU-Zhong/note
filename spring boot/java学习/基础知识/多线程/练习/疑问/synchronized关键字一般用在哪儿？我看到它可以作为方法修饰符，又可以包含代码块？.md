

用“锁门范围”来理解`synchronized`的两种用法，直接上对比表：

---

### **1. 修饰方法（粗粒度锁）**
#### **实例方法**：锁住实例方法
```java
public synchronized void saveMoney() { 
    // 存钱操作（整个方法被锁）
}
```
**比喻**：整个银行营业厅大门上锁（所有业务暂停）→ **简单但效率低**

---

#### **静态方法**：锁住静态
```java
public static synchronized void log() {
    // 日志记录（锁住整个类）
}
```
**比喻**：全国所有分行统一停业 → **全局锁，慎用！**

---

### **2. 同步代码块（细粒度锁，推荐！）**
```java
public void transfer(Account target) {
    // 非同步代码（可并发执行）
    
    synchronized (this) { // 只锁当前账户
        synchronized (target) { // 再锁目标账户（防死锁要排序）
            // 转账核心代码
        }
    }
}
```
**比喻**：只锁涉及的两个保险柜（其他业务照常）→ **精准控制，效率高**

---

### **选择依据（性能 vs 方便）**
| **场景**                  | **方法修饰符**              | **代码块**                  |
|--------------------------|---------------------------|---------------------------|
| **锁的范围**              | 整个方法                   | 自定义代码片段              |
| **锁对象**                | 实例方法锁this，静态方法锁类 | 可指定任意对象（更灵活）     |
| **适用情况**              | 简单场景（方法全部需要同步） | 复杂逻辑（部分代码需同步）   |
| **性能**                  | 较差（锁范围大）           | 较好（锁范围小）           |
| **死锁风险**              | 高（容易无脑加锁）         | 可控（精确管理多把锁）      |

---

### **最佳实践**
1. **优先用代码块**：精确控制锁的范围，比如只锁共享变量相关操作
2. **锁对象用final**：避免锁对象被修改导致混乱
   ```java
   private final Object lock = new Object(); // 专用锁对象
   public void method() {
       synchronized(lock) { ... }
   }
   ```
3. **不要锁字符串/基础类型**：可能因JVM优化导致意外锁冲突

---

### **错误示例（极简版）**
```java
// ❌ 错误1：锁对象不明确
synchronized("LOCK") { ... } // 字符串常量可能被其他类共享

// ❌ 错误2：锁方法导致性能问题
public synchronized void hugeMethod() { 
    // 只有1行需要同步，其他99行白锁了！
}
```

---

**总结口诀**：  
小锁代码块，大锁伤性能，静态锁全局，对象锁实例！ 🔒
