

在 Java 中，**封装（Encapsulation）** 是面向对象编程的核心原则之一，但实际使用中需要注意一些关键细节，否则可能导致代码漏洞、维护困难或设计缺陷。以下是封装时需要特别注意的要点和常见陷阱：

### **1. 访问修饰符的合理使用**
- **原则**：**最小暴露原则**，只暴露必要的接口。
- **注意事项**：
  - **默认用 `private`**：所有字段默认设为 `private`，只有确定需要时才开放更宽松的权限。
  - **谨慎使用 `protected`**：`protected` 允许子类访问，但可能导致父类内部细节泄露，破坏封装性。
  - **避免滥用 `public`**：公共方法或字段会成为类的“承诺”，后续修改可能影响调用方。

**错误示例**：
```java
public class User {
    public String password; // 公共字段直接暴露，安全隐患！
}
```

**正确做法**：
```java
public class User {
    private String password; // 设为私有
    public void setPassword(String password) {
        // 可添加密码加密逻辑
        this.password = encrypt(password);
    }
}
```

---

### **2. Getter/Setter 的设计陷阱**
- **不要无脑生成 Getter/Setter**：不是所有字段都需要暴露，避免过度开放。
- **控制修改权限**：某些字段只读（只提供 Getter，不提供 Setter）。
- **添加逻辑保护**：在 Setter 中校验数据有效性。

**示例**：
```java
public class BankAccount {
    private double balance;

    // 只提供 Getter，禁止外部直接修改余额
    public double getBalance() {
        return balance;
    }

    // 存款方法代替 Setter，添加业务逻辑
    public void deposit(double amount) {
        if (amount > 0) {
            balance += amount;
        }
    }
}
```

---

### **3. 避免暴露可变对象的引用**
- **防御性拷贝（Defensive Copy）**：当返回对象引用时，返回拷贝而非原始对象，防止外部修改内部状态。

**错误示例**：
```java
public class Student {
    private List<String> courses = new ArrayList<>();
    
    // 直接返回内部集合引用，外部可修改内部数据！
    public List<String> getCourses() {
        return courses;
    }
}
```

**正确做法**：
```java
public class Student {
    private List<String> courses = new ArrayList<>();
    
    // 返回不可修改的副本
    public List<String> getCourses() {
        return new ArrayList<>(courses); // 防御性拷贝
    }
}
```

---

### **4. 继承对封装的破坏**
- **子类可能破坏父类封装**：子类可访问父类的 `protected` 字段，需谨慎设计。
- **解决方案**：
  - 尽量用 `private` 而非 `protected`。
  - 使用 `final` 限制可继承性。
  - 通过组合（Composition）代替继承。

**示例**：
```java
// 父类
public class Shape {
    protected int width; // 子类可修改，可能破坏逻辑
}

// 正确做法：通过方法控制访问
public class SafeShape {
    private int width;
    protected void setWidth(int width) {
        if (width > 0) this.width = width; // 添加校验
    }
}
```

### **5. 构造方法中的封装**
- **强制合法性**：构造方法应确保对象初始状态合法，避免创建非法对象。
- **避免在构造方法中调用可覆盖方法**（如非 `final` 或 `private` 方法），防止子类重写导致逻辑错误。

**错误示例**：
```java
public class Animal {
    public Animal() {
        init(); // 子类可能重写 init()，导致构造逻辑混乱
    }
    public void init() {}
}
```

---

### **6. 不可变对象（Immutable Objects）**
- **终极封装**：如果对象状态不需要改变，直接设计为不可变类（如 `String`）。
- **实现方式**：
  - 所有字段设为 `private final`。
  - 不提供 Setter。
  - 返回防御性拷贝。

**示例**：
```java
public final class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int getX() { return x; }
    public int getY() { return y; }
}
```

---

### **7. 使用 Lombok 时的注意事项**
- **不要依赖自动生成的 Setter**：Lombok 的 `@Setter` 会暴露所有字段的修改权限，需手动控制。
- **自定义逻辑覆盖**：如果需要校验逻辑，手动编写 Setter 并禁用 Lombok 生成。

**示例**：
```java
public class Product {
    @Getter // 自动生成 Getter
    private String id;
    
    private double price;

    // 手动编写带有校验的 Setter
    public void setPrice(double price) {
        if (price >= 0) {
            this.price = price;
        }
    }
}
```

---

### **8. 单元测试验证封装性**
- **测试字段不可直接访问**：确保测试代码通过公共方法操作对象，而不是反射暴力破解私有字段。
- **验证数据保护逻辑**：例如测试 Setter 的校验规则是否生效。

---

### **总结：核心原则**
1. **暴露最小接口**：只开放必要的操作。
2. **保持数据完整性**：通过校验和不可变性防止非法状态。
3. **防御性编程**：防止外部修改内部数据。
4. **谨慎处理继承**：避免子类破坏封装。
