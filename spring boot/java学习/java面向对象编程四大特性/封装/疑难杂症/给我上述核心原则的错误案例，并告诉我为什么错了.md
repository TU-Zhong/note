### **1. 错误：过度暴露字段（违反最小暴露原则）**
```java
public class User {
    public String username; // 公共字段直接暴露
    public String password; // 致命问题：密码可被外部直接修改！
}

// 外部代码可以随意修改密码
User user = new User();
user.password = "123456"; // 直接赋值，无任何校验
```
**为什么错**：  
- 字段直接暴露，外部代码可绕过业务逻辑（如密码加密）直接修改数据，导致安全隐患。
- 违背了“**最小暴露原则**”，破坏了封装性。

---

### **2. 错误：无脑生成 Getter/Setter**
```java
public class Order {
    private double totalAmount; // 订单总金额
    private List<String> items; // 订单项

    // 自动生成所有 Getter/Setter
    public void setTotalAmount(double totalAmount) {
        this.totalAmount = totalAmount;
    }
    public List<String> getItems() {
        return items;
    }
}

// 外部代码可以随意修改金额和订单项
Order order = new Order();
order.setTotalAmount(-100); // 金额为负数，非法！
order.getItems().clear();   // 直接清空订单项！
```
**为什么错**：  
- `totalAmount` 本应通过计算得出，允许直接设置会导致逻辑错误。
- 返回 `items` 集合的引用，外部可直接修改内部数据（如清空列表）。
- **正确做法**：  
  - 移除 `setTotalAmount`，通过业务方法（如 `addItem()`）计算金额。  
  - 返回集合的防御性拷贝（如 `new ArrayList<>(items)`）。

---

### **3. 错误：暴露可变对象的引用**
```java
public class Company {
    private List<Employee> employees = new ArrayList<>();

    // 直接返回内部集合的引用！
    public List<Employee> getEmployees() {
        return employees;
    }
}

// 外部代码直接修改内部数据
Company company = new Company();
company.getEmployees().add(new Employee()); // 看似方便，实则危险！
company.getEmployees().remove(0);           // 可能破坏业务逻辑！
```
**为什么错**：  
- 外部代码可直接修改 `employees` 集合，绕过了类内部的业务逻辑（如添加员工前的权限校验）。  
- **正确做法**：返回不可修改的副本或不可变视图（如 `Collections.unmodifiableList(employees)`）。

---

### **4. 错误：继承破坏封装**
```java
public class BankAccount {
    protected double balance; // 允许子类直接访问余额

    public void withdraw(double amount) {
        if (amount <= balance) {
            balance -= amount;
        }
    }
}

public class HackedAccount extends BankAccount {
    // 子类直接修改 balance，绕过了校验逻辑！
    public void hackBalance() {
        balance = 1000000; // 随意设置余额
    }
}
```
**为什么错**：  
- 父类的 `protected` 字段暴露给子类，子类可直接修改字段，导致余额被非法篡改。
- **正确做法**：将 `balance` 设为 `private`，子类必须通过父类方法操作数据。

### **5. 错误：构造方法中调用可覆盖方法**
```java
public class Animal {
    public Animal() {
        init(); // 调用可覆盖的方法
    }
    public void init() {
        System.out.println("Animal initialized");
    }
}

public class Cat extends Animal {
    @Override
    public void init() {
        System.out.println("Cat initialized");
    }
}

// 创建 Cat 实例时，输出 "Cat initialized"，但父类构造方法未完成！
// 可能导致子类依赖未初始化的字段
```
**为什么错**：  
- 父类构造方法调用被子类重写的 `init()`，此时子类尚未完全初始化，可能导致空指针或其他逻辑错误。  
- **正确做法**：将 `init()` 设为 `private` 或 `final`，禁止子类覆盖。

### **6. 错误：伪不可变对象**
```java
public class ImmutablePerson {
    private final String name;
    private final List<String> friends = new ArrayList<>();

    public ImmutablePerson(String name) {
        this.name = name;
    }

    public List<String> getFriends() {
        return friends; // 返回可变集合的引用！
    }
}

// 外部代码可以修改 friends 列表
ImmutablePerson person = new ImmutablePerson("Alice");
person.getFriends().add("Bob"); // 破坏了不可变性！
```
**为什么错**：  
- 虽然 `friends` 字段是 `final`，但返回的是其内部引用，外部仍可修改集合内容。  
- **正确做法**：返回防御性拷贝（如 `new ArrayList<>(friends)`）或不可变集合（如 `List.copyOf(friends)`）。

---

### **7. 错误：滥用 Lombok 的 @Setter**
```java
@Getter
@Setter // 自动生成所有 Setter
public class Product {
    private String id;
    private double price;
}

// 外部代码随意修改价格
Product product = new Product();
product.setPrice(-100); // 价格为负数，非法！
```
**为什么错**：  
- 使用 `@Setter` 自动生成 `setPrice()`，未添加价格校验逻辑，导致非法数据。  
- **正确做法**：手动编写 `setPrice()`，添加校验逻辑，并禁用 Lombok 的自动生成。

### **8. 错误：单元测试绕过封装**
```java
public class Account {
    private double balance;

    private void calculateInterest() {
        // 复杂计算逻辑
    }
}

// 测试代码使用反射强制调用私有方法
@Test
public void testInterest() throws Exception {
    Method method = Account.class.getDeclaredMethod("calculateInterest");
    method.setAccessible(true); // 暴力破解私有方法
    method.invoke(account);
}
```
**为什么错**：  
- 测试直接操作类的私有方法，导致测试与实现细节耦合。若未来私有方法被修改，测试会失败。  
- **正确做法**：通过公共方法间接测试私有逻辑，保持封装性。

---

### **总结**  
封装的核心是**控制访问权限**和**保护数据完整性**。上述错误案例的共同问题是：  
1. 暴露了不应暴露的数据或方法。  
2. 允许外部绕过业务逻辑直接修改状态。  
3. 忽略了继承带来的封装破坏风险。  

通过对比错误和正确做法，可以更深刻地理解封装的重要性。希望这些例子对你有帮助！ 😊
