好的，我们来详细说说**模板方法模式 (Template Method Pattern)**。

这是一种**行为型设计模式**，它在一个方法中定义一个算法的骨架（即模板），而将算法中某些可变步骤的实现延迟到子类中。模板方法使得子类可以在不改变算法整体结构的情况下，重新定义算法中的某些特定步骤。

---

### 核心思想与目的

- **定义算法骨架**：在抽象父类中定义一个“模板方法”，这个方法通常是 `final` 的（防止子类覆盖算法流程），它按顺序调用一系列基本操作（包括抽象方法和具体方法）。
- **延迟实现可变步骤**：将算法中会变化的部分（即基本操作）声明为抽象方法，强制子类去实现它们。
- **复用固定逻辑**：算法中固定不变的部分则在父类中直接实现。
- **控制子类扩展**：父类控制着算法的流程和结构，子类只能在父类规定的“坑”里（即实现的抽象方法或覆盖的钩子方法）填充具体逻辑。

可以把它想象成制作一道菜的食谱：

- **食谱（抽象父类）**：规定了做菜的整体步骤，比如“1. 准备食材；2. 切菜；3. 炒菜；4. 调味；5. 装盘”。这个整体流程是固定的（模板方法）。
- **具体步骤（抽象方法/钩子方法）**：
    - “准备什么食材？”、“怎么切？”、“用什么调料？”这些具体操作可能会因菜品不同而不同，食谱会把这些步骤留空，让具体的厨师（子类）去填写。
    - 有些步骤可能是可选的，比如“是否加辣椒？”（钩子方法，父类可以提供默认不加）。
- **具体的菜品（具体子类）**：比如“宫保鸡丁厨师”和“麻婆豆腐厨师”，他们都遵循食谱的大流程，但在“准备食材”、“切菜”、“调味”等具体步骤上会有自己的实现。

---

### 主要参与者

1. **抽象类 (AbstractClass)**：
    
    - 定义了一个或多个**抽象方法 (Primitive Operations)**，这些是算法中的可变步骤，由具体子类实现。
    - 实现了一个**模板方法 (Template Method)**，该方法定义了算法的骨架。模板方法会调用抽象方法以及其他在抽象类中已实现的方法（具体方法或钩子方法）。模板方法通常被声明为 `final`，以防止子类修改算法的执行顺序。
    - 可能还包含一些**具体方法 (Concrete Operations)**，这些是算法中固定不变的步骤，由父类直接实现，供模板方法调用。
    - 可能还包含一些**钩子方法 (Hook Operations)**，这些是父类中提供的具有默认实现（通常是空实现或返回默认值）的方法。子类可以选择性地覆盖这些钩子方法，以在算法的特定点插入自定义行为，从而影响算法的流程。
2. **具体类 (ConcreteClass)**：
    
    - 继承自抽象类。
    - 实现父类中声明的抽象方法，以完成算法中与特定子类相关的步骤。
    - 可以选择性地覆盖父类中的钩子方法。

---

### 工作流程

1. 客户端代码调用抽象类的模板方法。
2. 模板方法按照预定义的顺序执行算法的各个步骤。
3. 当执行到抽象方法时，会调用具体子类中实现的版本。
4. 当执行到钩子方法时，会调用子类覆盖的版本（如果被覆盖）或父类的默认版本。

---

### 优点

1. **代码复用**：将算法中不变的部分提取到父类的模板方法中，避免了在多个子类中重复编写相同的逻辑。
2. **封装变化**：将算法中可变的部分（通过抽象方法和钩子方法）隔离出来，由子类实现，使得系统更易于扩展和维护。
3. **控制反转 (Inversion of Control - IoC) / 好莱坞原则 ("Don't call us, we'll call you")**：父类（框架）调用子类（具体实现）的方法，而不是子类调用父类。这使得父类可以控制算法的整体流程。
4. **清晰的结构**：定义了算法的框架，使得整个算法的结构清晰易懂。

---

### 缺点

- **子类数量可能增多**：如果算法的可变步骤非常多，或者组合情况复杂，可能会导致需要创建大量的子类。
- **父类对子类的约束**：子类必须遵循父类定义的算法框架，可能在一定程度上限制了子类的灵活性。

---

### 适用场景

1. 当多个类有共同的行为，并且这些行为可以用一个统一的算法骨架来描述，但其中某些特定步骤在不同类中有不同的实现时。
2. 当需要控制子类的扩展时，可以通过将模板方法声明为 `final`，并提供受保护的抽象方法或钩子方法供子类实现。
3. 当重构代码时，如果发现多个类中存在重复的控制逻辑，可以考虑使用模板方法模式将其提取到父类中。

---

### Java 中的典型代码示例思路

Java

```
// 抽象类：AbstractBeverageMaker.java
abstract class AbstractBeverageMaker {

    // 模板方法，定义了制作饮料的整体流程
    // 通常声明为 final，防止子类覆盖算法骨架
    public final void prepareBeverage() {
        boilWater();         // 1. 烧水 (具体方法，所有饮料都需要)
        brew();              // 2. 冲泡 (抽象方法，具体饮料冲泡方式不同)
        pourInCup();         // 3. 倒入杯中 (具体方法)
        if (customerWantsCondiments()) { // 4. 是否加调料 (钩子方法，子类可选覆盖)
            addCondiments();     // 5. 加调料 (抽象方法，具体调料不同)
        }
        hookOperation();     // 6. 钩子操作 (可选，子类可覆盖以添加额外步骤)
    }

    // 具体方法
    private void boilWater() {
        System.out.println("烧开水");
    }

    private void pourInCup() {
        System.out.println("将饮料倒入杯中");
    }

    // 抽象方法，由子类实现
    protected abstract void brew();
    protected abstract void addCondiments();

    // 钩子方法，子类可以选择性地覆盖它
    // 默认返回 true，表示需要加调料
    protected boolean customerWantsCondiments() {
        return true;
    }

    // 另一个钩子方法，默认什么也不做
    protected void hookOperation() {
        // 空实现
    }
}

// 具体子类：CoffeeMaker.java
class CoffeeMaker extends AbstractBeverageMaker {
    @Override
    protected void brew() {
        System.out.println("用沸水冲泡咖啡粉");
    }

    @Override
    protected void addCondiments() {
        System.out.println("加糖和牛奶");
    }

    // 覆盖钩子方法，例如咖啡默认不加其他东西了
    @Override
    protected void hookOperation() {
        System.out.println("咖啡制作完毕，请慢用！");
    }
}

// 具体子类：TeaMaker.java
class TeaMaker extends AbstractBeverageMaker {
    @Override
    protected void brew() {
        System.out.println("用沸水浸泡茶叶");
    }

    @Override
    protected void addCondiments() {
        System.out.println("加柠檬");
    }

    // 覆盖钩子方法，例如茶不默认加调料
    @Override
    protected boolean customerWantsCondiments() {
        System.out.println("顾客是否需要为茶加调料？(yes/no)");
        // 简单模拟用户输入
        String answer = getUserInput(); // 假设这个方法获取用户输入
        return answer.toLowerCase().startsWith("y");
    }

    private String getUserInput() {
        // 模拟用户输入 "yes" 或 "no"
        // return "yes";
        return "no";
    }
}

// 客户端代码
public class TemplateMethodClient {
    public static void main(String[] args) {
        System.out.println("制作咖啡...");
        AbstractBeverageMaker coffeeMaker = new CoffeeMaker();
        coffeeMaker.prepareBeverage();

        System.out.println("\n制作茶...");
        AbstractBeverageMaker teaMaker = new TeaMaker();
        teaMaker.prepareBeverage();
    }
}
```

在这个例子中：

- `AbstractBeverageMaker` 是抽象类，`prepareBeverage()` 是模板方法。
- `boilWater()` 和 `pourInCup()` 是具体方法。
- `brew()` 和 `addCondiments()` 是抽象方法，必须由子类实现。
- `customerWantsCondiments()` 和 `hookOperation()` 是钩子方法，子类可以按需覆盖。
- `CoffeeMaker` 和 `TeaMaker` 是具体子类，它们实现了抽象方法并可能覆盖了钩子方法，从而定制了各自的饮料制作过程，但整体流程（烧水、冲泡、倒杯、加调料）是由父类的模板方法控制的。

Java 框架中也广泛使用模板方法模式，例如 `java.io.InputStream` 中的 `read(byte b[], int off, int len)` 方法内部会调用抽象的 `read()` 方法，Servlet 中的 `HttpServlet` 的 `service()` 方法会根据 HTTP 请求类型调用 `doGet()`、`doPost()` 等方法。
