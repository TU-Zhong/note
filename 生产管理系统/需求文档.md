# [PRD] 全域工业知识图谱引擎 (IKGE) 生产级需求文档

**文档版本：** v3.0 (Enterprise Edition)

**状态：** 评审准备中

**核心理念：** 元数据驱动、全域互联、异步协同、安全隔离。

---

## 1. 文档概览与背景

### 1.1 需求背景

- **业务痛点：** 1. **信息孤岛：** 生产、采购、销售数据存在于不同 ERP/MES 中，无法进行跨部门的因果溯源（如：客诉如何关联到具体原材料批次）。
    
    2. **隐性知识流失：** 老员工的维修、谈判、避坑经验随离职消失，缺乏结构化沉淀。
    
    3. **协作冲突：** 多人同时维护设备文档时，容易发生覆盖导致数据丢失。
    
- **目标用户：** 全厂人员。包括一线工人（记录者）、班组长/主管（审核者）、决策层（溯源分析者）。
    

### 1.2 核心目标

- **知识资产化：** 实现 100% 物理资产（设备/SN）与逻辑资产（SOP/合同）的图谱挂载。
    
- **溯源效率：** 跨部门（如销售到采购）的质量溯源时间从“天级”降低至“秒级”。
    
- **激励闭环：** 通过积分榜，使员工主动参与知识贡献率提升 30% 以上。

## 2. 流程图示 (Logic Flow)

### 2.1 核心主路径：知识协同与图谱点亮

1. **建模阶段：** 超级管理员在后台定义“实体模型（Schema）”与“关系约束”。
    
2. **提报阶段：** 员工 A 在 Wiki 编辑页面修改内容并新增关联（如：该设备依赖某零件）。
    
3. **冲突检测：** 系统检测当前基线版本。若 A 提交时，已有他人修改并发布，触发 **Reviewer-side Merge**。
    
4. **审核阶段：** 部门主管在待办池进行 Diff 审计，执行“通过/驳回/修剪后通过”。
    
5. **落库阶段：** 审核通过 $\rightarrow$ PostgreSQL 更新 Wiki 快照 $\rightarrow$ Neo4j 同步建立关联边 $\rightarrow$ 增加个人积分。
    

### 2.2 逆向与异常流程

- **驳回流程：** 审核不通过 $\rightarrow$ 文档退回草稿箱 $\rightarrow$ 员工收到站内信告知驳回理由。
    
- **恶意惩罚：** 审核员勾选“恶意提交” $\rightarrow$ 系统执行扣分并限制该用户 24 小时内的提交权。
    
- **断网/超时：** 前端本地缓存草稿，待网络恢复后自动重试并校验版本号。

## 3. 功能详细说明 (Functional Requirements)

### 3.1 模块 A：元数据驱动引擎 (The Lego Base)

- **前置条件：** 管理员权限登录。
    
- **字段定义：**
    
    - `Entity_Type`: String (20), 唯一标识（如 Machine, Supplier）。
        
    - `Properties`: JSONB, 动态属性列表（名称、类型、必填项、脱敏等级）。
        
    - `Relations`: Array, 允许的连线目标及类型。
        
- **后端规则：** 采用 Schema-on-Read 模式。所有的动态字段必须在 PostgreSQL 的 JSONB 字段中索引化，确保查询效率。
    

### 3.2 模块 B：跨部门协同 Wiki (Collaborative Wiki)

- **页面元素：**
    
    - **左侧：** Markdown 实时预览编辑器，支持图片/视频拖拽上传至 MinIO。
        
    - **右侧：** 局部拓扑图预览，展示当前节点及其 1 度关联。
        
- **交互逻辑：** * **@ 引用：** 输入 `[[` 触发实体搜索弹窗，选择后建立图谱连线意图。
    
- **后端规则：** 严格执行 **乐观锁 (Optimistic Locking)**。提交请求必须携带 `base_version`。若 `db_version > base_version`，接口返回 409 错误。
    

### 3.3 模块 C：单级审批工作流 (Approval Workflow)

- **前置条件：** 用户具备当前部门的 `REVIEWER` 角色。
    
- **审核逻辑：**
    
    - **Diff 视图：** 系统必须以高亮方式（绿色加、红色减）展示本次修改与当前线上版本的差异。
        
    - **修剪发布：** 审核员可直接在审核界面修改工人的错别字或删除涉密图片后再发布。

## 4. 异常处理 (Exception Handling)

|**异常场景**|**处理规则**|
|---|---|
|**并发冲突**|审核员通过 A 时，若 B 也在审，B 提交通过时，系统自动执行 `Merge`。若冲突无法自动解决，弹窗要求审核员手动选择。|
|**空状态展示**|首次进入无 Wiki 的设备页面时，展示“该资产尚未录入经验，点击创建”引导。图谱区展示孤立节点。|
|**权限限制 (ABAC)**|非采购部人员查看“供应商”Wiki 时，系统自动过滤标记为 `Clearance_Level > 1` 的 JSON 字段。|
|**网络波动**|上传大型视频文件时，采用 **MinIO 断点续传**，防止车间 Wi-Fi 信号不稳定导致失败。|

## 5. 非功能需求

### 5.1 数据埋点

- **PV/UV：** 统计各部门最活跃的 Wiki 词条（评估设备关注度）。
    
- **转化路径：** 统计从“扫码”到“成功提交修改”的漏斗转化率。
    
- **积分分布：** 统计全厂月度积分均值，辅助绩效考核。
    

### 5.2 性能要求

- **图谱查询：** Neo4j 执行 3 层关系下钻耗时必须 $\le 300ms$。
    
- **并发支撑：** 支持全厂 500 人同时在线（考虑到工厂倒班制，瞬时并发约 50-100）。
    

### 5.3 安全性

- **数据脱敏：** 核心参数、采购单价等字段在 API 层进行动态过滤，严禁明文传输至前端后再隐藏。
    
- **存储加密：** MinIO 存储的附件采用私有读协议，通过后端生成有时效的临时签名 URL 进行访问。
