这是一个非常好的问题！它正好触及了 Python 设计哲学的核心。

答案是：**并非所有内置类都实现了所有操作符，而是每个类都根据自身的“天性”和用途，实现了逻辑上合理且有意义的操作符。**

Python 的设计哲学是实用主义和“符合直觉”。一个操作符是否被实现，取决于这个操作对于该数据类型来说是否有明确、无歧义的意义。

---

### 核心原则：“操作要有意义”

如果一个操作对于某个数据类型来说没有直观的意义，或者可能产生歧义，Python 倾向于不实现它，从而避免混淆。

我们来看一个包含“支持”和“不支持”的对比表格，您会立刻明白这个设计思想。

### 深入理解几个有趣的设计选择

#### 1. 为什么 `set` (集合) 不支持 `+` 操作符？

- **问题**：`+` 通常意味着“加法”或“拼接”，但对于集合来说，最常见的操作是“并集 (union)”。
    
- **设计决策**：Python 的设计者认为，使用 `+` 来表示并集可能会让不熟悉集合论的人感到困惑。因此，他们选择了一个在数学上更明确无误的操作符：`|` (管道符)。
    
    Python
    
    ```
    set1 = {1, 2, 3}
    set2 = {3, 4, 5}
    
    # 正确的求并集方式，使用 | 操作符
    union_set = set1 | set2  # 结果是 {1, 2, 3, 4, 5}
    
    # 如果用 + 会直接报错 TypeError
    # error_set = set1 + set2 
    ```
    
    这样做让代码的意图更加清晰：看到 `|` 就知道是在做集合运算。
    

#### 2. 为什么 `dict` (字典) 以前不支持 `+`，现在用 `|`？

- **问题**：合并两个字典比拼接列表复杂。如果两个字典有相同的键，值应该怎么办？是前者覆盖后者，还是后者覆盖前者，或者报错？
    
- **历史决策**：因为存在歧义，很长一段时间里 Python 都不支持用操作符来合并字典。你需要调用 `.update()` 方法。
    
- **现代演进 (Python 3.9+)**：社区最终决定，像 `set` 一样，为字典引入 `|` 操作符来表示合并（后面的字典会覆盖前面的同名键）。
    
    Python
    
    ```
    dict1 = {'a': 1, 'b': 2}
    dict2 = {'b': 3, 'c': 4}
    
    # 在 Python 3.9+ 中，可以用 | 合并字典
    merged_dict = dict1 | dict2  # 结果是 {'a': 1, 'b': 3, 'c': 4}
    ```
    

#### 3. 为什么 `int` (整数) 没有 `len()`？

这看起来很明显，但它完美地印证了“操作要有意义”的原则。`len()` 函数的意义是“获取一个容器中元素的数量”。而一个整数（比如 `123`）是一个单一的、不可分割的值，它不是一个容器，所以它没有“长度”。

### 总结

所以，您的问题的答案可以总结为：

**Python 的内置类并非“为了统一而统一”地实现所有操作符。它们是在“有意义、无歧义”的前提下，有选择性地、深思熟虑地实现了相应的特殊方法。**

这种设计哲学使得 Python 既灵活强大，又保持了代码的清晰和可预测性。当你使用一个操作符时，它的行为通常和你对该数据类型的直觉是一致的。
