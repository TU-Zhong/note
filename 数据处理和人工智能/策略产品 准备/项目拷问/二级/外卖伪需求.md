### 场景一：针对“Redis 实现购物车”的极限施压

- **【业务方提出的需求】**
    
    “运营总监说，为了防止用户流失，我们要把C端用户的购物车数据**永久保存**。即使用户卸载了App，一年后再重新登录，他去年加进购物车的黄焖鸡米饭也必须还在！”
    
- **【可行性与伪需求分析（5 Whys）】**
    
    - **可行性：** 极低。Redis 是内存数据库，极其昂贵。把上百万僵尸用户的历史购物车数据全放在 Redis 里做持久化，不仅浪费极高，而且一旦宕机恢复极慢。
        
    - **伪需求：** 外卖是一个**“即时性、高冲动、短周期”**的消费场景。用户点外卖是为了立刻吃，不是像淘宝买衣服那样用来做备忘录的。一年前的黄焖鸡店铺可能早就倒闭了，价格也变了，保留下来毫无业务价值。
        
- **【如何验证与最终方案】**
    
    - **验证思路（数据打点）：** 拉取数据库历史日志，统计当前购物车数据的“生命周期”。计算一下加购后超过 24 小时未下单的购物车，最终的转化率是多少？（大概率趋近于0）。
        
    - **你的破局方案：** 拒绝永久保存。对于未登录的游客，购物车暂存本地缓存（Local Storage）；对于登录用户，存入 Redis 但设置 **TTL（过期时间，如 3 天）**。用低成本的技术手段匹配真实的外卖业务节奏。
        

### 场景二：针对“MyBatis 复杂条件查询”的性能博弈

- **【业务方提出的需求】**
    
    “老板觉得用户挑菜品太麻烦了。要求在C端首页加一个‘超级筛选器’：要求能同时根据【距离我 500米内 + 销量前十 + 口味偏辣 + 碳水低于 300g + 正在做秒杀促销】进行多表联合精准查询。”
    
- **【可行性与伪需求分析（5 Whys）】**
    
    - **可行性：** 灾难级。MyBatis 写个五六张表的连表查询不难，但在外卖午高峰的极高并发下，这种全表扫描+多维排序的动态 SQL 会瞬间把 MySQL 数据库的 CPU 打满，导致整个系统雪崩。
        
    - **伪需求：** 真的有吃货点个外卖要设 5 个极其苛刻的条件吗？“饥饿的用户没有耐心”。
        
- **【如何验证与最终方案】**
    
    - **验证思路（假门测试）：** 不要直接开发这个复杂的 SQL 逻辑。先在前端做一个只有按钮但没有实际功能的“超级筛选”入口。统计有多少人会点进去？如果点击率不到 1%，直接砍掉需求。
        
    - **你的破局方案：** 引导业务方做“降级”。用 Elasticsearch 做离线的标签倒排索引，或者退一步，只保留最核心的“距离+销量”排序，其它的口味偏好通过首页的个性化推荐信息流（Feed流）来解决，而不是让用户自己去搜。
        

### 场景三：针对“JWT 认证与权限”的商业逻辑陷阱

- **【业务方提出的需求】**
    
    “风控部门说，为了防止‘账号共享’，要在拦截器里加一个强校验：**C端用户只允许单设备登录。如果在 iPad 上登录了，手机端立刻被强制踢下线（类似微信的逻辑）。**”
    
- **【可行性与伪需求分析（5 Whys）】**
    
    - **可行性：** 技术上完全可行。利用 Redis 存储用户 ID 和最新设备的 Token，拦截器每次比对即可。
        
    - **伪需求：** 外卖平台的核心商业模式是什么？是卖出更多的饭！账号共享对平台不仅没有损失（不像 Netflix 会损失订阅费），反而如果全家共用一个账号点餐，还能增加平台 GMV 和用户粘性。为什么要花研发成本去阻止用户给你送钱？
        
- **【如何验证与最终方案】**
    
    - **验证思路（探寻根本动机）：** 对风控提 5 Whys：“为什么要防账号共享？” $\rightarrow$ “因为怕羊毛党用小号批量领新人券，然后把账号卖给别人。”
        
    - **你的破局方案：** 既然痛点是“领券防刷”，那我就不该限制【登录模块】的设备数。正确的做法是在【领券/下单模块】增加校验机制（同设备号/同微信号 限制领券），既不影响正常用户多设备点餐体验，又封死了黑产的羊毛路径。
        

### 场景四：针对“AOP 公共字段自动填充”的衍生需求

- **【业务方提出的需求】**
    
    “B端商家经常手滑误删菜品。为了防止客诉，商家端运营要求：**给所有的删除操作加一个‘一键撤回（Undo）’功能，允许商家在 24 小时内无条件恢复被删的菜品和价格。**”
    
- **【可行性与伪需求分析（5 Whys）】**
    
    - **可行性：** 开发一套完整的“快照与回滚”系统（Event Sourcing）成本极高。
        
    - **伪需求：** 商家真的需要高大上的“一键撤回时间机器”吗？他们只是怕“点错”。
        
- **【如何验证与最终方案】**
    
    - **验证思路（低保真原型）：** 在给商家加上沉重的回滚功能前，先在前端加一个简单的二次确认弹窗：“您正在删除招牌菜【农炒肉】，删除后历史销量将清零，是否确认？”看看误删率是否能下降 80%。
        
    - **你的破局方案：** 结合你项目里的 AOP 和 MyBatis 插件机制，向面试官展示优雅的底层解法：“我不需要做复杂的撤回功能。我可以在后端利用 AOP 和 MybatisPlus，把所有物理删除 `DELETE` 操作，在底层自动拦截并转换为逻辑删除 `UPDATE is_deleted = 1`。这样前端虽然删除了，但在数据库里数据还在。如果真有重大误操作，运营可以找 DBA 提工单恢复，既满足了安全兜底，又没增加前端的开发负荷。”
        

### 场景五：针对“Redis 缓存热点数据”的过度设计

- **【业务方提出的需求】**
    
    “为了营造双十一那样的抢购紧迫感，营销部门要求：**在C端首页，给每一个热卖套餐加上精确到秒的‘剩余库存倒计时’（比如：仅剩 15 份，数字实时跳动）。**”
    
- **【可行性与伪需求分析（5 Whys）】**
    
    - **可行性：** 极差。如果你为了一个花哨的倒计时，让几十万C端用户的手机每秒钟轮询一次后端的 Redis 去查库存，哪怕是 Redis 也会被巨大的 QPS（每秒查询率）瞬间打满击穿。
        
    - **伪需求：** 外卖不是茅台，不存在全网几百万人抢 10 份盒饭的场景。库存通常只管“有”和“无”。
        
- **【如何验证与最终方案】**
    
    - **验证思路（A/B 测试）：** 对一批用户展示静态的标签“热销中！库存紧张”，另一批用户什么都不加。看看光靠一个静态图片（不需要后端查库存）是否已经足够提升转化率了？
        
    - **你的破局方案：** 作为懂架构的 PM，坚决驳回实时的秒级库存轮询。建议改为**“阶梯式缓存延迟展示”**：正常浏览时，Redis 里的库存每 5 分钟同步一次到本地缓存；只有当用户真正点击【加入购物车】那一瞬间，系统才去后端的 Redis 或 MySQL 做精确的库存扣减校验。用异步的思路解决性能与营销的矛盾。
