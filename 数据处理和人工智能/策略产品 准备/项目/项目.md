### 1. 化解高频交互与网络开销的矛盾：前端 Diff Pool 与脏检查机制

- **业务痛点 (Situation)：** 传统的 Web 增删改，往往是修改一行就要发一次请求，这在面向数据库开发的高频场景下体验极差；如果仅在前端做数组替换，又很容易产生“假修改”导致无意查页面义的网络开销，甚至误覆盖真实数据。
    
- **定位与分析 (Task/Analysis)：** 我意识到，要实现媲美桌面端 DataGrip 的“双击内联编辑”体验，核心难点在于**精准追踪数据的生命周期状态（新增、修改、删除）**，并过滤掉无效的修改。
    
- **解决策略 (Action)：** 我首创了前端 **Diff Pool（差异状态追踪池）**。在数据拉取时生成深度快照（Deep Copy）；在用户输入失焦时，底层引擎会进行严格的**脏检查（Dirty Check）**。特别针对 JavaScript 中弱类型隐式转换的坑（如 `null`、`undefined` 和空字符串），我设计了全字段转 String 类型的像素级比对算法，确保只有真正改变的数据才会进入 `updated` 池。
    
- **成果与复盘 (Result)：** 实现了纯前端的“假删”和行级变色预警，将所有的改动打包在点击“提交”时一次性合并发送。**复盘来看**，目前的深拷贝使用的是 `JSON.parse`，在遇到超大表或特殊数据类型（如 Blob/Date）时存在性能瓶颈，后续我计划引入 `lodash.cloneDeep` 或 `structuredClone` 来进一步优化。
    

### 2. 攻克巨型“上帝组件”带来的状态污染：基于 Pinia 与 Hooks 的高可用架构

- **业务痛点 (Situation)：** 在项目初期，由于交互极其复杂，侧边栏（包含数百行的树形数据渲染、弹窗、右键菜单）和工作区（多表数据表格）耦合严重，导致代码极难维护，且出现了“在这个标签页修改数据，影响了另一个标签页”的严重 Bug。
    
- **定位与分析 (Task/Analysis)：** 通过 Debug 排查，我定位到问题的根源在于**状态隔离失效**和**逻辑揉杂**。Vue 3 原生的响应式变量在多标签页场景下没有被正确地划分作用域。
    
- **解决策略 (Action)：** 我对系统架构进行了“刮骨疗毒”般的重构。
    
    1. 引入 **Pinia** 构建多标签页的内存与状态隔离架构，确保多表并发操作互不干扰。
        
    2. 运用**“高内聚、低耦合”**原则，将巨型组件拆解。例如，将侧边栏拆分为纯 UI 层的 `useSidebarResize`、交互层的 `useContextMenu`，以及核心网络与数据层的 `useDatabaseTree`。
        
- **成果与复盘 (Result)：** 彻底解决了多标签页状态污染问题。各个 Hook 独立负责单一职责，视图组件退化为极轻量的“调度中心”，系统可维护性和可测试性呈指数级上升。
    

### 3. 突破单一数据源限制：底层 JDBC 驱动包的动态隔离加载

- **业务痛点 (Situation)：** 传统的后端应用通常在 POM 文件中写死数据库驱动（如 MySQL 8）。但作为一个数据库 IDE 平台，用户可能会同时连接 MySQL 5、MySQL 8 甚至 SQL Server，这会导致极其严重的类冲突（Class Conflict）或“找不到驱动”的致命报错。
    
- **定位与分析 (Task/Analysis)：** 深入 JVM 的类加载机制后，我发现默认的 `AppClassLoader` 是全局单例的，无法同时加载同名但不同版本的驱动类。
    
- **解决策略 (Action)：** 我深入底层 JDBC 规范，放弃了传统的直连模式。采用反射机制配合自定义的 `URLClassLoader`，为每一个不同版本的数据库驱动包分配一个独立的类加载器实例，实现了**物理级别的类隔离加载**。
    
- **成果与复盘 (Result)：** 成功支持了多类型、多版本数据库的动态接入，用户只需提供驱动 jar 包路径即可建立连接。**复盘反思**：在动态加载类后，如果没有正确释放资源，容易引发 JVM 的 PermGen/Metaspace 内存泄漏。目前我在设计驱动卸载模块，确保不用的 ClassLoader 能够被 GC 回收。
    

### 4. 守卫数据更新的准确性：精准的 SQL 条件组装与主键溯源

- **业务痛点 (Situation)：** 在将前端的 Diff Pool 数据提交给后端执行 `UPDATE` 或 `DELETE` 操作时，如果 SQL 语句的 `WHERE` 条件不严谨，极易造成“误伤”（即改了第一行，结果全表同名的数据都被改了）。
    
- **定位与分析 (Task/Analysis)：** 我分析发现，难点在于前端需要动态适应“有主键表”和“无主键表”两种极端情况，且不能手动修改数据库自动维护的自增列（Identity）。
    
- **解决策略 (Action)：** 我在前后端协同上设计了严格的防御机制。
    
    1. 前端提取元数据，对于自增列直接拦截双击编辑并给出弹窗警告。
        
    2. 对于提交的 `oldRow`（修改前的数据），如果表存在主键，前端只提取主键传给后端，实现最高效的索引更新；如果无主键，则传入所有旧字段值，后端遍历拼接 `WHERE col1=? AND col2=?` 进行精确匹配兜底。
        
- **成果与复盘 (Result)：** 构建了极具质感的安全交互体验，100% 避免了业务数据的误更新。
    

### 5. 构建 Web 端多库直连的信任链路：基于 JWT 的多级会话隔离

- **业务痛点 (Situation)：** 在传统的单体应用中，一个 Session 对应一个用户。但在我们的 IDE 中，一个用户（浏览器）可能同时开着 3 个标签页，分别连着本地测试库、公司开发库和云服务器库。如何确保查询请求不会发错数据库？
    
- **定位与分析 (Task/Analysis)：** 前端必须精准记住“哪一个 Tab 页、哪一个节点对应着哪一个数据库连接的鉴权凭证”，且决不能将数据库账号密码明文保留在前端。
    
- **解决策略 (Action)：** 我设计了一套多级会话隔离方案。
    
    1. 后端在连接成功后，签发包含该特定数据库会话标识的 **JWT Token**。
        
    2. 前端侧边栏树形节点使用 `dbId` 前缀进行绝对溯源。每次双击打开新标签页时，都会顺着组件树向上寻找到所属数据库的专属 Token，并将其注入到 Pinia 的 Tab 参数中。
        
    3. 结合 Axios 拦截器，在每次请求时动态携带当前激活 Tab 的特定 Token。
        
- **成果与复盘 (Result)：** 保障了多库连接凭证的安全传输与会话隔离，用户在不同数据库实例间无缝切换，请求精准送达。
