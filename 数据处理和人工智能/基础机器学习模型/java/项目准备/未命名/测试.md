### 🧪 项目全链路测试清单

#### 模块一：动态多数据源与底层元数据 (JDBC & 类加载)

_测试重点：不同数据库方言的兼容性、类隔离、大数据量渲染_

- [ ] **多环境驱动加载测试**
    
    - [ ] 通过 `URLClassLoader` 动态引入 MySQL 8.x 和 MySQL 5.x 的驱动包，测试能否隔离加载，确保不会报 `ClassNotFoundException` 或引发类冲突。
        
    - [ ] 动态加载其他异构数据库（如 PostgreSQL, SQL Server）的驱动并尝试连接。
        
- [ ] **元数据提取精准度验证**
    
    - [ ] 针对包含复杂数据类型（如 JSON, BLOB, DateTime）的表，提取的列属性是否完全准确。
        
    - [ ] 复合主键、外键约束、唯一索引、普通索引等层级关系，是否能在左侧元数据树形视图中精确还原。
        
- [ ] **极限性能与稳定性测试**
    
    - [ ] 连接一个包含海量表（如 500+ 张表或数千个字段）的数据库，测试树形组件的渲染时间。是否存在卡顿？（可作为后续引入虚拟列表 Virtual List 的优化回顾点）。
        

#### 模块二：数据工作台与多标签页隔离 (Pinia 状态架构)

_测试重点：状态不串台、内存管理、并发操作_

- [ ] **多标签页并发隔离**
    
    - [ ] **交叉操作测试：** 打开表 A 的视图并在第 3 行修改数据（触发 Diff Pool），切换到表 B 的视图进行过滤搜索。再切回表 A，检查第 3 行的修改状态和高亮变色是否完好无损。
        
    - [ ] **SQL 控制台并发：** 在控制台 Tab 1 执行一个需耗时 5 秒的慢查询，立刻切到控制台 Tab 2 执行另一个查询。验证结果是否准确返回到各自的 Tab，没有发生数据串台。
        
- [ ] **生命周期与内存释放**
    
    - [ ] 连续打开 20 个表视图标签页，然后全部关闭。打开浏览器 DevTools 的 Memory 面板，触发垃圾回收（GC），观察 Pinia 中缓存的表数据和 Diff Pool 状态是否被彻底清理，验证是否存在内存泄漏。
        

#### 模块三：Diff Pool 核心引擎与数据编辑 (前端核心交互)

_测试重点：响应式精度、脏检查拦截、沉浸式编辑体验_

- [ ] **沉浸式双击内联编辑**
    
    - [ ] 双击任意单元格，能否顺畅唤起输入框；点击输入框外部（失去焦点）或按 Enter 键，能否自动保存到内存并退出编辑模式。
        
- [ ] **智能脏检查与无效更新拦截（重点）**
    
    - [ ] **原值复原测试：** 双击单元格将原值 `Apple` 改为 `Banana`（行变色预警触发），再次双击改回 `Apple`。系统应当通过深拷贝快照比对，智能发现数据未实质变更，自动**撤销**行变色预警并将其移出 Diff Pool 更新队列。
        
- [ ] **三态追踪（新增 / 修改 / 假删）**
    
    - [ ] **状态叠加冲突：** 点击“新增一行”（绿色预警），在该行输入数据，随后立刻选中该行点击“删除”。系统应直接将其从视图中抹去（因为还未入库），而不是加入假删队列。
        
    - [ ] **存量数据假删：** 选中一条原本存在的数据库记录点击“删除”。该行应被打上删除线或红色高亮（假删），且**绝对不能**在此时向后端发送 DELETE 接口。
        
- [ ] **批量同步与 Payload 精度**
    
    - [ ] 在当前视图中混合进行：新增 2 行、修改 3 行、假删 1 行。点击“提交同步”按钮，抓包查看 Axios 请求。
        
    - [ ] 验证发往后端的 JSON 快照是否**极其干净**（仅包含发生变动的这 6 行数据，而非整个表格），且分类明确（InsertList, UpdateList, DeleteList）。
        

#### 模块四：安全隔离与 RESTful 交互 (Axios & JWT)

_测试重点：连接凭证隔离、异常捕获_

- [ ] **多库凭证路由安全**
    
    - [ ] 连接数据库 DB1（生成 JWT-1）和 DB2（生成 JWT-2）。
        
    - [ ] 在 DB1 的标签页执行查询，查看 Axios 请求头，确保携带的是 JWT-1。
        
    - [ ] 在 DB2 的标签页执行操作，确保 Axios 拦截器能动态获取当前激活 Tab 的上下文，并准确切换为 JWT-2（防止跨库越权操作）。
        
- [ ] **拦截器全局异常处理**
    
    - [ ] 模拟数据库断开连接或 JWT 过期，触发任意操作。验证前端能否全局捕获 `401 Unauthorized` 或数据库异常，并弹出友好的桌面级 Notification/Message 提示，而不是导致页面白屏或崩溃。
        

#### 模块五：桌面级 UI/UX 质感体验 (Element Plus 深度定制)

_测试重点：视觉反馈、人机交互_

- [ ] 窗口缩放适配：改变浏览器窗口大小，IDE 的侧边栏、工具栏、数据表格区域的 Flex 布局是否自适应，表格是否出现合理的原生滚动条而没有破坏整体结构。
    
- [ ] 交互细节：按钮的 Loading 状态（提交数据时防止连点）、单元格编辑时的边界处理、右键菜单的定位与自动消失机制是否符合真实桌面 IDE 的直觉。
    

#### 模块六：SQL 注入防御与底层数据安全 (安全攻防测试)

_测试重点：动态 SQL 拼接引擎的安全过滤、标识符转义、恶意 Payload 拦截_

- [ ] **DDL 标识符注入拦截测试（测试 `quoteIdentifier` 引擎）**
    
    - [ ] **极端命名测试：** 尝试在新建表或新建列时，输入纯数字名称（如 `12345`）、包含空格的名称（如 `my new table`）以及 SQL 关键字（如 `SELECT`, `TABLE`）。系统应能通过方括号 `[]` 等安全转义机制，成功建表且不报语法错误。
        
    - [ ] **闭合字符攻击：** 尝试将列名或表名命名为 `]; DROP TABLE users;--` 或包含单引号 `'` 的字符串。验证后端动态拼接 DDL 语句时，是否能正确转义（如将 `]` 替换为 `]]`），确保这些恶意字符只被当作普通的物理名称，而不会截断执行逻辑。
        
- [ ] **DML 数据流注入拦截测试（测试 Diff-Pool 同步安全性）**
    
    - [ ] **万能密码/越权尝试：** 在表数据视图中新增一行，在 `varchar` 类型的单元格中输入经典的注入 Payload，如 `' OR '1'='1` 或 `'; DELETE FROM table_name;--`。点击提交后，验证底层是否使用了 `PreparedStatement` 进行参数化预编译绑定，确保这些字符串被原样作为数据存入数据库，而非被当作 SQL 指令执行。
        
    - [ ] **类型溢出与畸形数据：** 尝试在 `int` 类型的列中提交超长数字或特殊符号，测试前端表单与后端的双重校验机制能否有效拦截并返回友好的错误提示，防止底层数据库直接抛出原始的 SQL Exception 暴露系统信息。
        
- [ ] **后端会话与越权操作拦截**
    
    - [ ] **越权执行测试：** 抓包拦截一个对 `表 A` 执行数据修改的合法请求，将其 Payload 中的 `tableName` 强行篡改为 `表 B`。系统（JWT 解析与 dbTokenMap）应能正确验证当前连接凭证是否有权操作目标表，或通过严密的上下文校验拒绝该越权请求。

### 💡 答辩与简历亮点话术（安全篇）

在介绍项目的安全性时，你可以这样总结：

> "考虑到传统 ORM 无法满足动态未知的表结构操作，本系统基于原生 JDBC 构建了**动态 DDL/DML 生成引擎**。针对最严峻的 SQL 注入风险，我在底层实现了基于 `quoteIdentifier` 的标识符安全隔离机制，并配合 `PreparedStatement` 预编译参数绑定。这套防御体系不仅彻底解决了纯数字列名、特殊关键字导致的语法崩溃，更从根本上阻断了 DDL 阶段的恶意闭合攻击和 DML 阶段的逻辑注入，达到了工业级数据管理工具的安全标准。"
