好的，我们来对这篇文章进行评价，并给出一套更现代化、更可靠的死锁排查通用方法。

### 对文章的评价

这是一篇不错的入门级文章，为初学者介绍了几个用于实时查看 SQL Server 状态的工具。但从专业死锁排查的角度来看，它存在一些**核心的误导性**和**方法上的过时**问题。

#### 优点 (Strengths)

1. **结构清晰**：文章按照“识别 -> 获取信息 -> 定位语句 -> 解决”的逻辑组织，易于理解。
    
2. **工具入门**：介绍了 `DBCC INPUTBUFFER` 和 `KILL` 这两个在紧急情况下确实会用到的命令。
    
3. **代码直观**：提供的代码示例简单明了，可以直接运行。
    

#### 主要问题与改进建议 (Major Issues and Suggestions for Improvement)

1. **【核心问题】混淆了“阻塞 (Blocking)”与“死锁 (Deadlock)”**
    
    - **文章的原理是错误的**：文章第二节的核心查询 `SELECT ... FROM sys.dm_tran_locks WHERE resource_type = 'OBJECT'` **查找的是锁和阻塞，而不是死锁**。
        
    - **阻塞 (Blocking)**：进程 A 持有资源 X 的锁，进程 B 想要访问资源 X，于是 B **等待** A 释放锁。这是一个线性的等待关系（A -> B）。
        
    - **死锁 (Deadlock)**：进程 A 等待进程 B 的资源，同时进程 B 也在等待进程 A 的资源。这是一个**循环等待**（A -> B -> A），谁也无法继续，导致僵局。
        
    - **结论**：按照文章的方法，您找到的只是一些持有锁的会话（SPID），它们可能是阻塞者，但**几乎可以肯定不是一个完整的死锁场景**。直接 `KILL` 这些 SPID 可能会误杀无辜的进程。
        
2. **使用了过时或不推荐的工具**
    
    - `sp_who` 和 `sp_lock` 是非常古老的系统存储过程，从 SQL Server 2005 开始就有了功能更强大、信息更准确的动态管理视图 (DMV) 来替代它们。
        
    - **`sp_who` 的替代品**：`sys.dm_exec_sessions` 和 `sys.dm_exec_requests`，它们提供了更丰富的会话和请求信息。
        
    - **`sp_lock` 的替代品**：`sys.dm_tran_locks`（文章中也用到了，但用法不对）和 `sys.dm_os_waiting_tasks`（可以清晰地看到谁在等待什么资源）。
        
3. **缺乏捕获和根源分析的根本方法**
    
    - 文章的方法是“事中”处理，即在问题发生时手动去查。但死锁往往是瞬间发生，SQL Server 会自动选择一个“牺牲者” (victim) 并终止其事务来打破僵局。当你手动去查时，死锁可能已经结束了，你查到的只是当时的阻塞。
        
    - **真正的死锁排查是“事后”分析**，即捕获死锁发生时的完整信息，然后进行根源分析（Root Cause Analysis）以防止其再次发生。

### 找出死锁的现代化通用方法 (The Recommended Modern Approach)

专业的死锁排查不依赖于手动抓取，而是依赖于 SQL Server 自动记录的**死锁图 (Deadlock Graph)**。

#### 第一步：捕获死锁信息 (Enable and Find Deadlock Information)

这是最重要的一步。幸运的是，从 SQL Server 2008 开始，系统默认通过**扩展事件 (Extended Events)** 自动捕获死锁信息。

1. **找到死锁日志**：
    
    - 在 SSMS (SQL Server Management Studio) 中，连接到你的实例。
        
    - 展开 **管理 (Management)** -> **扩展事件 (Extended Events)** -> **会话 (Sessions)**。
        
    - 找到名为 **`system_health`** 的会话，它是默认开启的。
        
    - 展开 `system_health`，双击 `package0.event_file` 查看捕获的事件。
        
2. **筛选死锁事件**：
    
    - 打开事件文件后，数据会很多。点击工具栏上的 **筛选 (Filter)** 按钮。
        
    - 在字段中选择 `name`，运算符为 `equal` 或 `contains`，值为 `xml_deadlock_report`。
        
    - 点击确定，此时列表中只剩下死锁报告。
        

_（**备用方法**：如果扩展事件被禁用，可以启用跟踪标志 1222，它会将死锁信息以文本形式输出到 SQL Server 的错误日志 (Error Log) 中。`EXEC sp_readerrorlog` 可以查看。但扩展事件是首选。）_

#### 第二步：分析死锁图 (Analyze the Deadlock Graph)

双击 `xml_deadlock_report` 事件，你会在下方看到一个 XML 格式的死锁报告，并且 SSMS 会为你生成一个可视化的**死锁图**。

这个图是分析死锁的核心，它包含了你需要知道的一切：

- **受害者 (Victim)**：被一个叉标记的进程，这是 SQL Server 选择 `KILL` 掉以解决死锁的 SPID。
    
- **进程 (Process)**：圆圈代表参与死锁的每个进程（SPID）。
    
- **资源 (Resource)**：矩形代表死锁争夺的资源（如某个表的某个键、某一页等）。
    
- **箭头**：表示锁和等待关系。从进程指向资源的箭头表示“持有锁”，从资源指向进程的箭头表示“等待锁”。
    

**从死锁图中你需要解读的关键信息：**

1. **哪些 SPID 参与了死锁？**
    
2. **它们在争抢什么资源？** (例如：`OBJECT: 6:757577737:0` 表示某个数据库的对象锁；`KEY: 6:72057594046316544 (1a39e8756abe)` 表示某个表的键锁)。
    
3. **每个 SPID 当时正在执行什么 SQL 语句？**
    
    - 在 XML 报告中，找到每个 `<process>` 节点下的 `<inputbuf>` 标签，这里面就是造成死锁的**原始 SQL 语句**。这比 `DBCC INPUTBUFFER` 更准确，因为它记录的是死锁发生**那一刻**的语句。
        

#### 第三步：解决问题 (Resolve the Issue)

分析完死锁图后，你就知道了“谁”和“谁”因为“什么代码”在“什么资源”上产生了循环等待。解决方案通常有以下几种：

1. **优化查询和索引**：
    
    - 检查死锁语句的执行计划。是不是因为缺少合适的索引导致了表扫描 (Table Scan) 或大范围的索引扫描 (Index Scan)，从而锁定了太多资源？
        
    - 为 `WHERE`、`JOIN` 子句中的列添加索引，可以大大减少锁的范围和持有时间。
        
2. **统一资源访问顺序**：
    
    - 这是最经典的死锁原因。如果一个事务更新 `TableA` 再更新 `TableB`，而另一个事务更新 `TableB` 再更新 `TableA`，就极易产生死锁。
        
    - **解决方案**：规定应用程序中所有事务必须**以相同的顺序**访问这些表。
        
3. **缩短事务持有锁的时间**：
    
    - 保持事务简短，不要在事务中包含用户交互或其他耗时操作。
        
    - 尽快 `COMMIT` 或 `ROLLBACK` 事务。
        
4. **使用更优的事务隔离级别**：
    
    - 如果读写冲突是主要原因，可以考虑开启**快照隔离** (`SNAPSHOT ISOLATION`) 或**读已提交快照隔离** (`READ COMMITTED SNAPSHOT ISOLATION` - RCSI)。
        
    - 在 RCSI 级别下，“读”操作不会对数据加共享锁，而是读取行版本，从而彻底消除了“读”和“写”操作之间的阻塞和死锁。
        

### 结论

总的来说，您提供的文章介绍了一些基础的实时监控命令，但其核心的死锁排查逻辑是有缺陷的。专业的 SQL Server 死锁分析应该**以扩展事件捕获的死锁图为核心**，通过事后分析找到问题的根本原因，并通过优化索引、代码逻辑或隔离级别来彻底解决问题，而不是简单地在事中 `KILL` 掉某个进程。
