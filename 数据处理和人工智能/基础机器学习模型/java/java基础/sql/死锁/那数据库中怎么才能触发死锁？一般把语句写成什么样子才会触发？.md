当然可以。触发死锁的关键并不在于 SQL 语句本身有多复杂，而在于**多个事务（会话）对资源加锁的顺序**。

我们将这个原理总结成一个“死锁魔法公式”：

> **两个或以上的事务 + 以不同的顺序 + 争夺两个或以上的相同资源 = 死锁**

下面，我将为您展示几种在数据库中最常见的、能稳定触发死锁的 SQL 语句写法。您只需要打开两个 SSMS 查询窗口，模拟两个并发的用户会话，然后按步骤执行即可。

---

### 场景一：最经典的更新冲突死锁（行锁）

这是最教科书式的死锁场景。两个事务以相反的顺序更新两行数据。

#### 1. 准备工作

在您的数据库中执行以下脚本，创建一个简单的 `Products` 表并插入两条数据。

SQL

```
-- 如果表已存在，先删除
IF OBJECT_ID('Products', 'U') IS NOT NULL
    DROP TABLE Products;
GO

-- 创建一个简单的产品表
CREATE TABLE Products (
    ProductID INT PRIMARY KEY,
    ProductName NVARCHAR(50),
    Stock INT
);
GO

-- 插入两条数据
INSERT INTO Products (ProductID, ProductName, Stock) VALUES (1, N'苹果', 100);
INSERT INTO Products (ProductID, ProductName, Stock) VALUES (2, N'香蕉', 200);
GO
```

#### 2. 模拟死锁

您需要**打开两个新的查询窗口**（Query Window）来模拟两个并发的会话。

|**窗口 1 (模拟用户 A)**|**窗口 2 (模拟用户 B)**|
|---|---|
|**第 1 步**: 用户 A 开始一个事务，锁定产品 1（苹果）并减少库存。||
|`sql -- (在窗口1执行) BEGIN TRAN; UPDATE Products SET Stock = Stock - 1 WHERE ProductID = 1; -- 现在苹果被锁定了`||
||**第 2 步**: 用户 B 也开始一个事务，锁定产品 2（香蕉）并减少库存。|
||`sql -- (在窗口2执行) BEGIN TRAN; UPDATE Products SET Stock = Stock - 1 WHERE ProductID = 2; -- 现在香蕉被锁定了`|
|**第 3 步**: 用户 A 现在尝试锁定产品 2（香蕉）。因为它已经被用户 B 锁定，所以用户 A **进入等待状态**。||
|`sql -- (在窗口1执行) -- 这句会卡住，等待B释放锁 UPDATE Products SET Stock = Stock - 1 WHERE ProductID = 2;`||
||**第 4 步**: 用户 B 尝试锁定产品 1（苹果）。因为它已经被用户 A 锁定，所以用户 B **也进入等待状态**。|
||`sql -- (在窗口2执行) -- 这句会立即报错，因为死锁形成！ UPDATE Products SET Stock = Stock - 1 WHERE ProductID = 1;`|

**结果**：当您在窗口 2 执行第 4 步的语句时，SQL Server 会立刻检测到循环等待，并抛出死锁错误（通常是错误号 1205）。其中一个事务（窗口 2 的）会成为牺牲品并被回滚，从而让另一个事务（窗口 1 的）得以继续。

### 场景二：索引键范围死锁（插入/更新）

这种死锁更隐蔽。它不是锁定已存在的行，而是锁定索引中的一个“范围”或“间隙”，通常在插入数据时发生。

#### 1. 准备工作

创建一个带索引的预约表。

SQL

```
-- 如果表已存在，先删除
IF OBJECT_ID('Appointments', 'U') IS NOT NULL
    DROP TABLE Appointments;
GO

CREATE TABLE Appointments (
    AppointmentID INT IDENTITY PRIMARY KEY,
    PatientName NVARCHAR(50),
    AppointmentTime DATETIME
);
GO

-- 在预约时间上创建一个非聚集索引
CREATE INDEX IX_AppointmentTime ON Appointments(AppointmentTime);
GO
```

#### 2. 模拟死锁

同样，打开两个查询窗口。

|**窗口 1 (模拟护士 A)**|**窗口 2 (模拟护士 B)**|
|---|---|
|**第 1 步**: 护士 A 开始事务，准备插入一个 9:00 的预约。数据库会在 `AppointmentTime` 索引上锁定 `9:00` 附近的一个范围，以防其他人同时操作。||
|`sql -- (在窗口1执行) BEGIN TRAN; INSERT INTO Appointments (PatientName, AppointmentTime) VALUES (N'张三', '2025-10-18 09:00:00');`||
||**第 2 步**: 护士 B 也开始事务，准备插入一个 9:10 的预约。数据库同样会在索引上锁定 `9:10` 附近的范围。|
||`sql -- (在窗口2执行) BEGIN TRAN; INSERT INTO Appointments (PatientName, AppointmentTime) VALUES (N'李四', '2025-10-18 09:10:00');`|
|**第 3 步**: 护士 A 现在想插入 9:10 的预约。这个位置的索引范围已被护士 B 锁定，所以护士 A **等待**。||
|`sql -- (在窗口1执行) -- 这句会卡住 INSERT INTO Appointments (PatientName, AppointmentTime) VALUES (N'王五', '2025-10-18 09:10:00');`||
||**第 4 步**: 护士 B 又想插入 9:00 的预约。这个位置的索引范围已被护士 A 锁定。**死锁形成！**|
||`sql -- (在窗口2执行) -- 这句会报错 INSERT INTO Appointments (PatientName, AppointmentTime) VALUES (N'赵六', '2025-10-18 09:00:00');`|

**结果**：这同样会立即触发死锁。这个例子说明，即使操作的数据行完全不同，只要它们在索引上的“物理位置”很近，导致锁定了重叠的索引范围，也可能发生死锁。

---

### 如何避免这些死锁？

从上面的例子可以看出，触发死锁的核心是**“无序”**。因此，解决死锁的核心就是**“有序”**。

1. **统一资源访问顺序**：
    
    - **强制规定**：无论业务逻辑是什么，当需要同时操作多个资源时，都必须按照固定的顺序。例如，总是先操作 `ProductID` 小的，再操作 `ProductID` 大的。这样，所有事务都会先尝试锁 `ProductID=1`，就永远不会出现循环等待。
        
2. **保持事务简短**：
    
    - 事务越长，持有锁的时间就越久，与其他事务冲突的可能性就越大。尽快完成必要的操作，然后 `COMMIT`。
        
3. **使用更优的事务隔离级别**：
    
    - 对于 SQL Server，可以考虑开启**读已提交快照隔离 (RCSI)**。在这个级别下，“读”操作不会阻塞“写”操作，能极大地减少阻塞和死锁的发生。
        
    - `ALTER DATABASE [YourDBName] SET READ_COMMITTED_SNAPSHOT ON;`
        
4. **优化查询，减少锁的范围**：
    
    - 确保查询能用上合适的索引。一个全表扫描会锁定整张表，而一个索引查找只会锁定几行，死锁概率天差地别。
