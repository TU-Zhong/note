垃圾收集算法是理论基础，而**垃圾收集器**就是内存回收的具体落地实现。随着业务规模的扩大，收集器的发展经历了一个清晰的演进路线：从单线程到多线程，从追求吞吐量到追求极致的低延迟。

我们可以按照这个演进路线来梳理主流的垃圾收集器。对于企业级开发（尤其是很多还在使用 JDK 8 的金融后台或国企系统），掌握前面几种尤为重要。

### 1. 基础阶段（单线程）

- **Serial 收集器 (新生代) / Serial Old 收集器 (老年代)**
    
    - **特点：** 最古老的收集器。它是一个**单线程**工作的收集器。在进行垃圾回收时，必须暂停其他所有的工作线程（Stop The World），直到它收集结束。
        
    - **算法：** 新生代采用**标记-复制**，老年代采用**标记-整理**。
        
    - **适用场景：** 内存资源受限的客户端模式（Client 模式），比如早期的桌面应用。
        

### 2. 吞吐量优先阶段（多线程）

- **Parallel Scavenge 收集器 (新生代) / Parallel Old 收集器 (老年代)**
    
    - **特点：** 这是 **JDK 8 的默认垃圾收集器组合**。它们是多线程并行的。与其他收集器关注“如何缩短停顿时间”不同，Parallel 系列的目标是达到一个可控制的**吞吐量**（运行用户代码时间 / (运行用户代码时间 + 运行垃圾收集时间)）。
        
    - **算法：** 新生代采用**标记-复制**，老年代采用**标记-整理**。
        
    - **适用场景：** 后台运算量大、不需要太多用户交互的任务（例如批处理、报表计算）。
        

### 3. 低延迟突破阶段（并发收集）

- **CMS (Concurrent Mark Sweep) 收集器**
    
    - **特点：** 这是一个里程碑式的收集器，它的核心目标是**获取最短回收停顿时间**。它第一次实现了让垃圾收集线程与用户线程（基本上）**同时工作**。
        
    - **算法：** 针对**老年代**，采用**标记-清除**算法。
        
    - **致命缺陷：**
        
        1. **内存碎片：** 因为基于标记-清除，会产生大量碎片。
            
        2. **浮动垃圾：** 并发清理阶段用户线程还在运行，会产生新的垃圾，只能等下次 GC 清理。
            
    - **适用场景：** 对响应时间要求极高的互联网站点或 B/S 系统的服务端。
        

### 4. 区域化与全能阶段（现代主流）

- **G1 (Garbage-First) 收集器**
    
    - **特点：** 开创了收集器**面向局部收集**的设计思路。它不再把堆内存严格划分为两个连续的新生代和老年代，而是把整个堆划分成多个大小相等的**独立区域 (Region)**。G1 会跟踪各个 Region 里面的垃圾价值（回收能获得的空间大小和所需时间的经验值），在后台维护一个优先级列表，**优先回收价值最大的 Region**。这是 **JDK 9 及以后的默认收集器**。
        
    - **算法：** 整体上看是**标记-整理**，局部（两个 Region 之间）看是**标记-复制**。完美解决了 CMS 的碎片问题。
        
    - **适用场景：** 大内存、多核心的服务端应用，要求高吞吐量且停顿时间可控。
        

### 5. 探索无停顿阶段（未来趋势）

- **ZGC (Z Garbage Collector) / Shenandoah**
    
    - **特点：** 极低延迟的垃圾收集器。无论堆内存是几百兆还是几 TB，它们都能把垃圾回收的停顿时间限制在 **10 毫秒以内（甚至 1 毫秒）**。
        
    - **核心技术：** 比如 ZGC 使用了**染色指针**和**读屏障**等黑科技，让对象在移动的同时，用户线程依然可以无感地访问它们。

### 核心对比总结表

掌握这几种收集器的核心定位，在做系统调优或面对技术交流时就能做到有的放矢：

| **收集器**               | **作用代**     | **核心算法** | **关注点**   | **典型应用场景**               |
| --------------------- | ----------- | -------- | --------- | ------------------------ |
| **Serial/Serial Old** | 新生代/老年代     | 复制 / 整理  | 简单高效      | 内存极小的环境                  |
| **Parallel 系列**       | 新生代/老年代     | 复制 / 整理  | **吞吐量**   | 后台批处理计算 (**JDK 8 默认**)   |
| **CMS**               | 老年代         | **清除**   | **低延迟**   | 响应时间敏感的 Web 服务           |
| **G1**                | 整堆 (Region) | 整理 + 复制  | 可控停顿与高吞吐  | 现代大内存服务端 (**JDK 9+ 默认**) |
| **ZGC**               | 整堆          | 带有色指针的整理 | **极致低延迟** | 超大内存、超高响应要求的系统           |
