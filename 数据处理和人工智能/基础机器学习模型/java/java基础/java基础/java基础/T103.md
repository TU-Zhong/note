好的，我们来详细地剖析一下 Java 中 `HashMap` 的 `put(K key, V value)` 方法的执行过程。这是一个面试中的高频问题，因为它能很好地考察面试者对数据结构和算法的理解。

`HashMap` 的 `put` 过程可以概括为：**计算位置 -> 检查位置 -> 插入/更新 -> 可能的结构调整**。

我们将主要基于 **JDK 8** 的实现来讲解，因为它引入了红黑树优化，这也是目前的主流版本。

---

### ## `HashMap` 的内部结构

在开始之前，先快速回顾一下 `HashMap` 的内部结构：它是一个 **“数组 + 链表 / 红黑树”** 的组合体。

- **数组**：主体结构，被称为哈希桶（Hash Bucket）。数组的每个元素都是一个“桶”的入口。
    
- **链表 / 红黑树**：当多个 Key 计算出的哈希位置相同时（哈希冲突），这些 Key-Value 对就会以链表或红黑树的形式存放在同一个桶里。
    

---

### ## `put` 方法执行全过程

下面是 `put(key, value)` 方法的详细步骤：

#### ### 步骤 1：计算哈希值和桶的索引 (定位)

当你调用 `put(key, value)` 时，`HashMap` 首先要决定这个键值对应该放在数组的哪个位置。

1. **处理 `null` Key**：`HashMap` 允许 Key 为 `null`。如果 `key == null`，它会被特殊处理，固定地存放在数组的第 `0` 个桶（`table[0]`）中。
    
2. **计算 `hashCode`**：如果 Key 不为 `null`，调用 `key.hashCode()` 方法获取原始哈希码。
    
3. **扰动计算 (Rehash)**：为了让哈希值分布更均匀，减少哈希冲突，`HashMap` 并非直接使用 `hashCode()`。它会进行一次“扰动”计算：将原始哈希码的高 16 位与低 16 位进行**异或（XOR）**操作。
    
    - `h ^ (h >>> 16)`
        
    - **目的**：这样做的目的是让哈希码的高位也能参与到索引计算中，即使 `hashCode()` 实现得很糟糕（例如，高位变化大，低位变化小），也能尽可能地降低冲突概率。
        
4. **计算数组索引**：使用扰动后的哈希值和数组的长度 `n` 进行与（`&`）运算，来确定桶的索引。
    
    - `index = (n - 1) & hash`
        
    - **前提**：`HashMap` 的数组长度 `n` 永远是 2 的幂。这使得 `(n - 1)` 的二进制表示是全 1 的（例如 `16 - 1 = 15`，二进制是 `00001111`）。`&` 运算相当于一次高效的取模（`%`）运算，但位运算的性能更高。
        

#### ### 步骤 2：检查桶内情况并插入/更新

定位到具体的桶（`table[index]`）后，根据桶内的情况进行处理：

**情况 A：桶是空的 (该位置没有任何元素)**

- 这是最简单的情况。直接创建一个新的节点（`Node` 对象），包含哈希值、Key、Value，并把它放入 `table[index]` 位置。操作完成。
    

**情况 B：桶不是空的 (发生哈希冲突)**

- 这时，需要遍历桶里的元素（链表或红黑树），判断是更新旧值还是添加新节点。
    
    1. **检查第一个节点**：首先比较桶里第一个节点的 Key 是否和我们要插入的 Key **完全相同**（哈希值相同，并且 `equals()` 方法返回 `true`）。
        
        - 如果相同，说明是**更新操作**。用新的 `value` 覆盖旧的 `value`，并返回旧值。操作完成。
            
    2. **检查后续结构**：如果第一个节点的 Key 不匹配，则需要检查后续的结构。
        
        - **如果该桶是红黑树结构**：调用红黑树的插入方法 (`putTreeVal`)。在树中查找是否存在相同的 Key。
            
            - 如果找到了，就更新对应的 Value。
                
            - 如果没有找到，就在树中新增一个节点。
                
        - **如果该桶是链表结构**：遍历整个链表。
            
            - 在遍历过程中，如果发现有某个节点的 Key 和我们要插入的 Key 完全相同，就更新该节点的 Value。操作完成。
                
            - 如果遍历到链表末尾仍然没有找到相同的 Key，说明这是一个新的键值对。将这个新节点**追加到链表的末尾**（尾插法）。
                

#### ### 步骤 3：后续处理 (结构调整)

在**成功添加一个新节点**（而不是更新）之后，`HashMap` 会进行两项检查。

1. **检查是否需要“树化” (Treeify)**
    
    - 在向链表追加新节点后，会检查该链表的长度是否达到了**树化阈值 `TREEIFY_THRESHOLD`（默认为 8）**。
        
    - 如果达到了 8，`HashMap` 会将这条过长的链表转换为一个**红黑树**。
        
    - **为什么？** 因为链表的查询时间复杂度是 O(N)，而红黑树是 O(logN)。当哈希冲突严重时，树化能极大地提高查询效率。
        
    - **注意**：树化还有一个前提条件，即 `HashMap` 的总容量（数组长度）必须大于等于 `MIN_TREEIFY_CAPACITY`（默认为 64）。如果容量太小，`HashMap` 会优先选择**扩容**而不是树化。
        
2. **检查是否需要扩容 (Resize)**
    
    - 每当成功添加一个新节点后，`HashMap` 的总元素数量 `size` 会加 1。
        
    - 接着会检查 `size` 是否超过了**扩容阈值 `threshold`**（计算方式：`capacity * loadFactor`，默认负载因子 `loadFactor` 是 0.75）。
        
    - 如果超过了阈值，就会触发 `resize()` 方法进行**扩容**。
        
        - **扩容过程**：通常是创建一个容量为**原容量两倍**的新数组。然后，将旧数组中的所有元素重新计算它们在新数组中的位置，并迁移过去。这是一个非常耗时的操作。
