下午好！在台北的周三下午，我们来做一个非常经典的面向对象概念大对比，把**普通类、抽象类和接口**这三者的异同点彻底梳理清楚。

这三者是 Java 中构建类体系的基石，理解它们的定位和区别，对于写出良好设计的代码至关重要。

---

### 核心区别一览表

==这张表格从多个维度对它们进行了最直接的比较：==

| 特性        | 普通类 (Concrete Class)                  | 抽象类 (Abstract Class)                  | 接口 (Interface)                             |
| --------- | ------------------------------------- | ------------------------------------- | ------------------------------------------ |
| **实例化**   | **可以** (`new MyClass()`)              | **不可以**                               | **不可以**                                    |
| **构造方法**  | **有**                                 | **有** (供子类调用)                         | **没有**                                     |
| **成员变量**  | 可以有各种变量                               | 可以有各种变量（实例/静态变量）                      | **只能有** `public static final` **常量**       |
| **抽象方法**  | **不可以**                               | **可以有**                               | **可以有** (Java 8 前只能有抽象方法)                  |
| **具体方法**  | **必须全部是**具体方法                         | **可以有**                               | **可以有** (`default` 和 `static` 方法, Java 8+) |
| **继承/实现** | 单继承，多实现 (`extends` 1, `implements` N) | 单继承，多实现 (`extends` 1, `implements` N) | (类)可以多实现 (`implements` N)                  |
| **设计目的**  | 一个具体的**“事物”**                         | 一类事物的**“共性/模板” (is-a)**               | 一种行为的**“契约/能力” (can-do)**                  |

### 详细解释与定位

#### 1. 普通类 (Regular Class) - 蓝图的最终成品

- **定位**：一个**完整的、具体的实现**。它是面向对象世界中可以直接拿来**创建实例（对象）**并使用的基本单元。
    
- **特点**：
    
    - **可实例化**：是三者中唯一可以使用 `new` 关键字创建对象的。
        
    - **完全实现**：类中的所有方法都必须有具体的方法体，不能有任何抽象方法。
        
- **用途**：作为程序中实际工作的“工兵”对象，代表一个具体的、可操作的事物，比如一个 `Student`、一个 `Order`。
    

#### 2. 抽象类 (Abstract Class) - 未完成的设计模板

- **定位**：一个**不完整的模板**，为一族**有紧密亲缘关系**的类提供一个共同的基类。
    
- **特点**：
    
    - **不可实例化**：不能用 `new` 来创建实例。
        
    - **可包含抽象方法**：可以只定义方法的签名而不提供实现，强制子类去提供具体的实现。
        
    - **可包含具体方法和成员变量**：这是它和接口的一个核心区别。抽象类可以为所有子类提供**共享的代码实现**和**共享的状态（成员变量）**。
        
- **用途**：当你发现多个类有共同的属性和行为时，可以将这些共同点向上提取到一个抽象类中，以实现**代码复用**。它完美地诠释了 **“is-a”（是一个）** 的关系。
    
    - **例子**：`Dog` (狗) 和 `Cat` (猫) 都“是” `Animal` (动物)。我们可以创建一个 `Animal` 抽象类，它包含所有动物共有的属性（如 `name`）和行为（如 `sleep()`），以及一个抽象的行为 `makeSound()`，因为不同动物的叫声不同，需要由子类自己去实现。
        

#### 3. 接口 (Interface) - 行为的契约与能力

- **定位**：一个**纯粹的行为规范和契约**，它只定义一个对象**“应该能做什么”**，而不关心“它是什么”或“它怎么做”。
    
- **特点**：
    
    - **不可实例化**。
        
    - **不能有实例变量**：这是它和抽象类在 Java 8 之后最根本的区别。接口不能为实现类提供状态。
        
    - **可被多重实现**：一个类可以同时实现多个接口，这意味着一个类可以同时具备多种不同的“能力”。
        
    - **Java 8 之后**，接口也可以通过 `default` 方法提供默认的方法实现，这增强了其灵活性。
        
- **用途**：定义一套**标准或规范**，让不同的、甚至毫无关系的类都可以遵守。它完美地诠释了 **“can-do”（能做什么）** 的关系。
    
    - **例子**：`Airplane` (飞机) 和 `Bird` (鸟) 在本质上完全不同，但它们都“能飞”（`Flyable`）。`AlarmClock` (闹钟) 和 `Dog` (狗) 也不同，但它们都“能发出声音”（`Audible`）。这时就应该用接口来定义 `Flyable` 和 `Audible` 这两种能力。

### 总结与如何选择？

用一句话来帮你做决定：

- 当你需要描述一个**具体的、可以工作的“事物”**时，使用**普通类**。
    
    > 例如：`ArrayList` 是一个可以直接使用的列表。
    
- 当你需要为一族**亲缘关系很近**的类**提取“共同的属性和行为”**，并且想**复用代码**时，使用**抽象类**。
    
    > 例如：`AbstractList` 为各种 `List` 实现提供了共同的基础代码。
    
- 当你需要定义一种**“能力”或“规范”**，让**任何不相关的类**都可以选择去实现它时，使用**接口**。
    
    > 例如：任何对象都可以通过实现 `Comparable` 接口来获得“可比较”的能力。
    

**现代编程建议**：

> **优先使用接口 (Favor Interfaces)**

在设计时，应优先考虑使用接口来定义类型和契约，因为接口提供了最大的灵活性，实现了完美的解耦。只有当你确定需要在多个子类之间共享代码和状态时，才应该考虑使用抽象类。
