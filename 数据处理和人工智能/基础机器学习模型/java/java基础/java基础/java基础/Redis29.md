你的记忆非常准确！你提到的“先删缓存再改DB、延迟双删”正是这个领域的经典战役。

保证数据库和 Redis 的双写一致性，是分布式系统中最让人头疼的并发难题之一。在探讨具体的先后顺序之前，业界有一个绝对的共识（**Cache Aside Pattern 旁路缓存模式**）：

**对于缓存，我们通常只做“删除（淘汰）”，绝对不去“更新（修改）”。**

_因为如果你去“更新”缓存，在并发写的情况下极其容易发生由于线程执行先后顺序导致的“脏数据”覆盖；而且很多缓存数据是经过复杂计算的，更新成本高，不如直接删了，让下一个读请求去自己算。_

既然确立了“写操作只删缓存”的基调，我们就只剩下两条路：**先删缓存**，还是**先改数据库**？我们逐一推演：

---

### 第一局：先删缓存，再改数据库（导致脏读的罪魁祸首）

很多人直觉上觉得这个方案好：我先清理旧数据，再写新数据，干干净净。但在高并发下，这会引发灾难。

- **初始状态：** DB 中的值是 `A`，缓存里的值也是 `A`。
    
- **致命并发场景：**
    
    1. **写线程** 要把数据改成 `B`。它第一步：**删除了缓存**。
        
    2. 这时，**写线程**还没来得及去改 DB，或者正在改 DB 还没提交。
        
    3. 突然杀出一个**读线程**，去查缓存，发现缓存为空（没命中）。
        
    4. **读线程**立刻去查 DB，此时 DB 里的值还是旧的 `A`。
        
    5. **写线程**终于把 DB 改成了 `B`。
        
    6. **读线程**把它刚才拿到的旧值 `A`，**写入了缓存**。
        
- **灾难结果：** DB 里是新值 `B`，缓存里是永远去不掉的旧值 `A`。**数据永久不一致！**
    

### 第二局：延迟双删（第一局的补丁方案）

为了解决上面那个“旧值被读线程重新塞回缓存”的问题，**延迟双删**诞生了。

它的核心逻辑就是你记忆里的那句：**先删缓存，再改数据库，然后休眠（等一定时间），再删一次缓存。**

- **为什么要等一定时间？**
    
    - 就是为了等那个捣乱的**读线程**把旧数据 `A` 写入缓存后，写线程再把它删掉。
        
    - 这个“休眠时间”必须大于一次“读请求查询 DB 并写回缓存”的时间。（通常设为 50ms - 500ms 不等）。
        
- **致命缺陷：**
    
    - 休眠会**严重拖慢写请求的接口响应时间（吞吐量暴跌）**。
        
    - 休眠时间的具体数值完全是“拍脑袋”决定的（玄学调参），很难精准控制。
        
    - **结论：** 这是一个时代的眼泪，现在大厂的新项目基本**不推荐**使用了。
        

---

### 第三局：先改数据库，再删缓存（业界的标准基线）

这也是大名鼎鼎的 **Cache Aside Pattern** 推荐的做法。

- **执行顺序：** 写线程先把 DB 改成 `B`，然后再把缓存里的 `A` 删除。
    
- **为什么它更好？**
    
    - 因为数据库的“写操作”通常比“读操作”慢得多。在这个模型下，引发不一致的条件极其苛刻（需要一个读线程在缓存刚好失效的瞬间去读 DB，并且在这个读线程写回缓存之前，有一个写线程飞速完成了修改 DB 和删除缓存的两步操作）。由于写慢读快，这种并发交错在物理上极难发生。
        
- **存在的问题：**
    
    - 如果**第二步“删除缓存”因为网络抖动失败了**怎么办？
        
    - DB 已经是新值了，但缓存没删掉，依然是旧值。数据又双叒叕不一致了！
        

---

### 第四局：最终极的杀器 —— Canal 订阅 Binlog + 消息队列

为了彻底解决“第二步删缓存失败”的问题，架构师们决定**把业务代码和缓存同步逻辑彻底解耦**。业务代码只负责“改数据库”，剩下的脏活累活全部交给底层的中间件去异步保证。

**它的工作流堪称艺术：**

1. **业务代码：** 只需要无脑更新 MySQL 即可，完全不用管 Redis。业务接口瞬间返回，性能极高。
    
2. **Canal 中间件：** Canal 伪装成 MySQL 的一个从库（Slave）。当 MySQL 发生了更新，会产生 Binlog。Canal 通过监听 Binlog，立刻知道了“哪张表的哪行数据被改了”。
    
3. **消息队列（MQ）：** Canal 把这个“变更消息”扔进 Kafka 或 RocketMQ。
    
4. **消费端同步：** 专门写一个微服务去消费 MQ，执行**删除 Redis 缓存**的操作。
    
    - **终极保障：** 如果删 Redis 失败了怎么办？没关系，MQ 天生自带**重试机制**！它会一直重试，直到删除成功为止。这就是所谓的**“最终一致性”**。
