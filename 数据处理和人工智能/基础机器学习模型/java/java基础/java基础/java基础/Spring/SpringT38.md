Spring AOP 的底层核心秘密可以用四个字来概括：**动态代理（Dynamic Proxy）**。

AOP 并不是什么魔法，它并没有真正去修改你写好的业务代码。相反，当 Spring 启动时，它会在内存中为你原本的类动态生成一个**“替身”（代理对象）**。

**打个比方：**

你的原始类是一个**“大明星”**（只负责唱歌表演），而代理对象是它的**“经纪人”**。

外界想要请明星唱歌，不能直接找明星，必须先找经纪人。经纪人在唱歌前谈好出场费（Before 通知），然后让明星唱歌（执行目标方法），唱完后再安排专车接送（After 通知）。

Spring AOP 在运行期生成这个“经纪人”时，主要依赖两种技术：**JDK 动态代理** 和 **CGLIB 动态代理**。

---

### 1. JDK 动态代理（基于接口）

这是 Java 原生自带的代理机制，在 `java.lang.reflect` 包下。

- **核心原理：** 它是基于**接口**来实现的。JDK 会在内存中动态生成一个全新的类，这个新类**实现了和你的目标类一模一样的接口**。因为大家实现了同一个接口，所以外界用起来感觉是一样的。
    
- **前提条件：** 你的目标类**必须实现了至少一个接口**。
    
- **核心组件：** `Proxy` 类和 `InvocationHandler` 接口。
    
- **局限性：** 只能代理接口中定义的方法。如果你目标类里有一些自己独有的、没在接口里声明的方法，JDK 动态代理是拦截不到的。
    

### 2. CGLIB 动态代理（基于继承）

CGLIB（Code Generation Library）是一个强大的第三方代码生成包（Spring 内部已经打包了一份它的源码）。

- **核心原理：** 它是基于**继承**来实现的。CGLIB 会在内存中动态生成一个**目标类的子类**。在这个子类中，它会重写父类（也就是你的目标类）的所有非 final 方法，并在重写的方法中织入你的 AOP 逻辑。
    
- **前提条件：** 目标类**不需要**实现任何接口。
    
- **核心组件：** `Enhancer` 类和 `MethodInterceptor` 接口（底层通过 ASM 框架直接操作字节码）。
    
- **局限性：** 因为是基于继承和重写的，所以它**无法代理 `final` 修饰的类，也无法代理 `final` 或 `private` 修饰的方法**。
    

---

### 3. 两者对比总结（核心面试题）

|**对比维度**|**JDK 动态代理**|**CGLIB 动态代理**|
|---|---|---|
|**实现机制**|实现目标类的**接口**|继承目标类，作为它的**子类**|
|**前提条件**|目标类必须实现接口|无需实现接口|
|**局限性**|无法代理没有在接口中声明的方法|无法代理 `final` 类和 `final/private` 方法|
|**性能表现**|创建代理对象较快。<br><br>  <br><br>但在 JDK 8 之前，运行性能略低于 CGLIB；JDK 8 之后性能已经非常接近甚至反超。|创建代理对象较慢（需要操作字节码）。<br><br>  <br><br>但运行期调用方法的性能通常较好。|
|**来源**|JDK 原生自带|第三方开源库（Spring 内置）|

---

### 4. Spring 是如何做出选择的？（版本差异注意）

在实际开发中，你通常不需要手动去选，Spring 框架在底层帮你自动决定了：

- **在传统的 Spring 框架（或较低版本）中：**
    
    - 如果你的类实现了接口，Spring **默认**使用 JDK 动态代理。
        
    - 如果你的类没有实现任何接口，Spring **被迫**退化使用 CGLIB 动态代理。
        
- **在 Spring Boot 2.0 及更高版本中：**
    
    - Spring Boot 为了避免一些由于代理模式切换导致的奇怪强转报错（`ClassCastException`），官方修改了默认行为。
        
    - **现在的默认配置是：不管你有没有实现接口，Spring Boot 默认全部使用 CGLIB 动态代理**（`spring.aop.proxy-target-class=true`）。
        

### 💡 避坑指南：代理失效的“自调用”问题

理解了 AOP 是基于代理对象的，就能明白一个最经典的 Bug：**同一个类中的方法互相调用，AOP 会失效。**

Java

```
@Service
public class UserService {

    // 方法 A 没有事务
    public void methodA() {
        System.out.println("执行 A...");
        // 这里直接调用 methodB，事务会失效！
        this.methodB(); 
    }

    // 方法 B 加了事务控制
    @Transactional
    public void methodB() {
        System.out.println("执行 B，操作数据库...");
    }
}
```

**原因分析：**

当你从外部调用 `methodA` 时，调用的是“代理对象”。但在 `methodA` 内部执行 `this.methodB()` 时，这个 `this` 指向的是**原始对象本身**，而不是“代理对象（经纪人）”。既然绕过了代理对象，AOP 定义的那些增强逻辑（如开启事务）自然就无法生效了。
