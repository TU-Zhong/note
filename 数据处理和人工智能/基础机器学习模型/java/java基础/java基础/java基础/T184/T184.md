JVM（Java 虚拟机）的垃圾回收（Garbage Collection, 简称 GC）机制是 Java 语言的核心特性之一。它让开发者不需要像写 C/C++ 那样手动释放内存，而是由 JVM 自动在后台寻找并清理那些不再使用的对象，从而腾出内存空间。

为了兼顾效率和性能，现代 JVM 的垃圾回收主要基于**分代收集理论（Generational Collection）**。这个理论建立在一个经验统计法则上：**绝大多数对象都是“朝生夕灭”的（用完立刻就成了垃圾），而熬过多次垃圾回收的对象通常会存活很久。**

基于这个理论，JVM 将堆内存（Heap）划分为了不同的区域，针对不同区域采用不同的回收策略。

### 1. 堆内存的“分代”划分

JVM 将堆内存主要分为两块：**新生代（Young Generation）**和**老年代（Old Generation）**。

#### A. 新生代 (Young Gen)

这里是新对象诞生的地方。因为这里的对象死亡率极高，所以垃圾回收非常频繁。新生代内部又细分为三个区，比例通常是 `8 : 1 : 1`：

- **Eden（伊甸园）区：** 绝大多数新创建的对象首先被分配在这里。
    
- **Survivor 0 (S0) 区** 和 **Survivor 1 (S1) 区：** 它们是两个大小相等的幸存者区。在任何时候，总有一个是空的，用于在垃圾回收时暂存存活的对象。
    

#### B. 老年代 (Old Gen)

这里存放的是经过多次新生代垃圾回收后依然存活的“老顽固”对象（比如长期存在的缓存、静态集合里的对象等），或者是占用内存特别大的大对象。这里的垃圾回收频率较低。

---

### 2. 垃圾回收的完整过程（一个对象的生命周期）

我们可以把这个过程想象成一个“漏斗”筛选机制。

1. **诞生：** 当你 `new` 一个对象时，它通常会被放入新生代的 **Eden 区**。
    
2. **触发 Minor GC（新生代 GC）：** 当 Eden 区的空间被填满，无法再分配新对象时，JVM 会触发一次 Minor GC。
    
3. **初次筛选：** 垃圾回收器会检查 Eden 区。不可达的“死对象”被直接清理。依然存活的对象会被移动到 **S0 区**，并且它的“年龄（Age）”记为 1。此时 Eden 区被完全清空。
    
4. **再次筛选与交替：** 随着程序继续运行，Eden 区再次满了，再次触发 Minor GC。这次，回收器会同时清理 Eden 区和 S0 区。存活下来的对象会被移动到那个空闲的 **S1 区**，年龄加 1。此时 Eden 和 S0 被清空。（S0 和 S1 每次 GC 后会互相交换角色，保证始终有一个是空的用来接收幸存者）。
    
5. **晋升老年代 (Promotion)：** 对象在 Survivor 区中每熬过一次 Minor GC，年龄就加 1。当它的年龄达到一定阈值（默认是 15 岁）时，它会被认为是长期存活的对象，从而被直接移入**老年代**。
    
6. **触发 Full GC（全量 GC）：** 随着时间推移，老年代慢慢也被填满了。这时候就会触发 Major GC 或 Full GC。Full GC 会清理整个堆内存（包括新生代和老年代）。**注意：** Full GC 通常会伴随较长的 "Stop-The-World"（暂停所有用户线程），导致系统卡顿，因此在系统调优时，我们要尽量减少 Full GC 的发生。

### 总结

JVM 的垃圾回收机制巧妙地利用了对象的生命周期特征。通过**新生代的频繁局部清理 (Minor GC)**，快速回收掉绝大多数短命对象，从而保证了效率；只有那些真正长期存活的对象才会被放入**老年代**，以此来降低耗时较长、影响性能的 **Full GC** 的发生频率。这就是现代垃圾回收器兼顾吞吐量和响应时间的核心秘密。
