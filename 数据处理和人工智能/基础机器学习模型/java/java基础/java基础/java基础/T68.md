下午好！在台北的周三下午，我们来聊聊 Java 中两个非常重要、也容易混淆的概念：**抽象类（Abstract Class）和接口（Interface）**。

抽象类和接口是 Java 中实现**抽象**的两种核心机制。它们都不能被直接实例化，都需要由子类或实现类来提供具体的实现。但它们的设计哲学和使用场景有着本质的区别。

我们可以用一个简单的比喻来抓住它们的核心思想：

- **抽象类 (Abstract Class)** 像是在描述 **“是什么” (is-a)**。它代表了一类事物的**共同本质和基础行为**。
    
- **接口 (Interface)** 像是在定义 **“能做什么” (can-do)**。它代表了一项**能力或契约**，任何类都可以获得这项能力。
    

---

### 对比总览表

这张表格可以让你快速了解它们的主要区别，特别是 Java 8 之后的变化。

|特性|抽象类 (Abstract Class)|接口 (Interface)|
|---|---|---|
|**继承规则**|**单继承** (一个类只能 `extends` 一个抽象类)|**多实现** (一个类可以 `implements` 多个接口)|
|**成员变量**|可以有各种类型的成员变量 (实例变量、静态变量，final 或非 final)|**只能有** `public static final` 类型的**常量** (不能有实例变量)|
|**构造方法**|**可以有**构造方法 (供子类调用)|**不能有**构造方法|
|**方法实现**|可以包含**抽象方法**和**具体方法**|**Java 8 之前**: 只能有抽象方法<br>**Java 8 之后**: 可以有**抽象方法**、`default` **默认方法**和 `static` **静态方法**|
|**访问修饰符**|成员可以是 `public`, `protected`, `private`, `default`|**Java 8 之前**: 方法隐式为 `public abstract`<br>**Java 8 之后**: 方法可以是 `public abstract`, `public default`, `public static`|
|**设计目的**|强调 **“is-a”** 的关系，用于代码复用和共享通用状态|强调 **“can-do”** 的关系，用于定义行为规范和契约|

### 详细解释与例子

#### 1. 继承规则：单继承 vs. 多实现 (最重要的区别)

- **抽象类**：Java 只支持单继承。一个类一旦继承了一个抽象类，就不能再继承任何其他类了。这体现了“is-a”关系的唯一性。
    
    - **例子**：`Dog` (狗) “是”一种 `Animal` (动物)。`Dog` 不可能同时又“是”一种 `Plant` (植物)。
        
    
    Java
    
    ```
    public abstract class Animal { ... }
    public class Dog extends Animal { ... } // 正确
    ```
    
- **接口**：Java 支持多实现。一个类可以同时实现多个接口，这意味着一个类可以同时具备多种不同的“能力”。
    
    - **例子**：`Airplane` (飞机) “能飞”（`Flyable`），同时它也“能发声”（`Audible`）。这两个能力是相互独立的。
        
    
    Java
    
    ```
    public interface Flyable { void fly(); }
    public interface Audible { void makeSound(); }
    public class Airplane implements Flyable, Audible { ... } // 正确
    ```
    

#### 2. 成员变量（状态）：可以有 vs. 只能是常量

这是 **Java 8 之后两者最根本的区别之一**。

- **抽象类**：可以定义实例变量，这意味着它可以为所有子类提供**共享的状态（State）**。
    
    Java
    
    ```
    public abstract class Animal {
        protected String name; // 实例变量，每个子类对象都有一份
    }
    ```
    
- **接口**：**不能定义实例变量**。它定义的任何变量都默认是 `public static final` 的，即全局常量。它只定义行为，不关心状态。
    

#### 3. 方法实现：部分实现 vs. 契约为主

- **抽象类**：天生就可以包含已经实现好的**具体方法**。这使得它非常适合用于代码复用，子类可以直接使用这些方法，无需重写。
    
- 接口：在 Java 8 之前，接口完全不能有方法实现。它的目的纯粹是定义一个契约，要求实现类必须提供所有方法的实现。
    
    在 Java 8 之后，接口可以通过 default 关键字来提供默认方法。这模糊了与抽象类的界限，使得接口也能提供一部分代码复用。但其主要目的仍然是定义契约，default 方法只是为了在不破坏已有实现类的情况下，为接口增加新功能。
    

#### 4. 构造方法：有 vs. 无

- **抽象类**：**可以并且通常都有**构造方法。虽然它自己不能被实例化，但它的构造方法是给子类调用的（通过 `super()`），用于初始化父类部分的成员。
    
- **接口**：**不能有**构造方法。它不是一个类，只是一个行为规范，自然也谈不上“构造”。

### 如何选择？

在实际开发中，可以遵循以下原则来做选择：

**选择抽象类 (Choose an Abstract Class) 的情况：**

1. **强烈的 “is-a” 关系**：当你描述的几个类在本质上是“同一类事物”时。例如，`Car`, `Truck`, `Bus` 都可以继承自抽象类 `Vehicle`。
    
2. **需要共享代码和状态**：你希望在多个子类中复用已经写好的方法或共享一些通用的成员变量。
    
3. **需要控制访问权限**：你想将某些方法或变量声明为 `protected`，只允许子类访问。
    

**选择接口 (Choose an Interface) 的情况：**

1. **定义“can-do”的能力**：当你想要定义一个类应该具备的某种**能力**，而不管这个类本身是什么。例如，`Comparable`, `Serializable`, `Runnable` 接口，任何类都可以实现它们来获得“可比较”、“可序列化”、“可运行”的能力。
    
2. **需要利用多重继承**：当一个类需要具备多种不相关的能力时。
    
3. **设计 API 契约**：当你需要为系统定义一套插件规范或模块间的通信标准时。接口是定义 API 的完美工具。
    

**现代编程建议：**

> **优先使用接口 (Favor Interfaces over Abstract Classes)**

在不确定时，优先选择接口。因为它更灵活，耦合度更低。只有当你确定需要复用子类间的代码和状态，并且它们之间存在明确的父子关系时，才考虑使用抽象类。
