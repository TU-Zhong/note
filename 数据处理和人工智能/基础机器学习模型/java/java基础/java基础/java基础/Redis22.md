Redis 的速度之所以快，是因为它的数据都存储在内存中。但内存是易失的，一旦服务器宕机或断电，数据就会彻底消失。为了解决这个问题，Redis 提供了持久化机制，将内存中的数据保存到硬盘上。

Redis 提供了两种主要的持久化机制：**RDB（Redis DataBase）** 和 **AOF（Append Only File）**。我们可以用一个通俗的比喻来理解：**RDB 像是在拍快照（照片），而 AOF 像是在做全过程录像（视频）。**

---

### 1. RDB (快照持久化)

RDB 是 Redis **默认**的持久化方式。它会在指定的时间间隔内，将内存中的数据集快照写入磁盘，生成一个紧凑的二进制文件（通常名为 `dump.rdb`）。

**工作原理：**

当触发持久化条件时，Redis 主进程会 `fork` 出一个子进程。主进程继续处理客户端的读写请求，而子进程负责将内存数据写入一个临时的 RDB 文件，写完后再替换掉旧的 RDB 文件。

- **优点：**
    
    - **恢复速度极快：** RDB 文件是极其紧凑的二进制文件，加载进内存的速度远超 AOF，非常适合灾难恢复。
        
    - **性能最大化：** 也是得益于 `fork` 子进程，主进程不需要进行任何磁盘 I/O 操作，保证了 Redis 的高性能。
        
    - **文件体积小：** 相比 AOF 记录每一条命令，RDB 只保存最终的数据状态，占用磁盘空间更小。
        
- **缺点：**
    
    - **容易丢失数据：** 因为是定时快照（比如每 5 分钟一次），如果 Redis 在两次快照之间宕机，这段时间内的所有修改都会丢失。
        
    - **`fork` 阻塞开销：** 如果数据集非常大，`fork` 子进程的过程可能会比较耗时，导致 Redis 瞬间卡顿（几十到几百毫秒）。
        

---

### 2. AOF (追加日志持久化)

AOF 机制通过记录 Redis 服务器所接收到的**每一条写命令**（如 `SET`、`HSET` 等）来实现持久化。当 Redis 重启时，它会通过重新执行这些日志文件中的命令来恢复数据。

**工作原理：**

所有的写命令会追加到 AOF 缓冲区的末尾，然后根据配置的策略（`always` 每次写入、`everysec` 每秒一次、`no` 交给操作系统决定）同步到磁盘的 `appendonly.aof` 文件中。通常推荐使用 `everysec`，在性能和数据安全之间取得平衡。

- **优点：**
    
    - **数据安全性高：** 默认配置下（每秒同步一次），最多只会丢失 1 秒钟的数据。
        
    - **文件格式可读：** AOF 文件是一个纯文本格式的日志文件，你甚至可以直接打开它查看或者在误操作（如执行了 `FLUSHALL`）后，通过删除最后一条命令来手动挽救数据。
        
    - **无损追加：** AOF 是在文件末尾追加，即使写入中途宕机，也不会破坏现有文件内容（Redis 提供了 `redis-check-aof` 工具修复不完整的写入）。
        
- **缺点：**
    
    - **文件体积庞大：** 记录每一条写命令会导致文件越来越大（虽然 Redis 有 AOF Rewrite 机制可以重写压缩文件，但整体体积仍大于 RDB）。
        
    - **恢复速度慢：** 重启时需要把成千上万条写命令重新跑一遍，数据量大时启动非常慢。
        
    - **轻微的性能影响：** 频繁的磁盘 I/O（如每秒同步）会对吞吐量产生一定影响。
        

---

### 3. 核心对比总结

|**特性**|**RDB (快照)**|**AOF (日志)**|
|---|---|---|
|**底层逻辑**|保存某一时刻的**数据结果**|保存所有的**写命令过程**|
|**文件大小**|小（紧凑的二进制）|大（纯文本追加）|
|**数据安全性**|较低（可能丢失几分钟的数据）|高（最多丢失 1 秒的数据）|
|**灾难恢复速度**|极快|较慢|
|**对性能影响**|只有在 `fork` 时有影响，平时极小|持续有轻微的磁盘 I/O 压力|

### 4. 应用场景与最佳实践

在实际生产环境中，我们通常不会只选其一。

#### A. 单一选择场景

- **只用 RDB：** 如果你的 Redis 只是用作纯缓存，数据丢失也无所谓（可以通过查数据库重新预热），追求极高的性能和极快的启动速度。
    
- **只用 AOF：** 极其罕见。通常只有在对数据完整性要求极高，且绝对不允许丢失超过 1 秒数据的场景下，才会作为主力方案。
    

#### B. 终极杀器：混合持久化 (Redis 4.0+)

这是现代 Redis 架构下的**绝对主流方案**。

在 Redis 4.0 之后，引入了 **RDB + AOF 混合持久化**模式（默认开启）。

- **它的做法是：** 当 AOF 文件太大了需要重写（Rewrite）时，Redis 会先将当前内存里的数据以 **RDB 的格式**写到新的 AOF 文件开头，然后再将这期间产生的新写命令以 **AOF 格式**追加到文件尾部。
    
- **结果：** 完美结合了两者的优点！既有了 RDB 极快的数据恢复速度，又保留了 AOF 秒级的数据安全性。文件的前半段是二进制的 RDB，后半段是文本的 AOF 日志。
