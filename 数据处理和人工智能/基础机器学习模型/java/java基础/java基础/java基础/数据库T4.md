这个问题是各大厂后端研发面试的**绝对高频必考题**。能把这个问题讲透，说明你不仅仅是“会用” MySQL，而是真正洞悉了关系型数据库的**存储灵魂**。

回答这个问题的核心破局点只有一个：**数据库的索引是存在磁盘上的**。

内存的读写是纳秒级的，而磁盘的读写是毫秒级的。对于数据库来说，最大的性能瓶颈就是**磁盘 I/O**。因此，评价一个索引结构好不好的最高标准是：**谁能用最少的磁盘 I/O 次数，快速找到数据，并且完美支持业务中最常见的“范围查询”。**

我们按“打怪升级”的思路，把这四个数据结构拉出来溜溜，看看为什么前三个会被无情淘汰。

### 1. 为什么不用 Hash 表？（极速但偏科）

- **优势：** 快！对于等值查询（比如 `WHERE id = 10`），Hash 的时间复杂度是 O(1)，极其霸道。
    
- **致命弱点：完全不支持范围查询。** 业务中大量充斥着 `WHERE age > 18` 这种 SQL。Hash 计算后的结果是毫无规律的散列值，18 和 19 在 Hash 表里的位置可能差了十万八千里。遇到范围查询，Hash 索引当场废掉，只能全表扫描。
    

### 2. 为什么不用红黑树 / 二叉搜索树？（太高太瘦）

- **优势：** 解决了 Hash 无法做范围查询的问题，数据是有序存放的。
    
- **致命弱点：树的层数（高度）太深了！** 在二叉树中，每个节点最多只有两个子节点。如果你有 1000 万条数据，这棵树的高度大约是 24 层。
    
- **磁盘灾难：** 树的每一个节点通常对应磁盘上的一个数据页。找一条数据要顺着树枝往下走 24 层，意味着最多要发生 **24 次磁盘随机 I/O**。在机械硬盘时代，几十次随机 I/O 足以让查询耗时达到秒级，这是绝对不可接受的。我们需要一种**“矮胖”**的数据结构。
    

### 3. 为什么不用 B 树 (B-Tree)？（差之毫厘）

为了让树变“矮胖”，计算机科学家发明了 B 树（多路搜索树）。它的一个节点可以有成百上千个分叉，这样 1000 万数据可能只需要 3 层就能装下，磁盘 I/O 骤降到 3 次！

这听起来很完美，**为什么还要演进到 B+ 树？**

- **致命弱点 1：节点太“重”。** 在 B 树中，**非叶子节点不仅存索引键，还存真实的整行数据**。MySQL 每次从磁盘读取数据是以“页（Page，默认 16KB）”为单位的。如果一页里塞满了真实数据，那它能放下的索引键就变少了。键变少了，分叉就变少了，树又会重新变高。
    
- **致命弱点 2：范围查询依旧拉垮。** 如果你要查 `age > 18`，在 B 树中找到 18 之后，为了找 19、20，你必须不停地在各个节点之间上下跳跃（中序遍历），这依然会引发大量的磁盘随机 I/O。
    

### 4. 终极王者：B+ 树（完美六边形战士）

B+ 树在 B 树的基础上做了两个极其天才的“微调”，直接封神：

1. **非叶子节点绝对不存数据，只存索引键。**
    
    - **收益：** 所有的真实数据都被赶到了最底层的叶子节点。这样一来，非叶子节点（目录页）变得极其“轻量”，一个 16KB 的页可以塞下上千个索引键。**这让树变得极度“矮胖”。** 一棵 3 层的 B+ 树，轻轻松松就能扛住两千万级别的数据量。不管表有多大，找任何一条数据，**最多只需要 3 次磁盘 I/O**。
        
2. **所有叶子节点用“双向链表”串联起来。**
    
    - **收益：** 这是针对范围查询的终极绝杀。找 `age > 18` 的数据？只需要 3 次 I/O 垂直降落到 18 所在的叶子节点，然后顺着底层的双向链表**平行滑过去**就行了。磁盘极其偏爱这种**顺序 I/O**，速度快到飞起。
