好的，这是一个非常核心的Java多线程问题。Java中的线程状态及其转换是理解并发编程的关键。

根据Java官方的定义（`java.lang.Thread.State` 枚举），一个线程在其生命周期中，总共会经历 **6 种状态**。

### 线程的 6 种状态

1. **新建 (NEW)**
    
    - **描述：** 当一个 `Thread` 对象被创建出来，但还没有调用 `start()` 方法时，它就处于这个状态。
        
    - **比喻：** 你已经买好了一辆车（`new Thread()`)，但它还停在车库里，引擎没有发动。
        
2. **可运行 (RUNNABLE)**
    
    - **描述：** 这是一个复合状态，它包含了操作系统线程状态中的 **“就绪”(Ready)** 和 **“运行中”(Running)**。
        
        - **就绪 (Ready):** 线程调用了 `start()` 方法后，进入线程队列，等待操作系统分配CPU时间片。它已经万事俱备，只欠CPU。
            
        - **运行中 (Running):** 线程获得了CPU时间片，正在执行 `run()` 方法中的代码。
            
    - **比喻：** 汽车已经发动引擎 (`start()`)，并且在赛道上排队等待出发（就绪）。一旦绿灯亮起（获得CPU），它就会立刻冲出去（运行中）。
        
3. **阻塞 (BLOCKED)**
    
    - **描述：** 线程正在等待一个**监视器锁（monitor lock）**。这通常发生在线程试图进入一个被其他线程占用的 `synchronized` 同步代码块或方法时。
        
    - **比喻：** 你想开车进一个**单行道的停车场**，但入口的栏杆被前面的车占着了 (`synchronized` 锁)，你只能在入口处停车排队，直到前面的车出来。
        
4. **等待 (WAITING)**
    
    - **描述：** 线程进入无限期等待状态。它需要被其他线程**显式地唤醒**，否则将永远等待下去。
        
    - **比喻：** 你的车停在路边等人，没有约定具体时间。你只能一直等，直到你的朋友打来电话（`notify()` / `notifyAll()`）让你去接他，你才能重新上路。
        
5. **计时等待 (TIMED_WAITING)**
    
    - **描述：** 线程进入一个有时间限制的等待状态。它可以在指定时间结束后**自动返回**，或者被其他线程**提前唤醒**。
        
    - **比喻：** 你的车停在路边等人，但你们约定了“最多等你10分钟”。如果10分钟内朋友来了（被唤醒），你就立刻出发；如果10分钟到了朋友还没来（超时），你也会自己开车走人。
        
6. **终止 (TERMINATED)**
    
    - **描述：** 线程的 `run()` 方法已经执行完毕，或者因未捕获的异常而退出。线程的生命周期到此结束。
        
    - **比喻：** 汽车已经到达终点，并且熄火了，旅程结束。

### 状态转换图

下面这张图清晰地展示了这6种状态之间的转换关系：

Code snippet

```
stateDiagram-v2
    [*] --> NEW
    NEW --> RUNNABLE: thread.start()
    RUNNABLE --> TERMINATED: run()方法执行完毕或异常退出
    RUNNABLE --> BLOCKED: 进入synchronized块/方法，但锁被占用
    BLOCKED --> RUNNABLE: 成功获取到synchronized锁
    RUNNABLE --> WAITING: 调用 Object.wait() / Thread.join() / LockSupport.park()
    WAITING --> RUNNABLE: 其他线程调用 Object.notify() / notifyAll() 或 LockSupport.unpark(thread)
    RUNNABLE --> TIMED_WAITING: 调用 Thread.sleep(t) / Object.wait(t) / Thread.join(t) / LockSupport.parkNanos(t) / LockSupport.parkUntil(t)
    TIMED_WAITING --> RUNNABLE: 等待时间结束 或 其他线程调用 Object.notify() / notifyAll() 或 LockSupport.unpark(thread)
    TIMED_WAITING --> BLOCKED: 被唤醒后需要获取锁，但锁被占用
    WAITING --> BLOCKED: 被唤醒后需要获取锁，但锁被占用
```

### 触发状态转换的具体方法

| 初始状态              | 触发条件/方法调用                                       | 目标状态                       | 描述                                         |
| ----------------- | ----------------------------------------------- | -------------------------- | ------------------------------------------ |
| **NEW**           | 调用 `thread.start()`                             | **RUNNABLE**               | 线程启动，进入就绪队列。                               |
| **RUNNABLE**      | CPU分配时间片                                        | **RUNNABLE (Running)**     | 线程正在执行。                                    |
| **RUNNABLE**      | CPU时间片用完                                        | **RUNNABLE (Ready)**       | 返回就绪队列等待下次调度。                              |
| **RUNNABLE**      | `run()`方法正常执行完毕或抛出未捕获异常                         | **TERMINATED**             | 线程生命周期结束。                                  |
| **RUNNABLE**      | 尝试进入`synchronized`代码块，但锁被其他线程持有                 | **BLOCKED**                | 等待获取监视器锁。                                  |
| **BLOCKED**       | 持有锁的线程释放了锁，并且该线程成功抢到锁                           | **RUNNABLE**               | 重新获得CPU执行资格。                               |
| **RUNNABLE**      | 调用 `object.wait()` (无参数)                        | **WAITING**                | 释放锁并无限期等待，直到被`notify`/`notifyAll`唤醒。       |
| **RUNNABLE**      | 调用 `thread.join()` (无参数)                        | **WAITING**                | 等待`thread`线程执行完毕。                          |
| **RUNNABLE**      | 调用 `Thread.sleep(long millis)`                  | **TIMED_WAITING**          | **不释放锁**，在指定时间内不会被调度。                      |
| **RUNNABLE**      | 调用 `object.wait(long timeout)`                  | **TIMED_WAITING**          | **释放锁**，等待指定时间或被唤醒。                        |
| **WAITING**       | 其他线程调用 `object.notify()` 或 `object.notifyAll()` | **RUNNABLE** 或 **BLOCKED** | 被唤醒后，需要重新竞争锁。如果抢到锁则进入RUNNABLE，否则进入BLOCKED。 |
| **TIMED_WAITING** | 1. 等待时间结束 2. 被`notify`/`notifyAll`唤醒            | **RUNNABLE** 或 **BLOCKED** | 同上，被唤醒或超时后，也需要重新竞争锁。                       |
