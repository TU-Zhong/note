好的，`HashMap` 的扩容机制（`resize`）是其核心操作之一，也是一个非常重要的知识点。它是一个相对耗时的过程，理解它的原理有助于我们写出更高性能的代码。

整个扩容机制可以围绕三个核心问题展开：**什么时候扩容？**、**扩容做什么？**、**如何高效地移动数据？**

---

### ## 1. 什么时候触发扩容（When）？

`resize()` 方法被调用的时机主要有三个：

1. **初始化时**：当你 `new HashMap()` 时，`HashMap` 内部的数组 `table` 还是 `null`。在第一次调用 `put` 方法时，会触发 `resize()` 来创建一个初始容量的数组（默认为 16）。
    
2. **达到扩容阈值时**：这是最常见的触发时机。在每次成功添加一个新元素后，`HashMap` 都会检查当前的元素数量 `size` 是否超过了**扩容阈值 `threshold`**。
    
    - `threshold = capacity * loadFactor`
        
    - `capacity` 是当前数组的容量（比如 16）。
        
    - `loadFactor` 是负载因子（默认为 0.75）。
        
    - 例如，默认情况下，当元素数量超过 `16 * 0.75 = 12` 时，就会触发扩容。
        
3. **链表树化前的检查**（JDK 8+）：当一个桶中的链表长度达到树化阈值 8 时，`HashMap` 准备将链表转换为红黑树。但在转换前，它会检查当前数组的容量 `capacity` 是否小于 `MIN_TREEIFY_CAPACITY`（默认为 64）。如果小于 64，它会认为当前数组太小，哈希冲突的根本原因是容量不足。因此，它会优先选择**扩容**，而不是树化。
    

---

### ## 2. 扩容具体做什么（What）？

`resize()` 方法的核心任务是：**创建一个更大的新数组，并将旧数组中的所有元素重新计算位置后，迁移到新数组中。**

1. **创建新数组**：
    
    - 计算新的容量 `newCapacity`，通常是旧容量的**两倍**（`oldCap << 1`）。
        
    - 计算新的扩容阈值 `newThreshold`，通常也是旧阈值的两倍。
        
    - 创建一个新的、容量为 `newCapacity` 的空数组 `newTable`。
        
2. **遍历旧数组**：
    
    - 逐个遍历旧数组 `oldTable` 的每一个桶（bucket）。
        
3. **迁移元素**：
    
    - 将每个桶中的元素（单个节点、链表或红黑树）迁移到 `newTable` 中。**这是整个扩容过程最核心、最复杂的部分。**
        

---

### ## 3. 如何高效地迁移数据（How）？(JDK 8 的精妙之处)

在 JDK 7 及以前，迁移数据的方式比较暴力：遍历所有元素，重新计算每个元素的哈希值和在新数组中的索引，然后放入新数组。

JDK 8 对此进行了巨大的优化，使得数据迁移变得非常高效。其核心思想是：**利用容量是 2 的幂这一特性，一个桶中的元素在扩容后，只可能去往两个新位置中的一个。**

**原理详解：**

- 扩容时，容量 `n` 变为 `2n`。计算索引的公式是 `(capacity - 1) & hash`。
    
- `n` -> `2n` 的变化，反映在 `capacity - 1` 的二进制上，就是**多了一位高位的 `1`**。
    
    - 例如：`capacity = 16`，`16 - 1 = 15` (二进制 `0000 1111`)
        
    - 扩容后 `capacity = 32`，`32 - 1 = 31` (二进制 `0001 1111`)
        
- 这意味着，一个元素在新数组中的索引，要么和**旧索引完全一样**，要么是**“旧索引 + 旧容量”**。
    
- 到底去哪个位置，完全取决于元素 `hash` 值的**新增的那个 bit 位**是 `0` 还是 `1`。
    

如何判断新增的 bit 位是 0 还是 1？

设计者用了一个非常巧妙的方法：if ((e.hash & oldCap) == 0)

- `oldCap` 的二进制表示中，只有一位是 `1`（例如 `16` 是 `0001 0000`），正好对应了新旧 `capacity-1` 掩码中差异的那一位。
    
- 如果 `(e.hash & oldCap) == 0`，说明 `hash` 值在那一位上是 `0`。那么这个元素在新数组中的索引**与旧索引相同**。
    
- 如果 `(e.hash & oldCap) != 0`，说明 `hash` 值在那一位上是 `1`。那么这个元素在新数组中的索引就是 **“旧索引 + 旧容量”**。
    

**迁移算法：**

基于以上原理，JDK 8 在迁移一个桶的数据时，不再是逐个元素计算新位置，而是：

1. 遍历桶中的链表（或红黑树）。
    
2. 创建两个新的“虚拟”链表：`low` 链表和 `high` 链表。
    
3. 再次遍历原链表，根据 `(e.hash & oldCap) == 0` 的判断结果，将节点分别挂到 `low` 链表或 `high` 链表的尾部。
    
4. 遍历完成后，`low` 链表中的所有元素都属于**位置不变**的，直接将 `low` 链表的头节点放在新数组的**原索引** `j` 处。
    
5. `high` 链表中的所有元素都属于**位置改变**的，直接将 `high` 链表的头节点放在新数组的**新索引** `j + oldCap` 处。
    

**这样做的好处：**

- **高效**：一次遍历就将一个复杂的桶拆分到两个新位置，避免了对每个元素进行重复的哈希计算。
    
- **保持相对顺序**：在拆分过程中，节点在 `low` 和 `high` 链表中的相对顺序与原链表保持一致，巧妙地解决了 JDK 7 中并发 `resize` 可能导致的环形链表问题。
    

### 总结

|特性|JDK 7 实现|JDK 8 实现|
|---|---|---|
|**数据迁移**|重新计算每个元素的 hash 和 index|通过 `(hash & oldCap)` 判断，将链表拆分为两个|
|**链表插入**|头插法|尾插法|
|**并发问题**|多线程下 `resize` 可能导致环形链表，CPU 100%|尾插法修复了此问题，但 `HashMap` 仍非线程安全|
|**效率**|效率较低，需要重新计算|效率更高，计算巧妙|

因此，`HashMap` 的 `resize` 是一个开销很大的操作。为了提高性能，在可以预估存储数据量的情况下，**建议在创建 `HashMap` 时指定一个合适的初始容量**，以减少或避免不必要的扩容。
