在主流的编程语言（如 Java、C# 以及 Python 等）中，判断一个对象是否可以被垃圾回收（Garbage Collection, GC），主要依赖两种核心算法：**可达性分析算法** 和 **引用计数算法**。

现代虚拟机（如 JVM）主要采用的是**可达性分析算法**，因为它能完美解决循环引用的问题。

### 1. 可达性分析算法 (Reachability Analysis) —— 绝对的主流

这是目前最常用、最严谨的判断方式。它的核心思想是：把所有活动的对象看作一棵树，从树根开始向下找，找不到的对象就是“死”对象，可以被回收。

- **GC Roots（根对象）：** 算法的起点。通常是一些确定还在被使用的引用，例如：
    
    - 虚拟机栈（线程局部变量）中引用的对象。
        
    - 方法区中静态属性或常量引用的对象。
        
    - 本地方法栈中（JNI）引用的对象。

- **判断过程：** 垃圾收集器从 GC Roots 开始，沿着引用链向下搜索。如果一个对象到 GC Roots 之间有任何一条路径相连（即可达），则认为该对象是**存活的**。如果一个对象到 GC Roots 没有任何路径相连（即不可达），则认为该对象**可以被回收**。
    

### 2. 引用计数算法 (Reference Counting)

这种算法比较简单直观，早期应用较多（比如 Python 内部也有使用，但配合了其他机制）。

- **判断过程：** 在对象中添加一个引用计数器。每当有一个地方引用它时，计数器值加 1；当引用失效时，计数器值减 1。任何时刻计数器为 0 的对象就是不可能再被使用的，可以被回收。
    
- **致命缺点（循环引用）：** 如果对象 A 内部引用了对象 B，而对象 B 内部又引用了对象 A，除此之外没有任何外部引用指向它们。此时它们的引用计数都为 1，导致它们永远无法被回收，从而造成内存泄漏。

### 补充知识：对象的“缓刑”

在 Java 中，一个对象即使在可达性分析中被判定为不可达，也并非“非死不可”。它至少要经历两次标记过程：

1. **第一次标记：** 对象不可达，被标记一次。并检查该对象是否覆盖了 `finalize()` 方法，或者该方法是否已经被调用过。
    
2. **第二次标记：** 如果对象有必要执行 `finalize()` 方法，它会被放入一个队列中。如果对象在 `finalize()` 方法中重新与引用链上的任何一个对象建立了关联（比如把自己赋值给某个类变量），它就能在第二次标记时“逃过一劫”。否则，它就会真正被回收。_(注：由于 `finalize()` 运行代价高且不确定性大，官方早已不推荐使用。)_
