您问到了一个非常核心的数据结构问题。`TreeSet` 之所以能够始终保持元素有序，其核心秘诀在于它底层使用了**一种名为“红黑树（Red-Black Tree）”的数据结构**来存储元素。

---

### 1. 底层数据结构：`TreeMap` 与红黑树

`TreeSet` 自身的代码其实非常少，它在实现上是一个非常巧妙的“包装”。

- TreeSet 内部包含一个 TreeMap：
    
    当你创建一个 TreeSet 时，它的内部实际上是创建了一个 ==TreeMap== 实例。你向 TreeSet 中==添加的元素，都被作为 TreeMap 的**键（Key）**来存储==。而 TreeMap 的==值（Value）则是一个固定的、无意义的虚拟对象。==
    
    Java
    
    ```java
    // TreeSet 源码简化版
    public class TreeSet<E> ... {
        // 内部实际上是一个 NavigableMap，其实现就是 TreeMap
        private transient NavigableMap<E,Object> m; 
    
        // 值是一个共享的虚拟对象
        private static final Object PRESENT = new Object();
    
        public boolean add(E e) {
            // 添加元素实际上是把元素作为 Key 放入内部的 TreeMap
            return m.put(e, PRESENT) == null;
        }
        // ...
    }
    ```
    
    所以，**`TreeSet` 的排序能力，完全来自于其内部 `TreeMap` 的排序能力**。
    
- TreeMap 的实现是红黑树：
    
    红黑树是一种自平衡的==二叉查找树==（Self-Balancing Binary Search Tree）。它有以下关键特性：
    
    1. **二叉查找树特性**：对于树中的任何一个节点，其左子树中所有节点的值都**小于**该节点的值，其右子树中所有节点的值都**大于**该节点的值。正是这个特性，保证了树内元素的**有序性**。
        
    2. **自平衡特性**：通过一系列复杂的规则（节点颜色、旋转、变色），红黑树能确保树的高度大致保持在 `log n`（n 为节点数）的水平，不会退化成一个长长的链表。这保证了其增、删、查操作的性能稳定性。

### 2. 排序的“标尺”：两种比较方式

既然红黑树需要不断地比较元素的大小来决定节点的位置，那么它比较的“标尺”是什么呢？这正是我们昨天讨论过的 `Comparable` 和 `Comparator`。

`TreeSet` 在创建时，会确定一种比较规则，并且在后续的所有操作中都严格遵守这个规则。

#### 方式一：==自然排序== (Natural Ordering) - `Comparable`

- **触发条件**：当你使用无参构造方法 `new TreeSet<>()` 创建实例时。
    
- **工作方式**：`TreeSet` 会假定你存入的所有元素都**实现了 `Comparable` 接口**。在比较两个元素大小时，它会调用元素的 `compareTo()` 方法。
    
- 如果元素没有实现 Comparable 接口，会发生什么？
    
    ==当你尝试添加第一个元素时不会出错，但当你添加第二个元素时，TreeSet 尝试调用 compareTo() 方法进行比较，此时会发现方法不存在，于是抛出 ClassCastException。==
    

#### 方式二：自定义排序 (Custom Ordering) - `Comparator`

- **触发条件**：当你使用带 `Comparator` 参数的构造方法 `new TreeSet<>(myComparator)` 创建实例时。
    
- ==**工作方式**：`TreeSet` 会**忽略**元素自身的 `compareTo()` 方法（即使有），而完全使用你在构造时传入的那个 `Comparator` 实例的 `compare()` 方法来进行所有的大小比较。==

### 3. 一个元素是如何被“插入并排序”的？

现在我们可以把整个流程串起来了。当你调用 `treeSet.add(newElement)` 时，内部发生了以下事情：

1. `TreeSet` 将 `add` 操作委托给内部的 `TreeMap`，相当于执行 `treeMap.put(newElement, PRESENT)`。
    
2. `TreeMap` 从红黑树的**根节点**开始。
    
3. 使用已确定的比较规则（`compareTo` 或 `compare`），将 `newElement` 与当前节点进行比较。
    
4. **比较结果**：
    
    - ==**小于 0**：说明 `newElement` 比较小，于是移动到当前节点的**左子节点**，继续重复第 3 步。==
        
    - ==**大于 0**：说明 `newElement` 比较大，于是移动到当前节点的**右子节点**，继续重复第 3 步。==
        
    - **等于 0**：说明 `newElement` 与树中的某个元素重复了。`Set` 不允许重复，所以 `add` 操作失败，方法返回 `false`，树的结构不发生改变。
        
5. ==这个过程一直持续，直到找到一个空的链接位置，然后将 `newElement` 作为新的叶子节点插入到树中。==
    
6. 插入新节点后，红黑树可能会**触发自平衡**操作（旋转、变色），以确保其结构依然满足红黑树的定义。
    

通过这个过程，`TreeSet` 确保了每次添加新元素后，它都会被放置在正确的排序位置上，从而使得整个集合始终保持有序。

### 总结

- **核心数据结构**：`TreeSet` 本质上是一个 `TreeMap` 的包装，而 `TreeMap` 是基于**红黑树**实现的。
    
- **排序依据**：依赖于两种比较机制之一 —— 元素的**自然排序**（实现 `Comparable` 接口）或在构造时传入的**自定义排序**（`Comparator` 对象）。
    
- **工作原理**：利用**二叉查找树**的特性，在插入每个元素时，通过不断比较大小，将其放置在树中的正确位置。
    
- **性能保证**：红黑树的**自平衡**特性，保证了 `add()`, `remove()`, `contains()` 等核心操作的时间复杂度稳定在 **O(log n)**。
