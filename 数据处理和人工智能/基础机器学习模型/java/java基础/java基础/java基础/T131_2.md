要彻底弄懂锁升级，我们必须先回答一个问题：**为什么 JDK 1.6 的设计者非要费这么大劲去搞锁升级？**

因为在 JDK 1.6 之前，`synchronized` 是个纯粹的**“重量级锁”**。只要你写了 `synchronized`，不管外面有几个线程竞争，它都会直接去向操作系统申请底层的 Monitor 锁。 这就好比，你只是想防一下舍友偷吃你的零食，结果你每次出门都去雇了一个全副武装的银行运钞车保安站在门口。 动用操作系统底层的锁，需要把线程从**“用户态”切换到“内核态”**，这个切换的代价极其高昂，甚至比你执行同步代码块本身的时间还要长！

后来，JVM 的大牛们经过海量数据分析发现了一个惊人的事实：**在大多数真实业务场景下，绝大多数的锁根本就不存在多线程竞争，往往是从头到尾都只有一个线程在反复进出。**

为了这个事实，JVM 引入了“按需分配”的锁升级机制（也就是对象头里 Mark Word 的变形记）：**无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁**。

我们来看看这四种状态是怎么一步步被逼出来的：

### 1. 偏向锁 (Biased Lock) —— “贴上专属标签”

- **场景：** 从头到尾只有**线程 A** 一个人在访问同步块。
    
- **机制：** 当线程 A 第一次进去时，JVM 发现没人竞争，干脆连锁都不加了，直接在对象的对象头（Mark Word）里**贴上线程 A 的 ID**。
    
- **收益：** 以后线程 A 再来，只要看一眼对象头上的 ID 是自己，直接长驱直入，连一点点同步的计算都不需要。性能等同于无锁！
    

### 2. 轻量级锁 (Lightweight Lock) —— “自旋不睡觉”

- **场景：** 这时**线程 B** 来了。虽然有 A 和 B 两个人，但他们是**交替来的**，或者 A 刚进去一瞬间就出来了，竞争并不激烈。
    
- **机制：** 发现有别的线程来了，JVM 立刻把偏向锁撤销，升级为**轻量级锁**。此时线程 B 发现门被 A 锁了，B **绝对不会去睡觉（阻塞）**，而是选择在门外原地打转（执行 **CAS 操作自旋**），不停地问：“A出来没？A出来没？”。
    
- **收益：** 因为 A 很快就出来了，B 原地打转两圈就进去了。虽然浪费了一丁点 CPU 在原地打转，但完美避开了最耗时的“用户态转内核态”阻塞排队操作。
    

### 3. 重量级锁 (Heavyweight Lock) —— “请出终极保安”

- **场景：** 竞争失控了。A 在里面待了很久很久，B 在门外原地打转（自旋）转得头晕眼花（超过了最大自旋次数）；或者此时 C、D、E 线程同时冲了过来，大家都在原地狂转，极其浪费 CPU。
    
- **机制：** JVM 看不下去了：“你们别转了，CPU 都快被你们空转烧干了！”。于是锁轰然膨胀为**重量级锁**。
    
- **结果：** 此时，我们上一轮聊过的底层老大哥 **Monitor** 正式接管。门外所有打转的线程，全部被一脚踹进内核态，强制进入 `_EntryList` 阻塞休眠（Blocked），乖乖等操作系统唤醒。
