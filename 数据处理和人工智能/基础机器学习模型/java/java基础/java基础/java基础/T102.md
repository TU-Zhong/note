下午好！在台北的周四下午，我们来深入 `HashMap` 的内部世界。

这是一个在 Java 面试中几乎必问的核心问题，因为它很好地考察了开发者对数据结构和性能的理解。

`HashMap` 的底层数据结构在 **Java 8** 是一个重要的分水岭。简单来说：

- **Java 8 之前：数组 + 链表**
    
- **Java 8 及之后：数组 + 链表 + 红黑树**
    

现代的实现（Java 8+）是为了优化旧版实现在某些极端情况下的性能问题。下面我们来详细解析这个现代的结构。

---

### 1. 核心组件：哈希表 (Hash Table)

`HashMap` 的本质是一个**哈希表**，也叫散列表。你可以把它想象成一个**酒店前台的存物柜（数组）**。

- **数组 (Array)**：这是 `HashMap` 的主体，数组的每一个格子被称为一个**“桶”（Bucket）**。
    
- **哈希函数 (Hash Function)**：当你需要存一个东西（键值对）时，你需要一个方法来决定它应该放在哪个柜子里。这个方法就是哈希函数，它根据你的物品（**Key**）的特征（`hashCode()`）来计算出一个独一无二的柜子编号（**数组索引**）。
    

---

### 2. Java 8 之前的实现：数组 + 链表

在 Java 8 之前，哈希表的实现比较简单。

**`put(key, value)` 的流程：**

1. 根据 `key` 的 `hashCode()` 计算出它应该存放在数组的哪个索引（哪个桶）下。
    
2. **如果没有冲突**：如果这个桶是空的，就直接把这个键值对（Entry）放进去。
    
3. **如果发生冲突**：如果这个桶已经有其他键值对了（即不同的 Key 计算出了相同的数组索引），这就是**哈希冲突**。此时，`HashMap` 会在这个桶里维护一个**链表（Linked List）**，把新来的键值对追加到链表的末尾。
    

缺点：

在极端情况下，如果大量的 Key 都计算出了同一个哈希值，那么这个桶对应的链表就会变得非常长。此时，如果你想在这个链表中查找、添加或删除一个元素，你就不得不从头到尾遍历整个链表。这会导致 HashMap 在这个桶上的操作时间复杂度从 O(1) 退化到 O(n)，性能急剧下降。

---

### 3. Java 8 及之后的进化：数组 + 链表 + 红黑树

为了解决上述问题，Java 8 对 `HashMap` 进行了重大优化。

核心改进：引入红黑树

基本的“数组 + 链表”结构保持不变，但增加了一个“升级”机制：

> 当一个桶中的**链表长度达到一个阈值（默认为 8）时，并且哈希表的总容量也大于等于一个阈值（默认为 64）时，这个链表就会被转换成一棵红黑树（Red-Black Tree）**。

- **红黑树**：一种自平衡的二叉查找树。
    
- **带来的好处**：树形结构的查找、插入和删除操作的**时间复杂度是 O(log n)**。这远远优于链表的 O(n)。
    

通过这个优化，即使在最坏的情况下（所有 Key 都冲突到同一个桶里），`HashMap` 的性能也能从 O(n) 提升到 O(log n)，保证了其稳定的查询性能。

“树退化为链表”：

相应地，当一个桶中的元素在删除后，红黑树的节点数量减少到一个阈值（默认为 6）时，它会重新退化成链表，以节省空间。

---

### 4. 另一个关键机制：扩容 (Resizing)

`HashMap` 的数组容量并不是无限的。当存入的元素越来越多，哈希冲突的可能性也越来越大。

- ==**加载因子 (Load Factor)**：一个表示哈希表填充程度的阈值，默认为 **0.75**。==
    
- **扩容时机**：当 `HashMap` 中元素的数量（`size`）超过了 `当前数组容量 × 加载因子` 时，就会触发**扩容**。
    
- **扩容操作**：
    
    1. 创建一个==新的数组==，其容量是**原数组的两倍**。
        
    2. ==将旧数组中的所有元素**重新计算哈希索引（Rehashing）**，然后移动到新数组的对应位置上。==
        

扩容是一个开销相对较大的操作，因为它需要遍历所有元素并重新分配位置。

### 总结

`HashMap` 在 Java 8 及之后的底层数据结构是**哈希表**，具体实现为：

1. 主体是一个**数组（`Node<K,V>[] table`）**，每个元素是一个“桶”。
    
2. 当多个 Key 的哈希值映射到同一个桶时，会发生**哈希冲突**。
    
3. 冲突的元素会以**链表**的形式存放在桶中。
    
4. 当某个桶中的链表长度超过阈值 **8**，并且总容量大于 **64** 时，该链表会**转换为红黑树**，以优化查询性能。
    
5. 当 `HashMap` 的元素数量超过 `容量 × 0.75` 时，会触发**扩容**，数组容量翻倍，并重新散列所有元素。
