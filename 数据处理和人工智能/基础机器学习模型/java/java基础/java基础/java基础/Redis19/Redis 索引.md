这个问题抓到了跳表（SkipList）的灵魂！要理解跳表，我们先要明白 Redis 为什么被“逼”着发明了这种结构。

### 1. 痛点：为什么底层单链表不够用？

假设我们有一个排好序的单向链表，存了 1000 个玩家的积分。

如果我们要找积分等于 `600` 的玩家，因为是链表，内存不是连续的，**我们无法使用二分查找**。我们只能从头节点开始，顺着指针一个一个往后遍历，时间复杂度是极其拉垮的 **$O(N)$**。对于追求极致性能的 Redis 来说，这是绝对不可接受的。

### 2. 破局：给链表建“高速公路”

既然单链表找得慢，那我们就模仿“二分查找”的思想，把链表变成立体的！

- **底层（第 0 层）：** 就像是**“国道”**。这里包含了所有的玩家节点，用指针串起来。
    
- **第一层索引：** 就像是**“省道”**。我们从底层节点中，随机抽出大约一半的节点，把它们提拔到第一层，再用指针连起来。
    
- **第二层索引：** 就像是**“高速公路”**。从第一层里再抽一半提拔上来。
    
- **最高层：** 就像是**“航空线”**。跨度极大，只有极少数几个节点。
    

### 3. 核心机制：怎么在高速公路上“跳”？

当我们要查找一个目标分数（比如 `70`）时，查找规则极其简单，只有两句话：

**从最高层的最左边开始寻找：**

1. **向右看：** 如果右边节点的分数 **小于或等于** 目标分数，那就**一直向右开**。
    
2. **往下跳：** 如果右边节点的分数 **大于** 目标分数，或者右边根本没有节点（`null`）了，说明目标肯定在当前节点和右边节点之间。此时，**原地垂直降落，往下一层跳**。
    

一直重复这个过程，直到降落到底层（第 0 层），就能精准锁定目标！

### 面试加分项：跳表是怎么建索引的？（抛硬币魔法）

在了解了查找逻辑后，很多面试官会紧接着抛出一个致命问题：“如果是 MySQL 的 B+ 树，为了维持树的平衡，插入新节点时需要发生复杂的**页分裂**。那跳表插入新节点时，怎么决定这个节点要建几层索引呢？难道每次插入都要重新打乱重排吗？”

**答案是：抛硬币（随机概率算法）！**

这就是跳表比 B+ 树和红黑树代码实现简单得多的原因：

当有一个新玩家加入 ZSet 时，Redis 的底层逻辑是这样的：

1. 先把这个节点插到底层的单向链表里。
    
2. 然后 Redis 开始**“抛硬币”**（执行一个随机函数）：
    
    - 如果是正面（比如 25% 的概率），就把这个节点拔高到第 1 层。
        
    - 继续抛，如果又是正面，就拔高到第 2 层。
        
    - 以此类推，直到抛出反面或者达到最大层数上限（通常是 32 层）为止。

这种看似儿戏的“纯靠运气”的建索引方式，在宏观大样本下，能极其完美地保持每一层的节点数是下一层的 1/4（或者 1/2，取决于设定的概率 p）。

**它彻底摒弃了复杂的树状平衡算法，用极简的随机数实现了极致的 $O(\log N)$ 插入和查询性能。** 这也是 Redis 作者为什么执意要用跳表而不用平衡树的终极原因。
