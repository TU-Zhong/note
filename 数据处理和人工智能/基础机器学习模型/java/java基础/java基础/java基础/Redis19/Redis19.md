面试官问这个问题，绝不是想听你像背说明书一样报出“String、List、Hash、Set、ZSet”这五个名字，而是想考察你是否懂得 Redis 能够快到起飞的底层秘密：**Redis 会根据当前数据量的大小和特征，在底层全自动地切换数据结构。**

这种为了压榨内存和 CPU 性能而做到极致的设计，才是大厂面试官最想听到的。下面我们逐一拆解这 5 种基本类型背后的 C 语言底层数据结构。

---

### 1. String (字符串)

这是最基础的类型，但底层大有乾坤。

- **底层结构：SDS（Simple Dynamic String，简单动态字符串）** 或 `int`。
    
- **核心原理：**
    
    - 如果你存的是个纯数字（比如 `set age 20`），Redis 底层会直接用一个 **`int`** 类型来存，连字符串转换都省了。
        
    - 如果存的是字符串，Redis **绝对不会**使用 C 语言原生的字符串，而是自己封装了 **SDS**。
        
- **为什么不用 C 原生字符串？（高频考点）**
    
    1. C 语言查长度要遍历整个数组（复杂度 $O(N)$），SDS 内部维护了 `len` 变量，查长度是 **$O(1)$**。
        
    2. 杜绝了缓存区溢出（SDS 修改前会自动检查并扩容）。
        
    3. **二进制安全**：C 字符串遇到 `\0` 就以为结束了，没法存图片、视频等二进制流。SDS 靠 `len` 来判断结尾，什么都能存。
        

### 2. List (列表)

常用于做消息队列、最新动态展示。

- **底层结构：QuickList（快速链表）**。_(注：在早期版本是 ZipList 和 LinkedList，Redis 3.2 之后统一合并为了 QuickList)_
    
- **核心原理：**
    
    - 普通的双向链表（LinkedList）每一个节点都有前后指针，指针本身占用的内存（64位系统下是 16 字节）甚至比存的数据还要大，这叫**内存碎片碎片化严重**。
        
    - **QuickList 的微创新：** 它是一个双向链表，但它的每一个节点不再是一个单一元素，而是一个 **ZipList（压缩列表）**。相当于把好几个元素像切糕一样紧紧压在一个连续的内存块里，外面再用指针把这些“切糕”连起来。
        
    - **收益：** 既保留了链表两端插入/弹出的极速 $O(1)$ 性能，又极致地压榨了指针带来的内存浪费。
        

### 3. Hash (哈希)

常用于存储对象，比如用户信息。

- **底层结构：ZipList（压缩列表） ➔ 升级为 HashTable（哈希表/字典）**。_(注：Redis 7.0 之后，ZipList 统一被替换为更优秀的 ListPack，但面试答 ZipList 依然通用)_
    
- **核心原理（动态切换）：**
    
    - **小数据量时（ZipList）：** 当 Hash 里的字段很少，且每个字段的字符串也很短时，Redis 会用极度紧凑的连续内存块（ZipList）来存。虽然查找是 $O(N)$，但因为数据极少，CPU 遍历内存连续块的速度甚至比算 Hash 值还快，而且**极其省内存**。
        
    - **大数据量时（HashTable）：** 一旦元素数量超过阈值（默认 512 个）或者某个字符串太长（超过 64 字节），底层会自动升级为纯正的 HashTable。此时查找变为绝对的 **$O(1)$**，用空间换时间。
        

### 4. Set (集合)

无序且不重复，常用于抽奖、点赞、共同好友（交集并集）。

- **底层结构：IntSet（整数集合） ➔ 升级为 HashTable（哈希表）**。
    
- **核心原理（动态切换）：**
    
    - **全是数字时（IntSet）：** 如果你往 Set 里塞的全是整数，且数量不多（默认小于 512 个），Redis 会拿一片连续内存按从小到大把它们排好。查找时直接用**二分查找**（$O(\log N)$），内存利用率 100%。
        
    - **出现非数字/数据变多（HashTable）：** 只要塞入一个非数字（比如 `sadd set "hello"`），或者数量超标，立马升级为 HashTable。此时**只用 Dict 的 Key 来存数据，Value 全都指向 Null**，以此保证元素的唯一性和 $O(1)$ 的判断速度。
        

### 5. ZSet (Sorted Set，有序集合)

这是 Redis 最引以为傲、也是面试中技术含量最高的数据结构。常用于排行榜、延时队列。

- **底层结构：ZipList ➔ 升级为 SkipList（跳表） + HashTable（哈希表）**。
    
- **核心原理：为什么是两个结构一起上？**
    
    ZSet 要求既能**通过成员名字光速查分数**，又要能**通过分数范围查前 10 名**。一个结构搞不定，Redis 直接上了双保险：
    
    1. **HashTable：** 负责存储 `成员 -> 分数` 的映射，实现 $O(1)$ 极速单点查询。
        
    2. **SkipList（跳表）：** 负责按分数排序。它在底层链表的基础上，抽出了多层“高速公路（索引）”。查范围时，从最高层的高速公路往下跳，极速锁定目标区间（复杂度 $O(\log N)$）。
        
- **终极考点：为什么 ZSet 用跳表，而不用 MySQL 的 B+ 树或红黑树？**
    
    因为跳表**极其容易实现并发无锁化**，并且在做范围查询时，跳表的最底层就是一个简单的单向链表，顺藤摸瓜非常快；而红黑树在插入/删除时需要复杂的左旋右旋来维持平衡，跳表只需要靠一个简单的“随机函数”决定层高，代码极简，速度极快。
