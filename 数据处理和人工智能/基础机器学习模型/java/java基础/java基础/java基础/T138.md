这是一个非常深入且关键的Java并发问题。简单回答是因为“Java规范就是这么规定的”，但这显然不能令人满意。

其背后深层的原因，是为了解决多线程协作中的**竞态条件（Race Condition）**，确保程序的**正确性和健起壮性**。

我们可以从两个层面来理解：**技术规范层面**和**设计思想层面**。

---

### 1. 技术规范层面：`IllegalMonitorStateException`

==从最直接的技术角度看，如果你不把 `wait()`, `notify()`, `notifyAll()` 放在同步代码块中，程序在运行时会直接抛出 `java.lang.IllegalMonitorStateException` 异常。==

为什么会抛这个异常？

因为这三个方法都依赖于一个叫做**“监视器锁（Monitor Lock）”**的东西，而在Java中，synchronized 关键字就是用来获取和释放这个监视器锁的。

- **`wait()`** 的一个核心动作是**释放当前持有的锁**，然后进入等待状态。如果你调用它的时候都没有持有锁，那它“释放锁”这个动作就无从谈起。
    
- **`notify()` / `notifyAll()`** 的作用是唤醒**正在等待这个锁**的其他线程。如果你都没有持有这个锁，也就无权去唤醒别的等待者。
    

所以，JVM的规范强制要求：**要操作一个对象的等待/唤醒机制，你必须先成为这个对象的“主人”（即持有该对象的锁）。**

---

### 2. 设计思想层面：防止“丢失的唤醒”(Lost Wakeup)

这是最核心、最本质的原因。`synchronized` 关键字确保了“检查条件”和“进入等待”这两个动作成为一个**原子操作**，从而避免了致命的竞态条件。

我们来看一个经典的**生产者-消费者**模型，如果没有 `synchronized` 会发生什么：

**场景**：一个任务队列，消费者线程发现队列为空时，就等待；生产者线程放入任务后，就唤醒消费者。

#### 错误示范（没有 `synchronized`）

Java

```
// 这是一个【错误】的实现，用于演示问题
public void consume() {
    // 步骤1: 消费者检查队列是否为空
    if (queue.isEmpty()) {
        // 【危险时刻】: 就在这里，线程可能被切换！
        try {
            // 步骤4: 消费者准备等待
            queue.wait(); 
        } catch (InterruptedException e) {}
    }
    // ...消费任务...
}

public void produce() {
    // 步骤2: 生产者放入一个任务
    queue.add(task);
    // 步骤3: 生产者发出唤醒通知
    queue.notify();
}
```

**“丢失的唤醒”发生过程：**

1. **消费者**线程进入 `consume()` 方法，执行到 **步骤1**，发现 `queue` 是空的。
    
2. ==就在它判断为`true`之后，准备执行 `wait()` 之前，**发生了线程上下文切换**，CPU被**生产者**线程抢占。==
    
3. **生产者**线程执行 `produce()` 方法，在 **步骤2** 向队列中添加了一个任务。
    
4. ==紧接着，生产者在 **步骤3** 调用了 `notify()`。但此时此刻，**没有任何线程在等待**，所以这个 `notify()` 信号就像是对着空谷呐喊，**被永远地丢失了**。==
    
5. 随后，CPU又切换回**消费者**线程，它从上次中断的地方继续执行，也就是 **步骤4**，调用 `wait()` 方法进入等待状态。
    

**结果：** ==消费者永远地等待一个已经发生过的、被错过的 `notify()` 信号，造成**程序死锁**。==

#### 正确实现（使用 `synchronized`）

Java

```
public synchronized void consume() {
    // 使用 while 是为了防止“虚假唤醒”，这是另一个最佳实践
    while (queue.isEmpty()) { 
        try {
            // "检查条件" 和 "调用wait()" 成为了一个原子操作
            queue.wait(); 
        } catch (InterruptedException e) {}
    }
    // ...消费任务...
}

public synchronized void produce() {
    queue.add(task);
    queue.notify();
}
```

**`synchronized` 如何解决问题？**

1. **消费者**线程进入 `consume()`，**获得了锁**。它检查 `queue.isEmpty()` 为 `true`。
    
2. ==消费者调用 `wait()`。这个方法非常特殊，它会**原子性地**做两件事：① **将当前线程放入等待队列**；② **释放锁**。==
    
3. 现在，**生产者**线程可以获得锁了，它进入 `produce()` 方法，添加任务，然后调用 `notify()`。
    
4. 因为消费者调用 `wait()` 和释放锁是原子性的，**生产者不可能在消费者“检查完条件”和“进入等待”的间隙中插入执行**。因此，`notify()` 调用绝对不会在 `wait()` 之前发生，**“唤醒”信号就不会丢失**。
    

---

### 总结

|为什么必须用 `synchronized`|解释|
|---|---|
|**技术规范要求**|`wait/notify` 操作的是对象的监视器(monitor)，你必须先获取该监视器的锁，否则会抛出 `IllegalMonitorStateException`。|
|**防止“丢失的唤醒”**|`synchronized` 保证了“检查条件”（如 `queue.isEmpty()`）和“进入等待状态”（调用 `wait()`）这两个步骤成为一个**原子操作**，不会被其他线程中断，从而避免了竞态条件。|
|**保证内存可见性**|`synchronized` 还能保证当一个线程修改了共享变量（如任务队列）后，这个修改对其他线程是可见的。|
