在开始敲命令之前，必须要纠正一个很多人都会犯的认知误区：**你刚才遇到的 `Lock wait timeout exceeded`（锁等待超时）和真正的“死锁（Deadlock）”是两码事。**

- **锁超时：** 窗口 2 等窗口 1，等太久没耐心了（单向排队）。
    
- **真正的死锁：** 窗口 1 拿着 A 锁等 B 锁，窗口 2 拿着 B 锁等 A 锁。两人互相僵持，**MySQL 只要一发现死锁，会瞬间主动杀掉（Rollback）其中一个事务**，根本不会去等那 50 秒的超时时间。
    

搞清楚了区别，下面是你在生产环境排查这两大元凶的“实战命令工具箱”。

### 一、 查杀“死锁”与“锁阻塞”的命令大全

#### 1. 抓取“真正死锁”的案发现场日志

死锁发生极其迅速，一闪而过。要想知道刚才到底是谁和谁撞车了，直接看 InnoDB 的底层状态引擎：

SQL

```
SHOW ENGINE INNODB STATUS\G
```

**怎么看：** 在打印出的一大堆日志中，按 `Ctrl+F` 搜索 **`LATEST DETECTED DEADLOCK`**（最近检测到的死锁）。这下面会清清楚楚地记录：引发死锁的两个事务 ID 是什么、执行的什么 SQL、分别持有什么锁、在等什么锁，以及 MySQL 最终“赐死”了哪一个。

#### 2. 排查“锁等待（超时）”的元凶 (MySQL 8.0 版本)

如果你用的 MySQL 8.0（目前大厂主流），当线上系统卡住时，依次敲下这三条命令进行“破案”：

- **第一步：看谁在等锁（受害者）**
    
    SQL
    
    ```
    SELECT * FROM performance_schema.data_lock_waits;
    ```
    
- **第二步：看系统里都有哪些锁被谁占着（嫌疑人）**
    
    SQL
    
    ```
    SELECT * FROM performance_schema.data_locks;
    ```
    
- **第三步：终极绝杀（找出不提交的那个老赖）**
    
    SQL
    
    ```
    SELECT * FROM information_schema.innodb_trx WHERE trx_state = 'RUNNING';
    ```
    
    找到那个 `trx_started` 时间极早，且一直 `RUNNING` 的事务，拿到它的 `trx_mysql_thread_id`，然后直接执行 `KILL [线程ID];`，警报解除。
    

_(注：如果你用的是较老的 MySQL 5.7，对应的系统表是 `information_schema.innodb_locks` 和 `innodb_lock_waits`)_

### 二、 开启与排查“慢查询 (Slow Query)”的命令

慢查询日志是用来抓出那些拖慢系统的“垃圾 SQL”的。默认情况下 MySQL 是关闭它的（为了省性能），但在生产环境通常会打开并设置一个阈值（比如执行超过 1 秒就算慢）。

#### 1. 数据库内的配置与查看命令

SQL

```
-- 1. 查看慢查询日志是否开启，以及日志文件存放在哪里
SHOW VARIABLES LIKE '%slow_query_log%';

-- 2. 临时开启慢查询日志（重启失效，永久生效需修改 my.cnf）
SET GLOBAL slow_query_log = 'ON';

-- 3. 查看当前的慢查询时间阈值（默认是 10 秒）
SHOW VARIABLES LIKE 'long_query_time';

-- 4. 把阈值改成 1 秒（运行超过 1 秒的 SQL 都会被记下来）
SET GLOBAL long_query_time = 1;
```

#### 2. 服务器终端的分析命令（非 SQL）

当慢查询日志跑了一天之后，文件可能会非常大。你不可能用肉眼去看，这时候要在 **Linux 命令行**里使用 MySQL 官方自带的日志分析神器 **`mysqldumpslow`**：

Bash

```
# 找出访问次数最多的前 10 条慢 SQL
mysqldumpslow -s c -t 10 /var/lib/mysql/hostname-slow.log

# 找出平均执行时间最长的前 10 条慢 SQL（面试常考实操）
mysqldumpslow -s t -t 10 /var/lib/mysql/hostname-slow.log
```

### 实战演练：慢查询定位后的终极杀招（EXPLAIN）

用上面的命令抓到那条执行了 5 秒的慢 SQL 后，**千万不要去猜它为什么慢**。

MySQL 提供了一个叫 **`EXPLAIN`** 的命令。你只需要把这个词加在那条慢 SQL 的最前面，MySQL 就会把它的**底层执行计划**全盘托出。你看懂了这个计划，就能瞬间知道是索引没命中，还是发生了极其耗时的全表扫描（filesort）。
