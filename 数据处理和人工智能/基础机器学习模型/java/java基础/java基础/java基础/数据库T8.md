你之前被问到的这个面试题，是一个极其经典的**“优化器陷阱”**！

我先直接回答你的疑惑：**假如创建了联合索引 `(a, b)`，在 `WHERE` 后面写成 `b = 1 AND a = 2`，索引绝对不会失效！它依然会完美命中 `(a, b)` 联合索引。**

很多初学者死记硬背“最左前缀法则”，以为 SQL 语句里的书写顺序必须和索引顺序一模一样。但实际上，MySQL 内部有一个非常聪明的**查询优化器 (Query Optimizer)**。当它看到 `b = 1 AND a = 2` 时，会在底层默默地把它**重写排列**成 `a = 2 AND b = 1`，以此来匹配联合索引。

_核心记住：最左前缀法则指的是**索引树上的排序规则**和**查询条件里有没有这个列**，而不是你在 SQL 语句里敲键盘的先后顺序。_

---

### 导致 MySQL 索引失效的“六宗罪”

在日常开发中，索引失效通常是因为我们在不知不觉中破坏了 B+ 树的查找规则。以下是最常见的 6 个翻车场景：

1. **违反最左前缀法则：** 联合索引 `(a, b, c)`，查询条件是 `WHERE b = 1 AND c = 2`。因为跳过了带头的 `a`，B+ 树无法从根节点开始二分查找，索引彻底失效。
    
2. **在索引列上做“手脚”（函数、计算）：** 比如 `WHERE age + 1 = 18` 或 `WHERE SUBSTRING(name, 1, 1) = '张'`。B+ 树里存的是原生的 `age`，你加上 1 之后，引擎就不认识了，只能全表扫描挨个算。
    
3. **隐式类型转换（致命暗坑）：** 比如用户的手机号字段 `phone` 是 `VARCHAR` 类型。你写了 `WHERE phone = 13800138000`（没加引号，传了个整型）。MySQL 会自动用隐式函数把字符串列转成数字再比对，等同于在列上加了函数，索引瞬间失效。
    
4. **LIKE 模糊查询“左边带 %”：** `WHERE name LIKE '%张'`。B+ 树是按字典序从左到右排的，你连第一个字是什么都不知道，根本无法走树形查找，只能全表扫描。（但 `LIKE '张%'` 是可以走索引的）。
    
5. **范围查询右边的列失效：** 联合索引 `(a, b, c)`，如果写 `WHERE a = 1 AND b > 2 AND c = 3`。`a` 和 `b` 能走索引，但因为 `b` 是范围查询，导致符合 `b > 2` 的那一堆数据里，`c` 是无序的，所以 `c` 用不到索引。
    
6. **OR 连接的“猪队友”：** `WHERE a = 1 OR b = 2`。如果 `a` 有索引，但 `b` 没有索引。优化器一想，反正为了查 `b` 也要全表扫描，干脆 `a` 也别走索引了，直接一次性全表扫描完事。
