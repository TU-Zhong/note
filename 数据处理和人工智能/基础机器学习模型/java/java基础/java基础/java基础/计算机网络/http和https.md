HTTP（超文本传输协议）和 HTTPS（安全超文本传输协议）是我们在浏览网页、开发接口时最核心的网络协议。

简单一句话概括：**HTTPS 就是穿上了“防弹衣”（SSL/TLS 加密层）的 HTTP。**

打个形象的比方：

- **HTTP 就像寄“明信片”：** 邮递员、分拣员（沿途的路由器、宽带运营商、黑客）都能清楚地看到明信片上写了你的账号密码，甚至可以偷偷涂改上面的内容（比如运营商给你强行插播网页广告）。
    
- **HTTPS 就像寄“密码箱”：** 你把信件放进密码箱锁死，只有收件人手里有钥匙（私钥）。沿途的人只能看到一个上了锁的铁箱子，既不知道里面写了什么，也无法篡改。

### 1. 核心区别对比（面试必考）

|**对比维度**|**HTTP**|**HTTPS**|
|---|---|---|
|**安全性**|**明文传输**。数据毫无遮拦，极易被窃听、篡改或劫持。|**密文传输**。具有防窃听、防篡改、身份防伪装的特性。|
|**默认端口号**|**80**|**443**|
|**证书要求**|不需要。|必须向 CA（证书颁发机构）申请 SSL/TLS 数字证书（有免费的也有几千块一年的）。|
|**连接速度**|**快**。只需完成 TCP 的三次握手即可开始传输数据。|**略慢**。除了 TCP 三次握手，还需要进行一套极其复杂的 SSL/TLS 握手来协商密钥。|
|**SEO 搜索引擎**|权重较低。|谷歌、百度等搜索引擎优先收录 HTTPS 网站，排名更靠前。|
|**浏览器标识**|Chrome 等现代浏览器会在地址栏直接亮红灯，提示**“不安全”**。|地址栏会显示一把代表安全的**绿色小锁**。|

### 2. HTTPS 是如何保证绝对安全的？（进阶原理）

很多人以为 HTTPS 就是简单地把数据加密一下，其实它的设计极其精妙，完美解决了网络安全中的三大痛点：

#### A. 防窃听（机密性）—— 混合加密机制

HTTPS 并没有死板地只用一种加密方法，而是结合了两种加密算法的优点：

1. **非对称加密（慢但安全）：** 在刚开始“握手”认识的时候，双方利用非对称加密（如 RSA 算法），在黑客眼皮子底下极其安全地协商出一个**“对称密钥”**。
    
2. **对称加密（快）：** 协商好密钥后，后续真正传输大量网页、视频数据时，为了保证极高的性能，双方改用刚才协商好的“对称密钥”进行加密传输。
    

#### B. 防伪装（真实性）—— CA 证书机制

你怎么证明你现在访问的淘宝网是真的淘宝，而不是黑客搞的一个长得一模一样的钓鱼网站？

- 服务器必须亮出它的**数字证书**（类似于网站的“电子身份证”，由具有公信力的国际 CA 机构颁发）。
    
- 你的浏览器内部天生内置了这些权威 CA 机构的公钥，一旦核对发现证书是伪造的，或者域名不匹配，浏览器就会立刻弹窗拦截，阻止你访问。
    

#### C. 防篡改（完整性）—— 数字签名

哪怕黑客破解不了密码箱，他把密码箱砸了，换个一模一样的假密码箱送过去行不行？

不行。HTTPS 会对每一条传输的信息生成一个唯一的**“数字摘要（MAC）”**。接收方收到数据后算一下摘要，如果和随信附带的摘要对不上（哪怕黑客只修改了一个标点符号），接收方就会立刻发现数据被篡改，直接丢弃。

### 3. 行业现状

在十年前，可能只有银行、支付页面才会用 HTTPS，因为加密解密比较消耗服务器 CPU 资源。

但在今天，随着硬件性能的过剩以及 HTTP/2 协议的普及，**HTTPS 已经是全网强制的标准配置**。苹果 iOS (ATS 机制) 和安卓甚至在操作系统底层就直接一刀切，强制要求 App 内所有的网络请求必须走 HTTPS，否则直接拒绝发送。
