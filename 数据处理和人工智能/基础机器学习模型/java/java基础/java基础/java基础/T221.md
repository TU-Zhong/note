**双亲委派模型（Parents Delegation Model）** 是 Java 虚拟机（JVM）中类加载器（ClassLoader）协作的一种核心机制。

简单来说，它的核心思想是：**“啃老”。** 当一个类加载器需要加载某个类时，它首先不会自己去尝试加载，而是**把这个任务委派给自己的“父类加载器”**去完成。每一层都是如此，直到把任务推给最顶层的“老祖宗”。只有当父类加载器明确表示“我找不到这个类，我加载不了”时，子加载器才会迫不得已自己去尝试加载。

### 1. 类加载器的层级结构

在了解工作过程前，需要先知道 Java 中默认的类加载器层级（从上到下）：

1. **启动类加载器 (Bootstrap ClassLoader)：** 最顶层，由 C++ 实现。负责加载 `<JAVA_HOME>\lib` 目录下的核心类库（比如 `rt.jar`，里面包含了 `java.lang.String`、`java.lang.Object` 等基础类）。
    
2. **扩展类加载器 (Extension ClassLoader)：** 负责加载 `<JAVA_HOME>\lib\ext` 目录下的扩展类库。
    
3. **应用程序类加载器 (Application ClassLoader)：** 也就是我们常说的 System ClassLoader。负责加载用户类路径（ClassPath）上指定的类，也就是**你自己写的业务代码**以及引入的第三方 Jar 包。
    
4. **自定义类加载器 (Custom ClassLoader)：** 开发者自己继承 `java.lang.ClassLoader` 实现的加载器，通常用于加载网络上的类、加密的类等。
    

*(注：这里的“父子”关系通常不是通过类的继承 `extends` 实现的，而是通过**组合（Composition）*_模式，将父加载器作为子加载器的一个属性来保存的。)_

### 2. 工作过程解析

双亲委派的工作流程可以分为两个阶段：**向上委派** 和 **向下尝试加载**。

- **步骤 1（向上委派）：** 当应用程序类加载器收到一个类加载请求（例如要加载 `com.tuzhong.Test`），它不会自己先找，而是把请求转交给父类（扩展类加载器）。
    
- **步骤 2：** 扩展类加载器收到请求后，也不自己找，继续转交给它的父类（启动类加载器）。
    
- **步骤 3（到达顶层）：** 启动类加载器收到请求，它已经没有父类了。于是它开始在自己的地盘（`<JAVA_HOME>\lib`）里寻找这个类。
    
- **步骤 4（向下尝试）：** * 如果启动类加载器找到了（比如你要加载的是 `java.lang.String`），它就直接加载返回，流程结束。
    
    - 如果没找到（比如你要找的是你自己写的 `com.tuzhong.Test`），它会把结果反馈给子类：“我这里没有，你自己找吧”。
        
- **步骤 5：** 扩展类加载器收到父类的反馈，开始在自己的地盘（`ext` 目录）找。找不到，再反馈给应用程序类加载器。
    
- **步骤 6：** 应用程序类加载器在 ClassPath 中寻找。这时候终于找到了你自己写的类，完成加载。

### 3. 为什么要设计这种模型？

在后端的面试或实际架构设计中，理解“Why”比“What”更重要。双亲委派模型主要解决了两个痛点：

1. **沙箱安全机制（核心防御）：** 防止核心 API 被篡改。假设有人写了一个恶意的类，包名和类名也叫 `java.lang.String`，里面植入了病毒代码。如果没有双亲委派，系统可能就加载了这个假 `String`。但有了双亲委派，请求一定会先送到最顶层的 Bootstrap。Bootstrap 一看是 `java.lang` 包下的，直接从核心库加载了正版的 `String` 类。那个**伪造的类永远得不到被加载的机会**。
    
2. **避免类的重复加载：** 当父加载器已经加载了该类时，子加载器就没必要再加载一次，保证了内存中类的**全局唯一性**。如果在内存中存在两个由不同 ClassLoader 加载的同一个类，JVM 会认为它们是两个完全不同的类，这会导致类型转换异常（ClassCastException）。
    

_(扩展思考：既然双亲委派这么好，有没有打破这个模型的场景？实际上，为了实现热部署、或者解决基础类需要调用用户代码接口的场景（如 JDBC Driver SPI），Tomcat、OSGi 以及 Java 本身的线程上下文类加载器，都采取了打破双亲委派机制的做法。这通常是进阶开发需要掌握的内容。)_
