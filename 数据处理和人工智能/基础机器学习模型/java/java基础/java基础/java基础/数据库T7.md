**索引下推（Index Condition Pushdown, 简称 ICP）**是 MySQL 5.6 引入的一项极其重要的查询优化技术。

要理解它解决了什么痛点，我们必须先弄懂在没有它之前，MySQL 是怎么干活的。它的核心痛点在于：**无谓的“回表”操作太多，白白浪费了大量的磁盘 I/O。**

我们通过一个经典的业务场景来拆解它。

### 痛点场景再现（没有 ICP 的时代）

假设你有一张用户表 `users`，里面建了一个**联合索引 `(name, age)`**。

现在你要执行一条查询：

SQL

```
SELECT * FROM users WHERE name LIKE '张%' AND age = 20;
```

根据联合索引的**最左前缀匹配原则**，`name LIKE '张%'` 可以用到索引，但由于它是范围查询（LIKE），导致后面的 `age = 20` **无法在索引树中用于精确匹配（走索引用来定位记录）**。

在 MySQL 5.6 之前（**关闭 ICP**），架构层的分工是非常死板的：

1. **存储引擎层（InnoDB）：** 只能根据 `name LIKE '张%'` 这个条件，去联合索引树里把所有姓张的记录找出来。
    
2. **回表（最痛的一步）：** 存储引擎找到一个姓张的，不管他几岁，**立刻拿着主键 ID 回表**（去聚簇索引查整行数据）。
    
3. **Server 层：** 存储引擎把查到的完整数据交给 MySQL Server 层。Server 层再根据 `age = 20` 进行过滤。
    

**痛点在哪里？**

假设全表有 10 万个姓“张”的人，但其中只有 10 个人是 20 岁。

为了找这 10 个人，InnoDB 必须做 **10 万次回表**（随机磁盘 I/O），把 10 万条完整记录传给 Server 层，最后 Server 层又扔掉了其中的 99990 条。这简直是灾难级的性能浪费！

### 解决方案：索引下推（ICP）

既然联合索引树里**本来就存了 `age` 的值**（联合索引包含 name 和 age），为什么存储引擎不自己先看一眼 age，而非要把过滤工作交给 Server 层呢？

于是 MySQL 5.6 引入了 **ICP**。开启 ICP 后，工作流程变成了这样：

1. **存储引擎层（InnoDB）：** 根据 `name LIKE '张%'` 找到记录。
    
2. **下推过滤（核心）：** Server 层把 `age = 20` 这个条件**“下推”**给了存储引擎。存储引擎在回表之前，直接看一眼联合索引里的 `age` 字段。
    
3. **精准回表：** 如果 `age != 20`，直接丢弃，**绝对不回表**！只有当 `age == 20` 时，才进行回表查出整行，交给 Server 层。
    

**结果：** 原本的 10 万次回表，骤降到了 10 次回表。查询速度实现了量级上的飞跃。

### 总结与应用

在执行 `EXPLAIN` 分析 SQL 语句时，如果你在 `Extra` 列中看到了 **`Using index condition`**，这就说明 MySQL 成功为你触发了索引下推优化。

需要注意的是，ICP 发挥作用是有前提的：

- 它只能用于**二级索引**（聚簇索引包含了整行数据，不需要回表，自然用不上 ICP）。
    
- 这个用于过滤的字段，**必须存在于你建立的那个索引之中**。如果条件里的字段不在索引里，存储引擎想提前看也看不到，还是只能乖乖回表。
