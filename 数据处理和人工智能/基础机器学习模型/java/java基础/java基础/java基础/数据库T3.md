这个问题问到了 MySQL 并发控制的“灵魂”！在面试中，能把 MVCC 的底层运作逻辑（特别是 Read View 的判断规则）条理清晰地讲出来，是区分普通开发者和高级开发者的重要分水岭。

**MVCC（Multi-Version Concurrency Control，多版本并发控制）**，顾名思义，就是通过维护数据的**多个历史版本**，让读写操作没有冲突。它的终极目标是实现**“读不上锁，读写不冲突”**，极大提升数据库的并发性能。

要彻底搞懂 MVCC 的工作原理，必须拆解它的三大核心基石：**隐藏字段**、**Undo Log（版本链）** 和 **Read View（读视图）**。

---

### 基石一：数据的“隐藏基因”（隐式字段）

在 InnoDB 中，你建表时定义的字段只是表象。MySQL 会在底层默默给每一行数据加上几个隐藏字段，最核心的是这两个：

1. **`DB_TRX_ID` (事务 ID)：** 记录**最后一次**插入或修改这行数据的事务 ID。你可以把它理解为这行数据的“版本号”。
    
2. **`DB_ROLL_PTR` (回滚指针)：** 指向 Undo Log 中这行数据的**上一个版本**的内存地址。
    

### 基石二：时光机（Undo Log 版本链）

当一个事务修改某行数据时，InnoDB 不会直接覆盖旧数据，而是：

1. 把旧数据拷贝到 Undo Log 中。
    
2. 修改当前数据，并把当前数据的 `DB_TRX_ID` 改为自己的事务 ID。
    
3. 把当前数据的 `DB_ROLL_PTR` 指向 Undo Log 中的旧数据。
    

如果有多个事务先后修改同一行数据，Undo Log 里就会形成一条**按时间倒序排列的“单向链表”**。链表头是最新的数据，顺着指针往下找，就是越来越老的数据。

### 基石三：上帝之眼（Read View 读视图）

版本链建好了，但当一个查询操作（普通的 `SELECT`）到来时，它应该读取链条上的哪一个版本呢？这就需要 **Read View** 来做决断。

Read View 就像是查询发起那一瞬间，数据库系统拍下的一张**“快照”**。这个快照里记录了当前系统里**所有还没提交的活跃事务**。

Read View 内部包含了四个极其关键的属性（**面试必考点**）：

- **`creator_trx_id`**：创建这个 Read View 的事务 ID（也就是当前发起查询的事务 ID）。
    
- **`m_ids`**：生成快照时，当前系统中**所有活跃（未提交）**的事务 ID 列表。
    
- **`min_trx_id`**：`m_ids` 列表里最小的事务 ID。
    
- **`max_trx_id`**：系统即将分配给下一个事务的 ID（也就是当前最大事务 ID + 1）。
    

---

### 核心运作机制：可见性判断规则（四步走）

当你的查询顺着 Undo Log 版本链往下找数据时，每遇到一个版本，就会拿出它的 `DB_TRX_ID`，按照以下规则与 Read View 对比，判断这个版本对自己**是否可见**：

1. **自己改的，自己当然能看：**
    
    如果被访问版本的 `DB_TRX_ID` == `creator_trx_id`。说明这个版本就是当前事务自己修改的，**可见**。
    
2. **老前辈改的，且已完结，可以看：**
    
    如果被访问版本的 `DB_TRX_ID` < `min_trx_id`。说明修改这个版本的事务在快照生成之前就已经提交了，**可见**。
    
3. **未来人改的，绝不能看：**
    
    如果被访问版本的 `DB_TRX_ID` >= `max_trx_id`。说明修改这个版本的事务是在快照生成**之后**才开启的，**不可见**。
    
4. **同时期的同辈，查户口（最复杂的情况）：**
    
    如果 `min_trx_id` <= `DB_TRX_ID` < `max_trx_id`。这时候需要判断这个 `DB_TRX_ID` 是否在活跃列表 `m_ids` 里面：
    
    - 如果在 `m_ids` 里：说明快照生成时，那个事务**还没提交**，所以它的修改**不可见**。
        
    - 如果不在 `m_ids` 里：说明快照生成时，那个事务**已经提交**了，**可见**。
        

如果当前版本不可见，查询就会顺着 `DB_ROLL_PTR` 找到上一个版本，再次套用这四条规则，直到找到第一个可见的版本为止。
