这四个问题可以说是 Redis 面试中**“出场率 100%”**的四大金刚。掌握它们不仅是为了应付面试，更是设计高并发系统时的必备防身技能。

我们把前三个问题（穿透、击穿、雪崩）统称为**“缓存异常三大战役”**，它们的核心矛盾都是同一个：**Redis 挡不住了，流量像洪水一样砸向了脆弱的 MySQL 数据库。**

下面我们逐一拆解，并在最后为你提供一个动态推演沙盘，让你彻底看清这三种灾难的发生过程。

### 23. 什么是缓存穿透？如何解决？（布隆过滤器的原理）

**定义：** 客户端请求的数据，在**缓存中不存在，在数据库中也不存在**。

**危害：** 既然数据库都没有，缓存里自然不可能有。这就导致每次请求都会“穿透” Redis，直接砸向 MySQL。如果黑客恶意伪造大量不存在的 ID（比如 `id = -1`）发起并发攻击，MySQL 会瞬间被查询拖垮。

**解决方案：**

1. **缓存空对象（简单粗暴）：** 只要发现数据库返回的是空，我也把这个“空结果”存到 Redis 里，并给它设置一个极短的过期时间（比如 5 分钟）。下次同样的假 ID 过来，Redis 直接返回 null，挡住攻击。
    
    - _缺点：_ 消耗额外内存；可能产生短期的缓存与数据库不一致。
        
2. **布隆过滤器 Bloom Filter（高端优雅，大厂必考）：** 在请求到达 Redis 之前，先加一道“安检门”。
    

**布隆过滤器底层原理：**

- **结构：** 它的底层是一个超大的二进制位数组（Bit Array，初始全为 0），以及几个不同的 Hash 函数。
    
- **写入（登记）：** 当我们往数据库插入一条数据（比如 `id=10`）时，经过几个 Hash 函数计算，算出几个数组的索引位置，把这些位置的 `0` 改成 `1`。
    
- **查询（安检）：** 当用户查询 `id=10` 时，同样经过这几个 Hash 计算，去查对应的位置。
    
    - **如果发现有个位置是 `0`：** 说明这个数据**绝对不存在**！直接拦截，请求连 Redis 都碰不到。
        
    - **如果发现所有位置都是 `1`：** 说明这个数据**可能存在**（因为存在 Hash 冲突，这几个 1 可能是其他数据撞运气涂黑的）。放行请求去查 Redis。
        
- **口诀：** **“布隆说没有，那就绝对没有；布隆说有，可能也没有。”**（宁可错杀一千，绝不放过一个）。

### 24. 什么是缓存击穿？如何解决？（互斥锁、逻辑过期方案）

**定义：** 缓存中**没有**，但数据库中**有**的数据。通常发生在一个**超级热点 Key**（比如秒杀商品、微博热搜）刚好过期失效的瞬间。

**危害：** 就在热点 Key 失效的这一秒内，成千上万的并发请求发现缓存没命中，于是所有人同时冲向 MySQL 试图重建缓存。这叫“单点爆破”，MySQL 会瞬间被压垮。

**解决方案（两种派别）：**

1. **互斥锁（Mutex Lock / 分布式锁）—— 保证强一致性**
    
    - **做法：** 发现缓存没命中时，大家不要一起去查 DB。而是必须要先抢一把锁（比如 Redis 的 `setnx`）。抢到锁的那**唯一一个线程**去查 DB 并重建缓存。没抢到锁的线程必须休眠一会儿，然后重试查缓存。
        
    - **优缺点：** 数据绝对一致；但高并发下大量线程阻塞排队，性能受损，甚至有死锁风险。
        
2. **逻辑过期方案 —— 保证高可用性（宁可脏读，绝不阻塞）**
    
    - **做法：** 针对热点 Key，我们**不设置物理过期时间（不使用 `expire` 命令）**，而是把过期时间作为一个字段存到 Value 里面（比如 `{"data": {...}, "expireTime": 16900000}`）。
        
    - **运行机制：** 线程查到数据后，判断一下这个时间戳，如果发现已经“逻辑过期”了，它会先**立刻把这份旧数据返回给前端**（保证响应极快），同时在后台偷偷开启一个异步线程，去抢锁、查 DB、更新缓存。
        
    - **优缺点：** 性能极高，没有任何阻塞等待；缺点是会产生短暂的数据不一致（前端会看到几秒钟的旧数据）。

### 25. 什么是缓存雪崩？如何解决？

**定义：** **大量 Key 在同一时间点同时过期**，或者 **Redis 节点直接宕机**。

**危害：** 击穿只是“单点爆破”，雪崩是“大面积崩塌”。海量请求直接砸给 MySQL，导致整个系统链路崩溃。

**解决方案：**

1. **打散过期时间（最常用）：** 给原本要设置的 TTL 加上一个随机值（比如原定 1 小时，改成 `1小时 + 随机1~5分钟`）。让 Key 的失效时间均匀散开，避免同一秒集体“阵亡”。
    
2. **Redis 高可用部署：** 搭建哨兵模式（Sentinel）或集群模式（Cluster），防止 Redis 单机宕机引发雪崩。
    
3. **多级缓存：** 增加 JVM 本地缓存（如 Caffeine），即使 Redis 挂了，本地缓存还能扛住第一波。
    
4. **服务降级/限流：** 如果事态彻底失控，启动 Hystrix 或 Sentinel 直接拒绝掉一部分请求，**牺牲部分用户体验，保住 MySQL 老命**。

### 26. Redis 的内存淘汰策略有哪些？（LRU、LFU 等的区别）

**背景：** 当 Redis 的内存使用量达到了 `maxmemory` 限制时，如果还有新数据要塞进来，Redis 必须无情地“干掉”一部分老数据。这就是淘汰策略。

Redis 提供了 8 种策略，但面试官真正在意的，是你能不能讲清楚 **LRU** 和 **LFU** 的核心区别：

1. **LRU (Least Recently Used - 最近最少使用)**
    
    - **核心逻辑：看“时间”。** 谁离上一次被访问的时间最久，谁就先被淘汰。
        
    - **盲区/缺陷：** 如果有个平时基本不用的冷门数据，就在 1 秒前恰好被查了一次，LRU 会认为它是“新鲜”的，从而让它存活；而一个平时天天用的热门数据，只是刚好这半小时没被查，反而可能被 LRU 冤杀。
        
2. **LFU (Least Frequently Used - 最不经常使用)**
    
    - **核心逻辑：看“频率”。** 谁在一段时间内被访问的次数最少，谁就被淘汰。_（Redis 4.0 之后引入的终极杀器）_
        
    - **优势：** 完美解决了 LRU 的盲区。刚才那个“1秒前碰巧被查了一次”的数据，因为总访问频率极低，会被 LFU 精准锁定并干掉，真正保护了高频热点数据。
        

**其他常规策略（了解即可）：**

- `noeviction`（默认）：不淘汰，内存满了再写直接报错。
    
- `volatile-ttl`：在设置了过期时间的 Key 中，淘汰最快要过期的。
    
- `allkeys-random` / `volatile-random`：闭着眼睛瞎淘汰（随机杀），一般没人用。
    

**面试满分小贴士：**

实际底层实现中，Redis 并没有维护一个庞大的时间链表去做真正的 LRU，因为那样太耗内存了。Redis 采用的是**“近似 LRU 算法”**：在对象的结构体里留了 24 个 bit 来记录最后一次访问的时间戳。每次淘汰时，随机挑 5 个 Key（数量可配），把其中时间最老的干掉，用极小的 CPU 成本达到了极其接近完美 LRU 的效果。这就是 Redis 压榨性能的艺术所在。
