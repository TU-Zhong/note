**可以，但是这里有一个极其重要的高级“坑”需要提前说明。**

在回答具体 SQL 之前，你必须先区分两个概念：**“锁等待（阻塞）”** 和 **“死锁”**。

1. **锁等待（Lock Wait）：** 事务 A 拿了锁，事务 B 在傻傻等待。这时候你**完全可以通过 SQL 查出来**谁在等谁。
    
2. **死锁（Deadlock）：** A 等 B，B 等 A。一旦发生这种情况，MySQL 的 InnoDB 引擎会在毫秒级内瞬间介入，**直接杀掉（回滚）其中一个事务**以打破僵局。
    

**致命后果：** 因为死锁发生的瞬间就被 MySQL “干掉”了一个，锁被释放了。所以，**你无法通过传统的 `SELECT` 表查询来抓到已经发生并结束的死锁记录**（查表只能查到当前正在发生的“锁等待”）。

如果你想用纯 SQL 排查**当前正在卡住的锁**，或者追踪**历史死锁**，这里是业界标准的排查套路：

---

### 第一招：排查“此时此刻”谁被谁锁住了（纯 SQL 查询）

在 MySQL 5.7 及以上版本（尤其是 8.0），强烈建议直接使用 `sys` 库，它提供了一个“上帝视角”的视图，把复杂的底层锁关系翻译成了人话。

**执行下面这条神级 SQL：**

SQL

```sql
SELECT 
    waiting_pid AS '被阻塞的线程ID',
    waiting_query AS '被阻塞的SQL',
    blocking_pid AS '罪魁祸首的线程ID',
    blocking_query AS '罪魁祸首正在执行的SQL',
    wait_age AS '已经等了多久',
    locked_table AS '被锁的表'
FROM sys.innodb_lock_waits;
```

- **实战用法：** 如果你发现系统卡住了，执行这条 SQL。看 `blocking_pid`（罪魁祸首），如果它占着茅坑不拉屎，你可以直接执行 `KILL 那个线程ID;` 来释放锁。
    

_(注：如果你用的是极其古老的 MySQL 5.6，你需要查 `information_schema.innodb_lock_waits` 和 `innodb_locks`；如果是 MySQL 8.0+ 底层表变成了 `performance_schema.data_locks`)_

---

### 第二招：排查“刚刚”发生的真正死锁

刚才说了，真死锁发生后就从系统表里消失了，无法用 `SELECT` 查表。你只能使用 MySQL 的引擎内置命令来查看**最近一次**的死锁案发现场：

SQL

```sql
SHOW ENGINE INNODB STATUS;
```

执行后，在返回的一大坨文本中搜索 `LATEST DETECTED DEADLOCK`。这里面会原封不动地保留最后一次死锁发生时的两个事务、它们执行的 SQL 以及抢夺的锁。

---

### 第三招：记录所有的历史死锁（用于复盘）

`SHOW ENGINE INNODB STATUS` 有一个缺点：它只记录“最后一次”。如果半夜连续发生了三次死锁，前两次就被覆盖了。

为了能事后复盘所有的死锁，你需要通过 SQL 开启一个全局开关：

SQL

```
-- 将所有的死锁信息打印到 MySQL 的 error log（错误日志）中
SET GLOBAL innodb_print_all_deadlocks = 'ON';
```

开启这个开关后，以后每次发生死锁，MySQL 都会把死锁的详细现场写入到它的 `error.log` 文件中。第二天上班时，你就可以去 Linux 服务器上翻看日志，把昨天夜里所有的死锁 SQL 全部揪出来优化。

**一句话总结：**

- 查**正在卡住的锁**：用 `SELECT * FROM sys.innodb_lock_waits;`
    
- 查**刚发生的死锁**：用 `SHOW ENGINE INNODB STATUS;`
    
- 查**历史所有死锁**：开启 `innodb_print_all_deadlocks` 去看错误日志。
