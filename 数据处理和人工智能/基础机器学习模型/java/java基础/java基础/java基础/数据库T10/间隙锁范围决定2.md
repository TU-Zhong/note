这个问题算是问到 InnoDB 锁机制的真正大动脉了！

上一轮我们弄懂了间隙是客观存在的物理空间（木桩之间的空地）。那么当你执行一条 SQL 时，MySQL 到底会选择给哪一片空地拉上警戒线呢？

核心决策规则只有两条：**“你的查询条件落在哪两根木桩之间”**，以及**“你用的是什么类型的索引”**。

为了让你彻底通透，我们直接把大厂面试中最爱考的 3 种加锁范围场景拆解开来（假设当前表里依然只有三行数据：`10, 20, 30`）：

### 规则一：等值查询一个“不存在”的值（精准封锁落点）

- **SQL 场景：** `SELECT * FROM table WHERE id = 15 FOR UPDATE;`
    
- **底层寻址：** MySQL 会顺着 B+ 树往下找，发现 15 应该排在 10 和 20 之间，但这里没有数据。
    
- **范围决定：** 为了防止幻读（也就是防止事务执行期间真的有人把 15 插进来），MySQL 会**直接锁住 15 所在的那个空隙**。
    
- **最终锁范围：** **`(10, 20)` 间隙锁**。别人想插 11、15、19 都会被阻塞，但插 25 没问题。
    

### 规则二：普通索引的等值查询（最凶狠的“前后双杀”）

这是开发中最容易踩坑，也是最容易引发大规模死锁的场景！

- **SQL 场景：** 假设这次 10, 20, 30 不是主键了，而是普通索引 `age`。执行 `SELECT * FROM table WHERE age = 20 FOR UPDATE;`
    
- **底层寻址：** 找到了 `age = 20` 的这根木桩。
    
- **范围决定：** 因为是普通索引，**它不唯一！** 这意味着别人还可以再插一条 `age = 20` 的数据进来。新的 20 按照 B+ 树的主键排序规则，可能会排在老 20 的左边空隙，也可能排在右边空隙。
    
- **最终锁范围：** 宁可错杀一千，绝不放过一个！MySQL 会把 20 **前面的间隙和后面的间隙全部锁死**。最终锁定范围是 `(10, 20]` 的临键锁 + `(20, 30)` 的间隙锁。加起来就是 **`(10, 30)`** 这个巨大的区间！在这个区间内，谁也别想动。
    

### 规则三：范围查询（顺藤摸瓜，沿途全锁）

- **SQL 场景：** `SELECT * FROM table WHERE id > 15 AND id < 25 FOR UPDATE;`
    
- **底层寻址：** 找起点 `id > 15`，落在 `(10, 20)` 间隙；顺着底部双向链表往后滑，找终点 `< 25`，落在 `(20, 30)` 间隙。
    
- **范围决定：** 只要是查询扫描经过的地方，不管有没有数据，统统加上临键锁。
    
- **最终锁范围：** `(10, 20]` 和 `(20, 30]`。
