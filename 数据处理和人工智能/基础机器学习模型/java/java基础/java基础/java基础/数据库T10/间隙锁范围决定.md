这是一个非常核心且极易懵圈的问题！你能问出这句话，说明你已经在思考 B+ 树底层的物理结构了。

很多教程里说“锁住 10 到 20 的间隙”，但这其实会给人一种错觉：好像 MySQL 提前在地上画好了格子一样。

其实根本不是这样！一句话破除迷雾：**间隙（Gap）是由当前索引树里【已经存在的真实数据】决定的！**

我们可以把当前表里已经存在的索引记录，想象成钉在泥地里的一根根**“木桩”**。而**木桩与木桩之间的空地，就是所谓的“间隙”**。

### 拆解“间隙”是怎么诞生的

假设你有一张表，主键 `id` 只有三行真实数据：`10`, `20`, `30`。

这三根“木桩”钉在 B+ 树的叶子节点上，会自动把整个数学空间硬生生切分成 **4 个间隙**：

1. **`(-∞, 10)`**：比最小木桩还小的前方空地。
    
2. **`(10, 20)`**：10 和 20 两根木桩之间的空地。
    
3. **`(20, 30)`**：20 和 30 两根木桩之间的空地。
    
4. **`(30, +∞)`**：比最大木桩还大的后方空地。（_底层原理：MySQL 会在每个数据页的最后偷偷加一个虚拟记录叫 Supremum，代表正无穷大。_）
    

**关键点来了：间隙是【动态】的！**

- **插入数据 = 钉下新木桩：** 如果你此时成功 `INSERT` 了一条 `id = 15` 的数据。那么原本 `(10, 20)` 的这个大间隙，瞬间就被劈成了两个小间隙：`(10, 15)` 和 `(15, 20)`。
    
- **删除数据 = 拔掉木桩：** 如果你把 `id = 20` 给 `DELETE` 掉了。那么原本的两个小间隙 `(15, 20)` 和 `(20, 30)`，就会瞬间合并成一个大间隙：`(15, 30)`。
    

这就是为什么 MySQL 的间隙锁能防住幻读：它锁的不是某个死板的数字范围，它锁的是**当前这两个数据节点之间的那段物理链表空间**。别人想在这段空间的链表上插新节点，就会被挡住。
