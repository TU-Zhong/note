这是 MySQL 面试中极其硬核的一个知识板块。很多开发者对锁的认知只停留在“行锁”和“表锁”这两个词上。如果想在面试中脱颖而出，你必须能够从**“颗粒度（范围）”**、**“兼容性（性质）”**以及 InnoDB 特有的**“行锁算法”**这三个维度，像庖丁解牛一样把它们拆解清楚。

我们一层一层来剥开 MySQL 锁的体系：

### 维度一：按“颗粒度（生效范围）”分类

这是最直观的分类方式，指的是这把锁一次性锁住了多少数据。

1. **全局锁 (Global Lock)**
    
    - **作用：** 直接把整个数据库实例锁死，只允许读，不允许任何写操作（增删改、建表改表全部阻塞）。
        
    - **命令：** `FLUSH TABLES WITH READ LOCK;` (FTWRL)
        
    - **场景：** 通常只用于**全库逻辑备份**（比如用 mysqldump 导数据时，保证备份期间数据的一致性）。
        
2. **表级锁 (Table-level Lock)**
    
    - **表锁 (Table Lock)：** 直接锁住整张表。由于颗粒度太大，极易引起阻塞，在 InnoDB 中极少主动使用。
        
    - **元数据锁 (MDL, Metadata Lock)：** 这是 MySQL 自动加的**隐式锁**。当你在对表执行 DML（增删改查）时，系统会自动加 MDL 读锁；当你在执行 DDL（比如 `ALTER TABLE` 加字段）时，会自动加 MDL 写锁。它的核心作用是：**防止你在查数据的同时，别的老六把表结构给改了（比如删了一列），导致查询崩溃。**
        
    - **意向锁 (Intention Lock)：** 这是一种非常有意思的“预告”锁。当一个线程要在表里的某一行加“行锁”时，它必须先在表级别挂一个“意向锁”。
        
        - _目的：_ 如果没有意向锁，当线程 B 想要给整张表加“表锁”时，它必须一行一行去遍历全表，看看有没有哪一行被别人加了行锁（极其低效）。有了意向锁，线程 B 只要看一眼表上有没有意向锁，就知道里面肯定有人持有了行锁，直接排队等待即可。**意向锁是用来提高加表锁效率的标志位。**
            
3. **行级锁 (Row-level Lock)**
    
    - **作用：** 颗粒度最细，只锁住特定的行，并发性能最高，极容易发生死锁。
        
    - **注意：** **行锁是 InnoDB 引擎独有的！** 像 MyISAM 引擎压根不支持行锁，只能锁全表。

### 维度二：按“兼容性（锁的性质）”分类

无论是表锁还是行锁，在性质上都可以分为这两种（也就是大名鼎鼎的读写锁）：

1. **共享锁 (Shared Lock / S 锁 / 读锁)**
    
    - **特性：** “读读共享，读写排斥”。我加了 S 锁，你可以读，你也可以继续加 S 锁；但谁都不能改（不能加 X 锁）。
        
    - **加锁方式：** `SELECT ... LOCK IN SHARE MODE;`
        
2. **排他锁 (Exclusive Lock / X 锁 / 写锁)**
    
    - **特性：** “写写排斥，读写排斥”。我加了 X 锁，意味着这行数据被我霸占了。你们谁都不能改，也不能用当前读，只能通过 MVCC 去看历史快照（这就是我们第一天聊的 MVCC 的作用）。
        
    - **加锁方式：** `UPDATE`, `DELETE`, `INSERT` 会自动加 X 锁。手动加锁：`SELECT ... FOR UPDATE;`
        

---

### 维度三：InnoDB 行锁的三大核心算法（面试绝对重点！）

前面说的行锁，听起来很简单：“锁住那一行就行了呗”。但实际上，InnoDB 的行锁**不是锁在数据记录上，而是锁在索引树上的！** 基于索引，InnoDB 演化出了三种极其精妙的行锁算法，专门用来对付并发幻读问题：

1. **记录锁 (Record Lock)**
    
    - **定义：** 严丝合缝地只锁住索引树上的**一条具体记录**。
        
    - **触发条件：** 精确匹配唯一索引或主键时（例如 `WHERE id = 10`）。
        
2. **间隙锁 (Gap Lock)**
    
    - **定义：** 不锁记录本身，而是锁住**两条索引记录之间的“空隙”**，禁止其他事务在这个空隙里插入新数据。
        
    - **目的：** **彻底消灭“幻读”！**（回忆一下幻读，就是别人在你查询的范围内 `INSERT` 了一条新数据）。把空隙锁住，幽灵数据就插不进来了。
        
    - **注意：** 间隙锁只有在 **RR（可重复读）** 隔离级别下才会生效。
        
3. **临键锁 (Next-Key Lock)**
    
    - **定义：** `Record Lock` + `Gap Lock` 的结合体。它既锁住目标记录本身，又锁住目标记录**前面的那段间隙**。（它锁定的是一个**左开右闭**的区间 `( , ]` ）。
        
    - **大白话：** 默认情况下，InnoDB 的行锁**其实都是临键锁**。只有当查询条件是唯一索引且精确命中时，它才会“退化”为仅仅锁住一行的记录锁（Record Lock）以提高性能。

### 总结串联

回想一下我们之前聊的事务隔离级别。

- 在 **RC（读已提交）** 级别下，只有记录锁（Record Lock），没有间隙锁。所以别的事务可以随意插入数据，这就会导致**幻读**。
    
- 在 **RR（可重复读）** 级别下，InnoDB 默认使用的是 **临键锁（Next-Key Lock）**，通过把数据之间的“缝隙”全部焊死，硬生生地在当前读的场景下，把幻读这个并发界的幽灵给彻底扼杀了！
