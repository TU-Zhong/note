这是一个在Java开发者中流传最广、也最容易引起争论的问题之一。

**标准答案是：Java 中只有**值传递 (pass-by-value)**，没有引用传递 (pass-by-reference)。**

之所以会产生巨大的困惑，是因为Java中传递对象时的“表现”看起来非常像引用传递。下面我们来彻底澄清这个概念。

### 什么是值传递？

**值传递 (Pass-by-value)**：在方法调用时，实际参数的**值**被复制一份，然后传递给形式参数。这意味着，在方法内部对形式参数的任何修改，都**不会影响**到方法外部的实际参数。

---

为了理解Java的行为，我们需要分两种情况来看：**传递基本数据类型**和**传递对象类型**。

### 场景一：传递基本数据类型 (Primitive Types)

这是最简单、最没有争议的情况，它清晰地展示了值传递的本质。

Java

```
public class Main {
    public static void main(String[] args) {
        int num = 10;
        System.out.println("调用方法前: " + num); // 输出: 10
        modify(num);
        System.out.println("调用方法后: " + num); // 输出: 10
    }

    public static void modify(int value) {
        value = 20; // 修改的是 value 这个副本
    }
}
```

**内存分析：**

1. `main` 方法中，变量 `num` 的值为 `10`。
    
2. 调用 `modify(num)` 时，Java 将 `num` 的**值（也就是10）复制了一份**，然后将这份副本赋值给了 `modify` 方法的参数 `value`。
    
3. 在 `modify` 方法内部，`value` 的值变成了 `20`。但这只是在修改那个**副本**，`main` 方法中的原始变量 `num` 毫发无损，依然是 `10`。
    

**结论**：对于基本类型，是纯粹的值传递，非常清晰。

---

### 场景二：传递对象类型 (Object Types)

**这是最容易混淆的地方。** 很多人看到在方法内部可以修改对象的属性，就误以为是引用传递。

让我们用一个比喻来解释：**酒店房间钥匙**

- **对象**：好比酒店里的一个**房间**。
    
- **对象的引用**：好比这个房间的**钥匙**。变量（如 `person`）里存的不是整个房间，而是能找到这个房间的钥匙。
    

**代码示例：**

Java

```
public class Person {
    private String name;
    public Person(String name) { this.name = name; }
    public void setName(String name) { this.name = name; }
    public String getName() { return this.name; }
}

public class Main {
    public static void main(String[] args) {
        Person person = new Person("张三");
        System.out.println("调用方法前: " + person.getName()); // 输出: 张三

        modify(person);

        System.out.println("调用方法后: " + person.getName()); // 输出: 李四
    }

    public static void modify(Person p) {
        // 通过钥匙副本进入房间，改变了房间里的东西
        p.setName("李四");
    }
}
```

看到这里，`person` 的名字从“张三”变成了“李四”，外部的 `person` 对象确实被修改了！这不就是引用传递吗？

**别急，这正是迷惑之处。让我们用“钥匙”的比喻来分析到底发生了什么：**

1. `main` 方法里的 `person` 变量，是一把指向“张三”这个 `Person` 对象（房间）的**钥匙**。
    
2. 调用 `modify(person)` 时，Java 遵循**值传递**原则，它复制了 `person` 变量的**值**。这个值是什么？是那把**钥匙**！
    
3. 所以，Java **复制了一把一模一样的钥匙**，并把这把**钥匙副本**交给了 `modify` 方法的参数 `p`。
    
4. 现在，`main` 方法里的 `person` 和 `modify` 方法里的 `p` 是**两把不同的钥匙**，但它们指向的是**同一个房间**。
    
5. 在 `modify` 方法里，`p.setName("李四")` 的操作，相当于拿着钥匙副本打开了那个唯一的房间，把里面的住户名字从“张三”改成了“李四”。
    
6. 方法结束后，钥匙副本 `p` 被销毁。但房间里的住户已经被永久改变了。所以 `main` 方法再用自己的原始钥匙 `person` 去看，发现住户已经变成“李四”了。
    

**到目前为止，它看起来和引用传递的效果一模一样。那么，如何证明它不是引用传递呢？**

让我们在 `modify` 方法里做一点小改动：

Java

```
public static void modify(Person p) {
    // 我们不再进入房间，而是让这把钥匙副本指向一个全新的房间
    p = new Person("王五");
    p.setName("赵六");
}
```

如果 Java 是引用传递，那么 `main` 方法里的 `person` 变量本身（也就是原始钥匙）应该被换掉，指向“王五/赵六”这个新房间。

**但实际运行结果是：**

```
调用方法前: 张三
调用方法后: 张三 
```

`main` 方法里的 `person` 毫发无损！

用“钥匙”比喻解释：

在 modify 方法里，你只是把你手里的那把钥匙副本扔掉了，然后从前台领了一把全新的、指向另一个房间（“王五”）的钥匙。你做的所有操作都是针对你手里的那把钥匙副本，这完全影响不到 main 方法手里持有的那把原始钥匙。它依然稳稳地指向最初的那个“张三”房间。

---

### 总结

1. **Java 永远是值传递**。
    
2. 当传递**基本类型**时，传递的是**值的副本**。
    
3. 当传递**对象类型**时，传递的是**引用的值的副本**（可以理解为**地址的副本**或**钥匙的副本**）。
    
4. 因为传递的是引用的副本，所以方法内外两个引用指向同一个对象。因此，在方法内**修改对象的内部状态**，会影响到方法外的对象。
    
5. 但是，在方法内**给参数重新赋值**，只是让这个引用的副本指向了一个新的对象，完全不会影响到方法外的原始引用。**这一点是证明Java是值传递的铁证**。
