下午好！在台北的周三下午，我们来聊聊面向对象设计中非常重要的 **SOLID** 五大原则之一：**开闭原则（Open-Closed Principle, OCP）**。

这是衡量一个软件设计是否“良好”的核心标准之一。

### “开闭原则”是什么？

开闭原则由 Bertrand Meyer 提出，其经典定义是：

> **一个软件实体（如类、模块、函数等）应该对扩展开放（Open for Extension），但对修改关闭（Closed for Modification）。**

这个定义听起来有点矛盾，我们来分解一下：

- “对修改关闭” (Closed for Modification)
    
    这是我们的核心目标。它意味着，一旦一个模块（比如一个类）的功能开发完成、经过测试、并已上线使用，我们就应该尽量避免去修改它内部的源代码。
    
    为什么？ 因为修改已经存在的、正在稳定运行的代码，风险非常高。你可能会无意中引入新的 Bug，并且还需要对这个模块及其所有依赖它的部分进行回归测试。
    
- “对扩展开放” (Open for Extension)
    
    这是我们达成目标所采用的手段。既然不能修改旧代码，那我们如何增加新功能呢？答案就是通过增加新代码的方式来扩展系统的行为。
    

简单来说，一个良好的设计应该允许你在**不改变现有代码**的前提下，轻松地**添加新功能**。

### 一个实例：从“坏设计”到“好设计”

让我们通过一个计算不同形状面积的例子，来直观地感受开闭原则。

#### 反面教材：违反开闭原则的设计

假设我们有一个 `AreaCalculator` 类，它根据传入的形状对象来计算面积。

Java

```
// 这是一个不好的设计
public class AreaCalculator {
    // 这个方法接收一个 Object，内部用 if-else 判断类型
    public double calculateArea(Object shape) {
        if (shape instanceof Rectangle) {
            Rectangle r = (Rectangle) shape;
            return r.getWidth() * r.getHeight();
        } else if (shape instanceof Circle) {
            Circle c = (Circle) shape;
            return Math.PI * c.getRadius() * c.getRadius();
        }
        return 0;
    }
}

class Rectangle { /* ... */ }
class Circle { /* ... */ }
```

问题在哪里？

这个设计完全违反了开闭原则。现在，如果我们需要增加一个新的形状，比如“三角形”（Triangle），我们必须回到 AreaCalculator 类中，修改 calculateArea 方法的源代码，增加一个 else if (shape instanceof Triangle) 的分支。

这就是“**对修改开放**”了，非常糟糕。每次增加新功能，都在修改核心的稳定代码。

#### 正确的姿势：遵循开闭原则的设计

如何改进呢？核心武器就是我们之前讨论过的**抽象**和**多态**。

第一步：创建抽象

我们定义一个抽象的 Shape 接口（或抽象类），它有一个计算面积的抽象方法。

Java

```
// 1. 创建一个抽象的 Shape 接口
public interface Shape {
    double calculateArea();
}
```

第二步：创建具体实现

让每一个具体的形状类都去实现这个 Shape 接口。

Java

```
// 2. 具体的类实现这个接口
public class Rectangle implements Shape {
    private double width;
    private double height;
    // ...
    @Override
    public double calculateArea() {
        return width * height;
    }
}

public class Circle implements Shape {
    private double radius;
    // ...
    @Override
    public double calculateArea() {
        return Math.PI * radius * radius;
    }
}
```

第三步：重构计算器

现在，AreaCalculator 不再关心具体的形状类型，它只依赖于 Shape 这个抽象。

Java

```
// 3. 计算器依赖于抽象，而不是具体实现
public class AreaCalculator {
    public double calculateArea(Shape shape) {
        // 直接调用 shape 自己的方法，让对象自己决定如何计算
        return shape.calculateArea();
    }
}
```

开闭原则的体现：

现在，如果我们想增加一个“三角形”，我们需要做什么？

1. **创建一个新的 `Triangle` 类**：`public class Triangle implements Shape { ... }`
    
2. **没了！**
    

我们**只需要增加新代码**（`Triangle` 类），而**完全不需要修改**任何已有的、稳定的代码（如 `AreaCalculator`, `Rectangle`, `Circle`）。

这就是：

- **对修改关闭**：`AreaCalculator` 类是封闭的，我们没有动它。
    
- **对扩展开放**：我们通过增加一个新的 `Shape` 实现类，轻松地扩展了系统的功能。
    

### 实现开闭原则的核心武器

- **抽象 (Abstraction)**：通过**接口**或**抽象类**来定义稳定的、不变的契约。
    
- **多态 (Polymorphism)**：允许高层模块（如 `AreaCalculator`）与抽象（`Shape`）交互，而具体的行为则由不同的子类在运行时决定。
    

遵循开闭原则，实际上就是在践行“**面向抽象编程，而不是面向实现编程**”这一重要的设计思想。

### 总结

开闭原则是构建稳定、可维护、可扩展的软件系统的核心设计原则之一。它的目标是通过**抽象**和**多态**，在面对需求变更时，尽量通过**增加新代码**来满足新需求，而不是通过**修改旧代码**，从而将变更带来的风险降到最低。
