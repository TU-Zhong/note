JVM 中真正执行内存回收的是垃圾收集器，而它们底层依赖的主要是三种核心的**垃圾收集算法**。这就好比打扫房间，针对不同的房间状况有不同的清扫策略。

主要有以下三种经典算法：

### 1. 标记-清除算法 (Mark-Sweep)

这是最基础的算法，分为“标记”和“清除”两个阶段。

- **过程：** 首先利用可达性分析，**标记**出所有需要回收的垃圾对象（或存活对象）；标记完成后，统一**清除**掉所有被标记的区域。
    
- **优点：** 逻辑简单，是最基础的思路。
    
- **缺点：** 会产生大量的**内存碎片**。清除后，内存中会留下许多不连续的小空闲区域（就像马路上到处是坑）。如果之后程序需要分配一个占用连续内存较大的对象（比如大数组），可能会因为找不到足够大的连续空间，而被迫提前触发下一次全量 GC。
    

### 2. 标记-复制算法 (Copying)

为了解决“标记-清除”带来的内存碎片问题，复制算法应运而生。上一篇提到的**新生代 Survivor 区**就是这种思想的典型应用。

- **过程：** 将可用内存按容量划分为大小相等的**两块**。每次只使用其中的一块。当这一块的内存用完了，就把里面还**存活**的对象按顺序**紧凑地复制**到另一块空闲内存上，然后再把原来那块已使用的内存一次性全部清空。
    
- **优点：** 每次都是对半区进行整体回收，分配内存时按顺序分配即可，**完全没有内存碎片**，运行极其高效。
    
- **缺点：** 空间浪费严重，**可用内存直接缩小为了原来的一半**。另外，如果内存里绝大多数对象都是存活的，复制的性能开销就会极其庞大（所以它只适合对象“朝生夕灭”的新生代）。
    

### 3. 标记-整理算法 (Mark-Compact)

针对老年代对象存活率高的特点，专家们提出了“标记-整理”算法。

- **过程：** 它的“标记”阶段与第一种算法一样。但后续步骤不是直接清理垃圾，而是让所有**存活的对象都向内存空间的一端滑动聚合**，然后直接清理掉存活边界以外的所有内存。
    
- **优点：** 既解决了“标记-清除”的内存碎片问题，又规避了“标记-复制”浪费一半空间的问题。
    
- **缺点：** 性能开销最大。因为在移动大量存活对象时，必须暂停所有的用户程序代码（这就是著名的 **Stop The World**, STW），并且底层还要去挨个更新所有指向这些对象的指针引用。

### 总结运用

现代商用 JVM 的垃圾收集器通常是**组合使用**这些算法的。

因为**新生代**每次存活的对象极少，复制开销小，所以采用**标记-复制算法**；而**老年代**对象存活率高，且没有额外的空间做后备担保，所以必须使用**标记-清除**或**标记-整理**算法。
