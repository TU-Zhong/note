好的，`getAndSet` 是 Redis 原子操作系列中的又一个非常有用的方法，它对应 Redis 的原生命令 `GETSET`。

它的作用，正如其名，是在**一个原子操作**中完成两件事：

1. **Set (设置)**：为一个键（Key）设置一个**新**的值。
    
2. **Get (获取)**：获取这个键在被设置新值**之前**的**旧**的值。
    

---

### ## 核心特性：原子性的“交换”

`getAndSet` 最关键的特性依然是**原子性**。

想象一下，如果你想实现同样的功能，你可能会分两步操作：

**非原子性的错误做法：**

Java

```
// 第一步：获取旧值
String oldValue = redisTemplate.opsForValue().get("mykey");

// 第二步：设置新值
redisTemplate.opsForValue().set("mykey", "newValue");
```

在高并发环境下，这种做法是**危险**的。在你的第一步和第二步之间，可能有另一个线程或服务已经修改了 "mykey" 的值。这样，你获取到的 `oldValue` 就不是你将要覆盖的那个“真正”的旧值了，它已经“过时”了。

而 `getAndSet` 把这两步合并成了一个**不可分割的操作**。当它执行时，Redis 会确保没有任何其他命令可以插足。你得到的值，**保证**就是在你设置新值那一瞬间，它所拥有的旧值。

通俗比喻：

getAndSet 就像一次物品交换。你递给对方一个新苹果（Set newValue），同时从对方手里拿走他原来的那个旧橘子（Get oldValue）。这个交换动作是在一瞬间完成的，不存在你先把苹果给他，他再慢悠悠把橘子给你的中间状态。

---

### ## 代码示例

让我们看一个实际的代码例子，比如更新系统的“上次访问时间”。

Java

```
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Service;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

@Service
public class SystemStatusService {

    @Autowired
    private StringRedisTemplate stringRedisTemplate;

    public void recordAccessAndGetLast() {
        String key = "system:last_access_time";
        String newTimestamp = LocalDateTime.now().format(DateTimeFormatter.ISO_DATE_TIME);

        // 原子性地设置新的访问时间，并获取上一次的访问时间
        String oldTimestamp = stringRedisTemplate.opsForValue().getAndSet(key, newTimestamp);

        if (oldTimestamp == null) {
            System.out.println("这是系统第一次被访问。当前时间已记录。");
        } else {
            System.out.println("系统上一次的访问时间是: " + oldTimestamp);
            System.out.println("当前访问时间已更新为: " + newTimestamp);
        }
    }
}
```

当你调用 `recordAccessAndGetLast()` 方法时：

- **第一次调用**：`key` 不存在，`getAndSet` 会设置新值，并返回 `null`。
    
- **后续调用**：`getAndSet` 会设置新的时间戳，并返回**上一次**存入的时间戳。

### ## 核心应用场景

`getAndSet` 在需要“获取并重置”或“获取并更新”的场景下特别有用。

#### **1. 实现“先取值，后重置”的计数器**

这是 `getAndSet` 最经典的应用场景。

需求：

假设你需要统计网站每分钟的请求量。你有一个计数器 requests:current_minute，它通过 increment 不断增加。现在你需要一个后台任务，每分钟准时来读取上一分钟的总请求量，并立即将计数器清零，以便开始下一分钟的统计。

错误做法：

get -> delete。如果在 get 和 delete 之间又有新的请求进来，这部分请求量就会丢失。

**正确做法**：

Java

```
public String getAndResetRequestCount() {
    String key = "requests:current_minute";
    
    // 原子性地获取当前计数器的值，并立即将其重置为 "0"
    String lastMinuteCount = stringRedisTemplate.opsForValue().getAndSet(key, "0");

    // 如果 lastMinuteCount 为 null，说明上一分钟没有请求，可以处理为 0
    return lastMinuteCount == null ? "0" : lastMinuteCount;
}
```

这个操作可以**精准地获取**到上一分钟的完整计数，同时**原子性地**将计数器归零，不会丢失任何数据。

#### **2. 管理“首次加载”的缓存**

需求：

一个数据需要从数据库加载并缓存。在多个请求同时到来时，只允许第一个请求去加载数据库，其他请求则等待。加载完成后，需要一个标识位告知后续请求缓存已就绪。

`getAndSet` 可以用来原子性地更新这个状态标识。

Java

```
String cacheStatusKey = "cache:products:status";
// 尝试将状态从 "NOT_READY" 更新为 "LOADING"
String previousStatus = redisTemplate.opsForValue().getAndSet(cacheStatusKey, "LOADING");

if ("NOT_READY".equals(previousStatus)) {
    // 成功抢到了加载权，因为我把状态从 NOT_READY 变成了 LOADING
    // ... 从数据库加载数据 ...
    // ... 加载完毕后，再将状态设置为 "READY"
    redisTemplate.opsForValue().set(cacheStatusKey, "READY");
} else {
    // 没抢到，说明其他人正在加载或者已经加载完毕，可以稍后重试获取缓存
}
```

### 总结

|对比|`get()` + `set()` (两步操作)|`getAndSet()` (一步操作)|
|---|---|---|
|**原子性**|**非原子性**，高并发下存在竞态条件，数据可能不一致|**原子性**，保证获取到的旧值就是被新值覆盖的那一刻的值|
|**核心作用**|简单的获取和设置|**原子性地交换值**，常用于“获取并重置/更新”的场景|
|**推荐场景**|低并发或对数据一致性要求不高的场景|**需要重置的计数器**、状态管理、需要获取前一个状态值的场景|

简单来说，当你需要**更新一个值，并且关心它更新前的那个旧值是什么**时，`getAndSet` 就是你的首选工具。
