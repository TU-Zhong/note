下面是结合你的项目回答为什么购物车使用 Redis 的原因，并给出与当前实现相关的注意点与对比。

核心理由

- 低延迟高并发：Redis 基于内存，读写非常快，能承受用户购物车频繁加减的高并发场景，避免数据库成为热点瓶颈。
- 易横向扩展：购物车请求无状态（用户ID来自 JWT 解析），服务节点可水平扩容，Redis 作为集中式存储让会话数据不依赖单机内存。
- 数据结构契合：购物车本质是“商品项 -> 数量”的键值映射， Hash 非常贴合这一模型，查询、修改都只针对一个 key 的 field，效率高。
- 原子操作与并发安全：Redis 原生提供原子自增/自减，天然避免数据库行锁竞争和复杂事务；购物车对一致性要求偏弱，最终一致即可。
- 业务属性匹配：购物车是短暂、易变的数据（用户经常增减或清空），不需要像订单那样强持久化，放在 Redis 更经济、简洁。
- 降压数据库：将高频的加入/减少操作从数据库移到 Redis，数据库主要承接“下单时落库”的关键写入，整体更稳。
结合当前项目实现

- 键设计：按用户分桶， key = cart:<userId> ；项键分别为 dish:<dishId> 或 combo:<comboId> ，值为数量。
- 主要操作：
  - add ：存在则数量+1，不存在则置为1（当前实现为 get + put ）。
  - list ： HGETALL 获取所有项，随后到数据库查询菜品名称、图片等展示信息。
  - sub ：数量>1则-1，否则删除该项； clean ：删除整个 cart:<userId> 。
- 关联代码位置： ShoppingCartServiceImpl 使用 RedisTemplate.opsForHash() 完成上述操作；店铺状态也用 Redis 缓存，说明项目已有稳定的 Redis 依赖与测试覆盖。
与仅用数据库的对比

- 数据库方案在高并发下需要良好表设计（唯一约束、索引、行锁控制）、复杂事务和大量写入，容易形成热点和锁竞争。
- Redis 将高频写入转为内存操作，吞吐更高；最终在“提交订单”这个关键节点一次性落库，既保证数据可靠又减轻数据库压力。
注意事项与优化建议

- 原子性优化：将 get + put 改为 HINCRBY 做数量增减，更简洁且原子。
- 过期策略：可对 cart:<userId> 设置 TTL，长期不活跃的购物车自动清理，控制内存占用。
- 价格与一致性：展示价格可以来自数据库或价格服务；实际成交价请在“下单”时快照并落库，避免 Redis 中价格被认为是强一致来源。
- 套餐信息：当前对 combo 展示略简，可补充套餐查询以完善名称/图片等回填。
- 宕机与持久化：购物车数据允许丢失（相对订单），如需更高可靠性可开启 AOF/RDB 或在关键节点做备份。
何时不建议只用 Redis

- 购物车被业务要求强一致、可审计、长期历史保留（例如合规追溯），更适合数据库持久化并用缓存做读优化。
总之：Redis 与购物车的读写模式天然匹配，能显著提升性能和扩展性，同时保持实现简单；你当前的代码已很好地利用了 Redis Hash 的优势，进一步用 HINCRBY 、TTL 和下单快照等策略即可让方案更稳健。
