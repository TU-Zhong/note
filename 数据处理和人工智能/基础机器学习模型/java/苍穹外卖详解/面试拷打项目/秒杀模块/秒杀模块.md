---

### 一、核心思想：重新认识流程

首先，我们必须明确一点：**秒杀不是常规的购物流程**。

- **常规流程**：浏览商品 -> 加入购物车 -> 提交订单 -> 付款。这个流程强调的是用户的从容选择和体验，每一步操作之间有延迟，并发压力不大。
    
- **秒杀流程**：用户在同一瞬间疯抢有限的商品。这个流程的本质是一个**“资格抢夺”**的过程。
    

因此，我们的核心设计思想是**“分离”**：将**“抢资格”**这个速度极快、并发极高的环节，与**“创建订单”**这个速度较慢、逻辑复杂的环节彻底分离开。

---

### 二、核心挑战

在设计时，我们必须时刻铭记秒杀的三大敌人：

1. **高并发 (High Concurrency)**：成千上万的请求在同一秒涌入。
    
2. **超卖 (Overselling)**：卖出的商品比实际库存多，这是灾难性的业务错误。
    
3. **数据库瓶颈 (Database Bottleneck)**：所有请求都直接操作数据库，会导致数据库锁竞争、响应缓慢，最终整个系统崩溃。
    

---

### 三、架构设计与关键技术

为了战胜这三大敌人，我们将采用**“缓存 + 消息队列”**的经典高并发架构。

- **主角 ① Redis (缓存)**：我们将用 Redis 这个内存数据库来充当**“资格争夺的战场”**。因为它操作内存，速度是操作磁盘的数据库的成千上万倍。
    
- **主角 ② 消息队列 (Message Queue - MQ)**：如 RabbitMQ 或 Kafka。我们将用它来充当**“成功抢到资格用户的等候区”**。它负责缓冲瞬时的高峰流量，让数据库可以从容地处理订单。
    
- **配角 ③ 数据库 (您的 SQL Server)**：它不再是承受所有压力的第一线，而是作为最终**“数据落地”**的地方，保证数据的最终一致性。
    

---

### 四、实现流程详解（三步曲）

#### 第一步：秒杀前 — 数据预热

在秒杀活动开始前（例如，提前5-10分钟），您需要运行一个后台任务，将秒杀数据从慢速的数据库“预热”到高速的 Redis 缓存中。

1. **加载库存**：从 SQL Server 中查出秒杀商品（比如商品ID为 `PID1001`）的库存数量（比如 `100` 件）。
    
2. **存入 Redis**：将库存数量存入 Redis。
    
    Code snippet
    
    ```
    # 键名设计：seckill:stock:{商品ID}
    SET seckill:stock:PID1001 100 
    ```
    
3. **准备用户名单**：创建一个空的 Set 集合，用于存放成功抢到资格的用户ID，防止重复抢购。
    
    Code snippet
    
    ```
    # 键名设计：seckill:users:{商品ID}
    DEL seckill:users:PID1001 
    ```
    

#### 第二步：秒杀中 — 闪电战（用户点击按钮的瞬间）

这是用户请求直接触达的后端接口，**这个接口的执行时间必须控制在几十毫秒以内**。

**接口核心逻辑如下：**

1. **基础校验**：判断用户是否登录、秒杀是否已开始等。
    
2. **执行 Redis Lua 脚本（保证原子性）**：这是整个秒杀功能的心脏。为了防止在“检查库存”和“扣减库存”这两步操作之间出现并发问题，我们必须使用 Lua 脚本，让多个命令在 Redis 服务端一次性、不可分割地执行。
    
    Lua
    
    ```sql
    -- check_and_decr_stock.lua 脚本
    
    -- 键名列表 (通过 KEYS 数组传入)
    local stockKey = KEYS[1]      -- 库存的键，例如 seckill:stock:PID1001
    local userSetKey = KEYS[2]    -- 成功用户名单的键，例如 seckill:users:PID1001
    
    -- 参数列表 (通过 ARGV 数组传入)
    local userId = ARGV[1]        -- 当前用户的ID
    
    -- 1. 检查用户是否重复抢购
    if redis.call('sismember', userSetKey, userId) == 1 then
        return 2  -- 返回2，代表重复抢购
    end
    
    -- 2. 检查库存是否足够
    local stock = tonumber(redis.call('get', stockKey))
    if stock <= 0 then
        return 0  -- 返回0，代表库存不足
    end
    
    -- 3. 扣减库存
    redis.call('decr', stockKey)
    
    -- 4. 将用户ID加入成功名单
    redis.call('sadd', userSetKey, userId)
    
    return 1  -- 返回1，代表抢购成功
    ```
    
3. **在 Java 中调用 Lua 脚本**：
    
    - 您的 Java 代码需要加载并执行上述 Lua 脚本，并传入相应的参数。
        
    - 根据 Lua 脚本的返回值进行判断：
        
        - 如果返回 `1` (成功)：执行第4步。
            
        - 如果返回 `0` (库存不足)：直接告诉用户“已售罄”。
            
        - 如果返回 `2` (重复抢购)：直接告诉用户“您已抢到，请勿重复操作”。
            
4. **发送消息到消息队列 (MQ)**：
    
    - 对于抢购成功的请求（脚本返回`1`），立即创建一个包含 `{ "userId": "xxx", "productId": "PID1001" }` 信息的**消息体**。
        
    - 将这个消息发送到您的消息队列中（例如，一个名为 `seckill-order-queue` 的队列）。
        
5. **立即向用户返回**：
    
    - 一旦消息成功发送到 MQ，这个接口的任务就完成了。
        
    - 立刻向用户的前端返回一个友好的提示：“恭喜您，抢购成功！订单正在处理中，请稍后在‘我的订单’中查看。”
        

至此，高并发的洪峰已经平稳度过。

#### 第三步：秒杀后 — 从容创建订单

现在，一个独立的、与前端接口完全解耦的**消息消费者服务**开始工作。

**消费者服务的逻辑：**

1. **监听队列**：这个服务持续地从 `seckill-order-queue` 队列中获取消息。
    
2. **处理消息**：每收到一条消息（例如 `{ "userId": "xxx", "productId": "PID1001" }`），就执行以下数据库操作。
    
3. **创建真实订单**：
    
    - 开启数据库事务。
        
    - **扣减数据库库存**：执行 `UPDATE products SET stock = stock - 1 WHERE id = 'PID1001' AND stock > 0;`。这一步是最终的库存扣减，也是防止超卖的最后一道保险。
        
    - **创建订单记录**：在您的 `orders` 表中插入一条新的订单数据。**这一步可以复用您现有的“提交订单”功能中的部分核心业务逻辑代码**，只是订单的来源标记为“秒杀活动”。
        
    - 提交数据库事务。
        
4. **通知用户**：订单在数据库中成功创建后，可以通过WebSocket、APP推送或短信等方式，给用户发送一个最终的“订单创建成功”通知。
    

---

### ## 与您现有功能的整合关系

- **商品浏览功能**：完全复用。秒杀商品也需要一个详情页来展示信息和倒计时。
    
- **购物车功能**：**完全绕过**。秒杀是为了追求极致的速度，不允许用户加入购物车再进行多步结算。点击“秒杀”按钮就是一次完整的资格抢夺。
    
- **提交订单功能**：**部分复用**。用户侧的提交订单页面被绕过，但后端服务中**负责将订单信息写入数据库的核心逻辑**，可以在上述的“第三步”中被消息消费者服务调用。
    

通过这套“Redis扛压 + MQ削峰 + 数据库落地”的架构，您就能构建一个既能应对高并发冲击，又能保证数据绝对准确的专业级商品秒杀功能。
