# 电商后台管理系统 · 实战 500 题

## 题单定位

这份题单面向**初级进阶**程序员：会用 Spring Boot 写简单接口，但没完整做过企业级项目。做完这份题单后，你将具备独立从零搭建一个电商后台、解决常见线上问题的能力。

**核心原则：**
- 每道题都是"做出来、跑得通"，不深入框架源码
- 代码写在哪、怎么验证，每道题都有明确产出
- 每个模块末尾有排错实战，训练你出问题自己查的能力
- SQL Server + Redis + RabbitMQ 三大中间件贯穿始终

**环境要求：**
- JDK 17+，Spring Boot 3.x，Maven
- SQL Server 2022 (Developer Edition)，通过 Docker 运行
- Redis 7.x，通过 Docker 运行
- RabbitMQ 3.x (Management)，通过 Docker 运行
- IntelliJ IDEA，Postman/Apifox，DataGrip 或 SSMS

## 模块二：SQL Server 索引与性能调优（60 题）

> **目标：** 打开执行计划不慌，看到慢查询知道怎么分析、怎么建索引、怎么写高效的 SQL。这是后端面试必考项。

### 2.1 SQL Server 执行计划入门（61-75）

61. 在 SSMS 中打开"包含实际的执行计划"（`Ctrl+M`），写一条 `SELECT * FROM goods WHERE price > 50`，观察执行计划中用了什么扫描方式（全表扫描 vs 索引查找）。
62. 解释执行计划中的常见操作符：`Table Scan`、`Clustered Index Scan`、`Clustered Index Seek`、`NonClustered Index Seek`、`Key Lookup`、`Nested Loops`、`Hash Match`、`Sort`。
63. 使用 `SET STATISTICS IO ON` 查看 SQL 语句的 IO 消耗：逻辑读取次数、物理读取次数。写两条同样结果但 IO 消耗不同的 SQL 对比。
64. 使用 `SET STATISTICS TIME ON` 查看 SQL 的 CPU 时间和总耗时。对比有无索引时的耗时差异。
65. 在 `goods` 表中插入 100 万条测试数据（编写存储过程或用在线数据生成工具），为后续调优练习提供数据基础。要求数据分布合理：价格在 1-10000 之间随机，分类 ID 在 1-100 之间随机，创建时间在最近一年内随机。
66. 在没有索引的情况下，查询"某个分类下价格大于 1000 的商品，按创建时间倒序"，记录执行计划和 IO 统计。
67. 为上述查询创建合适的索引（覆盖查询条件 + 排序），再次执行，对比索引创建前后的 IO 和耗时。
68. 演示 `索引覆盖`：查询仅需要 `goods_name`、`price`、`category_id` 三个字段，创建包含这三列的索引后，执行计划上会显示什么？Key Lookup 消失了吗？
69. 演示 `索引覆盖` 不生效的场景：`SELECT *` 时覆盖索引为什么不够用？对比 `SELECT 指定列` 和 `SELECT *` 的执行计划差异。
70. 使用 `DBCC DROPCLEANBUFFERS` 清除缓存后，对比"有索引"和"无索引"两种情况下同一查询的物理读取次数，加深对索引价值的理解。
71. 解释 `sys.dm_db_index_usage_stats` 视图：查询哪些索引从未被使用过，哪些索引被频繁扫描/查找。
72. 编写脚本统计当前数据库中缺失索引的建议（使用 `sys.dm_db_missing_index_details`），将缺失索引的创建 SQL 输出到结果集中。
73. 演示"错误的索引"带来的问题：为 `status` 字段创建索引（该字段只有 0/1 两个值，区分度极低），执行计划会选择这个索引吗？解释"索引选择性"的概念。
74. 演示联合索引的"左前缀"匹配规则：创建 `INDEX(category_id, price, create_time)` 联合索引后，分别测试 `WHERE category_id = 1`、`WHERE price > 100`、`WHERE create_time > '2025-01-01'` 三个查询中哪些能用到索引。
75. 使用 `OPTION(RECOMPILE)` 提示对比查询计划：为什么有时 SQL Server 会选错执行计划？什么场景下会用到这个 hint？

### 2.2 索引设计与优化实战（76-90）

76. 在 100 万数据下，对比"高选择性列"（如 `username` 唯一值很高）和"低选择性列"（如 `status` 只有 4 种状态）分别建索引的效果：索引大小、查询效率、是否被优化器选用。
77. 创建包含列索引（INCLUDE）：`CREATE INDEX idx_xxx ON goods(category_id) INCLUDE(goods_name, price)`。验证 INCLUDE 列不会影响索引键的排序，且可以覆盖查询。
78. 演示"索引查找 + 键查找(Key Lookup)"的完整过程：查询需要的列不完全在索引中时，SQL Server 先在非聚集索引找到主键，再用主键回聚集索引取数据。观察 IO 统计中逻辑读取的增加。
79. 优化 Key Lookup 的两种方式：① 在索引中加入 INCLUDED 列 ② 直接建覆盖索引。对比两种方式的索引大小和维护成本差异。
80. 分析 `ORDER BY goods_name LIMIT 20` 利用索引排序 vs filesort：对比有无 `INDEX(goods_name)` 时执行计划的 `Sort` 操作符差异。
81. 探究 SQL Server 的排序机制：什么情况下排序发生在内存中？什么情况会溢出到 `tempdb`？观察 `sys.dm_exec_query_stats` 中与排序溢出相关的指标。
82. 演示"排序 + 分页"的性能问题：`ORDER BY create_time DESC OFFSET 100000 ROWS FETCH NEXT 20 ROWS ONLY`。观察深层分页的执行计划，解释为什么越往后越慢。
83. 使用"延迟关联"优化深层分页：先用子查询获取前 100020 行的主键 ID，再 JOIN 原表取完整数据。对比优化前后的 IO 和耗时。
84. 探究"键值查找"(Key Lookup) 在批量查询中的风险：当查询返回行数较多时，优化器可能放弃索引直接全表扫描（tipping point）。在你的百万数据中测试并观察 tipping point。
85. 对比 `UNION` 和 `UNION ALL` 的执行计划差异：`UNION` 多了一个去重排序操作。在你的业务数据上验证两者耗时和 IO 的差距。
86. 使用 `EXISTS` 替代 `IN` 优化子查询：对比 `WHERE id IN (SELECT ...)` 和 `WHERE EXISTS (SELECT 1 FROM ... WHERE ...)` 的执行计划和性能。
87. 演示 `NOT IN`、`NOT EXISTS`、`LEFT JOIN ... IS NULL` 三种"差集"写法的性能差异，解释 `NOT IN` 遇到 NULL 时的陷阱。
88. 分析 `SELECT COUNT(*)` 和 `SELECT COUNT(1)` 和 `SELECT COUNT(column)` 的区别：在 SQL Server 中是否有性能差异？COUNT(column) 不统计 NULL 值这一特性在什么业务场景下有用？
89. 使用"筛选索引"(Filtered Index)：为 `WHERE deleted = 0 AND status = 0` 这种常见筛选条件创建筛选索引，对比普通全列索引的大小和性能差异。
90. 探究索引碎片：通过 `sys.dm_db_index_physical_stats` 查看索引的碎片率，当碎片率 > 30% 时，执行 `ALTER INDEX REORGANIZE` 或 `REBUILD`，对比重组前后的 IO 和扫描效率。

### 2.3 锁、阻塞与并发（91-100）

91. 使用 SQL Server Management Studio 开启两个查询窗口，模拟并发更新同一行（T1 更新 goods id=1 的价格，T2 也更新同一行但先不提交）。T2 会在哪里等待？观察 `sys.dm_exec_requests` 中的 `blocking_session_id`。
92. 使用 `sp_who2` 或 `sp_WhoIsActive` 存储过程，在阻塞发生时快速找到阻塞源头。
93. 使用 `sp_WhoIsActive` 定位锁等待链：A 阻塞 B，B 阻塞 C → 找出链头锁的持有者（阻塞源），查看其最后执行的 SQL。
94. 演示死锁：准备两个事务，T1 先更新 goods id=1 再更新 goods id=2，T2 先更新 goods id=2 再更新 goods id=1。同时提交两个事务，观察 SQL Server 的死锁图和错误信息。
95. 从 SQL Server 的死锁图中解读：受害者（被回滚的事务）是谁？幸存者（继续执行的事务）是谁？SQL Server 的选死策略是什么？
96. 阅读死锁 XML 报告：死锁发生时每笔事务持有的锁和请求的锁分别是什么类型的锁（X / U / S）？
97. 使用 `SET DEADLOCK_PRIORITY LOW` 降低某个会话的死锁优先级：在死锁发生时，指定某个不重要的业务做受害者被回滚。
98. 演示快照隔离级别（SNAPSHOT ISOLATION）：开启 `ALLOW_SNAPSHOT_ISOLATION` 后，读操作不再被写操作阻塞。编写并发读写测试验证。
99. 在商品表中加入一个 `version` 字段（乐观锁），在 Java 层实现乐观更新：先读当前 version → 更新时 WHERE version = 旧值 → 判断受影响行数 → 失败则重试。

### 5.4 RabbitMQ 进阶实战（256-275）

256. 实现"订单延迟取消"——方案一（死信队列）：创建消息时设置 TTL（`setExpiration("1800000")` 即 30 分钟），该队列没有消费者，消息过期后自动转发到死信队列 `order.cancel.queue`，由死信队列消费者处理取消订单逻辑。
257. 实现"订单延迟取消"——方案二（延迟插件 rabbitmq_delayed_message_exchange）：安装 `rabbitmq_delayed_message_exchange` 插件到 RabbitMQ 容器，创建 `x-delayed-type` 交换机，发送消息时设置 `x-delay=1800000`。对比方案一的精准度和维护复杂度。
258. 实现生产者确认（Publisher Confirm）：配置 `spring.rabbitmq.publisher-confirm-type=correlated`，在 `RabbitTemplate` 中设置 `ConfirmCallback`，如果消息发送到交换机失败，触发回调记录失败日志并做补偿。
259. 实现生产者回退（Publisher Return）：配置 `spring.rabbitmq.publisher-returns=true`，设置 `ReturnCallback`，如果消息路由到队列失败（Routing Key 写错了），触发回调通知。
260. 实现"消息本地表"可靠投递方案：在"支付回调"中先写入消息本地表 `t_pending_message`（状态为 `TO_SEND`）和订单数据在同一事务中，再由定时任务扫描 `TO_SEND` 的消息发送到 RabbitMQ，发送成功后标记为 `SENT`。即使服务挂了，定时任务也能补偿投递。
261. 实现消费者幂等性：消费者在收到消息时先判断该消息 ID `(messageId)` 是否在 Redis 中已处理过（`SETNX message:consumed:{messageId} 1 EX 3600`），已处理则直接确认跳过，防止重复消费。
262. 在数据库层做幂等兜底：在消费订单消息时，如果操作是"修改订单状态"，SQL 语句加条件判断 `WHERE status = 期望的旧状态`，防止重复更新。

### 6.4 秒杀优化与防护（346-360）

346. 实现"秒杀接口的前置校验"：在进入秒杀逻辑之前，先校验用户是否登录、活动是否在有效期、是否已达到限购数量，不合条件的请求快速拒绝不进入锁竞争。
347. 分析秒杀场景下的"热点 Key 问题"：100 万用户同时秒杀同一件商品，`seckill:stock:{skuId}` 这个 Redis Key 会成为单点热点。写出"热点 Key"的形成原因和至少 2 种解决方案。
348. 解决热点 Key——"库存拆分"：把秒杀库存拆为 N 份（如 50 份），分到不同的 Key（`seckill:stock:{skuId}:0` ~ `seckill:stock:{skuId}:49`），用户请求时随机路由到一个分片。验证：100 个并发请求能均分到各分片，无单点热点瓶颈。
349. 解决热点 Key——"本地缓存 + Redis 兜底"：在 JVM 内用 Caffeine 缓存秒杀活动信息和库存状态（如"库存是否归零"标记），大部分请求在本地缓存层就被过滤掉，少量请求穿透到 Redis。对比全走 Redis 的性能提升。
350. 实现"秒杀流量削峰"：秒杀入口处将用户请求放入 RabbitMQ 队列而非在线程中直接处理，等队列中的请求被消费者逐条处理。测试 5000 个并发请求全部入队且不会丢失，消费者顺序处理并返回结果。
351. 验证兜底规则：秒杀活动已结束但有人用之前的动态 URL 调用秒杀接口，后端应返回"活动已结束"。如果 Redis 中的活动状态 Key 已过期，也同样做兜底：去读 MySQL 活动表。
352. 实现"秒杀活动的安全开关"——热更新活动规则：通过 Nacos 配置中心动态修改限购数量和黑名单阈值，无需重启应用。
353. 探究"秒杀 URL 的动态化防刷"：每个用户拿到的秒杀链接都不同——后端用用户 ID + 活动 ID + 随机盐生成唯一 Token，有效期极短。相比固定 URL，为什么这能防刷？
354. 实现一个简单的验证码参与秒杀：秒杀开始前，前端弹出图片验证码，码正确才能调用秒杀接口。验证码答案存储在 Redis 中单次有效。
355. 探究秒杀的"CDN 缓存前置"思路：秒杀未开始前，商详页面的静态数据/价格等可以缓存到 CDN 不让源站承受所有压力。哪些内容可以 CDN 缓存？哪些必须实时？（纯分析题）
356. 测试高并发下的日志量：用 200 QPS 的 JMeter 脚本压测秒杀接口，观察应用日志量。发现每个请求打印了 5 条日志，200 QPS 就是每秒 1000 条日志。优化日志打印：只在异常或重要状态变更时打印日志，普通成功请求减少日志。配置异步日志 Appender。
357. 配置 Sentinel 限流规则：集成 Sentinel Dashboard，对秒杀接口的 QPS 做硬限流（如 200 QPS），超过则返回"系统繁忙，请稍后再试"。观察 Sentinel Dashboard 的实时监控曲线。
358. 配置 Sentinel 的熔断降级：调用秒杀服务超过一定失败比例（如 50% 以上请求因库存不足失败），自动降级返回"已售罄"，不再走完整的 Redis + MQ 链路（直接短路返回）。
359. 探究"异步下单的时间差"问题：用户秒杀成功（Redis 显示成功）后，MQ 消息还在处理中 → 用户刷新订单列表看不到刚秒杀的订单。展示如何处理这个时间差（用"排队中"状态安抚用户，或直接用 Redis 缓存的虚拟订单临时展示）。

### 6.5 模块排错实战（361-375）

361. **排错实战 1：** 秒杀报"库存不足"但数据库中库存仍为大量正数。排查：① 检查 Redis 中的秒杀库存是否大于真实库存 ② 活动预热脚本是否正确加载了库存 ③ 是否有多次预热/活动未清理的残留数据。在白板上画出排查路线。
362. **排错实战 2：** 秒杀结束后仍有人成功下单——怀疑是分布式锁失效。检查：① 锁的 value 是否使用了相同 ID（线程复用导致错误解锁）② 锁超时时间是否太短导致锁自动过期。重现然后彻底修复。
363. **排错实战 3：** 出现超卖（实际售出 > 秒杀库存）。这比超时和锁失效更严重。检查：① Lua 脚本的库存判断逻辑是否用 `>= 0` 而非 `> 0` ② 库存拆分后各分片的判断是否独立 → 分片之间没有协调？③ Redisson 的分布式锁是否有主从切换导致锁数据丢失。按照可能性逐个排查并打出日志。
364. **排错实战 4：** 秒杀活动开始瞬间，所有用户都看到"系统繁忙"，但服务器 CPU 才 20%，数据库连接也被正常。排查：是 Sentinel 的默认限流规则配得过低？是 Nginx 在前端限流了？是漏桶令牌桶参数不对？逐层恢复并重新压测。
365. **排错实战 5：** 热点 Key 导致 Redis 某个节点 CPU 飙高而其他节点空闲。使用 Redis 的 `INFO commandstats` 查看高频命令，用 `redis-cli --hotkeys` 定位热点 Key，进行库存拆分并重新报备。对比拆分前的热点 Key 指标和拆分后的均匀分布。

### 7.4 生产环境故障全链路排错实战（426-440）

426. **场景一：CPU 飙高到 100%。** 在 Linux 上复现（WSL2 或虚拟机）：写一个死循环代码 → 部署到容器 → 触发 → 用 `top` 找到高 CPU 进程 → `top -Hp` 找高 CPU 线程 → 转十六进制 → `jstack` 线程快照中定位到代码行。完整记录你的排查过程。
427. **场景二：堆内存 OOM。** 写一段代码持续向 `ArrayList` 中加对象直到爆堆内存。配置 JVM 参数 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof`，OOM 后用 MAT 打开 dump 文件，找出占用内存最大的对象和 GC Root 引用路径。
428. **场景三：数据库连接池耗尽。** 故意在代码中不关闭连接或设置连接池大小过小（如 `maximumPoolSize=2`），然后并发调接口直到 `Connection is not available`。用 HikariCP 的 `poolName` 和 MBean 观察连接池状态，最终排查到根因并调整连接池参数。
429. **场景四：慢 SQL 拖垮系统。** 制造一个缺少索引的慢查询（`SELECT * FROM order_info WHERE status = ? AND create_time > ? ORDER BY create_time DESC`，表含百万数据且无索引）。观察：接口响应时间超过 5s → 数据库 CPU 高 → 定位慢 SQL (Query Store / 扩展事件) → 加索引 → 恢复。记录整个定位和修复过程。
430. **场景五：Redis 缓存雪崩导致数据库被打满。** 模拟 50 个缓存 Key 同时过期 → 大量请求直打数据库 → 数据库 CPU 飙升。排查：先看 Redis 的缓存命中率（`INFO stats` 的 `keyspace_hits/misses`）→ 发现大量 miss → 确认数据库出现瞬时大流量 → 加上随机过期时间 + 缓存预热 → 恢复。
431. **场景六：消息丢失。** 生产者消息发出去了，但消费者从未收到。排查：① 检查 RabbitMQ 管理界面的交换机是否正确绑定了队列 ② exchange/queue/routingKey 是否一致 ③ 消息序列化器是否两方一致 ④ 消费者被异常 catch 导致手动 ACK 不执行。模拟其中一种场景并修复。
432. **场景七：消息积压。** 消费者处理每条消息耗时 500ms，生产者每秒发 1000 条。排查：在 RabbitMQ 管理界面看到队列堆积了数十万条消息 → 启动 20 个消费者实例分担处理 → 消息积压缓解。如果此时 Redis/数据库成为消费者端瓶颈怎么办？讨论两种方案：增加实例 + 优化消费者内部 SQL/缓存。
433. **场景八：缓存和数据库数据不一致。** 用延时双删跑了一段时间后发现仍有偶发不一致（缓存中是旧价格，数据库已更新）。用 Arthas `watch` 监控缓存更新的代码，定位到双删的第二次删除因网络抖动失败。把第二次删除改为入 RabbitMQ 异步重试，确保最终一致。
434. **场景九：应用启动失败。** 改配置后重启应用，但反复报 `APPLICATION FAILED TO START`。分析启动日志（`@SpringBootApplication` 下的自动配置报告、`CONDITIONS EVALUATION REPORT`），定位具体缺失的 Bean。人工制造：故意不用 `@EnableConfigurationProperties` 误删必需 Bean，然后修复。
435. **场景十：某个用户"支付了钱但订单仍是待支付"。** 分析：① 支付回调是否执行 ② 是否因网络超时导致回调丢失 ③ 消息表是否有补偿机制 ④ 手动补偿这条订单。在项目中实现一个"手动补偿"接口供客服使用（`PUT /admin/orders/{id}/manual-confirm-pay`），要求该接口被安全限权和记录操作人。
436. **场景十一：订单状态永远卡在"退款中"。** 分析：退款流程的设计是否有"等待第三方退款回调"的步骤 → 该回调未触发 → 订单永远卡住。排查：退款回调日志、退款表的更新语句是否有 WHERE 条件不匹配、是否有死锁导致更新失败。修复后手动刷新卡住的订单。
437. **场景十二：日志发现大量 `RedisConnectionFailureException`。** 时间线分析：每隔 10 分钟出现 1 次，持续 3-5 秒后自动恢复。排查：Redis 是否做了持久化（BGSAVE fork 导致短暂的连接中断）？Lettuce 的连接池是否配置了心跳和重连？调整 Lettuce 的连接超时和重试策略。
438. **场景十三：死锁定位。** 写两个互相等待的线程产生死锁。用 `jstack -l <pid>` 导出线程快照 → 分析 `Found 1 deadlock` 部分 → 定位是哪两个线程、争用的锁对象，并在代码中修复加锁顺序问题。
439. **场景十四：Metaspace 溢出的早期征兆。** 不断地动态生成类（如用 CGLib 大量生成代理），观察 Metaspace 使用不断上升。用 `jstat -gc <pid> 1000` 持续监控元空间使用 → 触发 OOM → 用 `-XX:MaxMetaspaceSize` 限制。解释哪些操作会导致元空间溢出：大量反射、CGLIB 生成类、JSP 编译等。
440. **场景十五：综合大排查。** 设计一个"雪崩"场景：A 接口调 B 服务（HTTP 调用），B 服务变慢 → A 的线程池队列满 → A 的接口超时 → 上游网关超时 → 用户刷页面 → 更多的请求压向 A → A 彻底无法响应。逐层排查：看网关日志 → 看 A 的线程 Dump → 看 B 的慢查询 → 临时熔断 → 修复 B → 恢复。复现这个场景并记录每一步操作和观察。
