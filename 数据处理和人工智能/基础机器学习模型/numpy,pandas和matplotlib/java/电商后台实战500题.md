# 电商后台管理系统 · 实战 500 题

## 题单定位

这份题单面向**初级进阶**程序员：会用 Spring Boot 写简单接口，但没完整做过企业级项目。做完这份题单后，你将具备独立从零搭建一个电商后台、解决常见线上问题的能力。

**核心原则：**
- 每道题都是"做出来、跑得通"，不深入框架源码
- 代码写在哪、怎么验证，每道题都有明确产出
- 每个模块末尾有排错实战，训练你出问题自己查的能力
- SQL Server + Redis + RabbitMQ 三大中间件贯穿始终

**环境要求：**
- JDK 17+，Spring Boot 3.x，Maven
- SQL Server 2022 (Developer Edition)，通过 Docker 运行
- Redis 7.x，通过 Docker 运行
- RabbitMQ 3.x (Management)，通过 Docker 运行
- IntelliJ IDEA，Postman/Apifox，DataGrip 或 SSMS

## 模块二：SQL Server 索引与性能调优（60 题）

> **目标：** 打开执行计划不慌，看到慢查询知道怎么分析、怎么建索引、怎么写高效的 SQL。这是后端面试必考项。

### 2.1 SQL Server 执行计划入门（61-75）

61. 在 SSMS 中打开"包含实际的执行计划"（`Ctrl+M`），写一条 `SELECT * FROM goods WHERE price > 50`，观察执行计划中用了什么扫描方式（全表扫描 vs 索引查找）。
62. 解释执行计划中的常见操作符：`Table Scan`、`Clustered Index Scan`、`Clustered Index Seek`、`NonClustered Index Seek`、`Key Lookup`、`Nested Loops`、`Hash Match`、`Sort`。
63. 使用 `SET STATISTICS IO ON` 查看 SQL 语句的 IO 消耗：逻辑读取次数、物理读取次数。写两条同样结果但 IO 消耗不同的 SQL 对比。
64. 使用 `SET STATISTICS TIME ON` 查看 SQL 的 CPU 时间和总耗时。对比有无索引时的耗时差异。
65. 在 `goods` 表中插入 100 万条测试数据（编写存储过程或用在线数据生成工具），为后续调优练习提供数据基础。要求数据分布合理：价格在 1-10000 之间随机，分类 ID 在 1-100 之间随机，创建时间在最近一年内随机。
66. 在没有索引的情况下，查询"某个分类下价格大于 1000 的商品，按创建时间倒序"，记录执行计划和 IO 统计。
67. 为上述查询创建合适的索引（覆盖查询条件 + 排序），再次执行，对比索引创建前后的 IO 和耗时。
68. 演示 `索引覆盖`：查询仅需要 `goods_name`、`price`、`category_id` 三个字段，创建包含这三列的索引后，执行计划上会显示什么？Key Lookup 消失了吗？
69. 演示 `索引覆盖` 不生效的场景：`SELECT *` 时覆盖索引为什么不够用？对比 `SELECT 指定列` 和 `SELECT *` 的执行计划差异。
70. 使用 `DBCC DROPCLEANBUFFERS` 清除缓存后，对比"有索引"和"无索引"两种情况下同一查询的物理读取次数，加深对索引价值的理解。
71. 解释 `sys.dm_db_index_usage_stats` 视图：查询哪些索引从未被使用过，哪些索引被频繁扫描/查找。
72. 编写脚本统计当前数据库中缺失索引的建议（使用 `sys.dm_db_missing_index_details`），将缺失索引的创建 SQL 输出到结果集中。
73. 演示"错误的索引"带来的问题：为 `status` 字段创建索引（该字段只有 0/1 两个值，区分度极低），执行计划会选择这个索引吗？解释"索引选择性"的概念。
74. 演示联合索引的"左前缀"匹配规则：创建 `INDEX(category_id, price, create_time)` 联合索引后，分别测试 `WHERE category_id = 1`、`WHERE price > 100`、`WHERE create_time > '2025-01-01'` 三个查询中哪些能用到索引。
75. 使用 `OPTION(RECOMPILE)` 提示对比查询计划：为什么有时 SQL Server 会选错执行计划？什么场景下会用到这个 hint？

### 2.2 索引设计与优化实战（76-90）

76. 在 100 万数据下，对比"高选择性列"（如 `username` 唯一值很高）和"低选择性列"（如 `status` 只有 4 种状态）分别建索引的效果：索引大小、查询效率、是否被优化器选用。
77. 创建包含列索引（INCLUDE）：`CREATE INDEX idx_xxx ON goods(category_id) INCLUDE(goods_name, price)`。验证 INCLUDE 列不会影响索引键的排序，且可以覆盖查询。
78. 演示"索引查找 + 键查找(Key Lookup)"的完整过程：查询需要的列不完全在索引中时，SQL Server 先在非聚集索引找到主键，再用主键回聚集索引取数据。观察 IO 统计中逻辑读取的增加。
79. 优化 Key Lookup 的两种方式：① 在索引中加入 INCLUDED 列 ② 直接建覆盖索引。对比两种方式的索引大小和维护成本差异。
80. 分析 `ORDER BY goods_name OFFSET 0 ROWS FETCH NEXT 20 ROWS ONLY` 利用索引排序 vs 额外排序：对比有无 `INDEX(goods_name)` 时执行计划的 `Sort` 操作符差异。
81. 探究 SQL Server 的排序机制：什么情况下排序发生在内存中？什么情况会溢出到 `tempdb`？观察 `sys.dm_exec_query_stats` 中与排序溢出相关的指标。
82. 演示"排序 + 分页"的性能问题：`ORDER BY create_time DESC OFFSET 100000 ROWS FETCH NEXT 20 ROWS ONLY`。观察深层分页的执行计划，解释为什么越往后越慢。
83. 使用"延迟关联"优化深层分页：先用子查询获取前 100020 行的主键 ID，再 JOIN 原表取完整数据。对比优化前后的 IO 和耗时。
84. 探究"键值查找"(Key Lookup) 在批量查询中的风险：当查询返回行数较多时，优化器可能放弃索引直接全表扫描（tipping point）。在你的百万数据中测试并观察 tipping point。
85. 对比 `UNION` 和 `UNION ALL` 的执行计划差异：`UNION` 多了一个去重排序操作。在你的业务数据上验证两者耗时和 IO 的差距。
86. 使用 `EXISTS` 替代 `IN` 优化子查询：对比 `WHERE id IN (SELECT ...)` 和 `WHERE EXISTS (SELECT 1 FROM ... WHERE ...)` 的执行计划和性能。
87. 演示 `NOT IN`、`NOT EXISTS`、`LEFT JOIN ... IS NULL` 三种"差集"写法的性能差异，解释 `NOT IN` 遇到 NULL 时的陷阱。
88. 分析 `SELECT COUNT(*)` 和 `SELECT COUNT(1)` 和 `SELECT COUNT(column)` 的区别：在 SQL Server 中是否有性能差异？COUNT(column) 不统计 NULL 值这一特性在什么业务场景下有用？
89. 使用"筛选索引"(Filtered Index)：为 `WHERE deleted = 0 AND status = 0` 这种常见筛选条件创建筛选索引，对比普通全列索引的大小和性能差异。
90. 探究索引碎片：通过 `sys.dm_db_index_physical_stats` 查看索引的碎片率，当碎片率 > 30% 时，执行 `ALTER INDEX REORGANIZE` 或 `REBUILD`，对比重组前后的 IO 和扫描效率。

### 2.3 锁、阻塞与并发（91-100）

91. 使用 SQL Server Management Studio 开启两个查询窗口，模拟并发更新同一行（T1 更新 goods id=1 的价格，T2 也更新同一行但先不提交）。T2 会在哪里等待？观察 `sys.dm_exec_requests` 中的 `blocking_session_id`。
92. 使用 `sp_who2` 或 `sp_WhoIsActive` 存储过程，在阻塞发生时快速找到阻塞源头。
93. 使用 `sp_WhoIsActive` 定位锁等待链：A 阻塞 B，B 阻塞 C → 找出链头锁的持有者（阻塞源），查看其最后执行的 SQL。
94. 演示死锁：准备两个事务，T1 先更新 goods id=1 再更新 goods id=2，T2 先更新 goods id=2 再更新 goods id=1。同时提交两个事务，观察 SQL Server 的死锁图和错误信息。
95. 从 SQL Server 的死锁图中解读：受害者（被回滚的事务）是谁？幸存者（继续执行的事务）是谁？SQL Server 的选死策略是什么？
96. 阅读死锁 XML 报告：死锁发生时每笔事务持有的锁和请求的锁分别是什么类型的锁（X / U / S）？
97. 使用 `SET DEADLOCK_PRIORITY LOW` 降低某个会话的死锁优先级：在死锁发生时，指定某个不重要的业务做受害者被回滚。

### 4.3 缓存问题实战（216-230）

216. 模拟"缓存穿透"：故意请求一个数据库中不存在的 `goodsId = -1`（恶意查询），每次请求都穿透缓存直接打到数据库。在高并发下数据库扛得住吗？写一个 JMeter 或简单多线程测试脚本验证。
217. 解决"缓存穿透"——方案一（缓存空对象）：对不存在的数据也缓存一个标记（如 JSON null），过期时间设短一些（5 分钟）。写出完整代码并解释为什么过期时间要短。
218. 解决"缓存穿透"——方案二（布隆过滤器）：引入 Redisson 的 `RBloomFilter`，在查询之前先判断 Key 是否可能存在。将所有合法的商品 ID 初始化到布隆过滤器中。对比方案一的优劣。
219. 模拟"缓存击穿"：设置某个热门商品的缓存过期时间只有 1 秒，然后用 20 个并发线程同时请求这个商品的详情接口。观察数据库连接数和请求耗时。
220. 解决"缓存击穿"——方案一（互斥锁/双重检查锁）：在查数据库重建缓存前，先用 Redis 的 `SETNX` 获取一个锁 Key（如 `goods:lock:{id}`），拿到锁的线程负责查库并重建缓存，没拿到锁的线程 sleep 一小段时间后重新读缓存。
221. 解决"缓存击穿"——方案二（逻辑过期）：缓存永不过期（或过期时间很长），在 Value 中保存一个"逻辑过期时间"字段。获取到缓存后判断逻辑时间是否已过，若是则返回旧值同时起一个异步线程去更新数据库并刷新缓存。
222. 模拟"缓存雪崩"：在同一时间点让 Redis 中 500 个缓存 Key 同时过期，观察对数据库的压力。写一个脚本批量 `SETEX` 500 个 Key 过期时间都是 60 秒，60 秒后用并发请求模拟雪崩。
223. 解决"缓存雪崩"：为过期时间添加随机抖动（如基础过期时间 30 分钟 + 随机 1-10 分钟），避免大量缓存同时失效。
224. 实现"缓存预热"：系统启动时（`CommandLineRunner` 或 `@PostConstruct`），预先将热门分类、热门商品的信息加载到 Redis 中。这样用户第一次请求也很快。
225. 实现"缓存+数据库一致性"——延时双删策略：更新数据库前先删除缓存，更新数据库后等待 500ms 再次删除缓存。为什么是双删？500ms 的延迟是怎么估算出来的？
226. 实现"缓存+数据库一致性"——Canal/Binlog 方案原理理解（仅理解题）：如果不用延时双删，如何在数据库变更后自动同步更新缓存？解释 Canal/Binlog 方式的基本原理。
227. 使用 Redis `SUBSCRIBE/PUBLISH` 实现缓存更新通知：当商品数据被修改，发布一条消息到 Redis 频道，其他服务实例收到通知后清除本地缓存。
228. 实现"缓存空窗期"的兜底方案：在 Redis 故障（宕机或连接超时）时，自动切换到直接查数据库，并在 Redis 恢复后自动切回缓存。验证：手动停掉 Redis → 请求仍能正常返回数据（不报错误）→ 启动 Redis → 缓存自动恢复。

> **目标：** 实现完整的订单生命周期管理，引入 RabbitMQ 做异步解耦和削峰，掌握消息可靠投递、重复消费、死信队列等企业级方案。

### 5.1 订单核心流程（271-290）

271. 设计订单状态枚举：`PENDING_PAYMENT(待支付)`、`PENDING_DELIVERY(待发货)`、`PENDING_RECEIPT(待收货)`、`COMPLETED(已完成)`、`CANCELLED(已取消)`、`REFUNDING(退款中)`。画出状态流转图。
272. 实现"创建订单"接口 `POST /api/orders`：从购物车选中商品生成订单，包含：校验库存是否充足、扣减库存、生成订单号（雪花算法或日期+序号）、计算总金额、保存订单和订单明细。
273. 为订单表建立合适的索引，包括：`order_no` 唯一索引、`user_id` 索引、`status` + `create_time` 联合索引。解释为什么 `status` + `create_time` 的联合索引能加速"查询待支付订单"。
274. 测试雪花算法的 ID 生成是否有边界问题（50 个并发线程同时生成订单号），验证：无重复、大致有序、不包含特殊字符。
275. 实现"订单列表查询"接口：支持按订单号模糊搜索、按状态筛选、按时间范围筛选、按金额排序，返回分页数据。
276. 实现"订单详情查询"接口：返回订单基本信息 + 订单明细列表 + 收货人信息 + 状态变更记录。
277. 实现"取消订单"接口：只有 `PENDING_PAYMENT` 状态的订单可以取消。取消时需要把库存加回去（恢复库存）。
278. 实现"支付回调"接口（模拟支付）：`POST /api/order/pay-callback`，接收 `{orderNo, payAmount, payTime, transactionId}`。更新订单状态为"待发货"，同时发送一条 RabbitMQ 消息通知用户"订单已支付"。
279. 实现"用户端订单查询"接口：用户只能看到自己的订单，按创建时间倒序排列。在 SQL 中强校验 `user_id = 当前登录用户`。
280. 实现"卖家发货"接口：管理员填写物流公司和物流单号，将订单状态更新为"待收货"。发货时必须校验订单处于"待发货"状态，否则抛出业务异常。
281. 实现"确认收货"接口：用户收到货物后点击确认收货，状态变为"已完成"。在确认收货时自动触发一个定时打款给卖家的模拟逻辑（打印日志即可）。
282. 实现"订单状态变更记录"：每次订单状态变化都写入 `order_status_log` 表，记录变更前状态、变更后状态、操作人、操作时间、备注。订单详情接口一并返回这个变更记录列表。
283. 实现"申请退款"接口：用户在"待发货"或"待收货"状态下可申请退款，状态变为"退款中"。退款需要管理员审核。
284. 实现"退款审核"接口：管理员审核退款申请，同意 → 恢复库存 + 更新状态为"已取消"，拒绝 → 更新状态回之前的状态。
285. 实现"订单评价"功能：订单"已完成"后 15 天内可评价，超时则关闭评价入口。评价后订单打上已评价标记，商品的好评率更新。
286. 实现"重复下单检测"：用户在 3 秒内连续点击两次提交订单，通过 Redis `SETNX` 防重：`order:submit:{userId}:{timestamp}`，3 秒内相同 key 只允许创建一个订单。
287. 实现"订单金额计算"的完整逻辑：商品总价 − 优惠券金额 + 运费 = 实付金额。先在接口注释中定义计算公式，再用代码实现。目前暂不需要真正的优惠券模块，用模拟数据替代。
288. 实现"收货地址管理"：`user_address` 表含 `id`、`user_id`、`receiver_name`、`receiver_phone`、`province`、`city`、`district`、`detail`、`is_default`。支持用户增删改查收货地址，默认地址自动切换。
289. 在创建订单接口中增加"使用优惠券"参数（先用 Redis 中存储的模拟优惠券数据替代）：检查优惠券是否在有效期内、是否满足最低消费金额，满足则扣减订单金额。
290. 实现"订单失效提醒"：待支付订单距离超时还有 5 分钟时，通过 RabbitMQ 延迟消息发送一条提醒给用户（打印日志模拟短信或推送通知）。

294. 测试 `@Transactional` 失效的第 3 个场景：在事务方法中 try-catch 了异常没有往外抛。验证事务提交了而非回滚。解释为什么 Spring 只有感知到未捕获的异常才会标记回滚。

299. 对比定时任务扫描和 RabbitMQ 死信队列（延迟消息）两种超时取消方案的优劣：时效性、系统开销、实现难度。写一段分析注释。
300. 实现"下单后库存预留的完整事务链"：下单 → 预扣库存（t_inventory_reserved）→ 创建订单 → 三个操作在同一个事务中完成，任何一个失败都要全部回滚。
301. 测试事务传播中的"挂起"行为：方法 A（REQUIRED）调用方法 B（REQUIRES_NEW）时，A 的事务被挂起，B 开启新事务。验证 B 提交后 A 再提交，即使 B 已提交，A 回滚也不会影响 B 的数据。
302. 测试 `PROPAGATION_NESTED`（SQL Server 的 Savepoint 支持）：NESTED 内部事务回滚只回滚到 Savepoint 位置，不影响外部事务。验证 NESTED 和 REQUIRES_NEW 的回滚范围差异。
303. **排错实战：** 一个订单的库存已经恢复，但订单状态还是"待支付"未改成"已取消"。排查：① 检查定时任务是否在执行 ② 检查 SQL 的 WHERE 条件是否匹配 ③ 检查事务是否提交。定位并修复。
304. 模拟并排查"长事务导致锁持有时间过长"：在一个事务中先更新订单，然后 Thread.sleep(30000)，再提交。同时另一个线程查询该订单，观察被阻塞。用 `sp_who2` 定位阻塞源。

### 5.4 RabbitMQ 进阶实战（321-340）

321. 实现"订单延迟取消"——方案一（死信队列）：创建消息时设置 TTL（`setExpiration("1800000")` 即 30 分钟），该队列没有消费者，消息过期后自动转发到死信队列 `order.cancel.queue`，由死信队列消费者处理取消订单逻辑。
322. 实现"订单延迟取消"——方案二（延迟插件 rabbitmq_delayed_message_exchange）：安装 `rabbitmq_delayed_message_exchange` 插件到 RabbitMQ 容器，创建 `x-delayed-type` 交换机，发送消息时设置 `x-delay=1800000`。对比方案一的精准度和维护复杂度。
323. 实现生产者确认（Publisher Confirm）：配置 `spring.rabbitmq.publisher-confirm-type=correlated`，在 `RabbitTemplate` 中设置 `ConfirmCallback`，如果消息发送到交换机失败，触发回调记录失败日志并做补偿。
324. 实现生产者回退（Publisher Return）：配置 `spring.rabbitmq.publisher-returns=true`，设置 `ReturnCallback`，如果消息路由到队列失败（Routing Key 写错了），触发回调通知。
325. 实现"消息本地表"可靠投递方案：在"支付回调"中先写入消息本地表 `t_pending_message`（状态为 `TO_SEND`）和订单数据在同一事务中，再由定时任务扫描 `TO_SEND` 的消息发送到 RabbitMQ，发送成功后标记为 `SENT`。即使服务挂了，定时任务也能补偿投递。
326. 实现消费者幂等性：消费者在收到消息时先判断该消息 ID `(messageId)` 是否在 Redis 中已处理过（`SETNX message:consumed:{messageId} 1 EX 3600`），已处理则直接确认跳过，防止重复消费。
327. 在数据库层做幂等兜底：在消费订单消息时，如果操作是"修改订单状态"，SQL 语句加条件判断 `WHERE status = 期望的旧状态`，防止重复更新。
328. 实现"订单消息的优先级"方案：低优先级的批量订单通知用普通队列，高优先级的退款通知用一个专用消费者（或多一个消费者实例）快速消费。
329. 使用 RabbitMQ 的 `Prefetch` 参数（`spring.rabbitmq.listener.simple.prefetch=10`）限制消费者每次从队列预取的消息数，防止某个消费者拿太多消息处理不过来而其他消费者空闲。
330. 模拟并解决"消息积压"：快速发送 10000 条消息（用 for 循环快速 publish），消费者故意 `Thread.sleep(100)` 模拟处理慢。观察 RabbitMQ 管理界面的消息堆积，然后增加 2 个消费者实例（多写两个 `@RabbitListener` 或增加并发消费者数 `concurrency = "3-5"`），观察消息堆积下降。
331. 探究 RabbitMQ 的消息持久化：声明队列时设置 `durable=true`，发送消息时设置 `MessageProperties.PERSISTENT_TEXT_PLAIN`。停掉 RabbitMQ 容器再重启，检查消息是否仍然存在。
332. 实现"消息的序列化与反序列化"：配置 RabbitMQ 使用 JSON 序列化器（`Jackson2JsonMessageConverter`），发送自定义 Java 对象而非字符串。消费者中反序列化回来，验证对象内容一致。
333. 实现"消费者异常重试的退避策略"：第一次失败后等 1 秒重试，第二次等 3 秒，第三次等 9 秒……使用 `RetryTemplate` 配置指数退避，避免快速反复重试。
334. 使用 RabbitMQ 的 `Alternate Exchange` 备胎交换机：创建一个 AE 交换机，当消息无法被路由到正确队列时，自动转发到备胎交换机和队列，避免消息无声丢失。

430. 设计秒杀的"兜底补偿"流程：MQ 消息发送失败 → 暂时未处理订单 → 定时任务扫描秒杀成功但订单未创建的记录 → 重新发起创建。写出完整代码并进行补偿测试。

### 6.5 模块排错实战（431-440）

431. **排错实战 1：** 秒杀报"库存不足"但数据库中库存仍为大量正数。排查：① 检查 Redis 中的秒杀库存是否大于真实库存 ② 活动预热脚本是否正确加载了库存 ③ 是否有多次预热/活动未清理的残留数据。画出排查路线。
432. **排错实战 2：** 秒杀结束后仍有人成功下单——怀疑是分布式锁失效。检查：① 锁的 value 是否使用了相同 ID（线程复用导致错误解锁）② 锁超时时间是否太短导致锁自动过期。重现然后彻底修复。
433. **排错实战 3：** 出现超卖（实际售出 > 秒杀库存）。这比超时和锁失效更严重。检查：① Lua 脚本的库存判断逻辑是否用 `>= 0` 而非 `> 0` ② 库存拆分后各分片的判断是否独立 → 分片之间没有协调？③ Redisson 的分布式锁是否有主从切换导致锁数据丢失。按照可能性逐个排查并打出日志。
434. **排错实战 4：** 秒杀活动开始瞬间，所有用户都看到"系统繁忙"，但服务器 CPU 才 20%，数据库连接也正常。排查：是 Sentinel 的默认限流规则配得过低？是 Nginx 在前端限流了？是漏桶令牌桶参数不对？逐层恢复并重新压测。
435. **排错实战 5：** 热点 Key 导致 Redis 某个节点 CPU 飙高而其他节点空闲。使用 Redis 的 `INFO commandstats` 查看高频命令，用 `redis-cli --hotkeys` 定位热点 Key，进行库存拆分并重新测试。对比拆分前的热点 Key 指标和拆分后的均匀分布。
436. **排错实战 6：** 秒杀接口压测到 1000 QPS 时，出现 `RedisConnectionPoolExhaustedException`。排查 Lettuce 连接池配置：`max-active`、`max-idle`、`min-idle` 是否合理，连接池大小是否与 QPS 匹配。调大连接池并重新压测，找到当前配置的吞吐上限。
437. **排错实战 7：** 秒杀成功的订单在数据库中不存在（Redis 有记录但 MySQL 没有）。排查：MQ 消费者是否将消息入库？消费者入库的事务是否被回滚？秒杀成功但订单未创建的补偿定时任务是否执行？逐环节确认。
438. **排错实战 8：** JMeter 压测秒杀接口时，发现前 100 个请求全部成功，之后的请求全部超时。排查：连接池（数据库/Redis/RabbitMQ）是否被前 100 个请求占满，导致后续请求拿不到连接而排队。检查各连接池的 `max-wait` 和 `timeout` 配置。
439. **排错实战 9：** Lua 脚本执行报 `NOSCRIPT No matching script. Please use EVAL`。排查：Redis 重启后缓存的脚本 SHA 失效，但代码中仍尝试用 `EVALSHA` 而非 `EVAL` 执行脚本。修复：捕获 `RedisScriptingException`，fallback 到 `EVAL` 重新加载脚本。
440. **排错实战 10：** 秒杀活动结束后，部分 Redis Key 未被清理导致下一次活动预热时数据污染。排查：活动清理脚本是否正确扫描了所有待清理 Key、有无使用通配符（`SCAN + DEL`）、有无不同活动的 Key 命名冲突。写一个完整的活动结束后清理脚本，包含清理前备份 Key 值的逻辑。

### 7.4 生产环境故障全链路排错实战（486-495）

486. **场景一：CPU 飙高到 100%。** 在 Linux 上复现（WSL2 或虚拟机）：写一个死循环代码 → 部署到容器 → 触发 → 用 `top` 找到高 CPU 进程 → `top -Hp` 找高 CPU 线程 → 转十六进制 → `jstack` 线程快照中定位到代码行。完整记录你的排查过程。
487. **场景二：堆内存 OOM。** 写一段代码持续向 `ArrayList` 中加对象直到爆堆内存。配置 JVM 参数 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/heapdump.hprof`，OOM 后用 MAT 打开 dump 文件，找出占用内存最大的对象和 GC Root 引用路径。
488. **场景三：数据库连接池耗尽。** 故意在代码中设置连接池大小过小（如 `maximumPoolSize=2`），然后并发调接口直到 `Connection is not available`。用 HikariCP 的监控观察连接池状态，最终排查到根因并调整连接池参数。
489. **场景四：慢 SQL 拖垮系统。** 制造一个缺少索引的慢查询（`SELECT * FROM order_info WHERE status = ? AND create_time > ? ORDER BY create_time DESC`，表含百万数据且无索引）。观察：接口响应时间超过 5s → 数据库 CPU 高 → 定位慢 SQL (Query Store) → 加索引 → 恢复。记录整个定位和修复过程。
490. **场景五：缓存雪崩导致数据库被打满。** 模拟 50 个缓存 Key 同时过期 → 大量请求直打数据库 → 数据库 CPU 飙升。排查：先看 Redis 的缓存命中率（`INFO stats`）→ 发现大量 miss → 确认数据库出现瞬时大流量 → 加上随机过期时间 + 缓存预热 → 恢复。
491. **场景六：消息丢失。** 生产者消息发出去了，但消费者从未收到。排查：① 检查 RabbitMQ 管理界面的交换机是否正确绑定了队列 ② exchange/queue/routingKey 是否一致 ③ 消息序列化器是否两方一致 ④ 消费者被异常 catch 导致手动 ACK 不执行。模拟其中一种场景并修复。
492. **场景七：消息积压。** 消费者处理每条消息耗时 500ms，生产者每秒发 1000 条。排查：在 RabbitMQ 管理界面看到队列堆积了数十万条消息 → 增加并发消费者数量 → 消息积压缓解。如果此时 Redis/数据库成为消费者端瓶颈怎么办？讨论两种方案。
493. **场景八：应用启动失败。** 改配置后重启应用，但反复报 `APPLICATION FAILED TO START`。分析启动日志（`CONDITIONS EVALUATION REPORT`），定位具体缺失的 Bean。人工制造然后修复。
494. **场景九：用户"支付了钱但订单仍是待支付"。** 分析：① 支付回调是否执行 ② 是否因网络超时导致回调丢失 ③ 消息表是否有补偿机制 ④ 手动补偿这条订单。实现一个"手动确认支付"接口供客服使用。
495. **场景十：综合大排查。** 设计一个"雪崩"场景：A 接口调 B 服务（HTTP 调用），B 服务变慢 → A 的线程池队列满 → A 的接口超时 → 上游网关超时 → 用户刷页面 → 更多的请求压向 A → A 彻底无法响应。逐层排查：看网关日志 → 看 A 的线程 Dump → 看 B 的慢查询 → 临时熔断 → 修复 B → 恢复。复现并记录每一步。
