在 Spring Boot 项目中，自定义业务异常类通常继承自 `RuntimeException`（运行时异常）。

这样做的主要原因是：Spring 的数据库事务（`@Transactional`）**默认只会在遇到 `RuntimeException` 或其子类时才会自动回滚**。如果继承普通的 `Exception`，发生错误时数据库可能不会回滚，这在电商系统中是极其危险的。

以下是 `BusinessException` 的标准实现代码：

Java

```
package com.你的包名.exception; // 替换为你的实际包路径

/**
 * 自定义业务异常
 */
public class BusinessException extends RuntimeException {

    // 错误码
    private Integer code;

    /**
     * 只传入错误信息的构造方法
     * （默认使用 500 作为通用业务错误码）
     */
    public BusinessException(String message) {
        // 调用父类 RuntimeException 的构造方法，把 message 传给它
        super(message);
        this.code = 500;
    }

    /**
     * 同时传入自定义错误码和错误信息的构造方法
     */
    public BusinessException(Integer code, String message) {
        super(message);
        this.code = code;
    }

    /**
     * 获取错误码
     */
    public Integer getCode() {
        return code;
    }

    // 注意：不需要重写 getMessage() 方法，
    // 因为父类 RuntimeException 已经通过 super(message) 帮我们管理了 message 属性。
}
```

### 这个类要怎么用？

写好这个异常类之后，你在业务代码（比如 Service 层）里遇到任何不符合预期的情况，就可以直接把它“抛”出去，不用再写繁琐的 `if-else` 去层层返回错误。

比如在“用户下单”的业务逻辑中：

Java

```
@Service
public class OrderService {

    public void createOrder(Integer userId, Integer stock) {
        // 1. 校验库存
        if (stock <= 0) {
            // 直接抛出业务异常，阻断程序继续往下走
            throw new BusinessException(400, "商品库存不足，无法下单！");
        }

        // 2. 校验用户状态
        if (userId == null) {
            // 只传 message，默认使用 500 错误码
            throw new BusinessException("用户信息校验失败");
        }

        // 3. 正常的下单逻辑...
        System.out.println("下单成功");
    }
}
```

**下一步通常的做法：**

现在你抛出了 `BusinessException`，但如果放任不管，前端看到的还是默认的一大堆可怕的 Java 报错堆栈。

为了配合你之前写的 `ApiResponse`，业界标准的做法是写一个 **“全局异常处理器”（`@RestControllerAdvice`）**。它的作用是专门在后台“接住”你抛出的 `BusinessException`，然后把它拆解成 `code` 和 `message`，优雅地包装进 `ApiResponse.error()` 里返回给前端。
