这些词汇是网络安全领域中，最经典的五种 **SQL 注入攻击手法**。

当安全测试工具（例如你在压测和防御测试中使用的 SQLMap）向服务器发送恶意请求时，它发送的那些带有特殊构造的字符串，就叫做“载荷（Payload）”。工具会轮番使用这五种载荷，来探测你的系统在哪个环节防线薄弱。

通俗来说，如果把数据库比作一个仓库，这五种手法就是五种不同级别的“撬锁和试探套路”：

### 1. 联合查询注入 (Union-based) —— “所见即所得”

- **原理**：利用 SQL 语法中的 `UNION` 关键字，强行把攻击者想查的数据，拼接到原本正常的查询结果后面。
    
- **现象**：这是最直白的一种。只要前端页面会把查询结果展示出来（比如商品列表），攻击者就能利用 `UNION SELECT` 让页面直接把数据库的账号、密码等核心数据像正常商品一样显示在屏幕上。
    

### 2. 报错注入 (Error-based) —— “借刀杀人”

- **原理**：攻击者故意构造一个语法极其怪异的 SQL 载荷，触发底层数据库（如 SQL Server 或 MySQL）的运行错误。
    
- **现象**：高明的攻击者会把“想偷取的数据”嵌入到会引发错误的函数中（比如 `extractvalue` 或除以 0 的操作）。如果后端的 Spring Boot 没有做全局异常拦截，直接把底层的原生报错信息抛到了前端，攻击者就能从这句报错提示中直接读出关键数据。
    

### 3. 布尔型盲注 (Boolean-based Blind) —— “盲人摸象”

- **原理**：当前端页面写死了，既不显示查询结果，也不抛出报错信息时，攻击者就变成了“盲人”。但他们发现，注入真条件（如 `AND 1=1`）和假条件（如 `AND 1=2`）时，页面的表现（如 HTTP 状态码、或者多出一个空行）会有微小的不同。
    
- **现象**：攻击者开始像玩“猜数字”游戏一样问数据库：“管理员密码的第一个字母的 ASCII 码是不是大于 100？”。如果页面是“真”状态，就继续缩小范围，通过写脚本成千上万次地试探，硬生生把数据一个字符一个字符地猜出来。
    

### 4. 时间型盲注 (Time-based Blind) —— “静默读秒”

- **原理**：这是最极端的情况，页面无论对错，返回的内容完全一模一样，连微小的差别都没有。此时布尔型盲注也失效了。
    
- **现象**：攻击者会在载荷中加入时间延迟函数（比如 MySQL 的 `SLEEP(5)` 或 SQL Server 的 `WAITFOR DELAY`）。他们问：“数据库名的第一个字母是不是 A？如果是，就沉睡 5 秒”。如果攻击者发现浏览器转圈卡顿了 5 秒才收到响应，就知道自己猜对了。这也是为什么有时自动化扫描工具会运行得极其缓慢的原因。
    

### 5. 堆叠查询 (Stacked Queries) —— “致命连击”

- **原理**：利用分号 `;` 作为语句结束符的特性，在一条合法的 SQL 语句执行完后，紧接着再强行塞入第二条、第三条完整的 SQL 语句。
    
- **现象**：前面四种方法通常只能用来“偷看”数据（属于 `SELECT` 级别）。而堆叠查询极其危险，攻击者可以在分号后直接跟上 `UPDATE` 篡改数据，或者 `DROP TABLE` 删表跑路。当你的系统在遭遇破坏性攻击测试时，往往就是碰到了恶意的堆叠查询载荷。
    

在现代架构中，通过 MyBatis 的 `#{}` 预编译处理、严格的 JDBC 配置（如默认关闭多条语句执行 `allowMultiQueries=false`），以及对输入参数的强校验，就能将这五种载荷的攻击路径彻底切断。
