首先，我必须纠正你一个概念上的错误：查阅你的源码，你的 `LIMIT 500` 和 `TOP 500` 截断逻辑是写在 `getTableData`（获取表内数据预览）方法里的，而不是获取表结构（表名、列名等）！你在论文里必须写清楚，我们压测的是“大表数据预览接口的并发性能”，学术表述必须绝对精准。

要完成这项量化测试，你需要按照以下五个标准步骤来执行：

### 第一步：压测环境与测试数据准备（基准线）

你不能用一张只有 10 行数据的表去测试 `LIMIT 500`。

1. **构造大表：** 在你的 Docker 环境中的 MySQL 和 SQL Server 里，分别创建一张测试表 `test_performance_table`。
    
2. **注入海量数据：** 写一个简单的 Python 脚本或存储过程，向这张表里插入至少 **10万行** 测试数据（包含各种数据类型：Varchar, Int, DateTime 等）。只有基数足够大，数据库在执行 `SELECT * ... LIMIT 500` 时才会产生真实的 I/O 开销和游标负担。
    
3. **获取 Token：** 启动你的 Spring Boot 后端。先使用 Postman 或你的前端页面调用 `/api/driver/connect` 接口建立连接，拿到那个 UUID 格式的 Token。这个 Token 将用于后续所有的并发请求。
    

### 第二步：JMeter 核心测试计划配置

下载并启动 Apache JMeter，按照以下层级结构建立你的测试计划（Test Plan）：

- **添加 HTTP Header Manager（HTTP 信息头管理器）：**
    
    - 你的 `/api/metadata/table/data` 接口强依赖请求头中的 `Authorization` 字段。
        
    - 在这里配置 Name: `Authorization`, Value: `你刚才获取的 Token`。这保证了所有的并发线程都能通过后端的拦截。
        
- **添加 HTTP Request Defaults（HTTP 请求默认值）：**
    
    - 配置你的服务器 IP（通常是 `127.0.0.1` 或你的服务器公网 IP）和端口（如 `8080`）。
        

### 第三步：设置阶梯式线程组（Thread Group）

你的目标是量化 50、100、500 并发下的表现。你需要创建三个独立的 Thread Group（或者使用 Ultimate Thread Group 插件做阶梯加压）：

1. **配置参数：**
    
    - **Number of Threads (users):** 分别设置为 50, 100, 500。
        
    - **Ramp-up period (seconds):** 设置为 1 或 2。这意味着 JMeter 会在 1-2 秒内瞬间启动所有这 500 个线程，模拟真实的极速高并发洪峰。
        
    - **Loop Count:** 设置为 50 或 100。不要只跑一次，跑几百次才能得出稳定的平均数据。
        
2. **配置 HTTP Request 取样器：**
    
    - 在线程组内添加 HTTP Request。
        
    - Method: `GET`
        
    - Path: `/api/metadata/table/data`
        
    - Parameters: 添加 `tableName` 参数，值为你准备好的 `test_performance_table`。如果测试 SQL Server，记得带上 `schema` 参数。
        

### 第四步：收集核心量化指标（添加 Listeners）

在线程组下方添加以下两个核心监听器：

- **Summary Report（汇总报告）：** 这是你论文数据的直接来源。你需要关注里面的：
    
    - **Throughput (TPS/QPS):** 系统每秒能处理多少个完整的预览请求。
        
    - **Average (RT - 平均响应时间):** 毫秒级。
        
    - **Error %:** 错误率。如果 500 并发时你的后端因为数据库连接池被打满而开始报错，这个数字就会飙升。
        
- **View Results Tree（察看结果树）：** 仅在调试时开启，确认响应 JSON 中确实返回了 500 条数据结构，一旦正式压测必须关闭它，否则会大量消耗你本机的内存。
    

### 第五步：如何在论文中呈现（学术化输出）

拿到数据后，绝不能直接把 JMeter 的截图贴在论文里，那是专科生的做法。你必须在 4.3 章节绘制严谨的图表：

1. **绘制折线图：** X 轴是并发用户数（50, 100, 200, 500），Y 轴需要有两根线：一根是吞吐量（QPS），一根是平均响应时间（RT，单位 ms）。
    
2. **数据分析结论：** 你需要在文字中分析：“如图 4-X 所示，在 10 万级数据表前，凭借后端底层的强制 `LIMIT 500` 截断保护，当并发量达到 100 时，系统 QPS 稳定在 XXX，平均 RT 为 XXX ms；当并发激增至 500 时，虽然 RT 上升至 XXX ms，但 Error 率保持在 0%，证明了该数据拦截机制在极端并发下有效防止了 JVM 内存溢出（OOM）与数据库全表扫描带来的死锁。”
