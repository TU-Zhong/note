### 第一部分：使用 Apifox 验证“降低网络请求”与“解决并发死锁”

结合你的 `TableDataSubmitController` 和 `TableDataSubmitService`，自研的 Diff Pool 本质上是把散碎的多次 DML 操作合并成一个巨大的 JSON 载体 (`DataSyncRequest`)，在同一个事务中执行。我们通过以下 10 组 Apifox 用例来获取量化数据。

#### 维度一：证明“网络请求数量大幅降低” (Diff Pool 机制对比)

我们需要对比“传统单条提交”与“Diff Pool 批量提交”的耗时和网络开销。

- **用例 1：传统模式基准测试 (单条请求)**
    
    - **配置**：循环发送 100 次 `POST /api/metadata/submit/sync`。每次请求只包含 1 条 `inserted` 数据。
        
    - **目的**：记录 100 次 HTTP 握手、建立连接、后端的 100 次事务开启/提交的**总耗时**。
        
- **用例 2：Diff Pool 模式测试 (合并请求)**
    
    - **配置**：发送 1 次 `POST /api/metadata/submit/sync`。在 `DataSyncRequest` 的 JSON 中，同时放入 50 条 `inserted`，30 条 `updated`，20 条 `deleted` 数据。
        
    - **目的**：证明 1 次 HTTP 请求就能完成用例 1 中 100 次请求的工作量。**图表素材**：在论文中画出“传统模式耗时 (ms)” vs “Diff Pool 模式耗时 (ms)”的柱状图，以及“网络吞吐量 (KB)”的对比。
        

#### 维度二：证明“降低高并发下的死锁风险” (并发与事务测试)

你在 `executeUpdated` 中通过比对 `oldRow` 实现了乐观锁机制（如果 `updateCount == 0` 则抛出异常），结合事务，这正是你解决并发冲突的底气。使用 Apifox 的“性能测试” (Performance) 模块配置以下用例：

- **用例 3：无冲突高并发写入 (吞吐量上限)**
    
    - **配置**：设置 100 个并发线程，持续 1 分钟。每个线程向 `sync` 接口发送完全不同主键的 `updated` 或 `inserted` 数据。
        
    - **目的**：测试后端和数据库的连接池极限，记录 TPS (每秒事务数) 和平均响应时间。
        
- **用例 4：高并发热点数据冲突测试 (核心锁机制验证)**
    
    - **配置**：设置 50 个并发线程。故意让这 50 个请求的 `updated` 数组中，包含对**同一张表的同一行数据**的修改，且 `oldRow` 传入相同的初始状态。
        
    - **目的**：证明你的乐观锁机制生效。由于并发，只有一个请求能成功，其余 49 个会被你的代码拦截抛出“更新失败，数据可能已被其他用户修改”，**从而避免了数据库底层的死锁 (Deadlock) 报错**。记录错误率和错误类型。
        
- **用例 5：读写混合并发测试 (死锁重灾区)**
    
    - **配置**：在 Apifox 自动化测试中混合配置：60% 的线程疯狂调用 `GET /api/metadata/table/data` 读数据，40% 的线程调用 `sync` 接口大批量写数据。
        
    - **目的**：证明在大批量读取（全表扫描）和频繁更新交替时，系统的事务隔离级别没有导致表级死锁。
        
- **用例 6：大事务超时与回滚能力测试**
    
    - **配置**：在一个 `DataSyncRequest` 中放入 1000 条正常插入的数据，和 1 条故意引发 SQL 异常的数据（比如主键冲突或类型错误）。
        
    - **目的**：证明你的 `conn.rollback()` 逻辑绝对可靠。系统在高并发下不会因为脏数据产生“幽灵锁”或部分提交。
        

#### 维度三：大单体数据包处理极限 (后端内存与解析)

- **用例 7：1000 级别 Diff 载荷压测**
    
    - **配置**：发送 1 个包含 1000 条 Update 数据的 JSON。
        
    - **目的**：测试 Spring Boot 解析巨大 JSON 和 JDBC 批量生成 SQL 的内存消耗与时间。
        
- **用例 8：10000 级别 Diff 载荷压测 (极端情况)**
    
    - **配置**：包含 10000 条变化数据的 JSON。
        
    - **目的**：找到后端的瓶颈点。如果这一步发生 OOM (Out Of Memory) 或者响应超过 30 秒，你可以在论文中坦承：“系统在单次 10000 条 Diff 提交时达到瓶颈，未来将引入分片上传机制”。**这种主动承认极限的态度，在答辩中非常加分。**
        
- **用例 9：Null 值比对逻辑压测**
    
    - **配置**：批量构造 `oldRow` 中包含大量 `null` 值的更新请求。
        
    - **目的**：验证你代码中精心编写的 `IS NULL` 逻辑 在大批量数据下是否会引起全表扫描（由于 Null 索引失效），记录数据库的慢查询日志。
        
- **用例 10：长连接保持与断开重连**
    
    - **配置**：模拟长时间不操作后，突然发送巨大 Diff 包；或者中途手动使 `ConnectionCathe` 中的 Token 对应连接失效。
        
    - **目的**：验证系统的鲁棒性，确保在各种异常网络状态下提交数据，不会引发连接池死锁。

### 第二部分：如何应对“十万级数据的网络前端内存泄漏”质疑？

如前所述，Apifox 测不了这个。你必须**离开 Apifox，打开 Chrome 浏览器**来做这部分数据的收集。老师的质疑非常精准：Vue/React 在维护十万级规模的响应式数组 (Reactive Arrays) 时，确实会卡爆甚至崩溃。

你需要按照以下步骤生成数据图表：

1. **打开 Chrome 开发者工具 (F12) -> Performance (性能) 面板 和 Memory (内存) 面板。**
    
2. **测试场景 A（未使用虚拟列表/普通响应式）：**
    
    - 让前端加载 10 万条数据，随意修改几条触发 Diff 收集。
        
    - 在 Memory 面板点击 "Take heap snapshot" (获取堆快照)。
        
    - **预期结果**：你会看到 JS Heap Size 飙升到几百 MB 甚至 1GB 以上，页面出现明显卡顿，这部分截图就是反面教材。
        
3. **测试场景 B（使用你优化的机制）：**
    
    - 如果你前端使用了**虚拟滚动 (Virtual Scrolling)** 或**分页 (Pagination)**，并且你的 Diff Pool 采用了非响应式的普通 Map/Set 来单独收集（而不是让 10 万条数据全部保持响应式监听）。
        
    - 再次获取堆快照。
        
    - **预期结果**：内存可能稳定在 30-50MB。
        
4. **制作图表**：将场景 A 和场景 B 的内存占用曲线放在一起，这就是强有力的“量化数据支持”。
