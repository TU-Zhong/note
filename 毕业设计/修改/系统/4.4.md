## 4.3 全局动态 SQL 防注入拦截体系实现

在开放式数据库管理平台中，动态拼接 SQL 语句是引发注入攻击（SQL Injection）的核心风险点[cite: 5, 6]。本系统构建了从请求实体校验、标识符方言转义到参数化预编译执行的三层防御体系，确保了 DDL 与 DML 操作的安全性[cite: 5, 6, 11]。

### 4.3.1 结构化请求对象与参数白名单校验

系统严格禁止前端直接传递原始 SQL 字符串，而是通过结构化的 DTO（Data Transfer Object）封装业务意图[cite: 12, 14]。

*   **强类型模型定义**：在建表与加列场景下，系统定义了 `CreateTableRequest` 和 `AddColumnRequest` 实体[cite: 12, 14]。例如，`ColumnDef` 静态类明确限定了字段名、数据类型、是否为空等元数据属性，从源头过滤了非法的指令片段[cite: 14]。
*   **正则表达式白名单**：针对 SQL 中无法使用占位符的参数（如字段长度或精度），系统在 `TableDdlService` 中引入了正则校验[cite: 5]。例如，长度参数必须匹配 `^[0-9,]+$`，有效阻断了利用长度字段植入恶意代码的路径[cite: 5]。

### 4.3.2 异构数据库方言标识符强制转义

针对 DDL 操作中涉及的表名、列名等标识符，系统基于策略模式实现了方言级别的转义处理，防止利用引号闭合进行的注入攻击[cite: 15, 16]。

*   **策略模式适配**：`DialectFactory` 根据当前物理连接的元数据自动匹配 `MySqlDialect` 或 `SqlServerDialect` 实例[cite: 15]。
*   **强制包裹机制**：所有标识符均需通过 `quoteIdentifier` 方法处理[cite: 16, 18]。
    *   **SQL Server 方言**：使用中括号 `[]` 包裹，并针对标识符内部可能存在的 `]` 符号进行双写转义（`]]`）[cite: 18]。
    *   **MySQL 方言**：使用反引号 `` ` `` 包裹，并处理内部反引号逃逸[cite: 17]。

**代码 4-5：SqlServerDialect 标识符绝对防御逻辑[cite: 18]：**
```java
public String quoteIdentifier(String name) {
    if (name == null || name.isEmpty()) return "";
    // 将 ] 替换为 ]] 以防止非法截断，并用 [] 包裹整体[cite: 18]
    return "[" + name.replace("]", "]]") + "]"; 
}
```

### 4.3.3 预编译执行与事务层级防御

对于涉及数据的 DML 操作（增删改），系统在 `TableDataSubmitService` 中全面采用了 `PreparedStatement` 预编译技术[cite: 4]。

*   **参数化查询**：在构建动态 SQL 时，系统仅拼接经 `quoteIdentifier` 处理后的安全列名，而所有的用户输入值均使用 `?` 占位符替代[cite: 4]。
*   **占位符注入**：通过 `ps.setObject(paramIndex++, value)` 将真实数据注入执行引擎[cite: 4]。由于预编译阶段已确定了 SQL 的语法树结构，用户输入的恶意字符串将被严格视为字面量，无法改变原有的执行逻辑[cite: 4]。

### 4.3.4 全局异常屏蔽与防御验证

为了防止攻击者通过报错信息（Error-based SQLi）探测系统底层架构，系统通过 `GlobalExceptionHandler` 实现了异常的统一脱敏处理[cite: 11]。

*   **堆栈信息屏蔽**：该处理器通过 `@ExceptionHandler(Exception.class)` 拦截所有底层驱动抛出的 SQL 异常[cite: 11]。
*   **友好结果返回**：系统仅向前端回传业务级的错误提示（如“操作失败：数据类型不匹配”），而将详细的错误堆栈记录于后端日志中，避免了敏感信息的外泄[cite: 11, 27]。
