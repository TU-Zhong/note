## 4.2 基于 Diff Pool 的差异同步引擎实现

为了解决 Web 端数据库管理系统在处理大规模离散数据编辑时产生的高频网络 I/O 痛点，本研究设计并实现了一套基于差异池（Diff Pool）的异步同步引擎。该引擎通过前端状态机追踪与后端事务控制的深度咬合，实现了数据变更的“感知即追踪，提交即闭环”。

### 4.2.1 差异追踪池设计与状态机模型

在前端逻辑层，系统利用 Vue 3 的 `reactive` 特性构建了一个名为 `diffPool` 的差异追踪仓库。该仓库包含 `inserted`、`updated` 和 `deleted` 三个独立的 Set 集合，用于分类存储待变更行的唯一标识。

- **前端虚拟主键机制**：考虑到部分数据库表可能未定义物理主键，或用户正在操作新增行，系统通过 `generateRowId` 方法为每一行动态注入一个以 `row_` 为前缀的虚拟主键（__row_id）。
    
- **状态转换逻辑**：当用户点击“添加行”时，系统生成新行并将其 ID 存入 `inserted` 池；当点击“删除”时，系统优先检查该行是否为新行，若不是则存入 `deleted` 池。

### 4.2.2 毫秒级字段比对算法与可视化反馈

为了实现单元格级别的精准感应，系统在 `useTableDiff` 中实现了 `checkRowDiff` 字段比对函数。

- **全字段扫描比对**：该函数通过遍历 `tableColumns` 集合，将当前行数据（tableData）与原始镜像数据（originalData）进行深度比对。为了规避 JavaScript 隐式类型转换带来的误差，算法强制将所有值转换为 `String` 类型进行严格匹配。
    
- **交互高亮反馈**：系统在 `useTableUI` 中定义了 `getRowClassName` 钩子函数，实时监听 `diffPool` 的变更。根据行 ID 所处的集合，动态为表格行绑定不同的 CSS 类名，实现“绿色-新增、黄色-修改、红色-删除”的可视化预警。
    

代码 4-3：毫秒级字段比对算法核心实现：

JavaScript

```
const checkRowDiff = (row) => {
    if (diffPool.inserted.has(row.__row_id)) return; // 豁免新增行
    const originalRow = originalData.value.find(r => r.__row_id === row.__row_id);
    let isChanged = false;
    for (const col of tableColumns.value) {
        // 统一转为 String 进行严格比对，防止 123 与 "123" 的误判
        const val1 = String(row[col] ?? '');
        const val2 = String(originalRow[col] ?? '');
        if (val1 !== val2) { isChanged = true; break; }
    }
    isChanged ? diffPool.updated.add(row.__row_id) : diffPool.updated.delete(row.__row_id);
}
```

### 4.2.3 结构化载荷打包与提交优化

在执行“保存更改”操作时，`useTableSync` 钩子负责将弥散的差异池数据进行结构化打包。为了减小网络传输压力并确保执行安全，系统进行了以下优化：

1. **自增列规避**：系统通过 `identityColumnNames` 过滤器自动识别并剔除数据库自增列，确保新增数据能正确由数据库底层生成标识。
    
2. **主键精简策略**：在处理删除（deleted）和修改（updated）操作时，系统优先提取物理主键字段（pkColumnName）作为定位条件，仅在无主键时才采用全字段匹配，极大降低了 Payload 体积。

### 4.2.4 后端策略适配与原子化事务控制

后端 `TableDataSubmitService` 接收到 `DataSyncRequest` 载荷后，进入最终的持久化阶段。

- **方言策略适配**：系统通过 `DialectFactory` 动态产出对应的 `SqlDialect` 实例，确保生成的 SQL 语句在标识符包裹、表名引用及参数占位符层面完全符合目标数据库（如 MySQL 或 SQL Server）的规范。
    
- **事务原子性保障**：后端强制开启手动事务管理（`setAutoCommit(false)`），按照“先删除、后插入、再更新”的严格顺序执行。若同步过程中发生任何异常（如外键约束冲突），系统将立即触发 `conn.rollback()`，并由 `GlobalExceptionHandler` 捕获异常反馈给前端，确保了数据同步的原子性与一致性。
    

代码 4-4：后端事务驱动同步核心实现：

Java

```
// 强制开启事务控制，确保数据一致性
conn.setAutoCommit(false);
try {
    // 依次执行删除、插入、更新逻辑[cite: 4]
    for(Map<String,Object> row : request.getDeleted()) executeDelete(conn, tableName, row, dialect);
    for(Map<String,Object> row : request.getInserted()) executeInserted(conn, tableName, row, dialect);
    for(UpdatePair pair : request.getUpdated()) executeUpdated(conn, tableName, pair.getOldRow(), pair.getNewRow(), dialect);
    conn.commit(); // 只有全部执行成功才提交事务[cite: 4]
} catch (Exception e) {
    conn.rollback(); // 任意步骤失败即回滚[cite: 4]
    throw e;
}
```
