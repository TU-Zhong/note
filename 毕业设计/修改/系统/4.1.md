## 4.4 基于 HikariCP 与 Caffeine 的并发会话调度实现

在多用户并发访问异构数据库的场景下，如何高效管理海量的物理连接池并防止内存溢出（OOM）是系统的核心挑战。本系统设计了一套结合身份校验、动态路由与资源自动回收的会话调度体系，确保了系统在高负载下的稳定性与安全性。

### 4.4.1 多级身份鉴权与会话隔离机制

为了保障多租户环境下的数据安全性，系统在应用层与资源层构建了双重隔离屏障。

- **平台级登录鉴权**：系统通过 `LoginController` 管理平台访问权限。用户登录成功后，系统生成 UUID 令牌（Token）并存入 `tokenStore` 中。
    
- **请求拦截校验**：通过 `WebConfig` 注册全局拦截器 `AuthInterceptor`，对除登录外的所有 API 请求进行预处理。拦截器从 HTTP Header 的 `Authorization` 字段提取 Token 并进行合法性校验，若 Token 失效则通过 `response.setStatus(401)` 拒绝访问，确保了操作指令的来源可靠。

### 4.4.2 基于 Caffeine 的高性能连接路由调度

系统核心调度中心 `ConnectionCathe` 负责管理“逻辑会话”与“物理连接池”的映射关系。

- **内存路由表设计**：系统采用 Caffeine 缓存库构建路由表，将用户连接令牌作为 Key，将对应的 `HikariDataSource` 实例作为 Value 进行托管。相比传统的 `HashMap`，Caffeine 提供了更优的并发竞争性能和丰富的缓存淘汰算法。
    
- **会话精准路由**：当业务请求到达时，系统依据 Header 中的 Token 实时从缓存中检索目标数据源实例。这种设计实现了“一次连接、多次复用”，避免了频繁建立 TCP 握手带来的性能损耗。

```
@PostConstruct
public void init() {
    poolMap = Caffeine.newBuilder()
        .maximumSize(100) // 最大缓存 100 个异构数据源实例[cite: 1]
        .expireAfterAccess(30, TimeUnit.MINUTES) // 核心：空闲 30 分钟自动淘汰
        .removalListener((String key, HikariDataSource ds, RemovalCause cause) -> {
            // 安全回收逻辑
            if (ds != null && !ds.isClosed()) ds.close(); 
        })
        .build();
}
```

### 4.4.3 物理资源生命周期管理与自动回收

针对异构数据库连接池可能导致的资源泄露风险，系统通过 HikariCP 与 Caffeine 的联动实现了资源的自动化闭环管理。

- **连接池动态伸缩**：每个 `HikariDataSource` 实例均配置了 `MaximumPoolSize: 20` 和 `MinimumIdle: 5`，允许连接池根据当前并发压力动态调整物理连接数。
    
- **“氧气管”自动拔除机制**：这是系统资源治理的关键。通过配置 `expireAfterAccess(30, Minutes)`，一旦某个数据库会话在半小时内无任何操作，Caffeine 缓存会自动将其剔除。此时，触发的 `removalListener` 监听器会显式调用 `ds.close()` 方法，彻底关闭底层的物理连接池并释放相关的 TCP 连接与内存资源，从根本上预防了“僵尸连接”积压导致的系统宕机。
