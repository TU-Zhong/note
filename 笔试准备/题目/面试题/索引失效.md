在数据库开发和后端面试中，理解索引失效的场景是非常核心的知识点。以最常用的 MySQL/InnoDB 为例，索引失效通常是因为查询方式破坏了 B+ 树的有序性，或者优化器认为全表扫描成本更低。

这里为你整理了 **13 个常见的索引失效例子及原因**，为了方便记忆，我们可以将它们分为几个大类：

### 一、 破坏了索引的有序性（函数与操作）

**1. 在索引列上使用函数**

如果对索引字段使用了内置函数，数据库需要把所有的索引值都拿出来执行一遍函数再比较，自然就无法使用 B+ 树的快速查找了。

- **失效示例**：`SELECT * FROM user WHERE DATE(create_time) = '2023-01-01';`
    
- **正确写法**：`SELECT * FROM user WHERE create_time >= '2023-01-01 00:00:00' AND create_time <= '2023-01-01 23:59:59';`
    

**2. 在索引列上进行算术运算**

与使用函数同理，改变了列的原始值。

- **失效示例**：`SELECT * FROM user WHERE age + 1 = 20;`
    
- **正确写法**：`SELECT * FROM user WHERE age = 19;`
    

**3. 隐式类型转换**

如果索引列是字符串类型（如 `VARCHAR`），但查询时传入了整型数字。MySQL 在比较时会默认将字符串转换为数字，这实际上相当于在索引列上加了 `CAST` 函数，导致索引失效。

- **失效示例**：`SELECT * FROM user WHERE phone = 13800138000;` （假设 phone 是 varchar）
    
- **正确写法**：`SELECT * FROM user WHERE phone = '13800138000';`
    

### 二、 联合索引使用不当

**4. 违反“最左前缀法则”**

假设有一个联合索引 `(a, b, c)`，B+ 树是先按 `a` 排序，再按 `b` 排序，最后按 `c` 排序。如果直接跨过前面的列去查后面的列，就无法利用索引。

- **失效示例**：`SELECT * FROM table WHERE b = 1 AND c = 2;` （跳过了 `a`，完全不走索引）
    

**5. 联合索引中使用了范围查询，导致右侧列失效**

在联合索引 `(a, b, c)` 中，如果对中间的列使用了范围查询（`>`, `<`, `BETWEEN`, `LIKE`），那么该列后面的列将无法使用索引。

- **半失效示例**：`SELECT * FROM table WHERE a = 1 AND b > 2 AND c = 3;` （`a` 和 `b` 会走索引，但 `b` 是范围查找，导致 `c` 无法使用索引）
    

### 三、 模糊查询与逻辑运算符

**6. `LIKE` 以通配符开头**

B+ 树是按照字符从左到右匹配的。如果左边是未知的，就无法在树中进行二分查找。

- **失效示例**：`SELECT * FROM user WHERE name LIKE '%三';` 或者 `LIKE '%三%'`
    
- **正确写法**：`SELECT * FROM user WHERE name LIKE '张%';` （这种情况会走索引）
    

**7. 使用 `OR`，且其中包含非索引列**

如果 `OR` 前后的条件中，有一个列没有索引，那么优化器为了保证数据查询完整，只能放弃索引进行全表扫描。

- **失效示例**：`SELECT * FROM user WHERE id = 1 OR description = 'test';` （假设 id 有索引，description 无索引）
    

### 四、 比较操作符的陷阱

**8. 使用不等于操作符 (`!=` 或 `<>`)**

在大多数情况下，查询不等于某个特定的值，往往意味着要扫描全表绝大部分数据。优化器计算成本后，通常会直接选择全表扫描。

- **失效示例**：`SELECT * FROM user WHERE status != 0;`
    

**9. 使用 `NOT IN` 或 `NOT EXISTS`**

与不等于的情况类似，由于需要排除特定的值，剩下的数据量如果很大，优化器也会直接放弃索引。

- **失效示例**：`SELECT * FROM user WHERE id NOT IN (1, 2, 3);`
    

**10. 使用 `IS NOT NULL` (视数据分布而定)**

很多时候 `IS NOT NULL` 会导致全表扫描，特别是当表中绝大部分数据都不是 NULL 时，走全表扫描比走索引回表更快。

- **失效示例**：`SELECT * FROM user WHERE email IS NOT NULL;`
    

### 五、 优化器与架构层面的原因

**11. 两表 JOIN 时字符集 (Charset) 或排序规则 (Collation) 不一致**

如果驱动表和被驱动表的关联字段字符集不一样（比如一个是 `utf8`，一个是 `utf8mb4`），在表连接时会发生隐式字符集转换，相当于调用了函数，导致被驱动表上的索引失效。

- **排查思路**：建表时务必统一所有表的默认字符集。
    

**12. `IN` 列表中的值数量过大**

虽然 `IN` 通常是会走索引的，但如果 `IN` 括号里面的值成千上万，优化器经过成本估算（Cost-based Optimization），可能会认为走普通索引后大量回表的代价大于直接全表扫描，从而引发全表扫描。

**13. 表数据量极小或数据分布极度倾斜**

MySQL 优化器非常“聪明”。如果一张表总共就几百条数据，或者你要查的数据占了整张表的 90% 以上，优化器会认为全表扫描的 I/O 成本比“查索引 -> 回表查询数据页”更低，从而主动忽略索引。
