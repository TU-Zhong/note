**但是！如果我们要做到 100% 的滴水不漏，以防遇到那种喜欢“广度扫射”的面试官，这个项目在“常规八股文结合业务”的方向上，还有 3 个隐藏的考点（作为防御性准备）：**

### 🛡️ 隐藏考点 1：设计模式与扩展性（面向对象设计）

- **面试官可能会问：** 你现在连的是 SQL Server，如果明天产品经理让你支持 MySQL、Oracle 或者 PostgreSQL，你的后端代码需要大改吗？你是怎么设计的？
    
- **准备思路：** 这里一定要扯到**设计模式**。你可以说你采用了**“策略模式（Strategy Pattern）”**或**“工厂模式（Factory Pattern）”**。定义一个统一的 `DatabaseDialect` 接口（包含获取表结构、分页查询、拼接方言 SQL 等方法），然后写不同的实现类（如 `SqlServerDialectImpl`, `MysqlDialectImpl`）。根据前端传来的数据库类型，动态路由到不同的实现类，符合“开闭原则（对扩展开放，对修改关闭）”。
    

### 🛡️ 隐藏考点 2：数据库连接池与资源泄露（JDBC 底层进阶）

- **面试官可能会问：** 你在前端连了一个数据库，后端是怎么建立连接的？每次点击都 `DriverManager.getConnection()` 吗？如果用户执行了一个非常耗时的 SQL（比如查了 10 分钟），导致连接一直不释放怎么办？
    
- **准备思路：** * **连接池：** 不能每次都直连，得提一下你动态维护了一个连接池（比如用 `ConcurrentHashMap` 缓存每个数据源的 `HikariDataSource`）。
    
    - **超时控制：** 必须要在 JDBC 层面设置 `Statement.setQueryTimeout()`，防止用户的慢 SQL 把你的 Tomcat 线程池和数据库连接池双双拖死。
        

### 🛡️ 隐藏考点 3：Spring 事务与批量操作（经典实战）

- **面试官可能会问：** 刚才咱们聊了你前端打包提交了 Insert、Update、Delete 三个数组。如果后端在执行到 Update 的时候，因为某个原因报错了（比如并发修改失败），前面已经执行成功的 Insert 会回滚吗？
    
- **准备思路：** 考察 `@Transactional` 的编程式事务或声明式事务。你需要强调，这三种操作在后端是放在**同一个 Spring 事务**（或者同一个 JDBC `Connection` 并设置 `setAutoCommit(false)`）里的。一旦中间报错，必须整体 `rollback()`，保证原子性，防止出现“半张表更新成功”的数据污染事故。
