### 第一维度：底层架构与 JVM 机制（极其硬核，必问！）

简历关键词：`深入底层 JDBC 规范`、`反射与 URLClassLoader`、`动态隔离加载`

**1. 拷打原题：你说你用 URLClassLoader 实现了驱动的动态加载，请问具体是怎么做的？如果我要连接 MySQL 5.7 和 MySQL 8.0，会不会发生类冲突？**

- **面试官意图：** 考察你对 JVM 类加载机制（双亲委派模型）的理解。
    
- **高分思路：** 解释通常的驱动都在 classpath 下，但为了动态支持不同版本，你自定义了 ClassLoader，打破了传统的双亲委派机制，为每个连接或不同版本的驱动实例化一个独立的 `URLClassLoader`。这样不同版本的 `com.mysql.cj.jdbc.Driver` 类的全限定名虽然一样，但由于加载它们的 ClassLoader 不同，在 JVM 中被视为完全不同的两个类，完美实现了“物理隔离”。
    

**2. 夺命追问：如果用户删除了某个数据库连接，这个 URLClassLoader 能被 JVM 的垃圾回收器（GC）顺利回收吗？有没有可能引发 Metaspace（元空间）内存泄漏？**

- **面试官意图：** 考察你对垃圾回收（GC Roots）和内存泄漏的实战认知。
    
- **高分思路：** 坦诚这是一个极易踩坑的点。向面试官说明，仅仅把 ClassLoader 置为 null 是不够的，因为 JDBC 的 `DriverManager` 底层会持有 Driver 实例的静态引用（GC Root）。必须在卸载前，通过反射调用 `DriverManager.deregisterDriver()` 强制注销，并确保相关的 Connection 全部关闭，才能彻底避免类和 ClassLoader 泄露导致的 OOM。

### 第二维度：前端状态与数据流转（自研亮点）

简历关键词：`首创前端 Diff Pool`、`Vue 3 reactive`、`深拷贝数据快照`

**3. 拷打原题：你的“前端 Diff Pool”具体是怎么设计的？它解决了传统表单提交的什么痛点？**

- **面试官意图：** 考察你对前端复杂状态管理的掌控力。
    
- **高分思路：** 传统方式每次修改都会发请求，或者提交时遍历整个表格，性能差且无法应对复杂操作。你的设计是维护三个 `Set`（inserted、updated、deleted）。通过在每行注入一个虚拟的 `__row_id` 作为唯一标识。在单元格失去焦点时触发“脏检查深对比算法”，只有真正改变了数据才会进入更新池。
    

**4. 夺命追问：如果用户做了一套极其变态的操作：先新增了一行，然后修改了它，最后又删除了它。你的 Diff Pool 是怎么流转的？提交给后端时会发送什么？**

- **面试官意图：** 考察状态机的逻辑严密性。
    
- **高分思路：** 展现你的严谨逻辑。新增时加入 `inserted` 池；修改时，系统判断它已经在 `inserted` 池中，所以**不加入** `updated` 池（因为新增的数据本身就是全量提交）；删除时，系统判断它在 `inserted` 池中，直接从本地数组剔除，并从 `inserted` 池中抹除，**绝对不进入** `deleted` 池。最终用户点击提交时，后端什么都不会收到（完美抵消），极大节省了网络开销。

### 第三维度：并发安全与极限场景（咱们刚才的实战精华！）

简历关键词：`智能拦截无效更新防误触`、`数据一致性`

**5. 拷打原题：如果两个用户同时用你的 IDE 打开了一张表，A 修改了某行并提交，B 稍后也修改了同一行并提交，你会把 A 的数据覆盖掉吗？怎么防并发？**

- **面试官意图：** 考察高并发下的数据一致性处理（乐观锁机制）。
    
- **高分思路：** 把咱们写的代码搬出来！前端提交时打包 `oldRow` 和 `newRow`。后端拼接 SQL 时，把 `oldRow` 作为 `WHERE` 条件。执行时判断 `executeUpdate()` 的返回值，如果为 0，说明数据已被他人修改，直接抛出异常并拦截。
    

**6. 夺命追问：用 `oldRow` 做 `WHERE` 条件是个好主意，但如果这行数据里包含了高精度的 `datetime2` 类型（包含 7 位小数）或者是自增主键（Identity），你在拼接 SQL 和执行更新时遇到了什么大坑？怎么解决的？**

- **面试官意图：** 只有真正手写过底层 IDE 的人，才会碰到并解决这个问题，这是你拉开与其他候选人差距的**绝杀题**！
    
- **高分思路（大招）：** 抛出“商业级 IDE 降维打击”理论。
    
    - **关于自增列：** 前端通过获取表元数据识别出 `identity` 类型的列，在提交时直接将其从 `newRow` 剔除，防止触发 SQL Server 的 `IDENTITY_INSERT` 报错。
        
    - **关于精度丢失与 WHERE 拼接：** 坚决不能把所有列都放进 `WHERE` 里比较。前端识别出 `primarykey`（主键），在组装 `oldRow` 时进行**精准瘦身**，只保留主键字段（如 `WHERE student_id = 2`），完美避开了浮点数和高精度时间的比对误差，同时大幅提升了 SQL 的执行效率！

### 第四维度：全局架构与安全性

简历关键词：`Pinia 高可用多标签页`、`JWT 会话隔离`

**7. 拷打原题：作为一个多 Tab 标签页的 IDE，用户如果在 Tab 1 连了测试库，在 Tab 2 连了生产库，你是怎么保证它们执行 SQL 时绝对不串库的？**

- **面试官意图：** 考察你对多数据源会话隔离的理解。
    
- **高分思路：** 讲解 Axios 拦截器结合 JWT 的机制。每个建立的连接都会生成一个包含连接唯一标识（或者加密的连接配置）的 Token。在发请求时，前端根据当前激活的 `activeTabId` 从 Pinia 中取出对应的 Token 放入 Header。后端接收到请求后，解析 Token 并在后端的连接池缓存（ConcurrentHashMap）中精准命中对应的 `Connection` 对象，从物理链路上杜绝了“串库”的可能性。
