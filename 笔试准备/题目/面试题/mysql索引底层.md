这道题绝对是后端开发面试中**数据库领域的“镇山之宝”**。不管是国企还是大厂，只要问到 MySQL，99% 会让你聊聊索引的底层数据结构。

理解索引底层，死记硬背数据结构很难受，我们直接从它的**设计初衷**出发来剖析。

### 一、 核心概念：什么是索引？

用大白话来说，**索引就是数据库的“目录”**。

想象一下，你要在一本 5000 页的字典里找“鼎”字，如果没有目录（全表扫描），你只能从第一页翻到最后一页，非常慢。如果有拼音目录（索引），你只需要查几页目录，就能直接定位到具体的页码。

在 MySQL 的 InnoDB 存储引擎中，这个“目录”底层采用的数据结构是 **B+ 树 (B+ Tree)**。

### 二、 为什么 MySQL 偏偏选中了 B+ 树？

面试官特别喜欢问：“为什么不用哈希表？为什么不用二叉树（红黑树）？为什么不用 B 树？”

我们可以通过对比来得出答案：

**1. 为什么不用 Hash 表？**

- Hash 表查找极快，时间复杂度是 $O(1)$。
    
- **致命缺点**：Hash 是无序的。在实际业务中，我们经常写 `WHERE age > 18` 这种**范围查询**。如果用 Hash，面对范围查询它直接瘫痪，只能去全表扫描。
    

**2. 为什么不用二叉树（或红黑树）？**

- 二叉树每个节点最多只有两个子节点。如果表里有 1000 万条数据，这棵树会变得**非常深（非常高）**。
    
- **致命缺点**：数据库的数据是存在磁盘上的，树里的每一个节点都对应一次磁盘 I/O（读写）。树太高，意味着查一次数据要进行几十次磁盘 I/O，速度极其缓慢。
    

**3. 为什么不用普通的 B 树，而要用 B+ 树？**

B 树和 B+ 树都是“矮胖型”的多路平衡查找树。它们的区别在于：

- **B 树**：树的所有节点（包括非叶子节点）都存储了真实的数据行。
    
- **B+ 树（MySQL 的选择）**：
    
    1. **非叶子节点只存索引键（Key）不存数据**。这就意味着一个 16KB 的磁盘页里，能塞下极其海量的索引键。这让整棵树变得**极度的“矮胖”**。通常 3 到 4 层的 B+ 树，就能存下千万级别的数据量（最多只要 3-4 次磁盘 I/O）。
        
    2. **所有真实的数据都放在最底层的叶子节点中**。
        
    3. **叶子节点之间形成了一个双向链表**。这个设计太绝了！当你要查 `age > 18` 的数据时，只要顺着树找到 18 这个节点，然后顺着底层的链表一路往后遍历拿数据就行了，极大地提高了范围查询的效率。
        

### 三、 真实表中的两种 B+ 树（极其重要）

在 InnoDB 引擎中，其实存在两种不同形态的 B+ 树，理解了它们，你就打通了 MySQL 索引的任督二脉：

**1. 聚簇索引（主键索引）**

- **特点**：**“索引即数据，数据即索引”**。
    
- **结构**：这棵 B+ 树的非叶子节点存的是主键 ID。而最底层的叶子节点里，**存的是整整一行完整的表数据**。
    
- **查找**：`SELECT * FROM user WHERE id = 10;` 只要顺着这棵树找到 id=10 的叶子节点，就把这行所有字段都拿出来了。
    

**2. 二级索引（非聚簇索引 / 普通索引）**

- **特点**：如果我们在 `name` 字段上建了一个索引，这就叫二级索引。
    
- **结构**：这棵 B+ 树的非叶子节点存的是 `name` 的值。最底层的叶子节点里，**只存了 `name` 的值和对应的主键 ID**，并没有存整行数据。

### 面试连招：什么是“回表”？

当我们执行 `SELECT * FROM user WHERE name = '张三';` 时，底层发生了什么？

1. 数据库先去 `name` 的二级索引 B+ 树里找，找到了 '张三' 所在的叶子节点。
    
2. 但是这个叶子节点里只有 name 和 主键 ID（假设 id=10），并没有其他的字段（比如 age, email）。
    
3. 于是，数据库拿到这个主键 ID=10，**回头再去主键的聚簇索引树里查一遍**，才能拿到完整的整行数据返回。
    

**这个“拿着 ID 再去主键索引树里查一遍”的过程，就叫作“回表”。** 咱们之前聊慢查询 `EXPLAIN` 的时候，如果走普通索引但依然很慢，往往就是因为回表的次数太多了！
