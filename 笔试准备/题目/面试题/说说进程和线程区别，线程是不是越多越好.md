这是一个非常经典的后端开发面试题。理解它们的区别以及如何合理使用线程，是构建高性能并发程序的基础。

### 一、 进程 (Process) 与线程 (Thread) 的核心区别

简单来说，**进程是操作系统进行“资源分配”的基本单位，而线程是“CPU调度和执行”的基本单位。** 我们可以把进程想象成一家公司，而线程是公司里的员工。

它们的主要区别体现在以下几个方面：

1. **资源拥有权与内存隔离：**
    
    - **进程：** 拥有独立的内存空间（堆、栈、代码段、数据段等）和系统资源（如文件描述符）。进程之间是相互隔离的，一个进程崩溃通常不会影响其他进程（系统稳定性高）。
        
    - **线程：** 线程是进程内的一个执行单元。同一进程内的多个线程**共享**该进程的内存资源（如堆和方法区），但每个线程有自己独立的运行栈（Stack）和程序计数器（PC）。由于共享内存，一个线程崩溃（如引发内存越界等未处理异常），往往会导致整个进程崩溃。
        
2. **切换与创建开销：**
    
    - **进程：** 创建、销毁和切换进程的开销非常大。因为操作系统需要为进程分配或回收独立的内存空间，并在上下文切换时保存和恢复大量的状态信息。
        
    - **线程：** 被称为“轻量级进程”。线程的创建、销毁和上下文切换的开销要小得多，因为它们不需要重新分配堆内存等大块资源，只需要保存和恢复少量的寄存器状态。
        
3. **通信方式：**
    
    - **进程间通信 (IPC)：** 比较复杂，需要借助操作系统的特定机制，如管道、消息队列、共享内存、Socket 等。
        
    - **线程间通信：** 非常方便，因为它们直接共享同一块内存空间，可以直接读写共享变量。但也正因如此，需要额外的同步机制（如锁）来保证数据一致性。

### 二、 线程是不是越多越好？

**绝对不是。** 虽然多线程可以提高系统的并发处理能力和 CPU 利用率，但如果盲目增加线程数量，不仅无法提升性能，反而会导致系统崩溃。主要原因有以下几点：

1. **上下文切换开销 (Context Switch Overhead)：** CPU 核心数量是有限的。当线程数量远超 CPU 核心数时，CPU 需要在不同线程之间频繁切换。每次切换都需要保存当前线程的状态并加载新线程的状态，这本身就是一个极其消耗 CPU 资源的操作。**如果线程过多，CPU 会把大量时间浪费在切换上，而不是执行真正的业务逻辑。**
    
2. **内存消耗殆尽 (OOM 风险)：** 虽然线程共享进程的堆内存，但每个线程在创建时都需要分配独立的虚拟机栈、本地方法栈和程序计数器（在 Java 中默认一个线程的栈大小通常是 1MB 左右）。如果无限制地创建线程，极易耗尽系统的内存，导致 `OutOfMemoryError`。
    
3. **激烈的资源竞争与死锁：** 线程越多，同时访问共享资源（如数据库连接、共享变量）的概率就越大。为了保证线程安全，必须大量使用锁（如 `synchronized` 或 `ReentrantLock`）。这会导致严重的锁竞争，增加线程阻塞和唤醒的开销，甚至引发死锁，让程序彻底卡死。

### 三、 实际开发中如何决定线程数量？

在后端开发中，我们通常不会手动频繁 `new Thread()`，而是使用**线程池 (Thread Pool)** 来复用和管理线程。确定合适的线程池大小，通常取决于任务的类型：

- **CPU 密集型任务（如复杂的加密、算法计算）：** 这类任务主要消耗 CPU 资源。线程数不宜过多，通常设置为 **CPU 核心数 + 1**。多出来的一个线程是为了在某个线程因为缺页中断或其他原因暂停时，保证 CPU 的利用率。
    
- **I/O 密集型任务（如读写数据库、网络请求、读写文件）：** 这类任务大部分时间都在等待 I/O 操作完成，CPU 大部分时间是空闲的。因此可以多配置一些线程，让 CPU 在等待期间去执行其他线程。常见的经验公式是：**线程数 = CPU 核心数 × 2**，或者更精确的公式：`线程数 = CPU 核心数 × (1 + I/O 等待时间 / CPU 计算时间)`。
