### 一、 案发现场：什么是缓存击穿？

如果说“缓存雪崩”是“大面积的盾牌同时碎裂”，那么**“缓存击穿”就是“某一块极其关键的防弹玻璃突然被子弹打穿了一个点”**。

想象一下这个场景：

在淘淘外卖平台，今天中午 12 点整，有一款“9.9元秒杀豪华烤肉套餐”。全网有 5 万个人都在疯狂点击这个套餐的详情页。

这个套餐的数据一直安稳地躺在 Redis 里，承受着极高的并发读请求。

但是，在 12:00:05 的那一毫秒，**这个爆款套餐的缓存刚好到期了（TTL 结束）**。

就在这一毫秒，有 1 万个并发请求同时到达。它们一查 Redis，发现数据没了（Cache Miss）。

因为没有任何阻拦，这 1 万个线程瞬间全部涌向底层 MySQL，去执行相同的 `SELECT` 语句试图重建缓存。

结果可想而知，数据库的某一张表或者某一行记录瞬间被这 1 万个连接锁死，甚至拖垮整个库。

### 二、 核心防御武器：互斥锁（Mutex Lock）

面对这种“单点爆破”，最经典的解法就是你提到的**利用 Redis 的 `setnx`（SET if Not eXists）指令来加锁**。

它的核心思想是：**排队，只放一个人进去查数据库，其他人等着。**

**具体的代码执行流程如下：**

1. **抢锁：** 1 万个线程发现缓存没命中，大家不去查数据库，而是第一时间去 Redis 里执行 `setnx lock:combo_99 1`。
    
2. **拿锁成功（天选之子）：** 因为 `setnx` 的原子性，1 万个线程里必定只有 1 个线程能执行成功（返回 1）。这个线程拿到了“令牌”，它被允许去查 MySQL，然后把查到的数据重新写回 Redis，最后释放这把锁（`del lock:combo_99`）。
    
3. **拿锁失败（群众演员）：** 剩下的 9999 个线程执行 `setnx` 失败（返回 0），说明已经有人去查数据库了。它们只好执行 `Thread.sleep(50)` 睡个 50 毫秒，醒来后**重新去查一次缓存**。大概率这时候那个“天选之子”已经把缓存重建好了，大家直接从缓存拿到数据，欢声笑语地返回。
    

> **⚠️ 致命陷阱（面试必考的“坑”）：**
> 
> 面试官一定会问：“如果那个拿到锁的线程，在查数据库的时候突然挂了（比如抛了异常，或者服务器停电），锁没被释放怎么办？”
> 
> **破局点：** 绝对不能只写 `setnx`。必须在加锁的同时，给这把锁**设置一个过期时间**。现在通常使用 `SET key value EX 10 NX` 这条原子命令，保证哪怕线程死了，锁在 10 秒后也会自动解开，防止死锁。

### 三、 高阶武器：逻辑过期（面试加分项）

除了加锁，业界还有一种“拿空间换体验”的高级玩法，特别适合那种**绝对不能卡顿、允许数据有短暂延迟**的页面。

- **做法：** 我们在把烤肉套餐存入 Redis 时，**不设置真正的物理过期时间（不设置 TTL）**，而是把过期时间作为一个普通的字段（比如 `expireTime: 1690000000`）存在 JSON 数据里。
    
- **效果：** 既然没有真实的 TTL，这个 Key 就永远不会在 Redis 里消失，绝对不会发生击穿。
    
- **运作：** 当线程查到数据时，会对比一下当前的系统时间。如果发现 `当前时间 > expireTime`，说明数据“逻辑上”过期了。此时，线程会**直接把这份旧数据返回给前端**（保证用户不卡顿），同时在后台偷偷开启一个**异步线程**去获取互斥锁、查询数据库并更新 Redis。

### 🗣️ 面试高分对答话术（直接背诵版）

面试官：“你能讲讲什么是缓存击穿，以及在项目中遇到热点数据是怎么处理的吗？”

**你可以这样层次递进地回答：**

> “缓存击穿是指，系统中**某一个被超高并发访问的热点 Key**，在突然过期的一瞬间，海量的并发请求因为没有命中缓存，同时去穿透查询底层的数据库，导致数据库由于局部压力过大而瘫痪。
> 
> 在处理诸如大促秒杀套餐这类热点数据时，我的首选防御方案是**使用互斥锁（Mutex Lock）**。
> 
> 具体来说，当缓存未命中时，我不会立刻去查数据库，而是利用 Redis 的 `SET ... EX ... NX` 命令去尝试获取一把分布式锁。只有成功拿到锁的那个线程，才被允许去查询 MySQL 并重建缓存。其他没拿到锁的线程则会短暂休眠后重试。这样就能确保无论并发多高，都只有一个请求打到数据库上。同时，利用 EX 参数给锁加上过期时间，也能完美避免因服务宕机导致的死锁问题。
> 
> 当然，如果是对用户体验要求极高，宁愿短暂展示旧数据也绝对不能阻塞请求的场景，我也会考虑采用**『逻辑过期』**方案。也就是不设置 Redis 键的物理过期，而是在 Value 中存放过期时间戳。一旦发现数据逻辑过期，先返回旧数据，再通过异步线程去更新缓存，以此来保证接口的极致响应速度。”

这段话术堪称完美：先定义问题，再给出标准解法（互斥锁），指出解法中的工程细节（防死锁），最后抛出高阶备用方案（逻辑过期）展示你的广度。
