这个问题简直是后端面试里的一道**“分水岭神题”**。只要简历里写了 Redis，面试官闭着眼睛都会问这道题。

它不仅考验你对缓存机制的理解，更是在试探你的**并发思维**。我们直接把你刚才提问里的那个“灾难现场”还原出来，你就能瞬间明白为什么业界有一个绝对的铁律。

### 一、 灾难推演：为什么绝对不能“先删缓存，再更新数据库”？

假设在你的外卖系统里，一份“豪华双人套餐”原来的价格是 50 元。现在商家觉得亏本，想涨价到 80 元。

如果你采用的代码逻辑是：**先删除 Redis 缓存，再去更新 MySQL。**

在高并发下，会发生极其可怕的**“并发读写交替”**：

1. **线程 A（商家更新操作）**：执行第一步，成功把 Redis 里的套餐缓存删除了。然后准备去更新 MySQL，但此时网络稍微抖动了一下，或者数据库有点卡，导致更新 MySQL 的操作卡顿了 50 毫秒。
    
2. **线程 B（大量吃货的查询操作）**：就在这致命的 50 毫秒内，一个用户打开了 APP 查看这个套餐。
    
3. **线程 B 查缓存**：发现 Redis 是空的（刚才被线程 A 删了）。
    
4. **线程 B 查数据库**：直接穿透到 MySQL，此时因为线程 A 还没把涨价的数据提交，所以线程 B 查到的**依然是旧价格 50 元**！
    
5. **线程 B 写回缓存**：线程 B 乐呵呵地把旧价格 50 元重新写回了 Redis。
    
6. **线程 A 终于完工**：过了几十毫秒，线程 A 终于把 MySQL 里的价格更新成了 80 元。
    

**💥 灾难后果：**

MySQL 里的真实价格已经是 80 元了，但 Redis 里刚才被线程 B 塞进去了一个旧的 50 元。在接下来的几个小时（直到缓存过期之前），所有的用户看到的都是 50 元，商家会被无休止地薅羊毛，系统数据遭遇**永久性的脏数据污染**。

---

### 二、 业界标准解法：Cache Aside Pattern（旁路缓存模式）

为了避免上面的灾难，业界 99% 的互联网公司（包括大厂）采用的标准策略是：

**👉 先更新数据库，然后再删除缓存。**

**为什么这种方案安全得多？**

同样的场景：商家把 50 涨到 80。

1. **线程 A（商家更新）**：先去操作 MySQL，把价格成功改成了 80 元。
    
2. **在这期间如果有查询**：其他用户查到的依然是 Redis 里的 50 元（旧数据）。这在业务上叫**“短暂的延迟”**，完全可以接受。
    
3. **线程 A（商家更新）**：MySQL 更新完后，立刻去删除 Redis 里的缓存。
    
4. **下一次查询**：新用户进来，发现 Redis 空了，去查 MySQL，拿到最新的 80 元，写回 Redis。完美同步。
    

> **💡 面试官的夺命小追问：“为什么是删除缓存，而不是更新缓存？”**
> 
> 答：“因为很多时候缓存里的数据不是简单从数据库查出来的，而是经过极其复杂的连表查询和计算得来的（比如关联了店铺优惠券、满减活动）。如果每次数据库有一点小改动就去重新计算一次缓存，极其浪费 CPU。直接删除缓存（Lazy Load 思想），让下一个真正需要用到这条数据的请求再去计算和加载，性能更好。”

---

### 三、 满分加薪项：如何追求极致的 100% 一致性？

其实“先更新数据库，再删缓存”在极其极端的情况下（比如删缓存那一步突然服务器断电失败了），依然会导致不一致。如果你在面试中能抛出下面这两个进阶方案，面试官会直接给你打出 S 级评价：

#### 进阶方案 1：延迟双删（低成本的万金油）

如果你非要“先删缓存”，或者担心主从数据库同步有延迟，可以使用这个策略：

1. 先删除 Redis 缓存。
    
2. 更新 MySQL 数据库。
    
3. **关键一步：** 线程休眠（Sleep）500 毫秒（具体时间看你业务的数据库读写耗时）。
    
4. **再删一次 Redis 缓存**。
    
    *目的是：*把那 500 毫秒内由于高并发乱入而写进 Redis 的脏数据，再清理掉一次。
    

#### 进阶方案 2：基于 Binlog + 消息队列的异步更新（大厂终极方案）

这是目前最优雅的解法。

- **做法：** 你的业务代码**根本不要去碰 Redis**，只管安心地修改 MySQL。
    
- **机制：** 在后台搭建一个中间件（通常用阿里的 **Canal**）。它会把自己伪装成 MySQL 的从库，实时监听 MySQL 的 `binlog`（数据变更日志）。一旦发现有数据被修改了，Canal 就把这条变更消息扔进 RabbitMQ 或 Kafka。
    
- **消费：** 专门写一个微服务去消费这个队列，稳稳当当地把 Redis 里对应的数据删掉。如果删失败了，利用消息队列的重试机制一直重试，直到成功。
    
- **优点：** 业务代码彻底解耦，再也不用管繁琐的一致性逻辑了。
