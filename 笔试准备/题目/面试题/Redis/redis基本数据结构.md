这个问题在面试中不仅考察你有没有背过 API，更是在考察你的**技术选型能力**。面试官想知道，当面对不同的业务需求时，你能不能挑出最省内存、效率最高的那把“武器”。

结合你做过的“淘淘外卖”项目，这简直就是你的主场。我们可以重点用外卖系统里的业务场景来举例，这会让你的回答非常有画面感。

Redis 最核心的基础数据结构有 5 种（常被称为“老五样”）。为你整理了高分对应的场景：

### 1. String（字符串）—— 最万能的打工仔

- **特点**：最基础的 Key-Value 结构。虽然叫 String，但也能存数字，甚至是一大坨 JSON。
    
- **外卖场景**：
    
    - **缓存简单的对象或字典**：比如缓存某个店铺的营业状态（0 休息，1 营业）。
        
    - **计数器**：利用 `INCR` 命令记录某个爆款套餐的“今日浏览量”。
        
    - **分布式锁**：就是咱们上一题刚讲过的防击穿利器 `setnx`。
        

### 2. Hash（哈希表）—— 购物车的神器

- **特点**：类似 Java 里的 `Map<String, Map<String, String>>`。它极其适合存储“对象”及其各个属性。
    
- **外卖场景（重点高亮区）**：**购物车**！
    
    - _传统做法（用 String）_：把整个购物车序列化成一个大的 JSON 存入 Redis。用户只要增加了一杯可乐，你都得把整个 JSON 取出来、反序列化、修改、再序列化塞回去，极其耗费 CPU 和带宽。
        
    - _高阶做法（用 Hash）_：`Key` 是用户的 ID，`Field` 是菜品 ID，`Value` 是购买数量。当用户增加一杯可乐时，只需执行 `HINCRBY cart:user_101 item_coke 1`，只改动了这一个字段，极其轻量和优雅。
        

### 3. List（列表）—— 排队与时间线

- **特点**：一个双向链表，支持从两端插入和弹出数据。
    
- **外卖场景**：
    
    - **最新的动态流**：比如展示外卖平台“最新入驻的 10 家店铺”，或者用户端展示“最新生成的 5 个订单”。
        
    - **轻量级消息队列**：利用 `LPUSH` 和 `BRPOP` 可以实现一个简单的阻塞队列（不过在真实的大型项目中，通常还是会交给专门的 MQ 来做）。
        

### 4. Set（集合）—— 去重与交并补

- **特点**：无序，且元素**绝对不能重复**。它天生支持求交集、并集和差集。
    
- **外卖场景**：
    
    - **用户标签去重**：给用户打上的偏好标签（如“重辣”、“免葱”），存入 Set 可以自动去重。
        
    - **抽奖系统**：外卖平台搞“点击参与免单抽奖”，把所有报名的用户 ID 塞进 Set 里，开奖时用 `SPOP` 命令随机弹出一个天选之子。
        

### 5. ZSet (Sorted Set，有序集合) —— 排行榜的绝对霸主

- **特点**：在 Set 的基础上，给每个元素绑定了一个浮点数分数（Score），Redis 会根据这个分数自动为你排序。
    
- **外卖场景（必答高光点）**：
    
    - **销量排行榜 / 热搜榜**：把套餐的 ID 作为元素，把它的月销量作为 Score。每卖出一份，就用 `ZINCRBY` 让分数加 1。获取销量前 10 名只需一句 `ZREVRANGE` 命令，极度丝滑。
        
    - _(面试加分提示：可以说出 ZSet 底层是用“跳表 SkipList”加“哈希表”实现的，查询效率极高。)_

### 🗣️ 面试高分对答话术（实战版）

当面试官问：“Redis 常见的数据结构有哪些？你平时是怎么用的？”

**你可以这样结合项目一气呵成：**

> “Redis 最核心的有五种基本数据结构：String、Hash、List、Set 和 ZSet。在我的外卖项目中，我主要根据不同的业务痛点选择了合适的结构。
> 
> 比如最基础的单值缓存、计数器以及实现防止缓存击穿的分布式锁，我使用的是 **String**。
> 
> 在实现**购物车**功能时，为了避免反复序列化庞大的 JSON 带来的性能损耗，我采用了 **Hash** 结构。以用户 ID 为 Key，菜品 ID 为 Field，数量为 Value，这样就可以极其细粒度地修改单个菜品的数量。
> 
> 对于平台首页的**热点套餐销量排行榜**，我使用了 **ZSet**。以销量作为 Score，Redis 底层的高效跳表机制能让我极其快速地实时取到 Top N 的榜单数据。
> 
> 至于 List 和 Set，我通常会用它们来处理一些简单的最新事件列表，或者做数据的快速去重。”

这样回答，不仅列出了知识点，而且每个点都稳稳地砸在了实际的业务痛点上，完美展现了你的实战能力。

在这个话题下，面试官经常会紧接着抛出一个探底的底层问题：**“既然 Redis 是单线程处理这些数据结构的，那它为什么还能这么快？”**（这关乎到 IO 多路复用和纯内存操作）。需要我帮你把这个底层原理用大白话拆解一下吗？
