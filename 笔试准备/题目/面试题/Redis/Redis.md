太棒了，恭喜你推进到 Redis 模块！

对于国企和银行的校招，Redis 的考察重点非常明确：**不追求花哨的超大规模集群架构，而是死磕“缓存的稳定性”和“数据一致性”**。因为金融系统和大型国企后台最怕的就是缓存突然挂掉，导致海量流量直接把底层的关系型数据库（MySQL）打死。

特别是你在做外卖业务时，用到了 Redis 来做高并发的支撑，面试官简直有一万种方式把底层的理论和你的业务结合起来。

以下为你整理了国企/银行校招 Redis 核心必考题单，按重要程度分为四大防线：

### 一、 缓存三大噩梦（绝对核心，必考必问）

这是校招 Redis 面试中出场率高达 99% 的考点，必须要能流利地区分它们，并给出兜底方案。

- **什么是缓存穿透？怎么解决？** （比如黑客疯狂请求一个外卖系统里根本不存在的套餐 ID，怎么用布隆过滤器或者缓存空对象来拦截？）
    
- **什么是缓存雪崩？怎么解决？** （大量外卖套餐的缓存在同一秒钟集体过期，导致数据库瞬间压力飙升，怎么通过把过期时间打散加随机值来预防？）
    
- **什么是缓存击穿？怎么解决？** （一个超级爆款套餐的缓存在过期的一瞬间，涌入了上万个并发请求，怎么用互斥锁 `setnx` 保证只有一个请求去查数据库？）
    

### 二、 数据库与缓存一致性（极度考验业务逻辑）

这类问题直击系统的痛点，关乎业务数据的准确性。

- **如何保证 Redis 和 MySQL 的数据一致性？** （比如商家在后台修改了套餐的价格，系统是该先更新数据库还是先删缓存？如果先删缓存，在高并发下会有什么灾难性后果？）
    
- **听说过“延迟双删”策略吗？为什么要延迟那几百毫秒？**
    
- **如果保证极端的强一致性？** （引入分布式锁，或者监听 MySQL 的 Binlog 异步更新缓存）。
    

### 三、 基础数据结构与应用场景（考察编码基本功）

面试官不会干巴巴地问你“有哪些数据类型”，而是会给你一个场景让你选型。

- **Redis 常见的数据结构有哪些？分别对应的使用场景是什么？** （重点想好：外卖系统里的“购物车”功能，你是用 `String` 存的一坨 JSON，还是用 `Hash` 结构存的？为什么？）
    
- **如果要做一个“销量排行榜”或者“热搜榜”，应该用哪种数据结构？** （必答 `ZSet` 有序集合，并能说出底层的跳表机制最好）。
    
- **Redis 为什么这么快？** （除了纯内存操作，一定要答出“单线程避免上下文切换”和“IO 多路复用机制”）。
    

### 四、 持久化与内存淘汰（底层运维意识）

银行非常看重数据安全，即便缓存宕机了，数据也不能全丢。

- **Redis 的持久化机制 RDB 和 AOF 有什么区别？** （RDB 是拍快照恢复快但容易丢最后几分钟数据，AOF 是记日志极其安全但恢复慢，企业里通常怎么组合使用？）
    
- **Redis 内存满了怎么办？有哪些内存淘汰策略？** （这里极大概率会和你之前复习的操作系统联动，让你详细解释 `allkeys-lru` 也就是 LRU 淘汰算法的具体表现）。

这四大块构成了校招 Redis 的绝对防线。如果你能把外卖项目中的**购物车缓存**和**套餐状态缓存**套用到这些题目里去讲，面试官会非常满意。
