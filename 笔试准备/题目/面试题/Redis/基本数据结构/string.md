### 场景一：最基础的缓存（单值缓存）

这是最常见的用法，比如缓存一个套餐的基本信息或者用户的 Token。

- **SET** (写入数据)
    
    Code snippet
    
    ```
    SET combo:1001 "烤肉套餐"
    ```
    
    _系统会返回：`OK`_
    
- **GET** (读取数据)
    
    Code snippet
    
    ```
    GET combo:1001
    ```
    
    _系统会返回：`"烤肉套餐"`_
    
- **DEL** (删除数据)
    
    Code snippet
    
    ```
    DEL combo:1001
    ```
    
    _系统会返回：`(integer) 1` (表示成功删除了1个键)_

### 场景二：超高并发计数器（极速点赞/浏览量）

如果用 MySQL 记录爆款套餐的浏览量，频繁的 `UPDATE` 会把数据库锁死。Redis 的 String 内置了对数字的原子化操作，极其高效且线程安全。

- **INCR** (每次加 1)
    
    Code snippet
    
    ```
    INCR views:combo:1001
    ```
    
    _系统会返回：`(integer) 1`。再执行一次会返回 `(integer) 2`。即便这个 key 之前不存在，Redis 也会自动把它当成 0 然后加 1。_
    
- **INCRBY** (一次加指定的数值)
    
    Code snippet
    
    ```
    INCRBY views:combo:1001 10
    ```
    
    _系统会返回：`(integer) 12`。比如一次性增加了 10 个浏览量。_
    
- **DECR** (每次减 1，常用于扣减库存的初步拦截)
    
    Code snippet
    
    ```
    DECR stock:combo:1001
    ```

### 场景三：防击穿的终极武器（分布式锁）

这就是我们上一题对付“缓存击穿”的绝对主力！它的核心机制是：**只有当这个 Key 不存在时，才能设置成功。**

- **SETNX** (老派分布式锁用法：SET if Not eXists)
    
    Code snippet
    
    ```
    SETNX lock:combo:1001 "thread_1"
    ```
    
    _如果此时没人查数据库，你会抢锁成功，返回：`(integer) 1`。_
    
    _如果紧接着另一个请求也执行这句命令，它会抢锁失败，返回：`(integer) 0`。_
    
- **SET ... EX ... NX** (🔥 **现代企业级标准用法，面试必说**)
    
    单纯的 `SETNX` 没法同时设置过期时间（非原子操作，中途宕机会死锁）。现在的大厂都要求用 `SET` 命令的复合参数：
    
    Code snippet
    
    ```
    SET lock:combo:1001 "thread_1" EX 10 NX
    ```
    
    _这句话的意思是：**如果**这个锁不存在 (NX)，**就**把它设置为 "thread_1"，**并且**给它 10 秒的自动过期时间 (EX 10)。这三个动作在 Redis 底层是一气呵成的绝对安全操作！_
