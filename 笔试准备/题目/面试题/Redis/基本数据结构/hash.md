安排！Hash（哈希表）可以说是 Redis 里处理**“对象”**的最强武器。

在前面的面试题里我们提到过：如果用普通的 `String` 来存购物车，你得把整个购物车的 JSON 拿出来、反序列化、修改、再全部序列化塞回去，这在高并发下极其浪费 CPU 和网络带宽。

但用了 `Hash`，一切都变得极其优雅！你可以把 `Hash` 想象成一个**大文件夹（Key）**，里面装了一堆**小文件（Field 和 Value）**。

咱们直接代入“淘淘外卖”的**购物车**场景，来体验下面这 5 个最常用的 Hash 核心命令：

### 场景还原：外卖购物车大作战

假设当前用户的 ID 是 `1001`，我们要操作他的购物车（Key 就叫 `cart:user_1001`）。

**1. 添加商品到购物车 (HSET)**

用户点了一份“烤肉套餐”（商品ID: `combo_99`，数量: `1`）和一杯“可乐”（商品ID: `item_coke`，数量: `2`）。

Code snippet

```
HSET cart:user_1001 combo_99 1 item_coke 2
```

_返回：`(integer) 2`（表示成功往这个 Hash 里塞了 2 个新字段）。_

**2. “再来一杯可乐！” (HINCRBY - 核心杀手锏)**

用户在页面上点击了可乐旁边的 `+` 号。此时后端**不需要**把整个购物车查出来，只需要精准打击“可乐”这个字段，给它加 1：

Code snippet

```
HINCRBY cart:user_1001 item_coke 1
```

_返回：`(integer) 3`（可乐的数量瞬间变成了 3）。这就是 Hash 性能爆表的原因：O(1) 复杂度，极省带宽。_

**3. 查看某件商品买了多少 (HGET)**

前端只针对性地查询可乐的数量，用来在页面上显示数字：

Code snippet

```
HGET cart:user_1001 item_coke
```

_返回：`"3"`_

**4. 去结算！获取购物车全部清单 (HGETALL)**

用户点击了“去结算”，我们需要把购物车里所有的菜品和对应的数量全都拿出来给前端展示：

Code snippet

```
HGETALL cart:user_1001
```

_系统会一行行地返回所有的商品 ID 和对应的数量。_

**5. 移除商品 (HDEL)**

用户觉得可乐太长胖了，点击了删除按钮，把可乐从购物车移出：

Code snippet

```
HDEL cart:user_1001 item_coke
```

_返回：`(integer) 1`。再用 HGETALL 查的时候，可乐就不见了。_
