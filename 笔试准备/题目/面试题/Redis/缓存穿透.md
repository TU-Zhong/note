这个问题是 Redis 面试里绝对的**“C位考点”**。你能主动结合“淘淘外卖”的业务场景来思考，说明你已经具备了高级开发的实战思维。

面试官问这个问题，本质上是在考验你**“面对恶意攻击时的系统防御能力”**。

我们把这个问题拆解成“案发现场还原”、“两大防御武器”以及“面试高分话术”三个部分，帮你把这个知识点吃透。

### 一、 案发现场：什么是缓存穿透？

正常情况下，用户的请求流程是：**查 Redis 缓存 ➡️ 缓存没有 ➡️ 查 MySQL 数据库 ➡️ 写入 Redis ➡️ 返回数据。**

**缓存穿透（Cache Penetration）**的案发现场是这样的： 假设你的外卖系统里，正常的套餐 ID 都是正整数（比如 1001, 1002）。 黑客写了一个脚本，疯狂地并发请求一个**根本不存在的套餐 ID**（比如 `id = -1` 或者 `id = 99999999`）。

1. 请求打到 Redis，Redis 里肯定没有这个 ID，缓存未命中（Cache Miss）。
    
2. 请求穿透 Redis，直接打到 MySQL。
    
3. MySQL 查了一圈，发现也没有这个 ID，返回 `null`。
    
4. **致命弱点出现了：** 因为数据库返回的是空，你的代码可能就不会把这个空结果写进 Redis。
    
5. 下一秒，黑客再次请求 `id = -1`，由于 Redis 还是没有，请求又一次结结实实地砸在 MySQL 上。
    

**结果：** 海量的恶意请求像利剑一样，瞬间穿透了 Redis 这层防弹衣，直接把底层的 MySQL 数据库打到宕机崩溃。

---

### 二、 两大防御武器（怎么解决？）

针对这种情况，业界有两套标准打法，各有优劣：

#### 武器一：缓存空对象（简单粗暴，适合应急）

- **怎么做：** 既然数据库查不到 `id = -1` 的套餐，那我就在 Redis 里强制存一个 `key = "combo:-1"`, `value = "null"`（或者特定的空 JSON `{}`），并给它设置一个很短的过期时间（比如 60 秒）。
    
- **效果：** 黑客再次请求 `id = -1` 时，Redis 就会直接返回那个假装的 `"null"`，请求就不会再打到数据库了。
    
- **优点：** 代码实现极其简单，防守见效快。
    
- **缺点：** 1. **浪费内存**：如果黑客用千万级不同的假 ID 攻击，Redis 会存入海量的无用空键。
    
    2. **短期数据不一致**：假如你刚把 `combo:999` 缓存为空，运营人员下一秒刚好上架了 ID 为 999 的真实套餐，用户在这 60 秒内依然会看到“套餐不存在”。
    

#### 武器二：布隆过滤器 (Bloom Filter) （高端局必备，面试加分项）

- **怎么做：** 布隆过滤器就像是 Redis 门前的一个**超级保安**。系统启动时，把你数据库里所有合法的套餐 ID 全都通过 Hash 算法映射到这个保安的“记忆（二进制位图）”里。
    
- **效果：** 当黑客请求 `id = -1` 时，请求先经过保安（布隆过滤器）。保安一查自己的记忆，发现绝对没有这个 ID，**直接在前端拦截丢弃**，连 Redis 都不用进，更别提 MySQL 了。
    
- **优点：** 极其节省内存（只存 0 和 1 的二进制位），拦截效率极高，是从源头掐断攻击。
    
- **缺点：**
    
    1. **存在一定的误判率（假阳性）**：布隆过滤器说某个 ID“不存在”，那它**绝对不存在**；但如果它说某个 ID“存在”，那它**可能不存在**（因为不同 ID 经过 Hash 可能会撞车映射到同一个位置）。
        
    2. **删除困难**：套餐下架了，很难从布隆过滤器里把它的记录单独抹掉。

### 🗣️ 面试高分对答话术（直接背诵版）

当面试官问出：“什么是缓存穿透？你在项目中怎么解决的？”

**你可以这样层次分明地回答：**

> “缓存穿透是指，客户端疯狂查询一个**在缓存和数据库中都不存在的数据**。因为数据库查不到就不会写入缓存，导致这类恶意请求每次都会直接穿透缓存去查数据库，如果有海量并发，极易导致数据库宕机。
> 
> 在我的外卖项目中，如果遇到黑客恶意伪造不存在的套餐 ID 发起攻击，我通常会考虑两种方案：
> 
> **第一种是缓存空对象。** 当从数据库查不到结果时，我依然会把这个空结果以及对应的假 ID 存入 Redis，并设置一个较短的过期时间（比如几分钟）。这样后续相同的恶意请求就会被 Redis 挡住。这种做法代码最简单，但缺点是如果遇到海量不同的假 ID，会浪费一定的 Redis 内存。
> 
> **第二种是引入布隆过滤器。** 可以在请求到达 Redis 之前加一层拦截。将所有合法的套餐 ID 提前加载到布隆过滤器中。请求过来时先判断，如果过滤器判断该 ID 绝对不存在，就直接返回拦截。这种方案极其节省内存，能从源头掐断恶意流量。虽然存在极小的误判率，但在防御恶意穿透的场景下是完全可以接受的。
> 
> 在实际的工程落地中，如果是访问量不是特别大的中小型项目，我倾向于直接用缓存空对象；如果是真正的高并发场景，我会采用布隆过滤器作为前置防御屏障。”

这段话术既描述了你在做外卖系统时的具体思考，又清晰地对比了两种方案的优缺点，还给出了你的“技术选型权衡”，面试官想不给你高分都难。
