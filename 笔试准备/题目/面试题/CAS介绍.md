在 Java 并发编程中，**CAS（Compare-And-Swap，比较并交换）** 是一种实现无锁（Lock-Free）同步的核心技术。它基于“乐观锁”的思想，允许在多线程环境下安全、高效地更新共享变量，而无需像 `synchronized` 或 `ReentrantLock` 那样阻塞线程。

Java 中 `java.util.concurrent.atomic` 包下的所有原子类（例如 `AtomicInteger`、`AtomicReference`）底层全部都是依赖 CAS 实现的。

---

### CAS 的核心原理

CAS 的操作包含三个核心参数：

1. **V (Value)**：内存地址中该变量的当前实际值。
    
2. **A (Expected)**：当前线程读取到的预期原值。
    
3. **B (New)**：当前线程想要更新进去的新值。
    

**执行逻辑：**

当且仅当内存位置的值（**V**）等于预期原值（**A**）时，才会将内存位置的值更新为新值（**B**）。

如果 **V** 不等于 **A**，说明在当前线程准备修改变量的这段时间里，有其他线程已经抢先修改了该变量。此时，CAS 操作失败，当前线程不会更新任何值，通常会选择重新读取内存中的最新值并再次尝试（这个不断重试的过程称为**自旋（Spinning）**），直到修改成功为止。

### Java 底层的实现支持

CAS 并不是在 Java 层面通过代码逻辑保证原子性的，而是直接依赖于**底层硬件的原子指令**。

- **Unsafe 类**：在 Java 中，CAS 操作主要是通过 `sun.misc.Unsafe` 这个类中的 native 方法（如 `compareAndSwapInt`）来调用的。
    
- **硬件级别支持**：这些 native 方法最终会映射到处理器底层的原子指令（例如 x86 架构下的 `lock cmpxchg` 指令）。因为是由 CPU 硬件在指令级别保证的原子性，所以在执行过程中绝不会被其他线程中断，从而保证了线程安全。
    

---

### CAS 的优缺点

**优点：**

- **性能高，无阻塞**：由于不涉及线程的挂起、上下文切换和恢复，在并发冲突不激烈的场景下，CAS 的性能远高于传统的悲观锁（如 `synchronized`）。
    

**缺点：**

1. **ABA 问题**：如果内存值原来是 A，被其他线程改成了 B，然后又被改回了 A。此时当前线程执行 CAS 时会发现值仍然是 A，从而判定为未被修改过并执行成功，但这中间其实发生了状态变化。
    
2. **CPU 开销大（自旋等待）**：如果在高并发、高冲突的环境下，多个线程都在抢占修改同一个变量，失败的线程会一直处于 `while` 循环的自旋状态，白白消耗大量的 CPU 资源。
    
3. **只能保证一个共享变量的原子操作**：标准的 CAS 只能对单一的变量生效。如果需要保证多个变量的原子性操作，传统的做法只能改用锁机制。
