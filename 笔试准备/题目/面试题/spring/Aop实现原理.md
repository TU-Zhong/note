这个问题绝对是面试里区分“背八股文选手”和“实战选手”的分水岭。

Spring AOP 的底层核心原理只有四个字：**动态代理（Dynamic Proxy）**。

为了在面试中拿高分，你可以用**“一个核心概念 + 两种实现方式 + 一个版本细节”**的逻辑来征服面试官。下面为你拆解这套高分话术：

### 第一步：用通俗的比喻解释“动态代理”

在抛出技术名词前，先用大白话定个基调，这会让面试官觉得你真的懂了。

> “Spring AOP 的底层是基于动态代理模式实现的。打个比方，目标类就像是**明星**，代理对象就像是**经纪人**。外界（业务代码）想找明星演戏，不能直接联系明星，必须经过经纪人。经纪人可以在演戏前谈好片酬（前置通知 `Before`），在演戏后安排车接送（后置通知 `After`），而明星只需要专注于演戏（核心业务逻辑）本身。”

### 第二步：深入两种核心底层机制（重点拷打区）

Spring 内部其实准备了两套“经纪人”方案，它会根据目标类的不同情况，智能切换：

**1. JDK 动态代理（Java 原生派）**

- **触发条件**：如果你的目标类**实现了一个或多个接口**（比如 `UserServiceImpl` 实现了 `UserService` 接口），Spring 就会默认使用 JDK 动态代理。
    
- **底层原理**：利用 Java 的反射机制（`java.lang.reflect.Proxy`）。它会在运行的时候，在内存中动态拼凑出一个“同父异母”的代理类，这个代理类和目标类实现了**相同的接口**。
    
- **局限性**：死板。目标类如果连接口都没有，JDK 动态代理就直接罢工了。
    

**2. CGLIB 动态代理（第三方硬核派）**

- **触发条件**：如果你的目标类**没有任何接口**，就是一个普普通通的独立类，Spring 就会退而求其次，调用 CGLIB 库来实现代理。
    
- **底层原理**：利用 ASM 字节码生成框架。它会在运行的时候，直接在内存中动态生成一个目标类的**子类**（继承关系），并在子类中重写父类的方法，从而实现拦截。
    
- **局限性**：因为是基于继承重写的原理，所以如果目标类被 `final` 修饰，或者里面的方法被 `final`、`private` 修饰，CGLIB 就无法继承和重写，AOP 就会失效。
    

### 第三步：秀出版本细节（绝杀加分项）

讲完上面两点已经能拿 80 分了，如果你再补充下面这个 Spring Boot 的版本变化，面试官会觉得你对底层了如指掌：

> “需要特别注意的是，虽然老版本的 Spring 优先使用 JDK 代理，但在 **Spring Boot 2.0 之后**，官方更改了默认策略。现在无论目标类有没有实现接口，Spring Boot 都会**默认强制使用 CGLIB 代理**（即 `spring.aop.proxy-target-class=true`）。官方这么做是为了防止开发者在依赖注入时，因为按类注入（`@Autowired UserServiceImpl`）而不是按接口注入而引发大面积的类型转换异常（ClassCastException）。”

---

### 💡 结合实战怎么聊？

面试官听完原理，通常会顺口问一句：“你在项目里是怎么用 AOP 的？”

这时候你可以自然地带入一些复杂的业务场景。比如在开发类似 Navicat 的 Web 端数据库管理后台时，前端 Vue 3 会频繁发请求让后端执行 SQL 查询或修改表结构。为了系统的安全性，你不可能在每一个 Controller 接口里都写一遍 JWT 鉴权和操作日志记录。

这时候，你就可以说：“**我会自定义一个 `@LogAction` 注解，然后利用 AOP 写一个切面类去拦截带有这个注解的方法。在请求执行前校验权限，在执行后自动把操作的表名、耗时记录到日志库里，完美做到了核心数据请求与安全监控的解耦。**”

原理和实战都串起来了，这个知识点基本就稳了。
