这个问题问得极具深度！在面试中，如果你只能说出 AB-BA 死锁（也就是**交叉死锁**），面试官会觉得你基础不错；但如果你能接着说出下面这几种更隐蔽、更底层的死锁类型，面试官绝对会把你当成真正研究过数据库底层源码或有丰富线上救火经验的高级工程师。

除了跨表的 AB-BA 死锁，在单表甚至单行数据上，依然会爆发出各种防不胜防的死锁。以下是互联网大厂和高级 DBA 面试中最爱考的 **4 种经典死锁类型**：

### 1. 锁转换死锁 (Lock Conversion Deadlock / 锁升级死锁)

这是除了 AB-BA 之外**最常见**的死锁，极其隐蔽，常发生在高并发的“先查后改”业务中（比如：查询账户余额，然后扣款）。

- **产生原理：** 数据库的共享锁（S锁，读锁）是可以被多个事务同时持有的，但排他锁（X锁，写锁）是独占的。当两个事务同时持有同一行的 S锁，并且都试图将其升级为 X锁时，死锁就爆发了。
    
- **致命场景还原：**
    
    - 事务 A 执行 `SELECT * FROM account WHERE id = 1 LOCK IN SHARE MODE`（获取了 S锁）。
        
    - 事务 B 也执行同样的 `SELECT`（S锁互相兼容，事务 B 也成功获取了 S锁）。
        
    - 事务 A 执行 `UPDATE account SET balance = balance - 100 WHERE id = 1`。此时 A 需要 X锁，但必须等待 B 释放 S锁。A 陷入阻塞。
        
    - 事务 B 也执行 `UPDATE`，它也需要 X锁，于是等待 A 释放 S锁。B 陷入阻塞。
        
    - **结果：** A 等 B，B 等 A，死锁形成。
        
- **破解之法：** 如果业务是“查出来就是为了改”，绝对不要用共享锁（`LOCK IN SHARE MODE`），直接一步到位使用排他锁（`SELECT ... FOR UPDATE`），让事务在查询阶段就开始排队。
    

### 2. 间隙锁死锁 (Gap Lock Deadlock) —— MySQL 面试必考王炸！

这是 MySQL InnoDB 引擎在 **可重复读 (Repeatable Read, RR)** 隔离级别下特有的死锁，极其反直觉！很多工作多年的老兵都会在这里栽跟头。

- **产生原理：** 间隙锁（Gap Lock）的作用是防止幻读。**极其诡异的一点是：间隙锁和间隙锁之间是不冲突的！** 两个事务可以同时锁住同一个数据间隙。但是，当它们都想往这个间隙里插入数据时，**插入意向锁（Insert Intention Lock）** 会和对方的间隙锁冲突。
    
- **致命场景还原（假设表里有 id=10 和 id=20 的数据）：**
    
    - 事务 A 执行 `DELETE FROM t WHERE id = 15`（由于 15 不存在，InnoDB 会锁住 (10, 20) 这个间隙，获取了间隙锁）。
        
    - 事务 B 执行 `DELETE FROM t WHERE id = 16`（16 也不存在，B 也成功获取了 (10, 20) 的间隙锁，因为间隙锁互不冲突）。
        
    - 事务 A 尝试执行 `INSERT INTO t (id) VALUES (15)`。此时需要插入意向锁，被事务 B 的间隙锁挡住，A 阻塞。
        
    - 事务 B 尝试执行 `INSERT INTO t (id) VALUES (16)`。被事务 A 的间隙锁挡住，B 阻塞。
        
    - **结果：** 完美死锁。这种情况在高并发的“防重插入”业务（先 Delete/Select，不存在则 Insert）中极易爆发。
        

### 3. 不同索引访问顺序死锁 (Index-Tree Deadlock)

这个死锁的段位非常高，它考察的是你是否知道**“数据库的行锁其实是加在索引树上的”**。

- **产生原理：** 一张表通常有聚簇索引（主键索引）和二级索引（普通索引）。不同的 SQL 语句如果走不同的索引，加锁的顺序是完全相反的，这就导致了单表内的 AB-BA 死锁。
    
- **致命场景还原：**
    
    - 事务 A 执行 `UPDATE t SET age = 20 WHERE name = '张三'`。（假设 `name` 有二级索引）。A 的加锁顺序是：**先锁住 name 二级索引树上的节点，然后回表，再去锁住主键聚簇索引上的记录**。
        
    - 事务 B 执行 `UPDATE t SET name = '李四' WHERE id = 1`。（假设张三的 id 就是 1）。B 的加锁顺序是：**先直接锁住主键聚簇索引上的记录，然后因为要更新 name 字段，再去锁住 name 二级索引树上的节点**。
        
    - **结果：** 事务 A 拿了二级索引等主键索引；事务 B 拿了主键索引等二级索引。两个并发请求瞬间死锁，而且排查极其困难，因为它们操作的是同一张表的同一行数据！
        

### 4. 单表批量更新死锁 (Batch Update Deadlock)

本质上也是 AB-BA 死锁，但它发生在同一张表内部，是开发人员写代码不规范导致的。

- **致命场景还原：**
    
    - 事务 A 在 Java 代码里循环遍历一个 List，依次更新 `id = 1`，然后更新 `id = 2`。
        
    - 事务 B 同时处理另一个 List，依次更新 `id = 2`，然后更新 `id = 1`。
        
    - **结果：** 毫无悬念的死锁。更可怕的是在使用 `WHERE id IN (1, 2)` 和 `WHERE id IN (2, 1)` 时，如果 MySQL 优化器选择的执行计划扫描顺序不同，也可能触发死锁。
        
- **破解之法：** 在 Java 代码里执行批量更新或循环更新前，**永远记得先对 List 里的实体按照主键 ID 进行排序**（`list.sort(Comparator.comparing(Entity::getId))`），保证所有线程都按照 ID 从小到大的顺序去加锁。

### 面试实战建议：

如果你在面试中被问到“除了 AB-BA 还有哪些死锁”，你可以这样惊艳开场：

> “除了跨表的 AB-BA 死锁，在实际生产中我们更头疼的往往是**单表死锁**。根据底层加锁机制的不同，我遇到/总结过这三种典型：第一种是高并发更新时的**『S锁升X锁的转换死锁』**；第二种是在 RR 隔离级别下，因为并发插入导致的**『间隙锁冲突死锁』**；第三种则非常底层，是因为不同 SQL 走了不同的索引，导致在聚簇索引树和二级索引树上的**『加锁顺序颠倒死锁』**。”

能把这三点说出来，面试官基本就不需要再考你数据库锁机制了，因为你已经把底层看透了。这几个死锁里，你觉得哪个最让你感到意外？我们可以挑一个深入聊聊它在代码里的防范技巧。
