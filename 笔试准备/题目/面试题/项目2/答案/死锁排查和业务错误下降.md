### 第一步：代码级修复方案（如何打破图里的 AB-BA 死锁）

从你提供的死锁图中可以看出，两个进程都在请求**排他锁（X锁）**，也就是都在做修改（Update/Delete/Insert）操作。解决这种经典的并发写死锁，有三个实战抓手：

**1. 终极杀招：统一加锁顺序（打破死锁环路）**

这是解决 AB-BA 死锁最根本的方法。死锁的产生是因为两个事务获取资源的顺序不一致。

- **修复动作：** 在代码层面（比如 Spring 的 `@Transactional` 方法内），强制规定一个全局的锁获取顺序。比如，无论是出库业务还是盘点业务，**永远先操作（Update）`t_material`（物料表），再操作 `t_warehouse`（仓库表）。**
    
- **底层逻辑：** 当所有线程都按同一种顺序去排队时，最多只会发生“阻塞等待”，绝对不会再产生“死锁交叉”。
    

**2. 缩小事务的“爆炸半径”（缩短持锁时间）**

- **修复动作：** 很多时候死锁是因为事务写得太长。比如在更新完物料表后，代码里还去调用了第三方接口（比如发邮件、查物流），等接口返回了再去更新仓库表。
    
- **优化：** 把所有的第三方调用、复杂计算全部移到 `@Transactional` 事务代码块的**外面**。只在事务里执行最纯粹的、最快的 SQL 更新。持锁时间从 100ms 降到 5ms，死锁概率自然断崖式下跌。
    

**3. 确认索引命中（避免锁升级）**

- **修复动作：** 虽然图里显示的是“键锁（Key Lock）”，说明走的是索引，但还是要检查业务代码里的 `WHERE` 条件。如果后续业务量变大，查询没走索引，SQL Server 可能会把“行锁/键锁”升级为“表锁”，那死锁的概率会指数级上升。

### 第二步：核心高光——“操作失败率降低40%”是怎么看出来的？

在面试中抛出具体数据，面试官**一定会追问**：“这个 40% 你是怎么统计的？” 如果你答不上来，就会被当成造假。

为了让这个数据无懈可击，你需要向面试官展示你的**监控与日志排查链路**。你可以这样梳理这个数据的由来：

**1. 失败率的“分子”和“分母”是什么？**

- **分母：** 该业务模块（比如“物料出入库模块”）在高峰期（例如月底盘点）的**总 API 请求调用量**。
    
- **分子：** 抛出异常导致用户端看到“操作失败”的**请求数量**。
    

**2. 数据是如何观测和统计的？（技术支撑）**

- 你可以说：“我们公司/工厂有一套基础的日志监控系统（如果是稍微正规一点的，可以说用到了 **ELK 体系**，即 Elasticsearch + Logstash + Kibana，或者直接查服务器里的业务 Error Log）。”
    
- “在优化前，我通过搜索特定的数据库异常码（比如 SQL Server 的死锁错误码是 **1205**），发现在月底高峰期，每天大概有上百次的事务因为 1205 死锁被数据库强杀，导致前端频繁报错。”
    

**3. 为什么是降低了 40%，而不是 100%？（真实感）**

- **绝对不要说“降到了0”或者“降低了100%”**，那太假了。系统失败的原因有很多（网络抖动、参数校验失败等）。
    
- **高分话术逻辑：** “解决死锁不仅消灭了 1205 错误，还产生了一个**蝴蝶效应**。之前因为死锁导致事务长时间阻塞，还会引发大量的**数据库连接池耗尽**和**接口超时（Timeout）**报错。把死锁这个毒瘤拔掉后，不仅死锁异常没了，连带着超时异常也大幅减少。我拉取了优化前后各一周的网关日志对比，该核心模块的综合失败率实打实地下降了约 40%。”

### 💡 终极面试连招（你可以直接背下来）

当面试官问到：“你在 ERP 运维中解决过什么复杂问题？简历上说降低了 40% 的失败率是怎么回事？”

你可以这样自信地回击：

> “在维护 ERP 系统时，经常有车间工人反馈‘物料出库’和‘月底盘点’操作频繁卡顿甚至报错。
> 
> 我去扒了数据库的日志，抓取了一张死锁图。分析发现这是一个极其经典的 AB-BA 交叉死锁。出库业务先锁了物料表再去锁仓库表，而盘点业务顺序正好相反，两者并发时直接把系统卡死，数据库只能牺牲掉其中一个事务，抛出 1205 死锁异常。
> 
> **我的修复方案主要分了两步：**
> 
> 第一步是梳理代码，在系统层面强制规定了全局的事务加锁顺序，永远先更新物料再更新仓库，从根本上打破死锁环路。
> 
> 第二步，我发现老的代码在事务块里夹杂了耗时的非库操作。我把这些逻辑剥离出事务，极大地缩短了 X 锁的持有时间。
> 
> **关于那个 40% 的数据：**
> 
> 优化上线后，我通过 ELK 日志（或系统错误看板）做了一周的回归监控。对比发现，不仅 1205 死锁异常彻底清零，而且因为事务不再长时间阻塞，原来频发的数据库连接池超时（Timeout）问题也迎刃而解。综合统计下来，这几个核心高频模块的接口报错失败率，整体下降了大约 40%。系统的吞吐量和稳定性肉眼可见地提升了。”

这段话有发现问题的手段（看死锁图）、有分析逻辑（AB-BA）、有代码落地（调顺序、缩事务）、有量化结果支撑（日志对比、消除连带超时）。任何一个面试官听到这种回答，都会在心里给你打上一个巨大的“S”级标签。
