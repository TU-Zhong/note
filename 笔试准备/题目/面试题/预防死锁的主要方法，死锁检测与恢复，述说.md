在后端开发的笔试和面试中，尤其是在银行、国企等非常看重计算机基础功底的单位，死锁问题几乎是必考的核心考点。它不仅涉及操作系统的底层理论，还直接关系到 Java 并发编程和数据库事务的实际稳定性。

要对付死锁，系统层面通常有三种策略：**预防、避免、以及检测与恢复**。这里我们重点梳理你问到的“预防”和“检测与恢复”。

---

### 一、 预防死锁 (Deadlock Prevention)

死锁的产生必须同时满足四个必要条件：**互斥、请求与保持、不可剥夺、循环等待**。预防死锁的核心思路就是**“釜底抽薪”——破坏掉这四个条件中的至少一个**。（由于“互斥”是由资源本身的物理特性决定的，比如一把锁同一时刻只能被一个线程持有，因此我们通常只能去破坏剩下三个条件）。

#### 1. 破坏“请求与保持”条件

- **策略：** 采用**静态资源分配法**。也就是要求一个线程（或进程）在开始执行前，必须一次性申请它所需要的全部资源。如果资源不够，就阻塞等待，直到所有资源都可用才开始运行。运行期间不再申请新资源。
    
- **优缺点：** 实现简单，绝对安全。但极其浪费资源，因为有些资源可能在代码运行到最后才用得到，却被早早霸占；同时也容易导致饥饿现象。
    

#### 2. 破坏“不可剥夺”条件

- **策略：** 当一个已经持有某些资源的线程，再去申请新的资源而得不到满足时，它必须**主动释放**它目前已经持有的所有资源。等以后需要时再去重新申请。
    
- **优缺点：** 这种方法常用于状态易于保存和恢复的资源（比如 CPU 寄存器状态）。但在后端业务逻辑或数据库事务中，强行剥夺资源（比如中断一个写了一半的文件或数据库操作）会导致数据不一致，实现起来非常复杂且开销极大。
    

#### 3. 破坏“循环等待”条件（最常用、最实用）

- **策略：** 采用**有序资源分配法**。将系统中的所有资源进行统一编号。规定所有线程必须严格按照**递增的顺序**去申请资源。如果需要申请编号较小的资源，必须先释放当前持有的所有编号较大的资源。
    
- **后端实践应用：** 这是在 Java 并发编程中最常用的防死锁手段。例如，当业务逻辑需要同时获取账户 A 和账户 B 的锁来进行转账时，我们可以规定永远先锁账号 ID 较小的那个，再锁账号 ID 较大的那个，这样就彻底杜绝了循环等待的发生。

### 二、 死锁检测 (Deadlock Detection)

如果系统为了追求高并发性能，没有采取严格的预防措施，那么死锁就可能发生。此时系统需要能够及时发现死锁。

1. **资源分配图 (Resource Allocation Graph)：** 操作系统内部会维护一张图，记录当前哪些线程持有哪些资源，哪些线程又在等待哪些资源。
    
2. **死锁定理：** 系统会定期运行死锁检测算法，去检查这张图里是否存在**环路（Cycle）**。如果存在不可化简的环路，就意味着发生了死锁。
    
3. **后端实践中的体现：**
    
    - **JVM 层：** 当 Java 程序卡死时，我们常用 `jstack` 命令导出线程快照。JVM 会自动在快照末尾帮你检测并打印出 `Found one Java-level deadlock` 的信息。
        
    - **数据库层：** 像 MySQL 的 InnoDB 存储引擎，内部就有一个高优先级的死锁检测机制（`wait-for graph`）。它会自动监控事务申请锁的等待关系，一旦发现环路，就会立刻采取行动。
        

---

### 三、 死锁恢复 (Deadlock Recovery)

一旦检测到死锁，系统不能干瞪眼，必须采取强硬手段打破僵局。常见的恢复手段有：

1. **资源剥夺法：** 挂起某些处于死锁状态的线程，强行抢走它们的资源，分配给其他急需的线程。
    
2. **撤销/终止进程法 (Kill Process)：** 强制终止死锁环路中的一个或多个线程。
    
    - **全盘否定：** 直接杀掉所有死锁线程，简单粗暴，但代价太大。
        
    - **逐个击破：** 按照某种优先级（比如谁占用的资源少、谁运行的时间短）逐个杀死线程，直到死锁环路被打破。
        
3. **数据库事务的经典恢复：**
    
    - 在 MySQL 中，如果 InnoDB 检测到死锁，它通常会自动选择一个**回滚代价最小（修改的数据行数最少）的事务进行 Rollback（回滚）**，并释放该事务所持有的全部锁，从而让另一个事务能够顺利执行下去。
