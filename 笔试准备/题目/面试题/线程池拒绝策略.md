当线程池的核心线程全满、任务队列已满，并且总线程数也达到了最大线程数（`maximumPoolSize`）时，如果还有新任务不断提交进来，线程池就会触发**拒绝策略（RejectedExecutionHandler）**。

Java 的 `ThreadPoolExecutor` 提供了 4 种内置的拒绝策略：

### 1. AbortPolicy（默认策略）

- **行为**：直接抛出 `RejectedExecutionException` 运行时异常，阻止系统正常工作。
    
- **适用场景**：这是线程池的默认行为。适用于那些绝对不能丢失任务的关键系统，通过抛出异常让上层业务代码或开发者及时感知到系统负载过高，从而触发报警或熔断机制。
    

### 2. CallerRunsPolicy（调用者运行策略）

- **行为**：不抛出异常，也不丢弃任务，而是将该任务直接交回给调用 `execute()` 或 `submit()` 方法的那个线程（通常是主线程或业务请求线程）去执行。
    
- **适用场景**：适用于不能丢弃任务且对响应时间要求不苛刻的场景。由于提交任务的线程去执行任务了，这会阻塞它继续提交新任务，从而起到一种天然的“限流”和“背压（Backpressure）”作用，给线程池一定的缓冲时间。
    

### 3. DiscardPolicy（丢弃策略）

- **行为**：默默地直接丢弃掉这个新提交的任务，不抛出任何异常，也不会执行它。
    
- **适用场景**：适用于那些无关紧要、即使丢失也不会影响核心业务逻辑的任务，例如某些不重要的数据打点、普通的日志记录或心跳检测等。
    

### 4. DiscardOldestPolicy（丢弃最老任务策略）

- **行为**：将任务队列（`workQueue`）中排在最前面（也就是最早进入队列、等待最久但尚未执行）的任务丢弃，然后再次尝试把当前的新任务加入队列。
    
- **适用场景**：适用于任务具有时效性的场景，或者是允许丢失部分历史数据的业务。比如系统定期获取并更新最新状态的任务，旧的状态数据即使不处理也没关系，只要保证最新提交的状态能被处理即可。
